## 1. 信号量 (Semaphore)

### 1.1 历史背景

1965 年，荷兰计算机科学家 Edsger W. Dijkstra 在开发 THE 多道程序操作系统时，提出了信号量概念——这是操作系统历史上第一个通用的进程同步原语。P 和 V 两个字母来自荷兰语：**P**roberen (尝试/测试) 和 **V**erhogen (增加/提高)。Dijkstra 因此获得 1972 年图灵奖。

### 1.2 信号量的本质

信号量 = **一个整数变量** + **一组等待队列** + **两个原子操作 (P/V)**。

它不是锁，而是更通用的同步工具——既可以实现互斥，也可以协调多个进程的执行顺序。

### 1.3 整型信号量 vs 记录型信号量

这是信号量演化的两个阶段，考试中常有"记录型信号量"的概念辨析。

#### 整型信号量 (Integer Semaphore)

Dijkstra 最早提出的形式——仅一个整数，P/V 用硬件原子指令实现，**没有等待队列**。

```c
// 整型信号量: 忙等版本
typedef int semaphore;

void P(semaphore s) {
    while (s <= 0)  // 没有资源, 循环测试
        ;            // ← 忙等! 不释放 CPU
    s--;
}

void V(semaphore s) {
    s++;
}
```

**致命缺陷**：当资源不足时，P() 操作进入**忙等 (busy waiting)**——"未让权等待"，违背了临界区管理的核心原则之一。在多道程序系统中，忙等浪费 CPU，且可能导致低优先级任务永远无法运行（如果高优先级任务在忙等中不释放 CPU）。

#### 记录型信号量 (Record / Blocking Semaphore)

在整型信号量基础上加入**等待队列**，当资源不足时进程主动阻塞（挂起），释放 CPU。这是现代操作系统中"信号量"的实际含义。

```c
// 记录型信号量: 让权等待版本
typedef struct {
    int value;                    // 可用资源数
    struct process_queue *wait;   // 等待队列 (阻塞的进程链表)
} semaphore;

void P(semaphore *s) {   // wait / down
    s->value--;
    if (s->value < 0) {  // 资源不足
        // 将当前进程加入 s->wait 队列
        // 将进程状态设为阻塞 (BLOCKED)
        block(s->wait);
        // 调用调度器, 切换到其他进程
        schedule();
    }
}

void V(semaphore *s) {   // signal / up
    s->value++;
    if (s->value <= 0) { // 有进程在等待 (value 仍 ≤0)
        // 从 s->wait 队列中取出一个进程
        // 将其设为就绪状态 (READY), 插入就绪队列
        wakeup(s->wait);
    }
}
```

**value 的含义** (记录型信号量)：

| value 的值 | 含义 |
|-----------|------|
| value > 0 | 当前可用的资源数量 |
| value = 0 | 无可用资源，且无等待进程 |
| value < 0 | 无可用资源，且 \|value\| = 等待进程数量 |

**辨析**：有些教材中 value 的初值为**资源总数**，P 操作先减后判断——"允许负值"的设计使得 value 既能表示可用资源数又能表示等待者数，是一种巧妙的编码。

### 1.4 P/V 操作的原子性保证 (核心)

P/V 操作必须**原子执行**——在 P 或 V 执行期间，其他进程不能对同一信号量执行 P/V。否则会出现两个进程同时通过 P() 进入临界区的竞态条件。

#### 1.4.1 单核 CPU：关中断

在单处理器系统中，保证原子性的最简单方式是**关中断**：

```
P(semaphore *s):
    cli()          // 关中断 (Clear Interrupt Flag)
    s->value--
    if (s->value < 0) {
        block(s->wait)
        schedule()
    }
    sti()          // 开中断 (Set Interrupt Flag)
```

为什么有效？
- 单核上，只有时钟中断或设备中断能让当前进程失去 CPU
- 关中断后，**没有任何事件能打断当前代码**——P/V 执行期间不会有其他进程被调度
- 这保证了操作的完整性

**局限**：关中断是特权指令，只能在**内核态**使用。用户态进程无法直接使用，需要系统调用。此外，关中断不能用于多核 CPU——一个核关中断不影响其他核的执行。

#### 1.4.2 多核 CPU：硬件原子指令

多核环境下，关中断只在本地核生效。需要**硬件提供的原子读-改-写指令**。最常用的是：

**TAS (Test-And-Set)**：

```c
// 硬件原子指令: 不可分割地执行读 + 写
bool TestAndSet(bool *lock) {
    bool old = *lock;
    *lock = true;   // 设置锁
    return old;     // 返回旧值
}
```

P/V 实现可以包裹在一个全局自旋锁内：

```c
void P_wrapper(semaphore *s) {
    while (TestAndSet(&p_global_lock))  // 获取保护 P 操作的全局锁
        ;                                 // 自旋等待
    // === 临界区: 原子执行 P ===
    s->value--;
    if (s->value < 0) {
        block(s->wait);  // 这里做上下文切换
        // 注意: 阻塞前需要先释放 global_lock!
    }
    *p_global_lock = false;  // 释放全局锁
}
```

**CAS (Compare-And-Swap)**——另一个常用指令：

```c
// 硬件原子指令: 如果 *ptr == old, 则设 *ptr = new, 始终返回旧值
int CAS(int *ptr, int old, int new) {
    int current = *ptr;
    if (current == old)
        *ptr = new;
    return current;
}
```

用 CAS 实现自旋锁：

```c
void spin_lock(int *lock) {
    while (CAS(lock, 0, 1) != 0)  // 尝试将 0→1
        ;                            // 失败则自旋
}
```

#### 1.4.3 现代内核的实际实现

在现代 Linux 内核中：

- **用户态信号量** (POSIX `sem_wait/sem_post`)：通过 `futex` (fast userspace mutex) 实现
  - 无竞争时：纯用户态原子操作 (CAS)，零系统调用
  - 有竞争时：通过 `futex()` 系统调用进入内核阻塞
  - 内核侧用自旋锁保护信号量数据结构
- **内核态信号量** (kernel `struct semaphore`)：直接用内核自旋锁或关中断保护

### 1.5 P/V 的阻塞与唤醒机制

#### P() 如何阻塞：等待队列

```c
void block(struct process_queue *q) {
    // 1. 将当前进程的 PCB 从就绪队列移除
    remove_from_ready_queue(current);

    // 2. 将 PCB 插入信号量的等待队列 (通常是链表)
    enqueue(q, current);

    // 3. 设置进程状态为 BLOCKED (或 WAITING)
    current->state = BLOCKED;

    // 4. 调用调度器, 选择下一个进程运行
    //    当前进程在此处"冻结", 直到被 V() 唤醒
    schedule();
}
```

#### V() 如何唤醒：出队策略

```c
void wakeup(struct process_queue *q) {
    // 1. 从等待队列中取一个进程
    struct process *p = dequeue(q);

    // 2. 将进程状态改为 READY
    p->state = READY;

    // 3. 插入就绪队列
    //    （此时不立即运行 p, 只是让它进入可调度状态）
    insert_ready_queue(p);
}
```

**出队策略**（即选择哪个被唤醒的进程）：

| 策略 | 说明 | 实现 |
|------|------|------|
| **FIFO** | 先阻塞的先唤醒 | 简单链表头部出、尾部入 |
| **优先级** | 等待者中优先级最高的先唤醒 | 优先队列 (堆) |
| **随机** | 任意选一个 | 不推荐 |
| **全部唤醒** | V 唤醒所有等待者, 它们重新竞争 | 类似 Mesa 语义中的 broadcast |

典型的 POSIX `sem_post` 唤醒策略是实现定义的——Linux 采用 FIFO。

### 1.6 三种信号量类型与完整工作实例

#### 类型 1：互斥信号量 (Mutex Semaphore)

- **初值 = 1**：信号量代表"临界区的访问许可"
- **本质**：二元信号量 (binary semaphore)，值只能是 0 或 1
- **模式**：进入临界区前 P(mutex)，离开后 V(mutex)——**必须成对**

**实例：两个进程互斥打印**

```c
semaphore mutex = 1;  // 初值 1: 打印机可用

// 进程 A
P(mutex);
使用打印机();   // 临界区 — 互斥保证
V(mutex);

// 进程 B
P(mutex);
使用打印机();   // 临界区 — 互斥保证
V(mutex);
```

**执行轨迹** (假设进程 A 先执行)：

```
时刻   进程A        进程B        mutex.value    说明
0      P(mutex)    —            0              A 获得锁
1      打印中      P(mutex)    -1             B 尝试, 阻塞 (value=-1, 1人等待)
2      打印中      (阻塞)      -1             A 继续
3      V(mutex)    —            0              A 释放, 唤醒 B (value=0, 0人等待)
4      —           打印中       0              B 获得锁, 进入临界区
5      —           V(mutex)     1              B 释放, 恢复初值
```

#### 类型 2：计数信号量 (Counting Semaphore)

- **初值 = N** (N ≥ 1)：表示有 N 个同类资源可用
- **用途**：控制有限资源的多个实例

**实例：5 个数据库连接，10 个并发请求**

```c
semaphore db_conn = 5;  // 5 个连接可用

// 每个请求处理线程：
P(db_conn);              // 申请一个连接
执行数据库查询();
V(db_conn);              // 归还连接
```

当第 6 个线程执行 `P(db_conn)` 时，value 从 0 变为 -1——该线程阻塞。最多同时 5 个线程持有连接。

#### 类型 3：同步信号量 (Synchronization Semaphore)

- **初值 = 0**：信号量本身不是"资源"，而是"事件通知"
- **用途**：保证两个进程/线程之间的**执行顺序**（前驱关系）

**实例：司机-售票员问题**

```c
// 约束: 司机必须等售票员关好门才能开车
//       售票员必须等司机到站停车后才能开门

semaphore can_start = 0;   // "可以开车了"的信号
semaphore can_open  = 0;   // "可以开门了"的信号

// 司机进程
while (true) {
    P(can_start);       // 等售票员说"门关了, 可以开车"
    开车();
    到站停车();
    V(can_open);        // 通知售票员"车停了, 可以开门"
}

// 售票员进程
while (true) {
    关车门();
    V(can_start);       // 通知司机"门关了, 开车吧"
    售票();
    P(can_open);        // 等司机说"到站了, 开门"
    开车门();
}
```

**核心模式**：P 等待一个条件 (前驱事件)，V 通知条件已满足。一个进程的 V 和另一个进程的 P 配对——形成**同步点**。

#### 类型 4：AND 型信号量 (Simultaneous Semaphore)

AND 型信号量由 Birch Hansen 和 Hoare 提出，用于解决**一次需要多个资源**的场景。

**核心思想**：`Swait(s1, s2, ..., sn)` 和 `Ssignal(s1, s2, ..., sn)` ——要么一次性获得所有需要的资源，要么一个也不占着，从而**避免死锁**。

```c
// AND 型信号量的 Swait (同时申请)
void Swait(semaphore *s1, semaphore *s2, ...) {
    // 要么全部成功，要么全部等待
    if (s1->value >= 1 && s2->value >= 1 && ...) {
        s1->value--;
        s2->value--;
        // ... 全部资源获取成功
    } else {
        // 将当前进程加入所有信号量的等待队列
        // 阻塞当前进程
        block_on_all(s1, s2, ...);
    }
}

// AND 型信号量的 Ssignal (同时释放)
void Ssignal(semaphore *s1, semaphore *s2, ...) {
    s1->value++;
    s2->value++;
    // ... 释放所有资源
    // 检查是否能唤醒某个进程 (它需要这些资源)
    try_wakeup();
}
```

**实例：哲学家进餐问题 —— AND 型解法**

```c
// 5 个哲学家人，5 根筷子，用信号量数组 chopstick[5]
// 每个哲学家需要同时拿到左右两根筷子

semaphore chopstick[5] = {1, 1, 1, 1, 1};  // 每根筷子初始可用

// 哲学家 i (AND 型信号量解法)
while (true) {
    think();
    // 注意: 同时申请左右两根筷子, 不会死锁
    Swait(chopstick[i], chopstick[(i+1) % 5]);
    eat();
    Ssignal(chopstick[i], chopstick[(i+1) % 5]);
}
```

**为什么能防止死锁？**
- 经典解法中，如果所有哲学家同时拿起左手筷子，再试图拿右手——全部阻塞，形成环路死锁
- AND 解法中，`Swait` 要么两根筷子都拿到，要么一根都不占——**原子性的全有或全无**
- 不会有进程"拿一根等另一根"的状态，死锁的环路条件被打破

### 1.7 信号量的局限性

| 问题 | 说明 |
|------|------|
| **使用错误编译器无提示** | 漏写 V()、P/V 顺序错误——编译器不会报错，运行时诡异 |
| **死锁风险** | 两个进程以不同顺序申请多个信号量时可能死锁 |
| **饿死风险** | 高优先级频繁 P/V 可能导致低优先级永远等不到 |
| **代码耦合度低** | 信号量散布在各处，修改时需要在代码中分散查找，不方便集中管理 |

## 2. 管程 (Monitor)

### 2.1 管程的设计动机

信号量是低级同步原语——强大、灵活，但也**容易出错**。一个 P/V 顺序颠倒就能导致死锁，编译器却一声不吭。

管程的目标：**把同步逻辑与共享数据封装在一起**，由编译器/运行时保证互斥，程序员只需在"哪里需要等待"这一点上做出正确决策即可。

### 2.2 管程的定义

> **管程 (Monitor)**：一个包含共享变量、操作这些变量的函数以及同步条件变量的语言级结构。同一时刻最多只有一个线程可以在管程内执行。

**三个组成部分**：

```
管程 Monitor_X
├── 局部共享变量 (private to the monitor)
│   → 只有管程内的函数能访问, 外界不可见
├── 入口函数 (entry procedures)
│   → 同一时刻只有一个线程能调用管程的任意入口函数
│   → 编译器自动插桩锁/解锁代码
└── 条件变量 (condition variables)
    → c.wait(): 当前线程在此条件上阻塞, 释放管程锁
    → c.signal(): 唤醒一个在此条件上等待的线程
```

### 2.3 Hoare 语义 vs Mesa 语义 (重点)

这是管程理论中最重要的概念区分，也是面试高频考点。

#### 2.3.1 Hoare 语义 (Hoare, 1974 — 原始论文)

**规则**：当一个线程执行 `c.signal()` 时：
1. 如果 c 上有等待线程，**被唤醒的线程立即获得管程锁并执行**
2. signal 者被**挂起**——它退出管程，等待被唤醒的线程完成后重新进入
3. 被唤醒的线程看到的状态**一定满足条件**——因为 signal 到它获得执行的这段时间内，没有其他线程能改变共享状态

```c
// Hoare 语义下的管程代码模式
monitor BoundedBuffer {
    Item buffer[N];
    int count = 0, in = 0, out = 0;
    condition not_full, not_empty;

    void put(Item item) {
        if (count == N)        // 缓冲区满
            not_full.wait();   // 等待不满条件
        buffer[in] = item;
        in = (in + 1) % N;
        count++;
        not_empty.signal();    // 通知: 非空
        // signal 后立即被挂起, 被唤醒者执行
    }

    Item get() {
        if (count == 0)        // 缓冲区空
            not_empty.wait();  // 等待非空条件
        Item item = buffer[out];
        out = (out + 1) % N;
        count--;
        not_full.signal();     // 通知: 非满
        return item;
    }
};
```

**为什么用 `if` 就够了？** 因为 signal 意味着"条件现在成立了"，且 signal 者立即交出执行权——被唤醒者一上来就能确认条件成立。

#### 2.3.2 Mesa 语义 (Lampson & Redell, 1980 — Xerox Mesa 语言)

**规则**：当线程执行 `c.signal()` 时：
1. signal 者**继续执行**——它不离开管程, 完成自己的管程调用后才释放锁
2. 被唤醒的线程被移到就绪队列, 但**不立即获得管程锁**
3. 被唤醒的线程最终获得锁再次进入管程时, 条件**可能已被其他线程改变**

```c
// Mesa 语义下的管程代码模式 — 注意 while!
monitor BoundedBuffer {
    Item buffer[N];
    int count = 0, in = 0, out = 0;
    condition not_full, not_empty;

    void put(Item item) {
        while (count == N)     // ← while, 不是 if!
            not_full.wait();
        buffer[in] = item;
        in = (in + 1) % N;
        count++;
        not_empty.signal();    // signal 后继续执行, 不走
    }

    Item get() {
        while (count == 0)     // ← while, 不是 if!
            not_empty.wait();
        Item item = buffer[out];
        out = (out + 1) % N;
        count--;
        not_full.signal();     // signal 后继续执行, 不走
        return item;
    }
};
```

**为什么必须用 `while` 而不是 `if`？** 考虑这个场景：

```
1. 消费者 A 发现缓冲区空, 执行 not_empty.wait() → 阻塞
2. 生产者 B 放入一个数据, 执行 not_empty.signal() → 唤醒 A
3. B 继续执行 (Mesa 语义), 又放入第二个数据
4. B 退出管程
5. 此时, 消费者 C 抢先进入管程, 取走了两个数据 → 缓冲区又空了
6. 消费者 A 终于获得锁进入管程, 从 wait 返回
7. 如果用的是 if: A 直接往下走取数据 → 缓冲区已空! 越界/逻辑错误!
8. 如果用 while: A 重新检查 count == 0 → 发现还是空 → 再次 wait
```

**Mesa 的黄金法则**：永远用 `while` 循环包围 `wait()` 调用，因为被唤醒不代表条件仍然成立。

#### 2.3.3 Hoare vs Mesa 对比总结

| 维度 | Hoare 语义 | Mesa 语义 |
|------|-----------|----------|
| signal 之后 | signal 者挂起, 被唤醒者**立即执行** | signal 者**继续执行**, 被唤醒者就绪等待 |
| 条件检查方式 | `if (condition) wait()` | `while (condition) wait()` |
| 被唤醒时条件 | **一定成立** (有保证) | **可能已改变** (需重检) |
| 上下文切换开销 | 高 (signal = 主动让权) | 低 (signal = 通知+继续) |
| 实现复杂度 | 高 | 较低 |
| 实际采用 | 很少 (理论原型) | **几乎所有现代系统** |

**实际系统都在用 Mesa**：
- Java `Object.wait()/notify()` — Mesa 语义
- POSIX `pthread_cond_wait()` — Mesa 语义
- C++ `std::condition_variable` — Mesa 语义
- Python `threading.Condition` — Mesa 语义

### 2.4 管程的实际实现：Java `synchronized` + `wait/notify`

Java 的管程实现是最典型、最广泛使用的例子。

```java
// Java 管程: synchronized + wait/notify
class BoundedBuffer {
    private int[] buffer = new int[10];
    private int count = 0, in = 0, out = 0;

    // synchronized = 管程的"入口互斥"
    public synchronized void put(int item) throws InterruptedException {
        while (count == buffer.length)   // ← while, Mesa 语义
            wait();                       // ← 释放管程锁, 阻塞在当前对象上

        buffer[in] = item;
        in = (in + 1) % buffer.length;
        count++;

        notifyAll();  // notify() 只随机唤醒一个, notifyAll() 唤醒所有
    }

    public synchronized int get() throws InterruptedException {
        while (count == 0)
            wait();

        int item = buffer[out];
        out = (out + 1) % buffer.length;
        count--;

        notifyAll();
        return item;
    }
}
```

**Java 管程与理论管程的映射**：

| 理论管程概念 | Java 对应 | 说明 |
|-------------|-----------|------|
| 管程 (入口互斥) | `synchronized` | JVM 自动插桩 monitorenter/monitorexit |
| 条件变量 | `wait() / notify() / notifyAll()` | 基于隐式条件变量 (每对象一个) |
| 等待条件 | `while (condition) wait()` | Mesa 语义, 必须 while |
| 管程锁 | 对象的 intrinsic lock (管程锁) | 每个 Java 对象都自带一把锁 |

**Java 管程的限制**：
- 每个对象只有一个"隐式条件变量"——粒度粗，所有 wait 都等在同一条件上
- `notify()` 随机唤醒一个——如果缓冲区有 reader 和 writer 两类等待者，`notify()` 可能唤醒"错误类型"的线程
- Java 5+ 引入 `java.util.concurrent.locks` (显式 Lock + 多个 Condition) 弥补了这些不足

## 3. 信号量 vs 管程：综合对比

| 维度 | 信号量 (Semaphore) | 管程 (Monitor) |
|------|-------------------|---------------|
| **抽象层次** | 低级同步原语 | 高级语言结构 |
| **互斥保证** | 程序员手动 P/V | 编译器/运行时自动保证 |
| **数据封装** | 无——共享变量与锁分离 | 有——共享数据封装在管程内部 |
| **易错程度** | **高**：P/V 遗漏、顺序错误、不成对 | **低**：编译器保证互斥, 程序员只写 wait 位置 |
| **灵活度** | 极高——可表达任意同步模式 | 中——受管程结构约束 |
| **死锁风险** | 使用者自己负责 | 逻辑死锁仍可能, 但更少见 |
| **条件同步** | 间接——通过信号量的值传递信息 | 直接——条件变量 `wait/signal` 语义清晰 |
| **代表实现** | POSIX `sem_*`, Linux `semaphore` | Java `synchronized`, Python `with Lock` |
| **适用场景** | 内核开发、嵌入式、简单生产者-消费者 | 应用层并发编程 (多线程服务器、GUI) |

### 选型建议

- **内核/底层** → 信号量 (性能关键, 直接控制)
- **应用层多线程** → 管程 (安全、可维护)
- **简单互斥** → mutex = 初值 1 的信号量 (或专门的 mutex 实现)
- **复杂同步** (多生产者多消费者、读写锁) → 管程 + 条件变量
