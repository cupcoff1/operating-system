## 1. 临界区问题 (The Critical Section Problem)

### 1.1 形式化定义

> **临界区 (Critical Section)**：进程中访问**共享资源**（共享变量、共享文件、共享设备等）的那段代码。
>
> **临界区问题**：设计一个协议，使得多个并发进程在执行各自的临界区时，共享数据的**一致性**得以保持。

**形式化模型**：每个参与进程的结构如下：

```
do {
    进入区 (entry section)    // 申请进入临界区的代码
        临界区 (critical section)  // 访问共享资源的代码
    退出区 (exit section)     // 释放临界区的代码
    剩余区 (remainder section)   // 不访问共享资源的代码
} while (true);
```

### 1.2 竞争条件 (Race Condition)

> **竞争条件**：多个进程并发访问同一共享数据，且最终结果依赖于各进程执行的**具体时序**（谁先谁后）。

竞争条件是临界区问题的反面——没有临界区保护，竞争条件就会发生。

**简单实例**：两个线程同时对 `counter` 做 `counter++`。

```c
// counter++ 在机器指令层面实际上是三条指令:
//   MOV  eax, [counter]    // 1. 从内存加载
//   ADD  eax, 1             // 2. 在寄存器中加1
//   MOV  [counter], eax    // 3. 写回内存
```

如果线程 A 执行完步骤 1 后被线程 B 抢占：

```
A: MOV eax, [counter]   → eax = 0
B: MOV ebx, [counter]   → ebx = 0  (A 还没写回)
B: ADD ebx, 1           → ebx = 1
B: MOV [counter], ebx   → counter = 1
A: ADD eax, 1           → eax = 1
A: MOV [counter], eax   → counter = 1  (应该是 2!)
```

预期 `counter = 2`，实际 `counter = 1`——丢失了一次更新。

### 1.3 临界区解必须满足的四个条件

| 条件 | 英文 | 形式化含义 |
|------|------|-----------|
| **1. 互斥 (Mutual Exclusion)** | Mutual Exclusion | 任何时刻最多只有一个进程在其临界区内 |
| **2. 空闲让进 (Progress)** | Progress | 如果无人在临界区且有进程想进入，则**必须能做出选择**让某个进程进入——不能无限期拖延，且选择不依赖剩余区中的进程 |
| **3. 有限等待 (Bounded Waiting)** | Bounded Waiting | 一个进程从提出进入请求到实际进入，等待次数有**确定的上限**——不能饿死 |
| **4. 让权等待** | CPU Relinquish | 在进入不了临界区时应**释放 CPU**——不忙等 |

**前三个条件**是所有临界区解法的**基本要求**；**让权等待**是操作系统环境下的额外期望——但硬件解法 (TAS, CAS) 和自旋锁并不满足这一条件。

## 2. 软件解法

### 2.1 演进历史 (三种有缺陷的尝试)

#### 尝试 1: 单标志法 (轮换算法)

```c
int turn = 0;  // 全局: 轮到谁进入

// 进程 P0
while (turn != 0) ;   // 等轮到自己
临界区;
turn = 1;              // 让给 P1

// 进程 P1
while (turn != 1) ;
临界区;
turn = 0;
```

**问题**：违反了**空闲让进**。如果 P0 进入临界区后长时间停留在剩余区 (不置 turn=1)，P1 永远无法连续两次进入——尽管临界区空着。

#### 尝试 2: 双标志法 (先举手再检查)

```c
bool flag[2] = {false, false};

// 进程 Pi
flag[i] = true;                // "我想进"
while (flag[j]) ;              // 等对方不想进
临界区;
flag[i] = false;               // "我出来了"

// 进程 Pj
flag[j] = true;
while (flag[i]) ;
临界区;
flag[j] = false;
```

**问题**：违反了**互斥**。两个进程可能同时将 flag 设为 true，然后都读到对方的 flag 为 true——两人都在 `while` 中互相等待，但在此之前可能同时越过 while 进入临界区（时序恰好错开时）。

#### 尝试 3: 双标志后检查法 (先礼让再检查)

```c
bool flag[2] = {false, false};

// 进程 Pi
flag[i] = true;
turn = j;                      // 礼让对方
while (flag[j] && turn == j) ; // 对方想进且轮到他
临界区;
flag[i] = false;
```

这个算法实际上已经正确了——它就是 **Peterson 算法的雏形**！下面展开完整分析。

### 2.2 Peterson 算法 (1981) -- 完整的双进程解法

Gary L. Peterson 于 1981 年发表，仅用两条机器指令的共享变量就解决了双进程临界区问题。

#### 完整代码

```c
int turn;           // 表示"轮到谁进入"
bool flag[2];       // flag[i] = true 表示 Pi 想进入临界区
// 初始: flag[0] = flag[1] = false; turn 任意

// ================ 进程 Pi (i=0 或 1) ================
flag[i] = true;                    // (1) 举手: "我想进入"
turn = j;                          // (2) 谦让: "你先"
while (flag[j] && turn == j)       // (3) 等待条件: 对方想进
    ;                               //     且轮到他 → 等待
                                   //     否则 → 进入

// ============== 临界区 ==============

flag[i] = false;                   // (4) 放手: "我出来了"
// ============== 剩余区 ==============
```

#### 2.2.1 互斥性证明 (Mutual Exclusion)

**要证明**：不可能 Pi 和 Pj 同时进入临界区。

**反证法**：假设 Pi 和 Pj 同时进入了临界区。

- Pi 进入了 → 在 Pi 的 while 条件中，**要么** `flag[j] == false`，**要么** `turn == i`
- Pj 进入了 → 在 Pj 的 while 条件中，**要么** `flag[i] == false`，**要么** `turn == j`

如果两者都进入了，则 Pi 读到的 `flag[j]` 和 Pj 读到的 `flag[i]` **至少一个为 false**——但两个进程在进入临界区前都执行了 `flag[自己] = true`。因此必然 `flag[i] = true` 且 `flag[j] = true`。

此时 Pi 能进入的唯一可能是 `turn == i`，Pj 能进入的唯一可能是 `turn == j`。但 `turn` 是一个整数变量，不可能同时等于 i 和 j。

**矛盾**。因此 Pi 和 Pj 不可能同时进入临界区。互斥性成立。

#### 2.2.2 空闲让进证明 (Progress)

**要证明**：如果临界区空且至少有一个进程想进入，则一定能选出一个进程进入，且选择不依赖于剩余区中的进程。

假设 Pi 和 Pj 都想进入 (`flag[i]=flag[j]=true`)。

- 如果 `turn == i`：Pj 的 while 条件为 `flag[i] && turn == i` → 成立 → Pj 等待。Pi 的 while 条件为 `flag[j] && turn == j` → 不成立 (因为 turn == i≠j) → Pi 进入。
- 如果 `turn == j`：对称地，Pj 进入。

如果只有一个进程想进入 (如 Pi)：`flag[j]==false`，Pi 的 while 条件为 `flag[j] && ...` → `false && ...` → **立即为 false** → Pi 直接进入。不阻塞。

**空闲让进成立。**

#### 2.2.3 有限等待证明 (Bounded Waiting)

**要证明**：一个进程从举手到进入，最多等一次。

如果 Pi 举手后需要等待，说明 Pj 在临界区内。Pj 退出临界区时执行 `flag[j] = false`。此时 Pi 的 while 条件变为 `false && ... = false` → Pi 可以进入。

即 Pi 最多等 Pj 的**一个临界区执行时间**。有限等待成立。

#### 2.2.4 Peterson 算法的现代局限性

**问题**：现代 CPU 的**乱序执行**和 **Store Buffer** 可能破坏 Peterson 算法的正确性。

考虑 CPU 将 `flag[i]=true` 和 `turn=j` 的执行顺序重排为：

```
turn = j;          // 先写 turn
flag[i] = true;    // 后写 flag  ← 被 CPU 重排, 实际上先执行了
```

这可能导致两个进程同时进入临界区。解决方案：使用**内存屏障 (memory barrier / fence)**。

```c
flag[i] = true;
// smp_mb()  // 内存屏障: 保证上面的 store 对下面可见
turn = j;
```

这也是为什么现代编程中很少直接使用 Peterson 算法，而更依赖硬件提供的原子指令 (包含隐式内存屏障)。

### 2.3 N 进程扩展：Lamport 面包店算法 (Bakery Algorithm)

Lamport 将 Peterson 算法推广到 N 个进程，核心思想类似面包店排队取号：

```c
bool choosing[N] = {false};  // 进程 i 是否正在取号
int  number[N]   = {0};      // 进程 i 的排队号码

// 进程 i 的进入区:
choosing[i] = true;
number[i] = max(number[0], ..., number[N-1]) + 1;  // 取号
choosing[i] = false;

for (int j = 0; j < N; j++) {
    while (choosing[j]) ;                              // 等 j 取完号
    while (number[j] != 0 &&                           // j 想进
           (number[j] < number[i] ||                   // j 号比我小
            (number[j] == number[i] && j < i)))        // 或者同号但 j 的 ID 小
        ;                                               // 让 j 先进
}

// 退出区:
number[i] = 0;
```

**关键设计**：(number, PID) 构成全序——保证了公平性和所有条件。

## 3. 硬件解法

软件解法 (Peterson) 在理论上优雅，但现代系统几乎全部依赖**硬件提供的原子指令**。

### 3.1 什么是硬件原子指令？

> **原子指令 (Atomic Instruction)**：由 CPU 硬件保证的、不可分割的读-改-写操作。执行期间，**任何其他 CPU 核心或总线设备都不能访问该内存位置**。

实现方式：
- **总线锁定 (Bus Lock)**：执行指令时锁定内存总线 (x86 的 `LOCK` 前缀)
- **缓存锁定 (Cache Lock)**：MESI 协议的升级版 — 将缓存行置为 `Modified` 或 `Exclusive` 状态，阻止其他核的并发访问，不锁定总线

### 3.2 Test-And-Set (TS / TSL)

```c
// 硬件原子指令：测试并设置
// 功能: 返回 lock 指向字节的旧值，同时将该字节设为 true
// x86 汇编: LOCK BTS (Bit Test and Set)
// ARM: LDREX / STREX 对 (Load-Link / Store-Conditional)

// C 语言等价描述 (由硬件保证原子性):
bool TestAndSet(bool *lock) {
    bool old = *lock;
    *lock = true;
    return old;   // 返回旧值
}
```

#### 用 TAS 实现自旋锁

```c
bool lock = false;  // false = 未锁, true = 已锁

// 进入区 (自旋等待):
while (TestAndSet(&lock))   // 如果旧值是 true (已锁), 循环
    ;                        // 什么都不做, 空转等待

// ===== 临界区 =====

// 退出区:
lock = false;  // 解锁 (写入是原子的, 因为一次 store 是原子的)
```

**分析**：

| 条件 | 满足？ | 说明 |
|------|--------|------|
| 互斥 | 是 | TAS 保证只有一个进程能将 lock 从 false 变为 true |
| 空闲让进 | 是 | lock=false 时第一个执行 TAS 的进程立即进入 |
| 有限等待 | 否 | 理论上可能饿死 (概率极低) |
| 让权等待 | **否** | 忙等——这是硬件解法的主要缺陷 |

### 3.3 Swap / Exchange (XCHG)

```c
// 硬件原子指令：交换两个内存位置的值
// x86 汇编: XCHG (隐含 LOCK 语义)

// C 语言等价描述 (由硬件保证原子性):
void Swap(bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;
}
```

#### 用 Swap 实现自旋锁

```c
bool lock = false;

// 进入区:
bool key = true;
while (key == true)
    Swap(&lock, &key);   // 原子交换 lock 和 key
// 退出循环时: lock 被设为 true, key 被设为 lock 的旧值

// ===== 临界区 =====

// 退出区:
lock = false;
```

**解读**：如果 lock 原本是 false，交换后 `key=false`，退出循环——加锁成功。如果 lock 原本是 true，交换后 `key=true`——继续自旋。

**TAS 与 Swap 等价**：两者都实现了原子的"读旧值 + 写新值"。两者在功能上没有本质区别，都可用于实现自旋锁。

### 3.4 Compare-And-Swap (CAS) -- 最强大的原子指令

CAS 是比 TAS/Swap **更强大**的原子指令 —— 它不仅能做简单的 set-to-true，还能做任意值的"比较并交换"。无锁编程 (lock-free programming) 的核心就是 CAS。

```c
// 硬件原子指令：比较并交换
// x86 汇编: LOCK CMPXCHG
// ARM: LDREX / STREX

// C 语言等价描述:
int CAS(int *ptr, int old_val, int new_val) {
    int current = *ptr;             // 原子的"读"
    if (current == old_val)         // 仅当值未改变
        *ptr = new_val;             // 原子的"写"
    return current;                 // 返回旧值 (方便调用者判断)
}
```

#### 用 CAS 实现自旋锁

```c
int lock = 0;  // 0 = unlocked, 1 = locked

// 进入区:
while (CAS(&lock, 0, 1) != 0)  // 尝试将 0→1, 失败则自旋
    ;

// ===== 临界区 =====

// 退出区:
lock = 0;
```

#### CAS 的独特优势：无锁数据结构

CAS 可以构建**无锁 (lock-free)** 的并发数据结构，比基于锁的方案有更好的伸缩性和故障容忍（一个线程崩溃不会阻塞所有线程）。

**无锁栈的 push 操作**：

```c
void push(Node **top, Node *new_node) {
    do {
        new_node->next = *top;             // 准备新节点指向当前栈顶
    } while (CAS(top, new_node->next, new_node) != new_node->next);
    // CAS 保证: 如果 top 仍然是 old_top, 则更新为 new_node
    // 如果 top 被其他线程改了, 重试
}
```

#### CAS 的 ABA 问题

**场景**：线程 A 读到值为 A，线程 B 将值改为 B 又改回 A，线程 A 的 CAS 执行时发现值仍是 A——误以为没有变化，CAS 成功——但链表的中间状态已不同。

**解决**：**带版本号的 CAS** (如 x86 `CMPXCHG16B` 使用 128 位中的部分位做版本号)，或使用指针引用计数、Hazard Pointer 等技术。

### 3.5 Fetch-And-Add (FAA)

```c
// 硬件原子指令：原子的 X += N, 返回旧值
int FetchAndAdd(int *ptr, int increment) {
    int old = *ptr;
    *ptr += increment;
    return old;
}
```

常用于实现**票据锁 (Ticket Lock)**——保证 FIFO 公平性，解决饿死问题。

## 4. 自旋锁 (Spinlock) 深度剖析

### 4.1 定义与本质

> **自旋锁**：请求者在锁不可用时**原地循环 (自旋)** 而不阻塞，直到锁变为可用。

行为特征：
- **忙等 (busy waiting)**：CPU 空转，不释放给其他线程
- **没有上下文切换**：不进入内核调度
- **非常适合极短的临界区**

### 4.2 自旋锁 vs 互斥锁的深层对比

| 维度 | 自旋锁 (Spinlock) | 互斥锁 (Mutex) |
|------|------------------|---------------|
| **等待行为** | CPU 空转循环 (忙等) | 进入内核阻塞 (挂起) |
| **上下文切换** | 无 | 有 (至少两次: 阻塞 + 唤醒) |
| **等待开销** | 浪费 CPU 时间 | CPU 开销在上下文切换上 |
| **临界区长度** | 必须**极短** (纳秒~微秒级) | 可较长 (毫秒级) |
| **可睡眠？** | 持有自旋锁期间**绝不能睡眠** | 持有 mutex 期间可以睡眠 (不推荐) |
| **中断上下文可用？** | **是** (中断上下文唯一选择) | **否** (中断上下文不能睡眠) |
| **多核 vs 单核** | 多核上合理; 单核上纯废话 | 单核也能用 |
| **内核实现** | 原子操作 + 抢占禁用 | Futex / 等待队列 |

### 4.3 为什么中断上下文只能使用自旋锁？

在中断处理函数 (ISR, Interrupt Service Routine) 中：
1. **中断上下文没有进程上下文**——没有 PCB，没有"当前进程"
2. 如果尝试 `mutex_lock()` 且在锁不可用时阻塞——调度器需要切换到一个新的"当前进程"，但在中断上下文中，**没有"当前进程"可以切换出去**
3. 中断必须快速执行完毕——阻塞在中断中是致命的

因此，中断处理函数若需要访问共享数据，**唯一的选项就是自旋锁**：短暂地自旋等待（假设另一个核上的持锁中断也在快速执行），获取锁后极快地操作完，然后释放。

### 4.4 Linux 内核自旋锁实现细节

Linux 内核中自旋锁的完整实现需要考虑**三个层面的保护**：

#### 层面 1：多核互斥 (原子操作)

```c
// Linux kernel: include/linux/spinlock.h
typedef struct {
    arch_spinlock_t raw_lock;  // 架构相关部分
} spinlock_t;

// x86 实现 (简化):
typedef struct {
    unsigned int slock;  // 1 = unlocked, 0 = locked
} arch_spinlock_t;

static inline void arch_spin_lock(arch_spinlock_t *lock) {
    // 使用原子递减 (LOCK DECL) 
    // 如果结果为零, 加锁成功
    // 否则在缓存行上自旋, 直到值变为正
    while (atomic_dec_return(&lock->slock) != 1) {
        // 在 MESI 协议下, 自旋读取缓存行——避免总线争用
        while (READ_ONCE(lock->slock) <= 0)
            cpu_relax();  // PAUSE 指令: 减少功耗 + 通知 CPU 这是自旋循环
    }
}
```

#### 层面 2：禁用内核抢占 (preempt_disable)

```c
// 为什么需要禁用抢占?
// 考虑: 内核线程 A 持有自旋锁, 被时钟中断抢占,
//       调度器切换到了内核线程 B, B 也想获取同一把自旋锁
//       → B 自旋等待 A, 但 A 被抢占无法运行 → 死锁!

#define spin_lock(lock)          \
    do {                         \
        preempt_disable();       \  // 先禁用抢占
        arch_spin_lock(lock);   \  // 再获取锁
    } while (0)
```

#### 层面 3：中断上下文中还要关中断 (spin_lock_irqsave)

```c
// 如果临界区可能被中断处理函数中的代码访问:
// 不仅要禁用抢占, 还要关闭本地中断——防止中断处理函数
// 在同一 CPU 上抢占当前线程并尝试获取同一把锁

#define spin_lock_irqsave(lock, flags)  \
    do {                                \
        local_irq_save(flags);         \  // 关本地中断 + 保存标志
        preempt_disable();             \  // 同时禁用抢占
        arch_spin_lock(lock);          \
    } while (0)
```

**规则总结 (Linux 自旋锁使用黄金法则)**：

| 谁会抢同一把锁 | 使用 |
|--------------|------|
| 仅其他进程/线程 (不在中断中) | `spin_lock(l)` |
| 进程 + 中断处理函数都可能 | `spin_lock_irqsave(l, flags)` |
| 仅中断处理函数 | `spin_lock_irq(l)` (不需要保存标志) |
| 仅不同 CPU 上的中断处理函数 | `spin_lock(l)` (中断已在执行, 同核不会重入) |

#### 单核 CPU 上的特殊处理

在单核 (CONFIG_SMP=n) 情况下：
- `spin_lock()` 退化为**仅 `preempt_disable()`**——不需要原子操作 (只有一个核)
- `spin_lock_irqsave()` 退化为**仅 `local_irq_save()`**——不需要原子操作，只需要关中断

## 5. 互斥锁 (Mutex) — 阻塞式锁

### 5.1 定义

> **互斥锁 (Mutex)**：请求者在锁不可用时**进入内核阻塞 (挂起)**，将 CPU 让给其他线程。锁变为可用后被唤醒并获取锁。

### 5.2 自旋锁与互斥锁的选择策略

```
临界区长度 < 两次上下文切换的时间 (约几微秒)?
    ├── 是 → 使用自旋锁
    │   理由: 自旋几圈比切换两次快
    └── 否 → 使用互斥锁
        理由: 长时间忙等是纯粹的 CPU 浪费
```

### 5.3 Linux Futex：用户态 mutex 的高效实现

> **Futex (Fast Userspace mutex)**：Linux 提供的一种混合锁机制，将用户态原子操作的速度与内核态阻塞的能力结合起来。

**核心思想**：
- **无竞争**：锁在用户态通过 `CAS` 操作获取和释放——**零系统调用开销**
- **有竞争**：通过 `futex()` 系统调用进入内核，内核将线程放入等待队列并调度走

```c
// 用户态 futex 加锁 (简化):
void futex_lock(int *futex_word) {
    // 尝试 CAS: 如果 lock == 0, 设为 1, 返回
    if (CAS(futex_word, 0, 1) == 0)
        return;  // 快速路径: 无竞争, 直接成功

    // 慢速路径: 有竞争
    while (true) {
        // 将 lock 从 1 设为 2 (标记有等待者), 然后内核阻塞
        if (CAS(futex_word, 1, 2) != 0)
            CAS(futex_word, 0, 2);  // 可能已被释放, 尝试直接标记

        // 进入内核等待
        sys_futex(futex_word, FUTEX_WAIT, 2, NULL);
        // 被唤醒后回到 CAS 循环
    }
}
```

这是 Linux 实现 POSIX `pthread_mutex_t` 的底层机制——也是 Java `ReentrantLock`、C++ `std::mutex` 在 Linux 上的最终实现基础。

## 6. 高级锁变体

### 6.1 读写锁 (Reader-Writer Lock / rwlock)

**动机**：许多场景下**读远多于写**——单纯的互斥锁让多个读者也不能并发，浪费读并行性。

**规则**：

| | 无持有者 | 读者持有 | 写者持有 |
|------|---------|---------|---------|
| **读者申请** | 允许 | 允许 (读共享) | 阻塞 |
| **写者申请** | 允许 | 阻塞 | 阻塞 |

**偏向策略**：

| 策略 | 行为 | 优势 | 劣势 |
|------|------|------|------|
| **偏向读** | 只要有一个读者在，新读者直接进来 | 读吞吐高 | 写者可能饿死 |
| **偏向写** | 写者等待时，新读者也排队 | 写延迟低 | 读并发度下降 |
| **公平** | FIFO 队列 | 公平 | 实现略复杂 |

### 6.2 读写自旋锁 (Reader-Writer Spinlock)

读写锁的内核版本——读者和写者都通过**自旋**等待，不阻塞。适用于持有时间极短、又需要读并发的内核场景。

```c
// Linux: rwlock_t
// 读者获取:
read_lock(&rwlock);     // 如果无写者持有, 读者计数+1 (原子操作)
                        // 如果有写者, 自旋等待
read_unlock(&rwlock);   // 读者计数-1; 如果是最后一个读者, 可能唤醒写者

// 写者获取:
write_lock(&rwlock);    // 自旋等待直到无读者且无写者
write_unlock(&rwlock);  // 释放
```

### 6.3 顺序锁 (Seqlock) -- 写者优先的极致设计

> **顺序锁 (seqlock)**：一种**写者永远不会被读者阻塞**的锁。读者无锁读取，但读完后检查是否有写者在读期间修改了数据——如果有，重试。

**核心数据结构**：

```c
typedef struct {
    unsigned int sequence;  // 序列号: 偶数 = 稳定, 奇数 = 写者正在操作
    spinlock_t   lock;      // 写者之间的互斥锁
} seqlock_t;
```

**操作语义**：

```c
// === 写者操作 ===
void write_seqlock(seqlock_t *sl) {
    spin_lock(&sl->lock);   // 写者之间的互斥
    sl->sequence++;          // 变为奇数: 通知读者"数据正在被修改"
    // 内存屏障: 保证 sequence 的更新在数据修改之前被其他核看到
    smp_wmb();
}
void write_sequnlock(seqlock_t *sl) {
    smp_wmb();               // 保证数据修改在 sequence 更新前被看到
    sl->sequence++;          // 变为偶数: 通知读者"数据已稳定"
    spin_unlock(&sl->lock);
}

// === 读者操作 ===
unsigned int read_seqbegin(const seqlock_t *sl) {
    unsigned int seq;
    do {
        seq = READ_ONCE(sl->sequence);  // 读序列号
        smp_rmb();                       // 内存屏障
        if (seq & 1)                     // 奇数 → 写者在操作
            continue;                    // 自旋等待写者完成
        break;
    } while (true);
    return seq;
}

bool read_seqretry(const seqlock_t *sl, unsigned int start_seq) {
    smp_rmb();
    return sl->sequence != start_seq;  // 序列号变了 → 重试
}
```

**读者的使用模式**：

```c
unsigned int seq;
do {
    seq = read_seqbegin(&sl);   // 读取操作前序列号
    // === 读取共享数据 (完全无锁!) ===
    data_copy = shared_data;
    // ==================================
} while (read_seqretry(&sl, seq));  // 序列号变了就重试
```

**seqlock 的适用条件**：

| 条件 | 说明 |
|------|------|
| 写极少, 读极多 | 否则读者反复重试 |
| 数据是简单类型 (int, ptr) | 不支持指针链/复杂结构——否则重试成本高 |
| 读者不能阻塞 (中断上下文) | seqlock 读者不会睡眠, 只是重试 |
| 不需要多个写者并发 | 写者间仍用自旋锁互斥 |

**Linux 内核中的典型用途**：
- `jiffies` (内核"心跳"计数器) ——写者极少 (每 tick 一次), 读者遍布内核各处
- 内存描述符 (VMA) 的快速遍历

### 6.4 RCU (Read-Copy-Update) — 简介

RCU 是比 seqlock 更高级的同步机制，核心思想：
1. 写者不直接修改原数据——而是**复制一份 → 修改副本 → 原子的更新指针**
2. 旧数据在所有读者离开后才释放

RCU 允许读者**完全无开销**地读取（甚至不需要读序列号），代价是写者更复杂、延迟回收内存。Linux 内核大量使用 RCU（如网络路由表、文件系统 dentry cache）。

## 7. 全维度对比总结

| 机制 | 等待方式 | 互斥粒度 | 读写区分 | 中断可用 | 内核态 | 让权等待 | 公平性 | 实现复杂度 |
|------|---------|---------|---------|---------|--------|---------|--------|-----------|
| **Peterson 算法** | 忙等 | 进程互斥 | 否 | 否 | 否 | 否 | 是 (交替) | 低 |
| **TAS / CAS 自旋锁** | 忙等 | 进程互斥 | 否 | **是** | 是 | **否** | 否 | 极低 |
| **互斥锁 (Mutex)** | 阻塞 | 进程互斥 | 否 | 否 | 是 | 是 | 否 (通常不保证) | 中 |
| **读写锁 (rwlock)** | 阻塞 | 读共享/写互斥 | **是** | 否 | 是 | 是 | 取决于策略 | 中 |
| **读写自旋锁** | 忙等 | 读共享/写互斥 | **是** | 是 | 是 | **否** | 取决于策略 | 中 |
| **顺序锁 (seqlock)** | 读者无等待 | 读无锁/写互斥 | **是** | 是 (读者) | 是 | 读者让权 | 写优先 | 中 |
| **AND型信号量** | 阻塞 | 多资源同时申请 | 否 | 否 | 是 | 是 | 否 | 中 |
| **RCU** | 读者无等待 | 读无锁 | **是** | 是 (读者) | 是 | 读者让权 | 写延迟回收 | 高 |

### 选型指南

```
是否需要读写区分?
├── 否 → 纯互斥需求
│   ├── 临界区 < 1μs 且在中断上下文? → 自旋锁
│   └── 临界区较长或在用户态? → 互斥锁 (Mutex)
└── 是 → 有读写区分需求
    ├── 读极多, 写极少, 读写都在内核? → 顺序锁 (seqlock)
    ├── 读多写少, 临界区短, 中断可用? → 读写自旋锁
    ├── 读多写少, 临界区长, 可阻塞? → 读写锁 (rwlock)
    └── 读极多, 复杂数据结构? → RCU
```
