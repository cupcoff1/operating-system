# 死锁条件与预防

## 知识框架

```
死锁 (Deadlock)
│
├── 定义：一组进程互相等待对方持有的资源，导致无限期阻塞
│
├── 四个必要条件（缺一不可）
│   ├── ① 互斥 (Mutual Exclusion)       ─ 资源不可共享，只能独占
│   ├── ② 持有并等待 (Hold and Wait)    ─ 占着已有的，等还没拿到的
│   ├── ③ 不可抢占 (No Preemption)      ─ 已分配资源不能被强制剥夺
│   └── ④ 循环等待 (Circular Wait)      ─ 进程间形成闭环等待链
│
├── 资源分配图 (Resource-Allocation Graph)
│   ├── 无环 → 无死锁（充分条件）
│   ├── 有环 + 单实例 → 死锁（充要）
│   └── 有环 + 多实例 → 可能死锁，也可能不死锁
│
├── 资源分类
│   ├── 可剥夺资源 (Preemptable)        ─ 可强制收回（如 CPU、内存页）
│   └── 不可剥夺资源 (Non-Preemptable)  ─ 强制收回导致数据损坏（如打印机、数据库锁）
│
├── 死锁处理四大策略
│   ├── 预防 (Prevention)   ─ 破坏一个必要条件，死锁永不可能发生
│   ├── 避免 (Avoidance)    ─ 动态决策，拒绝危险请求（银行家算法）
│   ├── 检测与恢复 (Detection & Recovery) ─ 允许发生，事后检测并补救
│   └── 鸵鸟算法 (Ostrich)  ─ 假装没有死锁这回事
│
└── 容易混淆的三个概念
    ├── 死锁 (Deadlock) ─ 互相等待，全部阻塞
    ├── 活锁 (Livelock) ─ 不断重试，但无实际进展（互相"谦让"）
    └── 饥饿 (Starvation) ─ 低优先级进程长期得不到资源，但不阻塞
```

---

## 死锁的形式化定义

> 一组进程中的**每一个**都在等待一个事件，而该事件只能由这组进程中的**另一个**来触发——结果谁也无法被唤醒。

四个必要条件**必须同时满足**，缺任意一个则死锁不会发生。这一结论同时指导了死锁预防的策略：破坏任何一个条件即可杜绝死锁。

---

## 四个必要条件详解（附破坏分析）

### 条件 1: 互斥 (Mutual Exclusion)

**定义：** 至少有一个资源必须**独占使用**。同一时刻，最多一个进程能使用该资源。如果有其他进程请求，请求者必须等待资源被释放。

**为什么要独占？**
考虑一台打印机。两个进程同时向打印流写入数据，最终输出会是两份文档交错混杂的废纸。打印机的物理特性决定了它只能被独占——这是**资源的固有属性**，不是操作系统能用软件完全改变的。

**为什么破坏它能防止死锁？**
如果所有资源都变成可共享的（像只读文件），任何进程随时都能访问所有资源，无需等待任何事情。没有"等待"就没有死锁的土壤。

**破坏方法：** 将独占资源虚拟化为逻辑共享。例如，`spooling`（假脱机）技术让打印机资源在逻辑上"共享"——所有输出先写入磁盘 spool 目录，由 printer daemon 统一调度打印，进程不直接竞争物理打印机。

**现实局限：** 不是所有资源都能共享化。内核互斥锁、某些硬件寄存器、数据库写锁仍然必须互斥。这是一个理论基础最薄弱、实践中最难破坏的条件。

---

### 条件 2: 持有并等待 (Hold and Wait)

**定义：** 进程在持有部分资源的同时，等待获取其他资源。它不释放已持有的资源，而是在"手握资源，眼望别处"的状态下阻塞。

**为什么破坏它能防止死锁？**
去掉了"等的时候还占着"这一步。如果进程在请求新资源之前先释放所有旧资源，就不会出现"你等我放 R2、我等你放 R1"的闭环——每个进程要么全占、要么全空，无中间状态。

**破坏方法：**
- **一次性全申请法：** 进程在开始执行前一次性请求所有可能需要的资源。申请不到则一个都不要（不出让任何资源）。优点：实现简单。缺点：资源利用率极低——进程可能在运行前 10 分钟就占着最后 5 分钟才用到的资源；且 Max 必须提前知道（与银行家算法的局限同源）。
- **两阶段法：** 进程释放所有资源后才提出新请求。类似要求进程"先归还再借"。

**现实局限：** 某些资源（如打开的文件描述符）一旦释放，之前的工作状态就丢了；而且动态资源需求往往不可预知。

---

### 条件 3: 不可抢占 (No Preemption)

**定义：** 已分配给进程的资源不能被**强制剥夺**，只能由持有进程自愿释放。

**为什么破坏它能防止死锁？**
如果可以强行从进程手中抢走资源，P0 就不需要等 P1 主动放手了——系统直接剥夺 P1 的资源分给 P0，等待链断裂，循环无法形成。

**破坏方法：**
- 如果一个进程请求资源失败，系统回收它**已持有的所有资源**（连同被抢占的，一起放回 Available），让该进程重新等待。
- 设置抢占协议：某些资源类型（如 CPU、内存）可以保存上下文→剥夺→恢复——这就是 **可剥夺调度 (Preemptive Scheduling)** 的基本思想。

**现实局限：** 不可抢占是很多资源的物理属性。抢走打印机 --> 打印一半的纸作废；抢走磁带机 --> 数据撕裂；抢走互斥锁保护的内核数据结构 --> 系统崩溃。这个条件的破坏只对 **可剥夺资源**（见下文）有意义。

---

### 条件 4: 循环等待 (Circular Wait)

**定义：** 存在一条进程-资源的闭合等待链：P0 等 P1 放资源，P1 等 P2 放资源，……，Pn 等 P0 放资源。

**为什么破坏它能防止死锁？**
四条件的"封顶"条件。即使前三项满足，只要没有环，就不会死锁（最多出现有限等待）。破坏环等价于要求所有等待关系构成一个**偏序**——不能有回路。

**破坏方法：按序分配 (Resource ordering)：**
1. 给每种资源分配一个唯一编号（如 R1=1, R2=2, ..., Rm=m）。
2. 每个进程只能按**编号递增**的顺序申请资源（即不允许申请编号 ≤ 当前持有的任何资源的编号）。
3. 证明：编号递增 → 不可能形成环（环要求存在"大的等小的"等关系，与递增矛盾）。

**缺点：** 全局编号未必总能合理定义；进程可能要先申请不急着用的高编号资源（为了随后合法申请低编号资源），反而降低利用率；资源类型多的系统中编号体系难以维护。

## 资源分配图 (Resource-Allocation Graph)

资源分配图是**死锁检测**的可视化工具，由两类节点和两类有向边构成：

| 符号 | 含义 |
|------|------|
| 圆圈 ○ | **进程节点** (Process) |
| 方框 □ | **资源节点** (Resource)；方框内的小圆点 ● 表示该资源的**实例数** |
| P → R (请求边) | 进程 P 正在等待资源 R 的一个实例 |
| R → P (分配边) | 资源 R 的一个实例已分配给进程 P |

**判断规则：**
- **无环** → 一定无死锁（充分条件）。
- **有环** → 视资源实例数决定：
  - 环上每种资源均只有 **1 个实例** → 必有死锁（充要条件）。
  - 环上存在资源有 **多个实例** → 可能死锁，也可能不死锁——要看环外的进程是否最终释放环内进程需要的实例。

---

### 示例 1: 无环 → 无死锁

```
     R1 ●━━━━━━━━━━→ P1

     R2 ●━━━━━━━━━━→ P2

     P3 ──→ R1 (请求边)

     R3 ●━━━━━━━━━━→ P3
                       |
                       └──→ R2 (请求边)
```

图中只有边：R1 → P1, R2 → P2, P3 → R1, R3 → P3, P3 → R2。

**检查环路：** 从任意节点出发均无法回到自身。图中**不存在闭合环路**，系统一定没有死锁。P3 在等待 R1 和 R2，但 P1 / P2 完成后会释放相应资源。

---

### 示例 2: 有环 + 多实例 → 可能无死锁

```
     R1 (2 个实例 ●●)

      ├──→ P1 ──→ R2 (1 个实例 ●)

      └──→ P3          │
                        │
      P2 ──→ R1      P2 ←── R2
             ▲             │
             └─────────────┘
           (P2 同时等 R1、持有?)
```

重新整理为一个具体的多实例场景：

- **R1** 有 **2** 个实例，R2 有 **1** 个实例。
- P1 持有 1 个 R1 实例，请求 R2。（R1 → P1, P1 → R2）
- P2 持有 R2 的 1 个实例，请求 R1。（R2 → P2, P2 → R1）
- **P3 持有 1 个 R1 实例**，不请求任何资源。（R1 → P3）

**环路分析：**
P1 → R2 → P2 → R1 → P1 构成一个**环**。

**但是否死锁？** P3 持有 R1 的另一个实例。当 P3 完成执行后释放该实例，P2 获得 R1 的实例，完成后释放 R2，P1 再获得 R2……系统可以推进。虽然有环，但因为 R1 有多个实例，环外进程 P3 能"救场"，所以**有环但无死锁**。

---

### 示例 3: 有环 + 单实例 → 必定死锁

```
     R1 ●━━━━→ P1 ──→ R2 ●

      ▲                │
      │                │
      │                ▼

     P3 ←── R3 ●  ←── P2
       │
       └──→ R1 (请求边)
```

整理为清晰的单实例闭环：

- R1、R2、R3 各只有 **1 个实例**。
- P1 持有 R1，请求 R2。（R1 → P1, P1 → R2）
- P2 持有 R2，请求 R3。（R2 → P2, P2 → R3）
- P3 持有 R3，请求 R1。（R3 → P3, P3 → R1）

**环路分析：**
P1 → R2 → P2 → R3 → P3 → R1 → P1 构成**一个完整的环**，且环上每种资源只有 1 个实例。

**结论：** 因为每个资源仅有一个实例，没有"环外的多余实例"能打破僵局。每个进程都在等下一个进程释放唯一的资源，而下一个进程也在等——**死锁已经发生。**

## 可剥夺资源 vs 不可剥夺资源

| 维度 | 可剥夺资源 (Preemptable Resource) | 不可剥夺资源 (Non-Preemptable Resource) |
|------|-----------------------------------|-----------------------------------------|
| **定义** | 系统可从进程手中强制收回，不会造成数据丢失或状态不一致 | 强制收回会导致工作丢失、数据损坏或系统状态错误 |
| **典型例子** | CPU（时间片到期剥夺）、内存页（换出到磁盘）、磁盘 IO 带宽 | 打印机（打印一半无法复原）、磁带机、数据库行锁、内核互斥锁持有的共享数据结构 |
| **被剥夺时的处理** | 保存上下文（保存寄存器 → PCB、写回交换区），之后可恢复 | 无通用恢复方法——剥夺意味着操作报废 |
| **与死锁的关系** | 如果所有资源均可剥夺 → 破坏条件③ → 永不出现死锁 | 不可剥夺资源是死锁的必要条件③的基础；存在不可剥夺资源才可能死锁 |
| **使用死锁预防** | 可对该资源类型应用抢占式死锁预防 | 只能通过破坏条件①②④来处理（互斥 / 持有并等待 / 循环等待） |
| **类比** | 图书馆借书，管理员可催你还书后借给别人 | 手术台上的手术刀——不能中途抢走 |

## 活锁 (Livelock) vs 饥饿 (Starvation) vs 死锁 (Deadlock)

这三个概念在考试和面试中经常被混淆。它们的核心区别在于**进程是否在运行**和**是否可能自愈**。

| 维度 | 死锁 (Deadlock) | 活锁 (Livelock) | 饥饿 (Starvation) |
|------|-----------------|------------------|--------------------|
| **进程状态** | 阻塞（Blocked），等待事件发生 | 就绪/运行（Running），不断主动重试 | 就绪（Ready），等着被调度 |
| **是否在做运算** | 否，CPU 切换到其他进程 | **是**，在 CPU 上执行循环、重试逻辑 | 否，没被调度上 CPU |
| **是否有进展** | 无——永久阻塞 | 无——反复执行但每次都因冲突而失败（"空转"） | 无——始终得不到所需资源 |
| **自愈可能** | 需要外界干预（杀进程 / 回滚） | 如果退避策略设计良好，可能自愈 | 如果调度器有 Aging 机制，可以自愈 |
| **发生原因** | 四个必要条件全部满足 | 冲突解决策略不当（如双方同时退让、同时重试） | 资源分配策略偏向某些进程（如优先级调度永不给低优先级 CPU） |
| **涉及等待关系** | 彼此等待对方释放资源 | 互相"让步"，都不真正占有资源 | 有进程一直得不到资源，但其他进程正常执行 |
| **经典比喻** | 两辆车在单行桥中间相遇，各不退让 | 两人在走廊迎面相遇，同时向左让、又同时向右让、反复不止 | 高级餐厅只招待 VIP，普通顾客永远排不上桌 |

### 活锁详细分析

```
进程 A:                            进程 B:
while (!acquired) {                while (!acquired) {
    lock(L);                           lock(L);
    if (resource_busy) {               if (resource_busy) {
        unlock(L);                         unlock(L);
        sleep(random());                   sleep(random());
        continue;  // 重试                  continue;  // 重试
    }                                  }
    // 使用资源                         // 使用资源
}                                    }
```

活锁的关键特征：**进程状态不是"阻塞"而是"运行"**，但它们反复执行相同的失败路径，永远到不了成功使用资源的代码。与死锁不同的是，活锁中的进程没有"卡在某个锁上"——它们甚至成功获得了锁、做了检查、释放了锁——只是每次都无功而返。

**解除活锁：随机退避 (Random Backoff)。** 上述代码中的 `sleep(random())` 如果换成固定的或同步的退避时间，两个进程仍可能同时醒来、同时检查、同时发现冲突、同时再次退让。解决方法：使用**指数退避 (exponential backoff)** + 引入随机因子，确保双方节奏迟早错开。以太网的 CSMA/CD 协议就是经典案例。

### 饥饿详细分析

饥饿的最典型场景：**优先级调度 + 不断有高优先级进程到达**。低优先级进程永远得不到 CPU，称为"饿死"。

更进一步：在死锁**检测与恢复**策略中，如果恢复手段是"杀进程"，且每次都选同一个"代价最小"的进程来杀——该进程可能被反复杀死、反复重启、永远无法完成工作。这构成另一种形式的饥饿（绕开死锁，代之以"被反复选为牺牲品"）。

**解除饥饿：Aging（老化）。** 进程等待时间越长，优先级越高。确保任何进程在有限时间内必被服务。

## 死锁预防总表

| 破坏条件 | 核心方法 | 优点 | 缺点 | 实用性 |
|----------|----------|------|------|--------|
| **互斥** | 资源共享化 (SPOOLing) | 理论上生效 | 打印机等物理资源不可共享；互斥锁必须互斥 | 低 |
| **持有并等待** | 一次性申请全部资源 | 简单，不需要抢占 | 利用率低；需预知 Max；可能导致饥饿（某些进程永远凑不齐全部资源） | 中 |
| **不可抢占** | 允许系统强制剥夺 | 有效消除等待链 | 只适用于可保存/恢复状态的资源；不适用于打印机等 | 低~中 |
| **循环等待** | 按序分配（编号递增） | 系统开销小，不需预知 Max | 全局编号不易；可能降低利用率 | 中~高 |
