# 银行家算法

## 知识框架

```
银行家算法 (Banker's Algorithm) — Dijkstra 提出
│
├── 定位：死锁避免 (Deadlock Avoidance)
│   └── 动态检查每个资源请求是否会导致系统进入不安全状态
│
├── 核心数据结构
│   ├── Available[j]  ─ 资源 j 当前可用数量
│   ├── Max[i][j]     ─ 进程 i 对资源 j 的最大需求（声明上限）
│   ├── Allocation[i][j] ─ 进程 i 已分配的资源 j 数量
│   └── Need[i][j]    ─ 进程 i 还需要的资源 j = Max - Allocation
│
├── 安全性算法 (Safety Algorithm)
│   │  判断系统当前是否处于**安全状态**
│   └── 流程：Work=Available → 找 Need≤Work 的进程 → 释放其资源 → 重复
│
├── 资源请求算法 (Resource-Request Algorithm)
│   │  每次请求时动态决策：同意 / 等待 / 拒绝
│   └── 流程：验证 Request≤Need → Request≤Available → 试探分配 → 安全性检查
│
└── 与其他策略对比
    ├── 死锁预防 ─ 静态，破坏必要条件，资源利用率低
    ├── 死锁避免 ─ 动态，须预知 Max，利用率中等 ★本条
    ├── 死锁检测 ─ 允许死锁发生，事后检测并恢复
    └── 鸵鸟算法 ─ 假装死锁不存在（Unix/Linux 实际策略）
```

---

## 术语定义

在进入算法之前，先定义每一个关键术语：

| 术语 | 英文 | 定义 |
|------|------|------|
| **安全状态 (Safe State)** | Safe State | 系统存在至少一个执行序列，能让所有进程顺利执行完毕而不会死锁。若不存在这样的序列，则系统处于**不安全状态 (Unsafe State)**。注意：不安全状态不一定会死锁，但死锁一定发生在不安全状态中。 |
| **安全序列 (Safe Sequence)** | Safe Sequence | 一个进程执行顺序 `<P_a, P_b, ..., P_n>`，按此顺序，每个进程在轮到执行时，其 Need 可以被当时的 Available 满足，执行完毕释放其持有的 Allocation，进而让后续进程也能满足。 |
| Available | Available | 系统中每种资源**当前空闲**的数量，随时变化。初始值 = 总资源量 - 所有 Allocation 之和。 |
| Max | Max | 进程 i 对资源 j 的**最大需求量**，在该进程整个生命周期中不变。必须事先声明——这也是银行家算法在实际系统中难以应用的主因。 |
| Allocation | Allocation | 进程 i **已分配**的资源量，随分配/释放而变化。 |
| Need | Need | 进程 i **仍欠缺**的资源量。**Need = Max - Allocation**。该值随进程执行递减。 |

---

## 数据结构

以表格形式给出多维资源情况。设系统有 **n 个进程**、**m 种资源**：

| 向量 / 矩阵 | 维度 | 含义 |
|-------------|------|------|
| `Available[m]` | 1 × m | 每种资源当前剩余可用数 |
| `Max[n][m]` | n × m | 进程 i 对资源 j 的最大需求 |
| `Allocation[n][m]` | n × m | 进程 i 已分得资源 j 的数量 |
| `Need[n][m]` | n × m | 进程 i 还需资源 j 的数量，恒等式：**Need = Max − Allocation** |
| `Finish[n]` | 1 × n | 布尔数组，安全性算法中使用，标识进程是否完成 |

核心约束（始终成立）：
$$
\text{Available}[j] + \sum_{i=0}^{n-1} \text{Allocation}[i][j] = \text{资源j总量}
$$

## 安全性检查算法 (Safety Algorithm)

> 用途：判断当前系统是否处于安全状态。每次资源请求的试探阶段都会调用一次。

**伪代码：**

```
Safety():
  1. Work[j] = Available[j]           for j = 0..m-1
     Finish[i] = false                 for i = 0..n-1

  2. 重复查找满足以下条件的进程 Pi:
       (a) Finish[i] == false
       (b) Need[i][j] ≤ Work[j]       for all j = 0..m-1

     如果找到：
       Work[j] = Work[j] + Allocation[i][j]     // 进程"归还"所有已分配资源
       Finish[i] = true                         // 标记该进程已经完成
       继续步骤 2

     如果找不到:
       跳至步骤 3

  3. 若所有 Finish[i] == true → 系统**安全**
     否则 → 系统**不安全**
```

**时间复杂度：** O(m × n²)。每轮遍历 n 个进程，最多扫描 n 轮；每轮比较 m 种资源。

**关键直觉：** 算法模拟"如果所有进程都能按某个顺序执行完毕"——找到一个就把它持有的资源释放回 Work 池，继续找下一个，直到所有进程都完成（安全）或卡住（不安全）。

## 资源请求算法 (Resource-Request Algorithm)

> 用途：当进程 Pi 发出资源请求 `Request_i[m]` 时，决定是否批准。

**伪代码：**

```
ResourceRequest(Pi, Request_i):

  // 阶段 1: 合法性检查
  if Request_i[j] > Need[i][j] for any j:
      报错："进程超过声明的最大需求"

  // 阶段 2: 可行性检查
  if Request_i[j] > Available[j] for any j:
      Pi 必须等待（系统当前无足够空闲资源）

  // 阶段 3: 试探分配 (Tentative Allocation)
  Available[j]  = Available[j]  - Request_i[j]
  Allocation[i][j] = Allocation[i][j] + Request_i[j]
  Need[i][j]      = Need[i][j]      - Request_i[j]

  // 阶段 4: 安全性复查
  if Safety() == true:
      正式分配，Pi 继续执行
  else:
      撤销试探分配（恢复之前的三组值）
      Pi 必须等待
```

**关键口诀（三个检查一个试探）：**
1. 要的不能超过声明的 → `Request ≤ Need`
2. 要的不能超过现有的 → `Request ≤ Available`
3. 假装给了，跑一遍安全性 → 安全才真给，不安全就回滚

## 完整演示例 (5 进程、3 资源类型 A/B/C)

### 初始状态

系统资源总量：**A=10, B=5, C=7**（可从初始 Available + sum(Allocation) 反推验证）

| 进程 | Max (A,B,C) | Allocation (A,B,C) | Need = Max − Allocation |
|------|-------------|--------------------|-------------------------|
| P0 | (7, 5, 3) | (0, 1, 0) | (7, 4, 3) |
| P1 | (3, 2, 2) | (2, 0, 0) | (1, 2, 2) |
| P2 | (9, 0, 2) | (3, 0, 2) | (6, 0, 0) |
| P3 | (2, 2, 2) | (2, 1, 1) | (0, 1, 1) |
| P4 | (4, 3, 3) | (0, 0, 2) | (4, 3, 1) |

初始 **Available = (3, 3, 2)**

验证：总资源 = (0+2+3+2+0+3, 1+0+0+1+0+3, 0+0+2+1+2+2) = (10, 5, 7) ✓

---

### 步骤 1: 计算 Need（已在上表完成）

Need = Max − Allocation，每个进程都需要按行相减。本步无决策，纯计算。


### 步骤 2: 运行安全性算法（推演安全序列）

**初始 Work = Available = (3, 3, 2)；Finish = [F, F, F, F, F]**

---

#### 第 1 轮扫描

| 进程 | Need | ≤ Work? (3,3,2) | 结果 |
|------|------|-----------------|------|
| P0 | (7,4,3) | 7 > 3 ✗ | 跳过 |
| **P1** | **(1,2,2)** | **1≤3, 2≤3, 2≤2 ✓** | **选中** |
| P3 | (0,1,1) | — (本轮已选中P1) | — |

**选中 P1：**
- Work = (3,3,2) + Allocation_P1(2,0,0) = **(5, 3, 2)**
- Finish[1] = T
- 安全序列候选: `< P1`

---

#### 第 2 轮扫描

| 进程 | Need | ≤ Work? (5,3,2) | 结果 |
|------|------|-----------------|------|
| P0 | (7,4,3) | 7 > 5 ✗ | 跳过 |
| P2 | (6,0,0) | 6 > 5 ✗ | 跳过 |
| **P3** | **(0,1,1)** | **0≤5, 1≤3, 1≤2 ✓** | **选中** |
| P4 | (4,3,1) | — | — |

**选中 P3：**
- Work = (5,3,2) + Allocation_P3(2,1,1) = **(7, 4, 3)**
- Finish[3] = T
- 安全序列候选: `< P1, P3`

---

#### 第 3 轮扫描

| 进程 | Need | ≤ Work? (7,4,3) | 结果 |
|------|------|-----------------|------|
| P0 | (7,4,3) | — | — |
| P2 | (6,0,0) | — | — |
| **P4** | **(4,3,1)** | **4≤7, 3≤4, 1≤3 ✓** | **选中** |

**选中 P4：**
- Work = (7,4,3) + Allocation_P4(0,0,2) = **(7, 4, 5)**
- Finish[4] = T
- 安全序列候选: `< P1, P3, P4`

---

#### 第 4 轮扫描

| 进程 | Need | ≤ Work? (7,4,5) | 结果 |
|------|------|-----------------|------|
| **P0** | **(7,4,3)** | **7≤7, 4≤4, 3≤5 ✓** | **选中** |
| P2 | (6,0,0) | — | — |

**选中 P0：**
- Work = (7,4,5) + Allocation_P0(0,1,0) = **(7, 5, 5)**
- Finish[0] = T
- 安全序列候选: `< P1, P3, P4, P0`

---

#### 第 5 轮扫描

| 进程 | Need | ≤ Work? (7,5,5) | 结果 |
|------|------|-----------------|------|
| **P2** | **(6,0,0)** | **6≤7, 0≤5, 0≤5 ✓** | **选中** |

**选中 P2：**
- Work = (7,5,5) + Allocation_P2(3,0,2) = **(10, 5, 7)**
- Finish[2] = T
- 安全序列候选: `< P1, P3, P4, P0, P2`

---

### 步骤 3: 结论

全部 `Finish[i] = true` → **系统处于安全状态。**

找到一个安全序列：**P1 → P3 → P4 → P0 → P2**（不唯一，另一合法序列为 P3 → P1 → P4 → P0 → P2）。

## 资源请求示例：P1 请求 (1, 0, 2)

某时刻，P1 发出请求 **Request = (1, 0, 2)**（需要 1 个 A、0 个 B、2 个 C）。

### 阶段 1: Request ≤ Need？

| 维度 | Request_P1 | Need_P1 | 比较 |
|------|-----------|---------|------|
| A | 1 | 1 | 1 ≤ 1 ✓ |
| B | 0 | 2 | 0 ≤ 2 ✓ |
| C | 2 | 2 | 2 ≤ 2 ✓ |

通过。请求没有超过 P1 声明的最大需求。

### 阶段 2: Request ≤ Available？

| 维度 | Request_P1 | Available | 比较 |
|------|-----------|-----------|------|
| A | 1 | 3 | 1 ≤ 3 ✓ |
| B | 0 | 3 | 0 ≤ 3 ✓ |
| C | 2 | 2 | 2 ≤ 2 ✓ |

通过。系统当前有足够的空闲资源。

### 阶段 3: 试探分配

假装批准请求，修改三组数据：

**Available 变化：**

| | A | B | C |
|------|---|---|---|
| 试探前 | 3 | 3 | 2 |
| 减 Request | −1 | −0 | −2 |
| **试探后** | **2** | **3** | **0** |

**Allocation[1] 变化：**

| | A | B | C |
|------|---|---|---|
| 试探前 | 2 | 0 | 0 |
| 加 Request | +1 | +0 | +2 |
| **试探后** | **3** | **0** | **2** |

**Need[1] 变化：**

| | A | B | C |
|------|---|---|---|
| 试探前 | 1 | 2 | 2 |
| 减 Request | −1 | −0 | −2 |
| **试探后** | **0** | **2** | **0** |

**试探后的完整快照：**

| 进程 | Allocation | Max | Need |
|------|------------|-----|------|
| P0 | (0, 1, 0) | (7, 5, 3) | (7, 4, 3) |
| **P1** | **(3, 0, 2)** | **(3, 2, 2)** | **(0, 2, 0)** |
| P2 | (3, 0, 2) | (9, 0, 2) | (6, 0, 0) |
| P3 | (2, 1, 1) | (2, 2, 2) | (0, 1, 1) |
| P4 | (0, 0, 2) | (4, 3, 3) | (4, 3, 1) |

试探后 Available = **(2, 3, 0)**

### 阶段 4: 重新运行安全性算法

**Work = (2, 3, 0)；Finish = [F, F, F, F, F]**

#### 第 1 轮

## 三种死锁处理策略对比

| 维度 | 死锁预防 (Prevention) | 死锁避免 (Avoidance) | 死锁检测与恢复 (Detection & Recovery) |
|------|----------------------|----------------------|---------------------------------------|
| **策略思想** | 运行前就破坏四个必要条件之一，让死锁根本不可能发生 | 运行时动态判断，拒绝任何可能通向不安全状态的请求 | 不阻止死锁；允许发生，用检测算法周期性扫描，发现后恢复 |
| **代表算法** | 一次性申请全部、按序分配 | **银行家算法 (Banker's)** | 资源分配图化简 / 等步检测算法 |
| **是否需要预知 Max** | 否 | **是**（进程必须事先声明最大需求） | 否 |
| **资源利用率** | 低（资源被提前锁定但闲置） | 中等（动态分配，但偏保守） | 高（尽可能满足请求，事后补救） |
| **吞吐量** | 低 | 中等 | 高（无预防开销时） |
| **检测/计算开销** | 无运行时开销（静态约束） | 每次请求 O(m·n²) | 周期性检测 O(m·n²)；恢复成本高（杀进程/回滚） |
| **恢复手段** | 不需要（死锁永不发生） | 不需要（死锁永不发生） | 终止进程、回滚事务、可剥夺释放 |
| **适用场景** | 小型嵌入式、实时系统、资源类型少 | 批处理系统、数据库系统、资源类型明确可控 | 大型通用操作系统（Linux/Windows 实际策略）、分布式系统 |
| **实际采用情况** | 少（互斥难以破坏，其余破坏代价大） | 少（Max 难以预知） | **最常用**（省去声明 Max 的负担，死锁概率本身不高） |
| **经典比喻** | "永远不让你上马路，因为可能撞车" | "看到前面会堵就让你等一等" | "撞了再叫警察来处理" |

## 银行家算法的优缺点

| | 说明 |
|------|------|
| **优点** | ① 不需要抢占或回滚（相比检测-恢复更温和）；② 资源利用率高于死锁预防；③ 保证系统永不进入死锁状态。 |
| **缺点** | ① 必须预知 Max，实际系统中进程很难事前声明；② 资源数和进程数必须固定（不允许热插拔）；③ 安全性检查 O(m·n²)，在大型系统中可能成为瓶颈；④ 过于保守——即使存在安全序列，也可能拒绝一些实际上不会导致死锁的请求。 |

## 常见考题

1. **给定 Alloc/Max/Available，求安全序列。** 按上述轮询法逐步推演即可。
2. **给定一个 Request，判断是否可批准。** 严格执行三个检查一个试探。
3. **是非判断："不安全状态一定会死锁。"** 错误。不安全状态不一定死锁；但死锁一定意味着不安全。
4. **银行家算法属于哪种死锁处理策略？** 死锁避免 (Deadlock Avoidance)。
