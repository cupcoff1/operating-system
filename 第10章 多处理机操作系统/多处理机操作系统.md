# 多处理机操作系统

> 对应慕课版第 10 章。本章讨论多处理机系统的架构分类（紧耦合/松耦合、SMP/ASMP、UMA/NUMA/COMA）、缓存一致性问题的原理与 MESI 协议、多处理机下的同步机制（自旋锁的底层分析和 RCU）、以及多处理机调度策略（负载均衡、亲和性、big.LITTLE）。

## 多处理机系统特征

| 特征 | 含义 | 与单处理机的区别 |
|------|------|-----------------|
| **并行性（Parallelism）** | 多个 CPU 物理上**同时**执行不同进程 / 线程 | 单处理机仅能并发（Concurrency，时间片轮转），非真正同时。多处理机实现真正的时间重叠执行。 |
| **分布性（Distribution）** | 任务（任务/线程）可被分配到不同的物理处理器上执行 | 引入负载均衡、调度域、NUMA 感知等问题。 |
| **通信性（Communication）** | 处理器间通过**共享内存**（紧耦合）或**消息传递**（松耦合，Message Passing）交换信息 | 共享内存通信需同步机制；消息通信需网络/总线支持。 |

## 系统架构分类

### 按耦合度分类

**耦合度（Coupling）** 是指多个处理器之间共享资源的紧密程度。

| 类型 | 架构 | 通信方式 | 扩展性 | 典型系统 |
|------|------|----------|--------|----------|
| **紧耦合（Tightly Coupled）** | 所有 CPU 共享同一物理内存和 I/O 子系统 | 共享内存（通过总线/交叉开关），信号量/互斥量/屏障同步 | 受总线带宽限制，通常 2-64 核不等 | 单台多核服务器、SMP 工作站 |
| **松耦合（Loosely Coupled）** | 每个 CPU 有独立本地内存，计算机之间通过网络互连 | 消息传递（Message Passing），如 MPI（Message Passing Interface） | 极强，可达数万节点 | 集群（Cluster）、分布式系统、MPP 超算 |

### 按对称性分类：SMP vs ASMP

#### SMP（Symmetric Multi-Processing，对称多处理）

**定义**：所有处理器地位完全平等，共享同一物理内存，每个处理器可以运行任何进程 / 线程，操作系统由任一处理器执行。

**核心特征：**
- 所有 CPU 有等价的硬件能力（同构，Homogeneous）
- 共享同一操作系统内核镜像（一份内核代码被所有 CPU 共享执行）
- 任何一个 CPU 都可以处理来自任何 I/O 设备的中断
- 内核必须保证：两个 CPU **绝不**同时执行同一段可能冲突的内核代码 —— 需要细粒度的内核锁

**具体例子**：一台运行 Linux 的 8 核 x86 服务器。每个核都能：
- 处理中断（irqbalance 分配或手动绑核）
- 执行任意用户进程
- 运行系统调用的内核路径
- 访问全部物理内存（通过系统总线）

#### ASMP（Asymmetric Multi-Processing，非对称多处理）

**定义**：有一个主处理器（Master）运行操作系统内核并分配任务，其余从处理器（Slave）仅执行主处理器分派的用户程序。

**核心特征：**
- 主 CPU 独享内核代码、I/O 处理、中断响应
- 从 CPU 只能运行用户态程序，不能执行内核代码
- 内核数据结构无需加锁（因为只有主 CPU 会修改），但可扩展性受限于主 CPU
- 操作系统设计相对简单

**具体例子**：早期 MP/M 操作系统、早期 SunOS 3.x 的 ASMP 模式、一些嵌入式异构 SoC（应用处理器 + 协处理器）。

| 对比维度 | SMP | ASMP |
|----------|-----|------|
| 处理器地位 | 全部平等、对等 | 一主多从 |
| 内核运行 | 任一 CPU 可执行内核代码 | 仅主 CPU 执行内核 |
| 中断处理 | 任何 CPU 可响应 | 仅主 CPU |
| 可扩展性 | 好（但锁竞争成瓶颈） | 差（主 CPU 成为单点瓶颈） |
| 内核复杂度 | 高（细粒度锁、并发控制） | 低（内核单线程、无需锁） |
| 负载均衡 | 可由任一 Core 发起调度 | 仅主 CPU 决策和分配 |
| 容错性 | 单 CPU 故障系统可能继续（取决于实现） | 主 CPU 故障 = 系统瘫痪 |
| 现状 | 主流，所有现代通用 OS | 仅限于特定嵌入式或遗留系统 |

## 内存访问模型

根据 CPU 访问内存的延迟是否一致，分为三种：

### UMA（Uniform Memory Access，统一内存访问）

- 所有处理器访问任意物理内存位置的延迟**完全相同**
- 共享一条系统总线或交叉开关（Crossbar）连接所有 CPU 和内存
- 实现简单，但内存带宽成为瓶颈 —— CPU 数量增多后，总线竞争急剧上升
- 典型系统：小型 SMP（2-8 核传统 x86 服务器，单内存控制器）

### NUMA（Non-Uniform Memory Access，非统一内存访问）

- 每个 CPU（或一组 CPU，称为 NUMA 节点 / Socket）配有**本地内存**（Local Memory）
- 访问本地内存快，访问其他节点的**远端内存**（Remote Memory）慢（通常 1.5-2 倍延迟）
- 解决了 UMA 的总线带宽瓶颈，但引入编程/调度复杂度
- 典型系统：多路 x86 服务器（2-8 Socket），每个 Socket 有自己的内存控制器
- **NUMA 感知调度**：OS 力求将进程/线程分配到其内存所在的本地 NUMA 节点，避免频繁跨节点访存

### COMA（Cache-Only Memory Access，仅缓存内存访问）

- 每个节点没有独立主存，只有大容量本地缓存
- 数据自动在各节点的缓存之间迁移和复制
- 编程模型简单（所有数据对任意节点的任意 CPU 都"看似"本地），但硬件实现极其复杂
- 属于学术/原型阶段多，代表系统：KSR-1、DDM（Data Diffusion Machine）

## 缓存一致性（Cache Coherence）

### 问题定义

在多处理机系统中，每个 CPU 有独立的**私有 Cache**（L1、L2 有时也私有）。若两个 CPU 的私有 Cache 中存有同一内存地址的副本，其中一个 CPU 对该地址执行写操作后，另一个 CPU 的 Cache 中该副本就变成了**过期数据（Stale Data）**。

**缓存一致性（Cache Coherence）** 就是确保这一矛盾的机制：对于同一个内存位置，所有 CPU 看到的写入顺序**看起来是一致的**。

```
时间线：
  CPU-A 读到地址 X 的值 = 5（缓存到 A 的私有 L1）
  CPU-B 读到地址 X 的值 = 5（缓存到 B 的私有 L1）
  CPU-A 写入 X = 42（仅更新了 A 的 L1，B 的 L1 仍是 5）
  CPU-B 读取 X → 仍是 5（读到了脏数据！）
```

### MESI 协议

MESI 是使用最广泛的**监听式缓存一致性协议（Snooping Cache Coherence Protocol）**。每个 Cache 行（Cache Line，通常 64 字节）处于以下四种状态之一，以该行当前所处的 MESI 状态决定 CPU 可以对该行执行哪些操作。

#### 四种状态

| 状态 | 全称 | 含义 | 该行是否仅在此 CPU 的 Cache 中？ | 该行与主存是否一致？（Dirty?） |
|------|------|------|--------------------------------|-----------------------------|
| **M（Modified）** | 已修改 | 该行数据仅在此 CPU 的 Cache 中，且已被修改（与主存不一致）。此 CPU 拥有该行的独占权，写回主存前必须独占。 | 是（该 CPU 独占） | 不一致（Dirty） |
| **E（Exclusive）** | 独占 | 该行数据仅在此 CPU 的 Cache 中，且与主存一致（Clean）。可以自由修改（转为 M）而不需通知其他 CPU。 | 是（该 CPU 独占） | 一致（Clean） |
| **S（Shared）** | 共享 | 该行数据可能存在于多个 CPU 的 Cache 中，与主存一致（Clean）。修改此状态的行必须先将其他 CPU 的副本**无效化**（Invalidation）。 | 否（可能被共享） | 一致（Clean） |
| **I（Invalid）** | 无效 | 该 Cache 行中的数据不可用（无效）。等效于"未缓存"。 | — | — |

#### MESI 状态转换图（简化）

```
M (Modified, Dirty, 独占)
  │ 其他 CPU 发起读 → 写回主存 → S (Shared, Clean)
  │ 其他 CPU 发起读且不写回 → 转发给请求方 → S
  └─────────────────────────────────────────────────
E (Exclusive, Clean, 独占)
  │ 本地写 → M (不需要总线事务！因为独占)
  │ 其他 CPU 发起读 → 转为 S（向其他 CPU 共享）
  └─────────────────────────────────────────────────
S (Shared, Clean, 可能多副本)
  │ 本地写 → 先发 Invalidate 总线事务使其他副本失效 → M
  │ 其他 CPU 发起写（监听到 Invalidate） → I
  └─────────────────────────────────────────────────
I (Invalid)
  │ 本地读（miss）→ 发总线 Read 事务
  │   若其他 CPU 无副本 → E（独占，Clean）
  │   若其他 CPU 有副本 → S（共享，Clean）
  │ 本地写（miss）→ 发总线 Read-For-Ownership (RFO) 事务
  │   使其他副本失效 → M（独占，Dirty）
```

**关键性能要点：**
1. **Invalidate 操作的开销**：CPU 写一个 S 状态的 Cache 行时，必须先向总线发出 Invalidate 事务，**等待所有其他 CPU 确认该行已失效**，才能完成写入。Invaldate 延迟直接影响写性能。
2. **RFO（Read-For-Ownership）**：CPU 写一个不在 Cache 中的行时，首先发出 RFO 请求。RFO 比普通 Read 更"重"，因为它同时要求"读取该行"和"独占所有权（使其他副本失效）"。RFO 的延迟是锁竞争场景的主要开销。
3. **总线监听（Bus Snooping）**：每个 CPU 的 Cache 控制器持续监听系统总线上的缓存一致性事务（Read, RFO, Invalidate, Writeback），并根据监听到的事务来更新本 Cache 行的状态。所有 CPU 共享一条总线，因此 Snooping 仅适用于总线互连的小规模 SMP。大规模 NUMA 系统使用**目录式（Directory-based）**一致性（缓存行状态存储在分布式目录中，而非广播到所有节点）。

#### 假共享（False Sharing）

如果两个 CPU 分别反复修改位于**同一 Cache 行（64 字节）内但不同的变量**，即使逻辑上这些变量相互独立，缓存一致性机制也会使该 Cache 行在两个 CPU 的 Cache 之间不断来回跳跃（"乒乓"效应，Ping-Pong），写性能急剧下降。

```
Cache Line (64 bytes):
  [ var_A (4B) ][ padding ][ var_B (4B) ][ ... ]

CPU-0 反复写 var_A ────┐
                        ├── 同一 Cache Line 在 M-I-M-I... 之间不断切换
CPU-1 反复写 var_B ────┘

解决方案：在频繁独立访问的字段之间插入填充（Padding）
使它们位于不同的 Cache Line（通常用 __cacheline_aligned 或手动 pad[64]）。
```

## 多处理机同步

### 关中断为何在多处理机中无效

在单处理机中，关中断可以防止同一 CPU 上的其他进程/中断处理程序抢占，从而保护临界区（临界区内的代码不会被中断或调度打断）。但在多处理机中，关中断**仅影响当前 CPU**：

- CPU-0 关中断 → CPU-0 不会被中断抢占
- 但 CPU-1 **仍然可以**执行任意代码，包括进入同一临界区

因此多处理机系统中必须有**真正的锁机制**（如自旋锁），以阻止其他 CPU 并发进入临界区。关中断在多处理机中仍需使用（防止同一 CPU 上的中断导致死锁），但仅是锁的辅助。

---

### 自旋锁（Spin Lock）深度解析

**定义**：自旋锁是一种忙等锁——当锁不可用时，请求锁的 CPU 在一个循环中反复检查锁的状态（"自旋"），而不是睡眠/阻塞/上下文切换。自旋锁适用于**临界区极短**的场景（如仅几行代码的链表操作），避免睡眠唤醒的开销。

#### 基础实现：Test-And-Set（TAS）

**Test-And-Set（TAS，测试并设置）** 是硬件提供的原子指令（x86 的 LOCK XCHG / LOCK BTS 等），做了两件事且原子完成：
1. 从内存读取一个值（测试）
2. 向该内存地址写入 1（设置）

```c
// TAS 语义：原子地读取 *addr 的旧值，然后将 *addr 设置为 1
// 返回旧值（0 = 锁空闲，1 = 锁已被占用）
int TestAndSet(int *addr) {
    int old = *addr;
    *addr = 1;
    return old;
}  // 此为伪代码，实际由一条 LOCK 前缀的原子指令完成

// 基于 TAS 的"笨"自旋锁
void spin_lock(int *lock) {
    while (TestAndSet(lock) == 1)
        ;  // 忙等
}
```

#### 问题：总线竞争（Bus Contention）

当 CPU-0 持有锁时，CPU-1 在一个循环中反复调用 TestAndSet。每次 TAS 是一次**带有 LOCK 前缀的写操作**，这会：

1. **锁住系统总线**（或缓存一致性链路），阻止其他 CPU 使用总线
2. 使 CPU-0 持有的 Cache 行（含锁变量）被无效化，迫使 CPU-0 回写
3. **大量总线流量**：多个 CPU 同时自旋，每个循环都触发 LOCK 总线事务

当锁被释放，持有锁的 CPU-0 要写回锁变量同时还要与其他自旋 CPU 的 TAS 竞争总线，导致**锁释放延迟**和严重的性能退化。这个就是 **"TAS 导致总线竞争"**。

#### 优化：Test-and-Test-and-Set

将"读"和"TAS（写）"分离：

```c
void spin_lock(int *lock) {
    while (1) {
        // 第一阶段：纯粹的读（普通 LOAD，不是 LOCK 前缀），测试锁是否空闲
        while (*lock != 0)
            ;  // 锁被占用时只读（Read-Only Spin），不产生总线事务
        // 第二阶段：锁看起来空闲了，再用 TAS 原子地尝试获取
        if (TestAndSet(lock) == 0)
            break;  // 获取成功
        // 若 TAS 失败（多个 CPU 同时看到空闲并竞争），回到第一阶段继续"
    }
}
```

**为什么更快：**
- 第一阶段的循环是纯粹的 **LOAD 操作**（只读）。Cache 行在 S（Shared）状态，多个等待的 CPU 各自在本地 Cache 中只读自旋。**零总线事务**。
- 只有当锁释放（*lock 变为 0，由持有者写回触发其他 Cache 的 Invalidate），所有等待 CPU 从 LOAD 循环中退出，然后才发出 TAS（独占写操作）去竞争。此时只有一条 RFO 总线事务在少数竞争 CPU 之间完成，大幅减少总线流量。
- 在锁释放到被再次获取之间，只有一个 CPU 的 TAS 成功（原子操作），其余 CPU 的 TAS 返回 1 然后回退到 LOAD 循环。

> Linux 内核的自旋锁实现（`arch_spin_lock`）在 x86 上正是基于此思想，但经过更多微体系结构调优（如考虑 PAUSE 指令降低功耗和减少乱序执行开销）。

### RCU（Read-Copy-Update，读-复制-更新）

**RCU** 是 Linux 内核提供的一种**读者完全无锁**的同步机制，专门优化**读多写少**的场景。

#### 核心思路

```
写者更新一个共享数据结构时：
  1. 复制一份副本
  2. 在副本上修改
  3. 原子地切换指针（将全局指针指向新副本）
  4. 等待所有已有的读者都完成（"宽容期"过后）
  5. 回收旧副本的内存
```

读者自始至终看到的是**要么旧版本、要么新版本**的完整一致性数据，不会看到中间修
改状态。读者也不需要获取任何锁。

#### 读者端（Reader Side）

```c
// 读者：无需锁，无任何原子操作
rcu_read_lock();               // 实际上在大多数配置中是空操作（仅禁止抢占）
ptr = rcu_dereference(global_ptr);  // 读取指针（在部分架构上是 READ_ONCE 语义）
if (ptr)
    data = ptr->value;         // 访问受 RCU 保护的数据
rcu_read_unlock();             // 标记读者完成（通常仅重新允许抢占）

// rcu_read_lock 和 rcu_read_unlock 在非抢占内核中几乎无开销。
// 在抢占内核中只是递增/递减一个 per-CPU 计数器。
// 读者不被阻塞、不被等待 —— 真正零开销。
```

#### 写者端（Writer Side）

```c
// 写者：
struct node *newp = kmalloc(sizeof(*newp), GFP_KERNEL);
// 1. 复制或构建新版本的数据
*newp = *oldp;
newp->value = 42;              // 修改
// 2. 原子地切换指针
rcu_assign_pointer(global_ptr, newp);
// 3. 等待所有已存在的读者完成（宽容期）
synchronize_rcu();              // 阻塞直到所有 CPU 至少经历了一次上下文切换
// 4. 安全回收旧数据（此时没有任何读者还在引用 oldp）
kfree(oldp);
```

**`synchronize_rcu()`** 的语义：阻塞直到所有 CPU 都至少经历了一次**静止状态（Quiescent State）**。这保证了在调用 `synchronize_rcu()` 之前已开始的所有 RCU 读侧临界区（即 `rcu_read_lock()` 的调用方）已经全部结束。因此旧数据可以安全释放。

非阻塞版：`call_rcu(&oldp->rcu_head, callback_function)` —— 不阻塞，当宽容期过后自动回调执行回收函数。

#### 宽容期（Grace Period）

```
时间轴：
CPU-0: [R1 开始] ................................ [R1 结束]
CPU-1: ......... [R2 开始] ... [R2 结束] .........
CPU-2: .................................. [R3 开始] ...

写者:  | <-------- 宽容期（Grace Period） --------> |
       指针切换                                    synchronize_rcu 返回

R1、R2：在宽容期开始前开始 → 可能在宽容期内结束 → 需要等待
R3：在宽容期开始后开始 → 看到的是新指针 → 不引用旧数据
宽容期结束时条件：所有 CPU 都通过了静止状态
```

#### 适用场景

| 场景 | 是否适用 |
|------|----------|
| 路由表（读多写少） | 极佳 — Linux 网络栈大量使用 RCU 保护路由表、邻居表 |
| 文件系统目录缓存 | 极佳 — dentry cache |
| RCU 链表 | 极佳 — Linux 内核的各种 RCU 链表替换读写锁 |
| 写多读少 | 不适用 — 写者间的同步和宽容期等待使 RCU 性能恶化 |

> RCU 的设计理念体现了"**读和写的不对称代价**"——读者的开销为零，代价转移到写者身上（副本创建 + 延迟回收）。

## 多处理机调度

### 负载均衡：推迁移 vs 拉迁移

**负载均衡（Load Balancing）** 的目标：将就绪线程尽可能均匀地分布到所有可用的处理器上，避免出现"一个 CPU 忙死、另一个 CPU 闲死"的情况。

#### 推迁移（Push Migration）

**由忙的 CPU 发起**。忙 CPU 上有过多就绪线程，主动将部分线程"推"到较空闲的 CPU 上。

- 触发：周期性负载检查（例如 Linux CFS 的 `rebalance_domains`），发现本 CPU 队列长度远超平均
- 操作：忙 CPU 选择该线程 → 从本队列出队 → 跨 CPU 将其入队目标 CPU 的就绪队列
- 优点：实现相对简单；调度域（Scheduling Domain）分级（如先 SMT 兄弟核 → 同物理核 → 同 NUMA → 跨 NUMA）
- 缺点：忙的 CPU 已经忙了，还要额外花时间去"找空闲 CPU 并迁移线程"——在负载极高的系统中是额外开销

#### 拉迁移（Pull Migration）

**由闲的 CPU 发起**。当 CPU 变为空闲（当前运行的线程阻塞/结束，无事可做），它主动检查其他 CPU 是否繁忙，从忙 CPU "拉"线程过来。

- 触发：CPU 的就绪队列为空，即将进入 Idle 状态
- 操作：空闲 CPU 遍历（找到负载最高的 CPU） → 从该 CPU 队列中"偷"一个线程 → 入本队列
- Linux 内核称此为 **Work Stealing（工作窃取）**，是 CFS 调度器的核心机制
- 优点：轻量级——只在 CPU 变为空闲时才检查，不额外给忙 CPU 加负担
- 缺点：多 CPU 同时变空闲时会发生**相互 steal 竞争**

| 维度 | 推迁移（Push） | 拉迁移（Pull / Work Stealing） |
|------|----------------|-----------------------------|
| 发起者 | 忙 CPU | 空闲 CPU |
| 触发时机 | 周期性检查（定时器触发） | CPU 变空闲时触发 |
| 对忙 CPU 的额外负担 | 有（已经忙了还要做迁移） | 无（空闲 CPU 自己完成） |
| 对空闲 CPU 的负担 | 无 | 有（空闲 CPU 检查其他 CPU） |
| 适用架构 | 对称小规模 SMP | 大规模/动态负载系统 |
| 现代实现 | Linux 的 `load_balance()` 函数同时使用两种（或结合运行队列统计信息） | |

### 处理器亲和性（Processor Affinity）

**处理器亲和性** 指尽量让一个线程在它之前运行过的 CPU 上继续运行的能力。

**为什么要亲和性？** 当线程在 CPU-A 上运行时，CPU-A 的私有 L1/L2 Cache 中充满了该线程的热数据（频繁访问的指令和数据）。如果下次调度把该线程迁移到 CPU-B，它将：
- 遭遇大量 Cache Miss，需从较慢的共享 L3/内存甚至远端 NUMA 节点重新加载
- 导致性能明显下降，直到热数据在 CPU-B 的 Cache 中重新"预热"

**两种亲和性：**

| 类型 | 含义 | 实现 |
|------|------|------|
| **软亲和性（Soft Affinity）** | OS 尽力将线程保持在原 CPU 上运行，但不做硬性保证 | Linux CFS 默认行为：调度器优先考虑同 CPU，仅在负载均衡需要时才迁移 |
| **硬亲和性（Hard Affinity）** | 用户显式将线程绑定（pin）到指定 CPU 上，OS 绝不会将线程迁移到其他 CPU | `sched_setaffinity()` / `taskset` 命令；适合 HPC、实时系统 |

> 亲和性是一把双刃剑：太强则负载可能不均（某些 CPU 过热），太弱则不断缓存失效。现代调度器在**亲和性与负载均衡之间寻找折中**。

### big.LITTLE 大小核架构

**big.LITTLE** 是 ARM 推出的异构多核（HMP, Heterogeneous Multi-Processing）架构，同一 SoC 上集成两种不同类型的 CPU 核心：

#### 核心类型

| 类型 | 微架构示例 | 特点 |
|------|-----------|------|
| **大核（big Core）** | Cortex-A76、Cortex-X2 | 乱序执行、超标量、深流水线；单线程性能极高；功耗极高（数瓦/核） |
| **小核（LITTLE Core）** | Cortex-A55、Cortex-A510 | 顺序执行（或浅乱序）、浅流水线；能耗比极佳（mW 级）；性能有限 |

#### 调度的目标

OS 调度器需要根据负载的实时强度动态决定将每个线程放在大核还是小核：

- **轻负载（后台任务、空闲工作、定时器）** → 小核：节能，性能足够
- **重负载（UI 渲染、游戏、编译、视频编解码）** → 大核：需要高性能
- **突发负载（用户交互）** → 快速从小核迁移到大核以降低响应延迟

#### 调度策略演进

1. **HMP（Heterogeneous Multi-Processing）schedule**（Android 早期）：
   - 简单策略：重负载 → 大核，轻负载 → 小核
   - 基于 CPU 负载阈值做迁移，不够智能

2. **EAS（Energy-Aware Scheduling）**（Android / Linux kernel 主线，5.x+）：
   - 引入**能量模型（Energy Model）**：每个 CPU 频率/性能/功耗表
   - 调度器在选择目标 CPU 时不仅看负载，还评估功耗开销
   - 考虑每个线程的"per-task util"（CPU 利用率估计），避免将轻负载错误地放到大核（浪费功率），或重负载放在小核（导致性能不足、功耗更差）

```
EAS 场景简化示例：
- 线程 A 利用率 50%，放到小核（800MHz，2mW）→ 优
- 线程 A 利用率 50%，放到大核（2000MHz，500mW）→ 劣（浪费功率）
- 线程 B 利用率 800%（需要大算力），放到小核 → 劣（小核全速 200% 才勉强跟上，延迟大）
- 线程 B 放到大核 → 优（大核 800% 只是正常轻载，延时低、功耗可控）
```

> 苹果 Silicon（M 系列）和 Intel（P-core / E-core）也采用类似的大小核架构，调度策略类似但各有优化（Intel Thread Director 等硬件辅助调度）。

### 调度域（Scheduling Domain）

Linux 内核将 CPU 拓扑结构抽象为分级**调度域（Scheduling Domain）**，负载均衡从最小域开始逐步扩大到更大的域：

```
层级（自底向上）：
SMT（超线程兄弟核）→ MC（同一物理核的多核组）→ DIE（同一 Die 的所有核）
→ NUMA 节点 → 跨 NUMA 节点

负载均衡规则：优先在最底层（开销最小）的调度域内平衡。
仅在低层域无法平衡时才上升到更高层域。
```

## 补充概念

### 连续调度（Gang Scheduling）

对于紧密协作的并行程序（如 MPI 任务或多线程屏障同步程序），若同一组协作线程在不同 CPU 上运行但调度不同步（一个线程被抢占而其他线程还在运行），则整个组都会因等待被抢占的线程而空耗 CPU（**阻塞式同步损失**，Blocking Synchronization Loss）。

**连续调度** 将一组协作线程同步调度：要么一组全部同时运行，要么都等待。时间片切换时整组一起换出。这尤其在科学计算的细粒度并行（频繁 barrier synchronize）中重要。

### MCS 锁（提及）

MCS（Mellor-Crummey-Scott）锁是一种**基于队列的自旋锁**，每个等待 CPU 在本地 Cache 行上自旋（而非在同一个锁变量上），消除了锁释放时的"惊群效应"和总线竞争。Linux 内核的 `qspinlock`（Queued Spinlock）就基于此思想实现。这对于大规模（>16 核）SMP 系统增加锁的可扩展性至关重要。
