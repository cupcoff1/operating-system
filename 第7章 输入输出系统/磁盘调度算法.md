# 磁盘调度算法

> **核心问题**：多个磁盘 I/O 请求同时到达，按什么顺序服务，让**寻道时间**最短？

## 知识框架

```
磁盘调度算法
├── 无优化
│   └── FCFS（先来先服务）
├── 贪心策略
│   └── SSTF（最短寻道优先）
├── 电梯类（单向扫描，折返）
│   ├── SCAN（走到底再折返）
│   ├── C-SCAN（单向到底，跳回0再单向）
│   └── LOOK / C-LOOK（到最后一个请求即折返）
└── 现代调度器
    ├── CFQ（完全公平队列）→ HDD
    └── NOOP → SSD
```

## 名词解释

| 术语 | 含义 |
|------|------|
| **寻道时间 (Seek)** | 磁头从当前柱面移动到目标柱面的时间，磁盘访问的**绝对主导**（2~20ms） |
| **旋转延迟** | 磁头到达目标磁道后，等待目标扇区转到磁头下（平均半圈） |
| **传输时间** | 实际读写数据的时间，占比最小 |
| **柱面 (Cylinder)** | 所有盘面上同一半径的磁道集合，切换柱面需移动磁头臂 |
| **饥饿** | 某些请求因为位置偏僻，长时间得不到服务 |

---

## 共用例题

> 磁头当前在 **53** 号柱面，请求队列：**98, 183, 37, 122, 14, 124, 65, 67**（数字 = 柱面号）

---

## 1. FCFS — 先来先服务

**策略**：按请求到达的顺序依次服务，不做任何优化。

### 逐请求追踪

```
初始位置 53
53 → 98  移动 45
98 → 183 移动 85
183 → 37 移动 146
37 → 122 移动 85
122 → 14 移动 108
14 → 124 移动 110
124 → 65 移动 59
65 → 67  移动 2
```

**总寻道距离**：640 柱面（最差）

> FCFS 仅作为基线参考。磁头反复跳远距离，平均寻道最长。**不做调度优化时就是 FCFS**。

---

## 2. SSTF — 最短寻道优先

**策略**：每次选择距离当前磁头**最近**的请求。贪心算法。

### 逐请求追踪

```
当前 53 → 距离最近的是：65（差12）
    65 → 最近：67（差2）
    67 → 最近：37（差30）还是 98（差31）？→ 37
    37 → 最近：14（差23）
    14 → 最近：98（差84）
    98 → 最近：122（差24）
    122 → 最近：124（差2）
    124 → 最近：183（差59）
    183 → 完成
```

### 服务序列

```
53 → 65 → 67 → 37 → 14 → 98 → 122 → 124 → 183
```

**总寻道距离**：(12+2+30+23+84+24+2+59) = **236**

| 维度 | FCFS | SSTF |
|------|------|------|
| 总寻道 | 640 | 236（-63%） |
| 饥饿 | 无 | **有**——远距离请求可能永远排不上 |

> **饥饿场景**：磁头在低柱面区域，远处的新请求不断有更近的请求插队，永远不服务远处的。这是 SSTF 的致命缺陷。

---

## 3. SCAN — 电梯算法

**策略**：磁头朝一个方向移动到底，沿途服务所有请求；到达最远端后折返，反方向同样处理。像电梯一样。

### 假设初始方向：向 0（减小）

```
当前 53，向 0 移动：
路径：53 → 37 → 14 → 0（假设到最远端 0）→ 折返
0 → 65 → 67 → 98 → 122 → 124 → 183 → 199（假设最大 199）
```

### 服务序列

```
53 → 37 → 14 → 0 → 65 → 67 → 98 → 122 → 124 → 183
```

**总寻道距离**：53 + 183 = **236**（加上到 0 端点的距离）

### 假设初始方向：向 199（增大）

```
53 → 65 → 67 → 98 → 122 → 124 → 183 → 199 → 折返
199 → 37 → 14
```

> 两种方向结果不同，但都没有饥饿——因为一定能等到磁头"扫过来"。

---

## 4. C-SCAN — 循环扫描

**策略**：SCAN 的变体。磁头**只单向服务**，到达最远端后**直接跳回**起点（0 柱面），过程中不服务任何请求。像单行街道扫地车。

**名词**：SCAN 的"SCAN"指扫描，"C-" 指 Circular（环形）。

```
当前 53，向 199 扫描：
53 → 65 → 67 → 98 → 122 → 124 → 183 → 199
跳到 0（瞬间跳转，不服务）
0 → 14 → 37
```

**总寻道距离**：(199-53) + (199-0) + (37-0) = 146 + 199 + 37 = **382**（加上跳回距离）

> **特点**：各位置请求的等待时间更均匀。边缘柱面不会像 SCAN 那样被"扫两次"。

### 为什么比 SCAN 均匀？

SCAN 中，磁头刚从某位置扫过后立刻折返，那个位置刚被服务——但磁头折返后回来，下一轮才再服务它，间隔可能很长。C-SCAN 单方向，服务间隔均匀。

---

## 5. LOOK / C-LOOK

**策略**：SCAN/C-SCAN 的实用优化版。**不到物理末端**，只走到当前方向上的**最后一个请求**即折返。

- **SCAN**：必须走到磁盘最末端（0 或 199）
- **LOOK**：走到请求队列在此方向上的最远端就折返（"看看"前面还有没有请求）

```
C-LOOK，当前 53，向大方向：

53 → 65 → 67 → 98 → 122 → 124 → 183（当前方向最后一个请求）
跳到 14（跳到最小请求，而非 0）
14 → 37（完成）
```

**总寻道距离**：(183-53) + (183-14) + (37-14) = 130 + 169 + 23 = **322**

---

## 6. FSCAN（冻结扫描）

**策略**：将请求分为**当前活跃组**和**新到达组**。磁头只服务当前活跃组，新到达的请求放入另一组，等活跃组全部完成后，两组交换，新的一组成为活跃组。

### 动机：防止磁头粘滞

在 SCAN/C-SCAN 中，磁头扫描路径上不断有同方向的新请求到达，磁头在该方向"黏住"，迟迟到不了远端。

```
SCAN 的粘滞问题：
磁头向 199 方向扫描中，在 100 附近服务完毕
→ 同时 105、110、108 到达（同方向）
→ 磁头继续服务 105、110、108
→ 又同时有 106、109 到达
→ 磁头被"黏"在 100 区域，远端 183 等不到
```

FSCAN 通过**冻结新请求**打破粘滞——磁头承诺完成当前组就调度新组。

> 实现复杂度中等，实际系统中不常用（N-Step-SCAN 和 Deadline 是更实用的替代）。

---

## 7. N-Step-SCAN

**策略**：类似 FSCAN，但将请求队列分为 **N 个子队列**。磁头一次只服务一个子队列，完成后处理下一个。

- N = 1 时退化为 FSCAN（只有两个队列轮换）
- N 较大时，每个子队列的请求量小，每次扫描更短，粘滞更弱
- 实际选择 N 与设备并发负载相关

### FSCAN vs N-Step-SCAN vs SCAN

| | SCAN | FSCAN | N-Step-SCAN |
|------|------|------|------|
| 新请求处理 | 同方向立即服务 | 冻结到下一组 | 放入其他 N-1 子队列 |
| 粘滞风险 | **有** | 无 | 很小 |
| 实现复杂度 | 低 | 中 | 中 |
| 实际使用 | 理解原理的基础 | 理论改进 | 理论改进 |

> 这两类算法是对电梯类扫描算法"磁头粘滞"问题的改进。在现代 Linux 中，通过 **Deadline 调度器**（读写请求各有截止时间）以更精细的方式解决类似问题。

---

| 算法 | 总寻道 | 策略 | 饥饿 | 实际可用 |
|------|--------|------|------|----------|
| **FCFS** | 640 | 按队列顺序 | 无 | 基线 |
| **SSTF** | 236 | 贪心最近 | **有** | 差 |
| **SCAN** | ~236+ | 电梯扫描 | 无 | 可 |
| **C-SCAN** | ~382 | 单向扫描 | 无 | 较均匀 |
| **C-LOOK** | ~322 | 到末尾请求即返 | 无 | **最优实用** |

> C-LOOK 是实际中最佳的经典算法：无饥饿、较均匀、不做无用的全行程扫描。

---

## 现代调度器

| 调度器 | 策略 | 适用 |
|--------|------|------|
| **CFQ（完全公平队列）** | 每个进程分配公平 I/O 带宽 | HDD（需寻道优化） |
| **Deadline** | 读/写请求各自截止时间，保证延迟上限 | 数据库类 |
| **NOOP** | 只合并相邻请求，无寻道优化 | **SSD**（无机械寻道） |
| **BFQ** | 类似 CFQ 的更精细公平调度 | 桌面环境 |

> **SSD 为什么用 NOOP**？SSD 没有机械磁头，随机访问和顺序访问一样快。寻道排序没有意义，越少干预越好。
