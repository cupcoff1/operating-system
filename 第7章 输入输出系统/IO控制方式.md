# I/O 控制方式

## 一、知识框架树

```
I/O 控制方式（按进化顺序）
├── 1. 程序直接控制（轮询）
│   └── CPU 全程等待，串行，效率极低
├── 2. 中断驱动 I/O
│   └── CPU 与设备部分并行，每字节/每字一次中断
├── 3. DMA（直接存储器访问）
│   ├── DMA 控制器内部结构
│   ├── 传输步骤
│   ├── 周期窃取 vs 成组传送
│   └── CPU 仅参与启动与结束
└── 4. 通道控制（大型机专用）
    ├── 字节多路通道
    ├── 数组选择通道
    ├── 数组多路通道
    ├── 通道程序 = 若干 CCW
    └── 三级控制：通道 → 控制器 → 设备
```

---

## 二、四种方式总览

| 方式 | 传送单位 | CPU 干预频率 | 数据通路 | CPU 与 I/O | 适用设备 | 典型场景 |
|------|----------|-------------|----------|------------|----------|----------|
| **程序直接控制** | 字 | 每字节/每字轮询 | CPU ↔ 设备 | 完全串行 | 极低速设备 | 简单嵌入式、传感器 |
| **中断驱动** | 字 | 每字节/每字中断 | CPU ↔ 设备 | 部分并行 | 低速字符设备 | 键盘、鼠标、串口 |
| **DMA** | 块 | 启动+完成各一次 | 主存 ↔ 设备（绕过CPU）| 高度并行 | 块设备 | 磁盘、SSD、网卡 |
| **通道控制** | 多块 | 极少（报告结果） | 通道程序控制 | 完全并行 | 多类设备 | 大型机、银行核心系统 |

---

## 三、程序直接控制（轮询）

**工作方式**：CPU 反复读取 I/O 控制器的状态寄存器，检查"就绪位"。设备未就绪时，CPU 持续空转等待。

```
CPU 执行伪代码:
  do {
    status = read_status_reg();   // 读状态寄存器
  } while (status.READY == 0);    // 忙则继续轮询
  write_data_reg(data);           // 设备就绪，写入数据
  // ... 继续下一个字节
```

**特征**：
- CPU 与 I/O **严格串行**。CPU 在轮询期间不能做任何其他工作。
- CPU 利用率取决于设备速度——设备越慢，CPU 浪费的时钟周期越多。
- 不需要额外硬件支持（无 DMA 控制器、无中断控制器），实现最简单。
- 现代 OS 几乎不用，仅在 Bootloader、嵌入式 MCU 裸机程序中存在。

---

## 四、中断驱动 I/O

**工作方式**：CPU 发出 I/O 命令后**立即返回做其他工作**。设备完成当前字节传输后，通过中断控制器向 CPU 发中断信号。CPU 响应中断，执行 ISR 传输下一个字节，然后再次返回。

```
流程:
  CPU 发 Read 命令 ──→ 设备开始读取
  CPU 切换到其他进程（调度）

  ... 设备读取完成 ...
  
  设备 → 中断控制器 → CPU: "数据就绪!"
  CPU 保存现场，进入 ISR:
    1. 从设备数据寄存器读 1 字节 → 主存
    2. 若还有数据待传,再次发命令 → 设备继续
    3. ISR 结束,恢复现场,返回原程序
```

**特征**：
- CPU 与 I/O 是 **部分并行** 的。CPU 发命令后可以计算，但每字仍需介入一次中断处理。
- 中断开销不可忽略：每次中断 ≈ 保存/恢复几十条指令的上下文（PSW、PC、寄存器）。
- 对高速设备（如磁盘），中断频率过高 → CPU 大量时间花在中断处理上而非实际计算。

**中断频率造成的 CPU 负担**：

假设磁盘数据率 1 MB/s，字大小 = 4 字节，则中断频率 = 250,000 次/秒。
每次中断处理耗时 10μs（保守估计，含上下文切换），则 CPU 花在 I/O 上的比例为：
`250,000 × 10μs / 1s = 25%`。

这只是 1 MB/s 的低速磁盘！对于 100 MB/s 的现代 SSD，同样的中断方案会吃掉 2500% 的 CPU —— 完全不可行。这就是 DMA 的必要性。

## 五、DMA（直接存储器访问）—— 深入详解

### 5.1 DMA 控制器的内部结构

DMA 控制器是一块独立的硬件芯片（或集成在芯片组中），内含以下寄存器：

```
DMA 控制器内部:
┌──────────────────────────────────────────┐
│ 地址寄存器 (Address Register / MAR)      │  ← 32/64 位：存放下次传输的主存目标地址
│ 计数寄存器 (Count Register)              │  ← 16/24 位：存放剩余待传输字节数
│ 控制寄存器 (Control Register)            │  ← 方向（读/写）、传输模式、中断使能
│ 状态寄存器 (Status Register)             │  ← 忙/空闲、错误标志、传输完成
│ 数据寄存器 (Data Register)              │  ← 暂存一个传输单元（通常 1-4 字节）
└──────────────────────────────────────────┘
```

| 寄存器 | 功能 | 大小（典型） |
|--------|------|-------------|
| **地址寄存器 (AR)** | 指向下一次传输在内存中的目标地址。每传一个单元自动递增（或递减）。| 32 位（可寻址 4GB）|
| **计数寄存器 (CR)** | 存储剩余字节数（或字数）。每传一个单元自动递减，减到 0 时表示传输完成。| 16 位（一次 DMA 最多 64KB）|
| **控制寄存器** | 配置方向（内存→设备 或 设备→内存）、传输模式（单字节/块/按需）、是否允许中断。| 8-16 位 |
| **状态寄存器** | 指示当前状态：忙 (Busy)、传输完成 (TC)、错误 (Error)。| 8 位 |

现代 DMA 控制器可能包含多个"通道"（Channel），每个通道有独立的一组 AR/CR，可以同时服务多个 I/O 设备。比如 Intel 8237A 有 4 个独立 DMA 通道。

### 5.2 DMA 传输的完整步骤

**第 0 步：CPU 准备 DMA 传输**

```
CPU 执行（通常在磁盘驱动程序中）:
  // 设置 DMA 控制器
  outb(DMA_ADDR_REG,  buffer_physical_addr);  // 步骤 0a: 写入主存目标地址
  outb(DMA_COUNT_REG, block_size_bytes);      // 步骤 0b: 写入传输字节数
  outb(DMA_CTRL_REG,  READ_FROM_DISK | INTERRUPT_ENABLE); // 步骤 0c: 配置方向和中断
  // 设置磁盘控制器
  outb(DISK_CMD_REG,  READ);                   // 步骤 0d: 发读盘命令
  outb(DISK_SECTOR_REG, target_sector);        // 步骤 0e: 指定扇区号
  // 此后 CPU 可以返回做其他事情
```

**第 1 步：设备就绪，DMA 申请总线**

```
磁盘控制器读取扇区数据到内部缓冲区后:
  磁盘 → DMA 控制器: DREQ (DMA Request) 信号拉高
    含义："我有数据要传，请求 DMA 服务"
```

**第 2 步：DMA 控制器向 CPU 申请总线控制权**

```
DMA 控制器 → CPU: HRQ (Hold Request) 信号拉高
  含义："请释放系统总线，我来控制"
  
CPU 完成当前总线周期后:
  CPU → DMA 控制器: HLDA (Hold Acknowledge) 信号拉高
  含义："总线已释放，你拿去用"
  CPU 同时将其地址/数据总线引脚置为高阻态（浮空），相当于拔掉 CPU 的总线插头。
```

**第 3 步：DMA 执行数据传输**

```
DMA 控制器:
  // 一个 DMA 传输周期
  MEMR (内存读) 或 IOR (I/O读) 使能 → 数据出现在总线上
  IOW (I/O写)   或 MEMW (内存写) 使能 → 数据写入目标
  AR ← AR + 1   (地址递增，指向下一个内存位置)
  CR ← CR - 1   (计数递减)
```

**关键选择：周期窃取 vs 成组传送**

- **周期窃取 (Cycle Stealing)**：DMA 每传一个字节就把总线还给 CPU。CPU 只在 DMA 传输的那个周期被暂停一个总线周期。对 CPU 的影响极小（几乎感觉不到）。适用于大多数外设。
- **成组传送 (Burst Mode)**：DMA 一次拿走总线，连续传完整块数据再释放。传输效率最高，但 CPU 在此期间完全无法访问内存（程序暂停执行）。适用于要求高吞吐的场景（如视频采集）。
- **透明传送 (Transparent Mode)**：DMA 只在 CPU 不使用总线时（比如 CPU 正在执行内部运算，不访问内存）才传输。对 CPU 完全透明，但实现复杂，需要硬件能精确判断 CPU 的总线使用时机。

**第 4 步：传输完成，DMA 释放总线**

```
当 CR = 0（计数归零）:
  DMA 控制器:
    HRQ 拉低（释放总线给 CPU）
    产生中断信号 → 中断控制器 → CPU
  CPU 响应中断:
    进入 ISR，检查 DMA 状态寄存器
    通知磁盘驱动程序：传输完成，缓冲区数据可用
    唤醒等待该数据的进程
```

**DMA 传输全程时序图**：

```
时间 →

CPU:    [设置DMA/磁盘] [做其他工作................................] [ISR处理完成]
               │                                              ↑
               ▼                                              │
DREQ:          └───┐         ┌───┐         ┌───┐    最后     │
                    │         │   │         │   │    DREQ     │
HRQ:                ├─────────┤   ├─────────┤   ├─────────────┤
                    │         │   │         │   │             │
HLDA:               ├─────────┤   ├─────────┤   ├─────────────┤
                    │         │   │         │   │             │
数据总线:           │传1字节  │  │传1字节  │  │传1字节  │     │
                    │         │   │         │   │             │
计数器:   N    →    │N-1      │...│   0     │  │             │
                                                 │
DMA中断:  ───────────────────────────────────────┘
```

### 5.3 CPU 利用率数值对比 —— 磁盘读取场景

**场景设定**：
- 从磁盘读取 4 KB 数据到内存
- 磁盘控制器读扇区本身需要时间（机械延迟 + 传输）：平均 10 ms
- 指令执行速度：1 GHz CPU（每条指令 ≈ 1 ns）
- 用于 I/O 处理的指令数估算（基于典型操作次数）

**计算方法**：

```
CPU 利用率 = (CPU 花在 I/O 上的时间) / (总 I/O 时间 + 可并发计算的时间)

具体来说:
  利用率(最大)  = CPU 做 I/O 工作的时间 / 总 I/O 操作耗时
  利用率(实际)  = 1 - (CPU 可用于其他工作的时间比)
  
这里我们计算: CPU 在做 I/O 操作期间"被绑定"不能做其他事的时间百分比
```

| 方式 | 每次 I/O CPU 工作 | I/O 过程中 CPU 干预次数 | 单次干预时间 | CPU 总耗时 | 总 I/O 耗时 | CPU 利用率 (被I/O绑定) |
|------|-------------------|------------------------|-------------|-----------|------------|----------------------|
| **轮询** | 设置命令 + 轮询等待 (10ms 全程等待) | 整个传输期间持续轮询 | 10,000,000 μs | **10,000,000 μs** | 10 ms | ≈ 100%（串行） |
| **中断驱动** | 设置命令 + N 次 ISR 处理 | 每传输 1 字中断 1 次（4KB / 4B = 1024 次）| 每次 ISR: ~5 μs | 5 + 1024×5 = **5,125 μs** | 10 ms | ≈ 51.25% |
| **DMA** | 设置 DMA + 磁盘命令 + 完成 ISR | 2 次（启动 + 完成） | 启动: 5μs + ISR: 5μs | **10 μs** | 10 ms | ≈ 0.1% |

**更直观的表述**：

| 方式 | CPU 用于 I/O 的时间 | 等效独立计算指令数 | 等效 CPU 占用比 |
|------|-------------------|-------------------|---------------|
| 轮询 | 10,000,000 μs | 10,000,000,000 条 | **100%** (整个 I/O 期间无法做任何事) |
| 中断驱动 | 5,125 μs | 5,125,000 条 | **~51%** (一半时间在处理中断) |
| DMA | 10 μs | 10,000 条 | **~0.1%** (几乎完全自由) |

**结论**：DMA 将 CPU 从 I/O 数据传输中彻底解放出来，使 CPU 利用率提升 500-1000 倍（相对于中断驱动方式）。对于一个 I/O 密集的服务器（如数据库服务器），DMA 是使并发处理成为可能的前提。

## 六、四种 I/O 控制方式深度对比

### 6.1 CPU 开销与吞吐量对比

| 维度 | 程序直接控制 (轮询) | 中断驱动 | DMA | 通道控制 |
|------|-------------------|----------|-----|----------|
| **CPU 每字节开销** | 极高（每条指令轮询） | 高（每次传输都进 ISR） | **极低**（仅启动/结束） | **最低**（几乎不干预） |
| **CPU 利用率** | 接近 0%（全部浪费在等待） | 30%-70%（设备越快越低） | 95%+ | 99%+ |
| **最大数据率** | 受限于 CPU 的 IN/OUT 指令速度 | 受限于中断处理速度 | 受限于总线带宽 | 受限于通道带宽 |
| **并发 I/O 支持** | 无 | 有限 | 中等（每 DMA 通道一台） | 强大（每通道可多台设备） |
| **适合设备** | 简单传感器、状态灯 | 键盘、鼠标（低速字符） | 磁盘、SSD、网卡（高速块） | 大型机磁盘阵列、磁带库 |
| **硬件复杂度** | 极简 | IC (中断控制器) | IC + DMAC | IC + IO 通道处理器 |
| **软件复杂度** | 极简 | 中等（ISR 编程） | 中等 | 高（通道程序编写） |
| **适用系统规模** | 嵌入式 MCU | 微型机、工控 | PC、服务器 | 大型机、企业核心 |

### 6.2 数据率边界

不同控制方式有各自的数据率上限（32 位系统，1 GHz CPU，每次 ISR 开销 5μs 估算）：

| 方式 | 理论上限 | 实际上限 | 瓶颈 |
|------|---------|---------|------|
| 程序直接控制 | 取决于 CPU 循环速度 | ~50 KB/s | CPU 在轮询，不能干别的 |
| 中断驱动 | 200,000 次中断/s × 4B ≈ **800 KB/s** | ~500 KB/s | 中断上下文切换开销不可压缩 |
| DMA | 总线带宽决定 ≈ 133 MB/s (PCI) | ~100 MB/s | 总线竞争、内存带宽 |
| 通道控制 | 通道级别带宽 | ~数百 MB/s | 通道本身 I/O 处理器速度 |

这个边界解释了一个反直觉的事实：**为什么 OS 不直接用中断驱动来读磁盘？**

答：因为中断驱动面对 MB/s 级别的磁盘时，CPU 被频繁中断压垮（每秒数十万次中断，每次 5-10μs），导致 CPU 几乎 100% 在服务中断，没有时间运行用户程序。DMA 通过硬件解放这条通路，使 CPU 只需要设置一次参数就撒手不管。

## 七、通道控制（大型机专用）

### 7.1 通道的本质

通道 = **一台专门执行 I/O 指令的 CPU**。它有自己的指令系统（通道命令字 CCW）、可以执行存放在主存中的通道程序、可以与 CPU 并行工作。

通道和 CPU 的关系：CPU 是"主 CPU"，通道是"从处理器"——但它们是并行执行的独立实体。

### 7.2 三级控制层次

```
        ┌─────────────┐
        │     CPU      │  ← 发出 I/O 指令，指定通道号和设备号
        └──────┬──────┘
               │ "帮我读设备 D 的 N 个扇区到地址 A"
               ▼
        ┌─────────────┐
        │    通道       │  ← 独立 I/O 处理器，执行通道程序
        │ (Channel)    │     从 CAW 取出第一条通道指令
        └──────┬──────┘
               │ 发出 CCW 指令
               ▼
        ┌─────────────┐
        │   控制器     │  ← 将通道命令翻译为设备电信号
        │ (Controller) │     控制器的 SDT/DCT 管理设备状态
        └──────┬──────┘
               │ 硬件信号（寻道、读扇区等）
               ▼
        ┌─────────────┐
        │   设备       │  ← 执行物理操作
        │  (Device)    │
        └─────────────┘
```

### 形象类比

```
CPU  ──→  通道  ──→  控制器  ──→  设备
老板      项目经理      工头          工人
```

| 角色 | 做什么 | 举例 |
|------|--------|------|
| **CPU（老板）** | 发号施令："把这份数据读进来"，发完就去干别的 | 执行进程，只发 I/O 指令，不盯执行 |
| **通道（项目经理）** | 领了任务自己拆解执行，干完汇报 | 独立执行通道程序，管理多个控制器 |
| **控制器（工头）** | 把上层指令翻译成设备能懂的电信号 | "磁头移到 50 号柱面"翻译成电机控制信号 |
| **设备（工人）** | 真正干活的 | 磁头移动、盘片旋转、读/写 |

### 7.3 相关寄存器

| 寄存器 | 全称 | 作用 |
|--------|------|------|
| **CAW** (Channel Address Word) | 通道地址字 | 存放通道程序在主存中的起始地址。CPU 启动 I/O 前设置。 |
| **CSW** (Channel Status Word) | 通道状态字 | 通道程序执行完毕后的状态报告（正常完成/异常类型/残余计数）。CPU 读 CSW 判断结果。 |
| **CCW** (Channel Command Word) | 通道命令字 | 单条通道指令，8 字节。包含：操作码、数据内存地址、传输字节数、标志位（链式/跳跃/SLI 等）。 |

### 7.4 工作流程

1. CPU 准备通道程序写入主存。
2. CPU 将通道程序首地址写入 CAW。
3. CPU 执行 `START I/O` 指令，指定通道号和设备号。
4. 通道从 CAW 所指的地址取第一条 CCW，执行。
5. 通道执行完一条 CCW 后，若该 CCW 设置了"链式标志"（Command Chaining），自动取下一条 CCW 继续执行。
6. 通道程序执行完毕（遇到无链式标志的 CCW 且传输完成），通道产生中断。
7. 通道将执行结果状态写入 CSW。
8. CPU 收到中断，读取 CSW 检查执行结果。

### 7.5 通道类型

| 类型 | 同时服务设备数 | 交叉方式 | 适用设备 | 核心特征 |
|------|-------------|----------|----------|----------|
| **字节多路** (Byte Multiplexer) | 多台低速 | 按**字节**交叉传送，各设备轮流占用一个字节时间片 | 打印机、读卡机、终端 | 微观上串行（单字节交叉），宏观上并行（多设备同时进行） |
| **数组选择** (Block Selector) | 一台高速设备 | **独占通道**，传完整个数据块再释放 | 早期磁盘、磁带 | 串行，效率低——通道在设备寻址期间空闲等待 |
| **数组多路** (Block Multiplexer) | 多台高速设备 | 按**数据块**交叉传送，各设备分时使用通道 | 现代磁盘阵列 | 最优方案——磁盘 A 寻道时通道服务磁盘 B 的数据传输 |

**三种通道的时间轴对比**：

```
字节多路:
设备A: [A1][A2][A3] ...
设备B:      [B1][B2] ...
通道:  [A1][B1][A2][B2][A3]...  ← 字节级交叉

数组选择:
设备A:  [══════ 整块数据 ══════]
设备B:                             [══════ 整块数据 ══════]
通道:   只有A                    只有B  ← 独占

数组多路:
设备A:  [寻道……………][══ 数据块 ═══]
设备B:       [══ 数据块 ═══][寻道……………]
通道:    服务B    服务A    服务A         ← 分时复用
```

### 7.6 通道 vs DMA 对比

| | DMA | 通道 |
|------|-----|------|
| **本质** | 专用数据传输硬件电路 | 独立 I/O 处理器（有自己的指令系统） |
| **指令能力** | 无指令，由硬件状态机控制 | 有完整的通道指令系统（CCW），可编程 |
| **控制粒度** | 一次传输一块数据 | 可执行复杂的 I/O 程序（条件判断、链式控制） |
| **并发设备数** | 一个或少量 | 数十台至上百台 |
| **CPU 干预** | 两次（启动 + 中断） | 一次（发出 START I/O） |
| **成本** | 低（芯片级） | 高（相当于一个小型 CPU） |
| **适用场景** | 微型机、小型机、PC | 大型机、企业核心系统 |

## 八、总结 —— I/O 控制方式的进化逻辑

```
程序直接控制 (轮询)
  │  问题: CPU 全程等待，利用率 ≈ 0%
  ▼
中断驱动 I/O
  │  改进: CPU 与 I/O 并行
  │  问题: 每字节一次中断，高速设备中断洪水
  ▼
DMA
  │  改进: 硬件搬运数据，CPU 仅启动/结束参与
  │  问题: 只能传数据，不能做复杂 I/O 管理
  ▼
通道控制
    改进: 独立 I/O 处理器，可编程控制
    特点: 仅在大型机中出现，PC/服务器用 DMA + 智能外设 (如 NVMe 的 SQ/CQ 模型)
```

> 现代 PC 大多运行在 DMA 层级，但高端 NVMe SSD 使用的"提交队列/完成队列"模型（SQ/CQ）在概念上已接近通道的思想——设备自己从队列取命令并执行，CPU 只需往队列写命令即可。
