## 二、I/O 系统层次结构

```
┌─────────────────────────────┐
│     用户层 I/O               │  ← 应用程序：printf(), scanf(), read(), write()
│  (User-level I/O Software)  │     用户看到的抽象（逻辑设备名、缓冲）
├─────────────────────────────┤
│  设备无关软件层              │  ← 核心转换层：命名、保护、阻塞/非阻塞、缓冲、错误处理
│  (Device-Independent SW)    │     管理 LUT，分配设备，统一缓冲策略
├─────────────────────────────┤
│     设备驱动程序             │  ← 厂商编写：翻译 OS 通用命令为设备特定指令
│  (Device Drivers)           │     每种设备一个驱动（.sys / .ko）
├─────────────────────────────┤
│     中断处理程序             │  ← 处理设备中断，唤醒等待进程
│  (Interrupt Handlers)       │     最小化 ISR 执行时间（上半部），延迟处理交给下半部
├─────────────────────────────┤
│     硬件                     │  ← 设备控制器 + 机电设备本身
│  (Hardware)                 │     控制器 = IO 芯片、DMA 控制器、中断控制器
└─────────────────────────────┘
```

各层职责：
- **用户层**：提供 C 库封装（`printf` → `write` 系统调用），逻辑设备名的使用。
- **设备无关层**：统一所有设备的接口、权限检查（"你是否有权写这个文件？"）、缓冲策略（内核缓冲区何时刷新？）。这是 OS 内核中代码量最大的 I/O 模块。
- **驱动程序层**：将通用读写请求翻译为设备寄存器操作序列。每个设备一个驱动，是 OS 中最容易出 Bug 的模块（运行在内核态且有直接硬件访问权）。
- **中断处理层**：响应设备完成信号，调度下半部处理（bottom half），唤醒驱动程序的等待队列。
- **硬件层**：设备控制器接收驱动程序的寄存器写操作，转换为电信号驱动设备动作。

## 三、引入缓冲的目的

| 目的 | 说明 |
|------|------|
| **缓解速度不匹配** | CPU 可执行 10^9 条指令/秒，磁盘约 10^2 次 I/O/秒（机械盘），相差 7 个数量级。缓冲充当速度适配器。 |
| **减少中断频率** | 没有缓冲时，每字节一次中断（见 I/O 控制方式章节分析）。有缓冲时，一个缓冲块填满才中断一次，中断频率降低 N 倍（N = 缓冲块大小），极大减少 CPU 上下文切换开销。 |
| **提高并行度** | 设备在填充满冲区 A 时，CPU 可以同时处理缓冲区 B 中的数据。双缓冲使得"设备传输"和"CPU 计算"这两个阶段完全重叠。 |
| **解决数据粒度不匹配** | 设备以字符/块传输，用户进程以逻辑记录处理。缓冲层做数据组装/拆分。 |

## 四、缓冲技术详解

### 4.1 单缓冲 (Single Buffering)

**结构**：系统在用户进程的内存空间中分配一个缓冲区（或在系统内核空间分配一个缓冲区）。

**工作方式**：

```
时间轴 →

设备:   [填BUF]                   [填BUF]                  [填BUF]
CPU :            [处理BUF]                 [处理BUF]                [处理BUF]
              ↑── M ──↑              ↑── M ──↑              ↑── M ──↑
                用户进程读取/复制
                数据到工作区
```

**核心约束**：设备和 CPU 不能同时使用同一缓冲区。设备将数据填入缓冲区时，CPU 必须等待；CPU 处理缓冲区数据时，设备不能写入。

**耗时公式**：

$$
T_{\text{单缓冲}} = \max(C, T) + M
$$

其中：
- **T**：设备将数据填入缓冲区所需时间。
- **C**：CPU 处理一个缓冲区块数据所需时间。
- **M**：将数据从系统缓冲区复制到用户工作区的时间。

**为什么是 `max(C, T) + M`？**

设 C < T（设备比 CPU 慢，典型场景）：
- 设备填完缓冲区耗时 T，这期间 CPU 空闲等待。
- 设备填完后，CPU 花费 M 时间把数据从缓冲区复制到用户区。
- 设备要等 CPU 释放缓冲区（即 M 结束）才能开始下一轮填充。
- 总计：T + M。此时 `max(C, T) = T`，公式成立。

设 C > T（CPU 比设备慢）：
- 设备很快填完缓冲区（耗时 T），但 CPU 还在处理上一块数据。
- 设备必须等待 CPU 处理完（耗时 C），再等 CPU 把数据搬走（耗时 M）。
- 总计：C + M。此时 `max(C, T) = C`，公式成立。

**数值示例**：

> 设备传输一块数据（4KB）耗时 T = 100 μs
> CPU 处理一块数据耗时 C = 50 μs
> 复制到用户区耗时 M = 10 μs

单缓冲吞吐：`T(块) = max(50, 100) + 10 = 110 μs/块`

等效数据率：4096 B × (1,000,000 / 110) ≈ **37.2 MB/s**

瓶颈：设备与 CPU 无法并行——设备填缓冲区时，CPU 闲置；CPU 复制数据时，设备闲置。两者形成了"互斥"关系。

### 4.2 双缓冲 (Double Buffering)

**结构**：分配两个缓冲区 BUF0 和 BUF1。

**工作方式**：

```
时间轴 →

设备:   [填BUF0][填BUF1][填BUF0][填BUF1]...
CPU :            [处理BUF0][处理BUF1][处理BUF0]...
                     ↑ C=50μs, M 被隐藏
```

**核心改进**：当设备填 BUF1 时，CPU 可以同时处理 BUF0。设备不用等 CPU 完成处理。

**耗时公式**：

$$
T_{\text{双缓冲}} = \max(C, T)
$$

M 被隐藏进 `max(C, T)` 的内部。原因是：设备填 BUF1 时，CPU 既在处理 BUF0 也在复制上一块数据到用户区——这两个动作被设备传输的时间所覆盖。

**数值示例（同上参数）**：

双缓冲吞吐：`T(块) = max(50, 100) = 100 μs/块`

等效数据率：4096 B × (1,000,000 / 100) ≈ **40.96 MB/s**

性能提升：110 μs → 100 μs，约 **9.1%** 提升。

但若 C = 150 μs（CPU 处理慢），则：

单缓冲：`max(150, 100) + 10 = 160 μs/块`
双缓冲：`max(150, 100) = 150 μs/块`
提升：150 μs/块，等效 27.3 MB/s。

双缓冲的真正价值在于：**设备与 CPU 完全并行**，不再互相等待。只要 C 和 T 匹配得当，效果接近理想吞吐。

### 4.3 三种缓冲的耗时对比

| 缓冲方式 | 耗时公式 | 设备-CPU 并行？ | 致命缺陷 |
|----------|---------|----------------|---------|
| **无缓冲** | T + C + M（三者串行） | 否 | 速度不匹配时严重浪费 |
| **单缓冲** | max(C, T) + M | 否（互斥） | M 成为固定开销 |
| **双缓冲** | max(C, T) | **是** | 若 C << T 或 T << C，则快的那一方频繁等待 |

**精确数值对比**（T = 100 μs, C = 50 μs, M = 10 μs）：

| 指标 | 无缓冲 | 单缓冲 | 双缓冲 |
|------|--------|--------|--------|
| 每块耗时 | 160 μs | 110 μs | 100 μs |
| 数据率 (4KB) | 25.6 MB/s | 37.2 MB/s | 40.96 MB/s |
| 相对无缓冲提升 | — | +45.5% | +60% |

### 4.4 循环缓冲 (Circular Buffer)

**结构**：多个大小相等的缓冲区连接成环形链表。两个指针：

```
              ┌───┐
              │ B4│ ←─── nextbuf (指向下一块可用缓冲)
              ├───┤
         ┌──→ │ B3│
         │    ├───┤
nexti ───┼──→ │ B2│ ← 正在被设备填入（输入指针）
         │    ├───┤
         │    │ B1│ ← 已填满，待 CPU 处理
         │    ├───┤
         └─── │ B0│ ← 正在被 CPU 处理（输出指针）
              ├───┤
              │...│
              └───┘
              
nexti  →  指向下一块空缓冲区（供设备填入）
nextg  →  指向下一块满缓冲区（供 CPU 取出处理）
```

**环形条件**：当 `nexti == nextg` 时，要么全部为空（设备无数据可填，暂不需要），要么全部为满（CPU 无空缓冲可用，需等待）。两个指针之间的缓冲区即为"正在使用中"的缓冲。

**应用场景**：生产者-消费者模型。设备是生产者（填缓冲），CPU 是消费者（取数据）。循环缓冲在两者生产/消费速率不一致时自然平衡，而不需要额外的同步等待（只要缓冲区数 > 2 且生产/消费速率波动不极端）。

## 五、缓冲池 (Buffer Pool) —— 操作系统的全局缓冲区管理

### 5.1 为什么需要缓冲池？

前面的单/双/循环缓冲区都是"专用"的——每个进程或每个设备各自管理自己的缓冲区。这有两个问题：
1. 内存浪费：100 个进程可能只有 10 个在做 I/O，但每个进程都预留了缓冲区。
2. 分配不均：有的进程缓冲区不够用，有的进程缓冲区闲置。

**缓冲池 = 操作系统统一管理的全局缓冲区集合**。进程和设备按需申请、用后归还。

### 5.2 三队列结构

缓冲池由三种 FIFO 队列管理缓冲区：

```
缓冲池全局结构:

┌──────────────────────────────────────────────────────┐
│  空缓冲队列 (Free Buffer Queue)                      │
│  空缓冲队首 → B5 → B6 → B7 → B8 → ...  → 空缓冲队尾│
│  这些缓冲区未被任何 I/O 使用，可直接分配              │
├──────────────────────────────────────────────────────┤
│  输入队列 (Input Queue)                              │
│  输入队首 → B1 → B2 → ...  → 输入队尾               │
│  这些缓冲区已装满从设备读入的数据，等待 CPU 取走处理  │
├──────────────────────────────────────────────────────┤
│  输出队列 (Output Queue)                             │
│  输出队首 → B3 → B4 → ...  → 输出队尾               │
│  这些缓冲区已装满进程要写出的数据，等待设备发送        │
└──────────────────────────────────────────────────────┘
```

**缓冲区结构**（每个缓冲区块内部）：

```
┌────────────────────────────┐
│  buf_next    → (队列链指针) │  ← 用于链入三队列之一
│  buf_prev    → (队列链指针) │
├────────────────────────────┤
│  数据区 (data area)         │  ← 实际存储数据的空间（如 4KB）
│  ...                       │
├────────────────────────────┤
│  设备号 (device number)     │  ← 归属哪个设备
│  数据块号 (block number)   │  ← 在设备上的逻辑块号
│  标志 (flags)              │  ← 脏位、忙位、错误位等
└────────────────────────────┘
```

### 5.3 缓冲区管理操作

**数据输入流程**（设备读 → 用户进程）：

```
1. 收容 (Takeup):
   从空缓冲队列取队首指针 → 分配一个空缓冲区 Bfree
   Bfree 链接到某个设备的输入队列队尾
   设备开始向 Bfree 填充数据

2. 提取 (Extract):
   设备数据填满 Bfree 后
   将 Bfree 从输入队列摘下
   交给等待数据的用户进程
   用户进程复制数据到自己的用户区后
   将 Bfree 归还空缓冲队列
```

**数据输出流程**（用户进程 → 设备写）：

```
1. 收容 (Takeup):
   从空缓冲队列取队首 → 分配空缓冲区 Bfree
   进程将数据写入 Bfree
   Bfree 链接到输出队列队尾（等待设备写出）

2. 提取 (Extract):
   设备从输出队列取队首缓冲区 Bdata
   设备将 Bdata 中的数据发送到物理设备
   发送完成后，Bdata 归还空缓冲队列
```

**四种基本操作**：

| 操作 | 源队列 | 目标 | 触发条件 |
|------|--------|------|----------|
| `get_buf(type)` | 空队列 → type 队列 | 从空队列取一块缓冲加入输入/输出队列 | 进程请求 I/O |
| `release_buf(type, buf)` | type 队列 → 空队列 | 用完后归还缓冲区 | 数据传输完毕 |
| `take_buf(type)` | type 队列 → 调用者 | 从输入/出队列取回就绪的缓冲区 | 设备/进程就绪 |
| `add_buf(type, buf)` | 调用者 → type 队列 | 将装满数据的缓冲区加入输入/出队列 | 设备/进程完成填装 |

**同步控制**（生产者-消费者）：

- 当空缓冲队列为空时，请求者（进程或设备）必须在空缓冲队列上**睡眠等待**，直到有缓冲区被释放。
- 当需要使用某类型队列的缓冲区但该队列为空时（比如进程想提取输入数据但输入队列为空），请求者必须在该队列上**睡眠等待**。

这本质上是经典的"有限缓冲区生产者-消费者问题"（Bounded Buffer Problem）在系统层的实现。

## 六、SPOOLing（假脱机 / 同时外围联机操作）

### 6.1 概念与动机

**SPOOLing** = **Simultaneous Peripheral Operations On-Line**（同时外围联机操作）。

**问题**：打印机、读卡机等设备是"独占设备"——一次只能给一个进程用。如果 P1 在打印一份 100 页报告，P2 也要打印一行日志，P2 必须等 P1 打完 100 页。这是不可接受的。

**思路**：把独占设备变成"看起来每个进程都独占的共享设备"。手段是——在磁盘上建立一个中间暂存区，所有进程都往磁盘"打印"，由后台守护进程把磁盘内容真正送往打印机。

```
   P1 -------→ 输出井(磁盘) -------→ 打印机守护进程 ------→ 打印机
   P2 -------→       │
   P3 -------→       │
               所有进程的"打印"在这里汇合
               磁盘充当速度匹配和排队缓冲区
```

这样就实现了：进程 A、B、C 同时"打印"（实际是往磁盘写），打印机守护进程按顺序把它们的内容送往物理打印机——每个进程看起来都"独占"了一台打印机。

### 6.2 系统组成

SPOOLing 系统由三大模块构成：

#### 6.2.1 输入井 / 输出井（磁盘上的暂存区）

```
磁盘上的 SPOOLing 区域划分:

┌──────────────────────────────┐
│      输入井 (Input Well)     │
│  ┌───────────────────────┐   │
│  │ 进程A_输入数据 (文件)  │   │  ← 从输入设备（读卡机/终端）预读的数据
│  │ 进程B_输入数据 (文件)  │   │
│  │ 进程C_输入数据 (文件)  │   │
│  └───────────────────────┘   │
│      (等待进程读取)          │
├──────────────────────────────┤
│      输出井 (Output Well)    │
│  ┌───────────────────────┐   │
│  │ 进程D_打印作业 (文件)  │   │  ← 进程写入的"打印"数据
│  │ 进程E_打印作业 (文件)  │   │
│  │ 进程F_打印作业 (文件)  │   │
│  └───────────────────────┘   │
│      (等待输出设备发送)      │
├──────────────────────────────┤
│  井目录 / 井管理元数据        │
│  记录每个井文件:              │
│    - 所属进程 PID             │
│    - 文件大小 / 剩余大小       │
│    - 状态(就绪/传输中/完成)   │
│    - 对应的设备号             │
└──────────────────────────────┘
```

#### 6.2.2 输入进程 / 输出进程（守护进程 Daemon）

**输入进程 (SPOOLing In-Daemon)**：

```
输入守护进程的生命周期:

  系统启动时创建，进入循环:
    ┌─────────────────────────────────┐
    │  while (系统运行) {              │
    │    1. 检查输入设备是否有数据就绪  │
    │    2. 若有，将数据从输入设备     │
    │       读入输入井（磁盘文件）      │
    │    3. 唤醒等待该输入数据的进程   │
    │    4. 休眠短暂时间（轮询间隔）   │
    │  }                              │
    └─────────────────────────────────┘
```

**输出进程 (SPOOLing Out-Daemon)**：

```
输出守护进程的生命周期:

  系统启动时创建，进入循环:
    ┌─────────────────────────────────┐
    │  while (系统运行) {              │
    │    1. 扫描输出井目录，找到       │
    │       状态为"就绪"的打印作业     │
    │    2. 若输出设备空闲，取一个作业 │
    │    3. 将作业数据从输出井读出     │
    │    4. 逐块发送到物理输出设备    │
    │    5. 作业完成，删除输出井文件   │
    │    6. 若输出井无作业，休眠等待   │
    │  }                              │
    └─────────────────────────────────┘
```

> 注意：输入/输出守护进程运行在**用户态**（非内核态）。它们只是具有特权的用户级后台进程，通过系统调用访问磁盘和 I/O 设备。

#### 6.2.3 井管理程序（API）

井管理程序是 SPOOLing 系统对外提供的接口，运行在内核态：

| API | 功能 |
|-----|------|
| `spool_in_open(dev)` | 进程请求从输入井读取数据。分配输入井文件，启动预读。 |
| `spool_in_read(buf, size)` | 进程从输入井读取已预读好的数据。若数据未就绪，阻塞。 |
| `spool_in_close()` | 释放输入井文件和缓冲。 |
| `spool_out_open(dev)` | 进程请求向输出井写入数据。在输出井创建作业文件。 |
| `spool_out_write(buf, size)` | 进程将数据写入输出井文件。 |
| `spool_out_close()` | 关闭作业文件，将其状态标记为"就绪"——输出守护进程将接手处理。 |

### 6.3 具体场景：打印机共享

**场景**：一台物理打印机，三个进程 P1、P2、P3 同时发出打印请求。

**步骤详解**：

```
时间 →

P1: [spool_out_open("/dev/printer")]  → 在输出井创建文件 /spool/print/job_1
P1: [spool_out_write(buf1)]           → 写入第一页数据到 job_1
P2: [spool_out_open("/dev/printer")]  → 在输出井创建文件 /spool/print/job_2
P1: [spool_out_write(buf2)]           → 写入第二页数据到 job_1
P3: [spool_out_open("/dev/printer")]  → 在输出井创建文件 /spool/print/job_3
P2: [spool_out_write(buf1)]           → 写入数据到 job_2
P1: [spool_out_close()]              → job_1 状态 → "就绪"
P3: [spool_out_write(buf1)]           → 写入数据到 job_3
P3: [spool_out_close()]              → job_3 状态 → "就绪"
P2: [spool_out_close()]              → job_2 状态 → "就绪"

此时输出井状态:
  job_1: [就绪] (P1, 2 页)
  job_2: [就绪] (P2, 1 页)
  job_3: [就绪] (P3, 1 页)

输出守护进程 (运行中):
  打印机空闲 → 取 job_1 → 逐页发送到打印机
  "打印机正在打印 P1 的作业..."

  当打印机空闲 → 取 job_2 → 发送
  "打印机正在打印 P2 的作业..."

  当打印机空闲 → 取 job_3 → 发送
  "打印机正在打印 P3 的作业..."
```

**关键点**：
- P1、P2、P3 **都认为自己独占打印机**——它们各自调用 `spool_out_open` 都成功返回了，没有被阻塞。
- 实际上它们是向磁盘写文件，没有竞争物理打印机。竞争被推迟到了输出守护进程的调度层面。
- 如果打印机是独占设备且无 SPOOLing，P2 在 P1 打印完成前调用 `open("/dev/printer")` 就会**阻塞**（或返回"设备忙"错误）。

### 6.4 SPOOLing 的效果

| 维度 | 无 SPOOLing | 有 SPOOLing |
|------|------------|-------------|
| 进程视角 | 独占设备，一次一个进程 | 虚拟设备，多进程共享 |
| 进程等待 | 进程必须等物理设备释放 | 进程写磁盘即返回（快） |
| 设备利用率 | 低（进程用完才释放） | 高（守护进程持续喂数据给设备） |
| 系统吞吐量 | 低 | 高 |

## 七、设备独立性（设备无关性）

### 7.1 概念

用户程序使用**逻辑设备名**（如 `/dev/printer`、`CON:`、`LPT1`）而非物理设备名（如 `端口 0x378`）。操作系统通过 **LUT（逻辑设备表，Logical Unit Table）** 完成逻辑名到物理设备的映射。

### 7.2 LUT 结构

```
逻辑设备表 (LUT):
┌─────────────┬──────────────┬─────────────────┐
│ 逻辑设备名   │ 物理设备名   │ 驱动程序入口指针 │
├─────────────┼──────────────┼─────────────────┤
│ /dev/tty0   │ 终端 #0     │ → tty_driver    │
│ /dev/sda1   │ SCSI disk 0 │ → sd_driver     │
│ /dev/lp0    │ 并口打印机  │ → lp_driver     │
│ /dev/null   │ 空设备      │ → null_driver   │
│ ...         │ ...         │ ...             │
└─────────────┴──────────────┴─────────────────┘
```

**LUT 的工作流程**：

1. 用户进程调用 `open("/dev/lp0")`
2. OS 在 LUT 中查找 `"/dev/lp0"`
3. 从 LUT 获取：物理设备为"并口打印机"，驱动入口为 `lp_driver`
4. OS 调用 `lp_driver->open()` 完成实际打开操作
5. 若物理设备不可用（如已经被独占分配），LUT 条目中可能标记为"忙"，进程阻塞等待

**设备独立性的真正意义**：
- 程序不依赖具体设备。同类设备可互换（比如从一台打印机换到另一台打印机，程序不用改）。
- 作业控制语言（JCL）中可以用逻辑名，管理员在系统配置层面完成物理映射。
- 如同文件系统的路径名 —— 你写 `"/home/user/file.txt"` 时并不需要知道数据在磁盘的哪个扇区。

## 八、设备分配数据结构

四层控制表，逐级查找，任一环节不可用即阻塞。

```
分配请求:
  ┌─────────┐
  │  SDT    │  ← 系统设备表：记录全系统所有设备
  │  系统级  │     每个设备一个条目（设备标识、DCT 指针）
  └────┬────┘
       │ 找到目标设备条目
       ▼
  ┌─────────┐
  │  DCT    │  ← 设备控制表：每台设备一张表
  │  设备级  │     记录设备状态（忙/闲）、等待队列、COCT 指针
  └────┬────┘
       │ 找到连接的控制器的 COCT
       ▼
  ┌─────────┐
  │  COCT   │  ← 控制器控制表：每个控制器一张表
  │  控制器级│     记录控制器状态（忙/闲）、等待队列、CHCT 指针
  └────┬────┘
       │ 找到连接控制器的通道的 CHCT
       ▼
  ┌─────────┐
  │  CHCT   │  ← 通道控制表：每个通道一张表
  │  通道级  │     记录通道状态（忙/闲）、等待队列
  └────┬────┘
       │ 全部可用
       ▼
    分配成功，启动 I/O
```

**阻塞示例**：

假设请求使用磁盘 X：
- SDT 中找到磁盘 X → DCT 指针有效 ✓
- DCT 中磁盘 X 状态 = "空闲" ✓
- DCT → COCT：磁盘 X 的控制器的 COCT 状态 = "忙"（正在服务磁盘 Y）
  → 进程在此 COCT 的等待队列上阻塞，等待控制器空闲

这就是 **"设备-控制器-通道"三级分配** 的思想：只有三者都可用，才能启动 I/O 操作。

| 结构 | 全称 | 作用 | 关键字段 |
|------|------|------|----------|
| **SDT** | 系统设备表 (System Device Table) | 系统级设备目录 | 设备标识、设备类型、DCT 指针 |
| **DCT** | 设备控制表 (Device Control Table) | 设备级状态和队列 | 忙/闲、等待队列头指针、COCT 指针 |
| **COCT** | 控制器控制表 (Controller Control Table) | 控制器级状态 | 忙/闲、等待队列、CHCT 指针 |
| **CHCT** | 通道控制表 (Channel Control Table) | 通道级状态 | 忙/闲、等待队列 |

## 九、中断处理详细步骤

标准中断处理八步流程（从前一文件移入此处以保持完整性）：

```
中断发生:
  ┌─ 1. 中断信号到达 CPU 的 INTR 引脚
  │
  ├─ 2. CPU 执行完当前指令，在下一个指令周期检查中断引脚
  │     （若 IF = 0 则不响应可屏蔽中断）
  │
  ├─ 3. 关中断 (CLI)，保存现场（硬件自动 + 软件辅助）
  │     硬件自动保存: 标志寄存器 EFLAGS、CS、EIP
  │     软件保存: 通用寄存器 push 入栈
  │
  ├─ 4. 查 IDT（中断描述符表），跳转到 ISR 入口
  │     中断向量号 N → IDT[N] → ISR 地址
  │
  ├─ 5. 根据需要开中断 (STI)
  │     （允许更高优先级的中断嵌套）
  │
  ├─ 6. 执行 ISR（中断服务程序）
  │     - 判断中断源（哪个设备发出的）
  │     - 调用对应的处理逻辑（设备驱动注册的回调）
  │     - 必要时设置下半部（softirq / tasklet）处理耗时操作
  │
  ├─ 7. 关中断 (CLI)，恢复现场
  │     弹出通用寄存器，恢复 EFLAGS/CS/EIP (IRET 指令)
  │
  └─ 8. 开中断 (STI)，返回原程序继续执行
```

> **上半部 vs 下半部**：步骤 6 中的 ISR 应该尽可能短（上半部）——只做最少量的关键操作（确认中断源、复制关键数据到内核缓冲区、设置标志）。时间敏感的操作（如大量数据复制、唤醒进程）延迟到下半部（softirq / tasklet / workqueue）执行，以保证中断响应不阻塞其他设备。

## 十、本章总览

```
                    ┌──── 缓冲 ────┐
                    │ 单/双/循环/池│
                    │ 速度匹配     │
                    │ 减少中断     │
                    └──────┬───────┘
   I/O 控制方式              │              设备分配
   (轮询/中断/DMA/通道)      │        (SDT→DCT→COCT→CHCT)
         │                  │                  │
         ▼                  ▼                  ▼
  ┌──────────────────────────────────────────────────┐
  │              设备无关软件层                       │
  │    (LUT 映射 / 缓冲策略 / SPOOLing / 权限检查)    │
  └──────────────────────────────────────────────────┘
         │                  │                  │
         ▼                  ▼                  ▼
    SPOOLing 假脱机   中断处理程序          设备驱动
    (独占→共享)      (上半部+下半部)      (厂商编写)
```

整个 I/O 子系统的核心设计原则是：**让 CPU 与 I/O 设备尽可能地并行工作，用缓冲和 SPOOLing 掩盖速度差异，用 LUT 隔离硬件的物理细节**。
