## Part 1：系统调用（System Call）

### 1.1 是什么

**系统调用（System Call）** 是用户态程序请求内核服务的**唯一合法入口**。用户程序不能直接调用内核函数（地址空间隔离 + 特权级隔离），只能通过 CPU 提供的"门"——系统调用指令——**主动陷入（Trap）**核心态，由内核代为完成操作后返回。

> **类比**：用户程序是银行的客户，内核是金库。客户不能自己进金库拿钱——他必须通过柜台（系统调用），填写申请表（传参），由银行职员（内核）进入金库完成操作，然后把结果交还给客户。

### 1.2 系统调用 vs 库函数 vs 普通函数

| 维度 | 系统调用 | 库函数（如 `printf`） | 普通函数 |
|------|---------|---------------------|---------|
| **运行态** | 跨用户态→核心态→用户态 | 用户态（内部可能调用系统调用） | 用户态 |
| **开销** | 大（数百到数千个 CPU 周期） | 中（如果内部调用了系统调用则大） | 极小（几条 CPU 指令） |
| **举例** | `write(1, buf, len)` — 直接对应 sys_write | `printf("hello")` — 格式化 + 多次调用 `write` | `strlen(buf)` — 纯用户态计算 |
| **地址空间** | 访问内核地址空间 | 访问用户地址空间 | 访问用户地址空间 |

> 许多标准 C 库函数（`printf`、`malloc`、`fopen`）**不是**系统调用——它们是用户态的封装函数，内部在必要时调用一个或多个系统调用。`printf("hello")` 最终会调用 `write(1, "hello", 5)`，这才是真正的系统调用。

### 1.3 系统调用分类

| 类别 | 说明 | POSIX 示例 |
|------|------|-----------|
| **进程控制** | 创建、终止、等待进程/线程 | `fork()` — 创建子进程；`execve()` — 替换进程映像；`exit()` — 终止进程；`waitpid()` — 等待子进程结束；`clone()` — Linux 特有的创建线程/进程的通用接口 |
| **文件操作** | 打开、读写、关闭、定位文件 | `open()`, `read()`, `write()`, `close()`, `lseek()`, `stat()`, `rename()`, `unlink()` |
| **目录操作** | 创建、删除、遍历目录 | `mkdir()`, `rmdir()`, `getdents()`, `chdir()` |
| **设备操作** | 控制设备参数 | `ioctl()` — I/O 控制（万能接口，用于设置终端属性、磁盘参数、网络配置等） |
| **信息维护** | 获取/设置系统或进程信息 | `getpid()`, `getuid()`, `time()`, `alarm()`, `sysinfo()` |
| **内存管理** | 动态分配/释放内存（大块） | `brk()`, `sbrk()` — 调整数据段大小（`malloc` 的底层依赖）；`mmap()` — 内存映射文件或匿名内存 |
| **通信** | 进程间通信（IPC） | `pipe()` — 匿名管道；`socketpair()` — socket 对；`shmget()` — 共享内存；`msgget()` — 消息队列；`semget()` — 信号量集 |
| **权限管理** | 修改进程权限 | `setuid()`, `setgid()`, `chmod()`, `chown()`, `capget()`/`capset()` |

### 1.4 系统调用完整链路 —— 逐步拆解

以 `read(fd, buf, 100)` 为例，在 **x86-64 Linux (glibc)** 上的完整执行路径。

#### Step 0：用户代码写调用

```c
// 用户程序 user_app.c
#include <unistd.h>

int main() {
    char buf[100];
    int n = read(0, buf, 100);  // 从 stdin (fd=0) 读 100 字节
    return 0;
}
```

#### Step 1：glibc 封装层

用户代码中调用 `read()`，实际调用的是 **glibc 提供的封装函数**（定义在 glibc 源码 `sysdeps/unix/sysv/linux/read.c` 或汇编宏中）。

glibc 中 `read` 的实现（简化后的伪代码）：

```c
// glibc 中 read() 的封装（简化）
ssize_t read(int fd, void *buf, size_t count) {
    // 1. 将系统调用号存入 rax 寄存器
    //    read 在 x86-64 Linux 上的系统调用号 = 0
    //
    // 2. 将参数依次存入 rdi, rsi, rdx
    //    rdi = fd (0)
    //    rsi = buf (缓冲区地址)
    //    rdx = count (100)
    //
    // 3. 执行 syscall 指令

    return (ssize_t) SYSCALL_WRAPPER(__NR_read, fd, buf, count);
}
```

glibc 通过汇编宏将上述逻辑展开为实际机器指令：

```asm
; glibc 中 read 的实际汇编逻辑（x86-64）
read:
    mov    eax, 0          ; rax = 0 (系统调用号: __NR_read)
    mov    edi, edi         ; 第一个参数已在 rdi 中（fd）
                             ; 第二个参数已在 rsi 中（buf）
                             ; 第三个参数已在 rdx 中（count）
    syscall                 ; ← 关键：触发核心态切换
    cmp    rax, -4096       ; 检查返回值：Linux 内核错误码用负值表示
    ja     .Lerror          ; 如果返回值在 [-4095, -1] 区间，则出错
    ret                     ; 正常返回，rax 中是实际读取的字节数
.Lerror:
    neg    eax              ; rax 取反 → 转为正数的 errno
    mov    DWORD PTR fs:errno, eax  ; 设置 errno
    mov    rax, -1           ; 返回 -1 表示出错
    ret
```

> **`cmp rax, -4096` 为什么是 -4096？** Linux 内核返回错误时，将负的 `errno` 值（-1 到 -4095）放在 `rax` 中。glibc 检查 `rax` 是否在这个范围内——如果是，说明出错，取反得到 `errno` 值，函数返回 -1。如果 `rax` 不在 [-4095, -1] 区间，说明是正常的正数返回值——直接返回。

#### Step 2：syscall 指令执行 —— CPU 硬件操作

当 CPU 执行到 `syscall` 指令时，硬件在**一个原子操作**中完成以下动作：

```
CPU 硬件自动执行（不可中断的原子操作）：

1. RCX ← RIP              保存用户态返回地址
   （即 save the return address: syscall 的下一条指令 = "cmp rax, -4096"）

2. R11 ← RFLAGS           保存用户态标志寄存器（IF, TF, IOPL 等关键标志）

3. RIP ← MSR_LSTAR        加载内核系统调用入口地址
   （MSR_LSTAR 在内核启动时被初始化为 entry_SYSCALL_64）

4. CS ← MSR_STAR[47:32] & 0xFFFC
   设置 CS 选择子 → 指向内核代码段（Ring 0, CPL = 0）

5. SS ← (MSR_STAR[47:32] + 8)
   设置 SS 选择子 → 指向内核数据段（Ring 0）

6. RFLAGS ← RFLAGS & ~MSR_SFMASK
   根据 MSR_SFMASK 屏蔽标志位（如清除 IF 关中断，或保留 IF 开中断）
   Linux 的 MSR_SFMASK 配置为仅清除某些调试标志，保留 IF（允许中断）

7. RSP ← 从 TSS（Task State Segment）加载内核栈指针
   每个进程有自己的内核栈（通常 4KB / 8KB / 16KB）
```

**关键 MSR 寄存器一览（x86-64）：**

| MSR | 地址 | 用途 |
|-----|------|------|
| `MSR_LSTAR` | `0xC0000082` | 64 位系统调用入口地址（RIP 加载源） |
| `MSR_STAR` | `0xC0000081` | 系统调用时的 CS/SS 选择子（高 32 位存 CS 基址，低 32 位存用户态 CS/SS 选择子，供 `sysret` 使用） |
| `MSR_SFMASK` | `0xC0000084` | 系统调用时对 RFLAGS 的屏蔽掩码（对应位设为 1 则清除该标志） |
| `MSR_CSTAR` | `0xC0000083` | 兼容模式（32 位程序在 64 位内核上）的系统调用入口 |

> **这些 MSR 什么时候设置的？** 在内核启动阶段，`syscall_init()` 函数（`arch/x86/kernel/cpu/common.c`）通过 `wrmsr` 指令写入上述 MSR。此后只要 CPU 执行 `syscall`，就会自动跳转到 `entry_SYSCALL_64`。

#### Step 3：内核入口 —— entry_SYSCALL_64

CPU 现在执行 `entry_SYSCALL_64` 处的代码（Linux 内核源码 `arch/x86/entry/entry_64.S`）。这是内核的第一个 C 函数调用前的"汇编序曲"。

```
entry_SYSCALL_64:
    swapgs                        ; GS 基址从"用户 TLS"切换到"内核 per-CPU 区域"
    mov    %rsp, PER_CPU(rsp_scratch)  ; 暂存用户栈指针到 per-CPU 区域
    mov    PER_CPU(cpu_current_top_of_stack), %rsp  ; 切换到内核栈
    pushq  $__USER_DS             ; 构造 iret 返回帧的一部分
    pushq  PER_CPU(rsp_scratch)   ; 保存用户栈指针
    pushq  r11                    ; 保存用户态 RFLAGS
    pushq  $__USER_CS             ; 保存用户态 CS
    pushq  rcx                    ; 保存用户态 RIP（返回地址）

    pushq  rax                    ; 保存系统调用号
    pushq  rdi, rsi, rdx, r10, r8, r9, rbx, rbp, r12-r15  ; 保存所有通用寄存器

    ; 至此，内核栈上构建了一个完整的 pt_regs 结构体
    ; 这既用于内核代码访问参数，也是将来 context switch 的基础

    mov    rdi, rsp               ; rdi = 指向 pt_regs 的指针（C 函数第一个参数）
    call   do_syscall_64          ; 进入 C 语言层
```

此时内核栈的布局：

```
高地址
┌──────────────────────┐
│  SS（用户态）          │  ← 来自 pushq $__USER_DS
├──────────────────────┤
│  RSP（用户态栈指针）    │
├──────────────────────┤
│  RFLAGS（用户态标志）   │  ← 来自 pushq r11
├──────────────────────┤
│  CS（用户代码段）       │  ← 来自 pushq $__USER_CS
├──────────────────────┤
│  RIP（返回地址）        │  ← 来自 pushq rcx
├──────────────────────┤
│  orig_rax（系统调用号） │  ← 来自 pushq rax
├──────────────────────┤
│  rdi, rsi, rdx, ...   │  ← 全部通用寄存器
│  r10, r8, r9, rbx,    │
│  rbp, r12-r15          │
├──────────────────────┤
│  （内核栈后续可用空间）  │
└──────────────────────┘ ← RSP 指向此处
低地址
```

#### Step 4：C 层 —— do_syscall_64

```c
// arch/x86/entry/common.c（简化）
__visible void do_syscall_64(unsigned long nr, struct pt_regs *regs)
{
    // nr = regs->orig_rax，即系统调用号
    if (likely(nr < NR_syscalls)) {
        // 从 sys_call_table 查表
        unsigned long (*fn)(struct pt_regs *) = sys_call_table[nr];
        // 调用具体处理函数
        regs->ax = fn(regs);   // 返回值存入 rax 字段
    }
    // 返回后，entry_SYSCALL_64 的汇编代码将 regs->ax 写入真实 rax 寄存器
}
```

**`sys_call_table`** 是一个函数指针数组，每个元素是一个系统调用处理函数：

```
sys_call_table 索引 (系统调用号)
┌──────┬────────────────────────────────┐
│   0  │ sys_read           → sys_read()│
│   1  │ sys_write          → sys_write()│
│   2  │ sys_open           → sys_open()│
│   3  │ sys_close          → sys_close()│
│ ...  │ ...                            │
│   9  │ sys_mmap           → sys_mmap()│
│  14  │ sys_rt_sigprocmask → ...       │
│  60  │ sys_exit           → sys_exit()│
│ 231  │ sys_exit_group     → ...       │
│ ...  │ ...                            │
└──────┴────────────────────────────────┘
```

> Linux 的 `sys_call_table` 定义在 `arch/x86/entry/syscalls/syscall_64.tbl`，总共约 450+ 个系统调用（随版本增长）。

#### Step 5：内核函数执行 —— 以 sys_read 为例

```c
// fs/read_write.c（简化逻辑）
SYSCALL_DEFINE3(read, unsigned int, fd, char __user *, buf, size_t, count)
{
    struct fd f = fdget_pos(fd);         // 根据 fd 查文件描述符表，找到 file 结构体
    if (!f.file)
        return -EBADF;                   // 错误的文件描述符 → -9 (errno)

    loff_t pos = file_pos_read(f.file);  // 获取当前文件偏移
    ssize_t ret = vfs_read(f.file, buf, count, &pos);  // VFS 层读取
    if (ret >= 0)
        file_pos_write(f.file, pos);     // 更新文件偏移
    fdput_pos(f);                        // 释放引用
    return ret;                          // 返回值存入 rax
}
```

`vfs_read()` 内部的分发逻辑：

```
sys_read()
  └→ vfs_read(file, buf, count, &pos)
       ├→ 检查权限（security_file_permission）
       ├→ 检查 file->f_op->read 是否存在
       │   ├→ 如果文件在 Page Cache 中有数据 → 直接从缓存 copy_to_user()
       │   └→ 如果缓存未命中 → 调用具体文件系统的 read
       │        ├→ ext4 文件 → ext4_file_read_iter()
       │        ├→ XFS 文件 → xfs_file_read_iter()
       │        ├→ NFS 文件 → nfs_file_read()
       │        └→ 块设备  → blkdev_read_iter() → 发起磁盘 I/O → 阻塞当前进程 → 调度
       └→ 返回实际读取字节数
```

**关键点**：当磁盘 I/O 需要等待时，内核将当前进程设为**阻塞状态**（TASK_UNINTERRUPTIBLE），放入等待队列，然后调用 `schedule()` 让出 CPU。等磁盘 I/O 完成（通过中断通知），再唤醒进程、把数据 copy_to_user、返回用户态。

#### Step 6：返回用户态

```
entry_SYSCALL_64 (返回路径，简化):
    ; 内核函数执行完毕，rax 中已包含返回值
    ; 检查是否需要处理信号（TIF 标志）
    ; 检查是否需要重新调度（need_resched 标志）

    popq    r15, r14, r13, r12, rbp, rbx  ; 恢复所有通用寄存器
    popq    r10, r9, r8, rdi, rsi, rdx
    popq    rax                            ; 不恢复 rax——保留内核返回值

    ; 以下判断是否有挂起的信号/调度请求，如果有则跳转到慢速返回路径
    ; （慢速路径上会调用 schedule() 或 do_signal()）

    swapgs                                 ; GS 从内核 per-CPU 切换回用户 TLS
    sysretq                                ; ← 返回用户态

───────────────────────────────────

CPU 硬件自动执行（sysretq）：
1. RIP ← RCX                恢复用户态返回地址（syscall 下一条指令）
2. RFLAGS ← R11             恢复用户态标志寄存器（包括 IF 中断使能位）
3. CS ← MSR_STAR[63:48]    恢复用户代码段（Ring 3）
4. RSP ← （用户栈指针恢复，由之前 push 的值决定）
```

#### Step 7：回到用户代码

```asm
; glibc 中 read() wrapper 的后半部分
    cmp    rax, -4096        ; rax = 内核返回值（正常时为正，错误时为负的 errno）
    ja     .Lerror
    ret                      ; 返回给用户代码，rax 中是读取字节数
```

整个链路完成。从用户调用 `read()` 到拿到返回值，涉及**两次特权级切换**（`syscall` + `sysret`）、一次完整的上下文保存/恢复、以及内核中 VFS → 文件系统 → 磁盘驱动的多层调用。

### 1.5 int $0x80 vs syscall 对比

| 维度 | `int $0x80` | `syscall` |
|------|------------|-----------|
| **架构** | x86 32 位（传统方式） | x86-64（快速系统调用） |
| **实现原理** | 通过中断门（Interrupt Gate），CPU 查 IDT[0x80] 获取处理程序地址 | 通过 MSR 寄存器直接跳转，无需查 IDT |
| **性能** | 慢——需要多次内存访问（查 IDT、权限检查、压栈） | 快——比 `int 0x80` 快约 2-3 倍 |
| **开关中断** | 自动关中断（通过中断门的 IF 位控制） | 通过 MSR_SFMASK 控制，Linux 保留 IF 打开 |
| **调用号** | 32 位的系统调用号表（约 380 个） | 64 位独立编号（与 32 位不同！） |
| **传参寄存器** | `eax`=调用号, `ebx, ecx, edx, esi, edi, ebp` | `rax`=调用号, `rdi, rsi, rdx, r10, r8, r9` |

> **注意**：同一个系统调用在 32 位和 64 位下的调用号**不同**。例如 `read`：32 位下调用号 = 3；64 位下调用号 = 0。这是因为两种模式使用独立的 `sys_call_table`。

## Part 2：中断（Interrupt）

### 2.1 中断的定义与本质

**中断（Interrupt）** 是指 CPU 在执行过程中，收到来自**外部设备**或**内部事件**的信号后，**暂停**当前程序的执行，转去执行一段特殊的**中断处理程序（Interrupt Service Routine, ISR）**，处理完毕后再**返回**被打断的程序继续执行。

**中断的本质**：让 CPU 从"被动轮询模式"变为"主动响应模式"。没有中断时，CPU 必须不断轮询设备状态（"数据好了吗？还没好？那现在呢？"），浪费大量 CPU 周期。有了中断，设备在数据就绪时**主动通知** CPU——CPU 在此期间可以继续执行其他任务。

> **类比**：没有中断 = 你每隔 5 秒去门口看快递来了没有（轮询）。有中断 = 快递员按门铃（中断），你听到后去开门。

### 2.2 中断分类

```
中断体系
├── 硬中断（Hardware Interrupt / External Interrupt）
│   ├── 可屏蔽中断（Maskable Interrupt, INTR）
│   │   ├── 时钟中断（Timer Interrupt）→ 调度、时间片到期
│   │   ├── 键盘中断 → 按键处理
│   │   ├── 网卡中断 → 数据包到达
│   │   ├── 磁盘中断 → I/O 完成
│   │   └── 鼠标、USB、GPU 等...
│   │
│   └── 不可屏蔽中断（Non-Maskable Interrupt, NMI）
│       ├── 硬件故障（内存 ECC 错误、总线错误）
│       └── 电源掉电告警
│
├── 软中断（Software Interrupt）
│   ├── int 指令显式触发（如 int $0x80 系统调用）
│   └── CPU 自身执行的软中断（如 Linux 的 softirq 机制——注意这是
│       内核概念，不同于 CPU 架构的软中断）
│
└── 异常（Exception / Internal Interrupt）
    ├── 故障（Fault）—— 可恢复
    │   ├── 缺页异常（#PF, Page Fault, 向量 14）
    │   ├── 设备不可用（#NM, 向量 7）—— 如 FPU 未初始化
    │   └── 一般保护故障（#GP, 向量 13）
    │
    ├── 陷阱（Trap）—— 故意的，用于调试
    │   ├── 单步调试（#DB, 向量 1, TF=1）
    │   └── 断点（#BP, 向量 3, int3 指令）
    │
    └── 中止（Abort）—— 不可恢复
        ├── 双重故障（#DF, 向量 8）—— 处理异常时又发生异常
        ├── 机器检查异常（#MC, 向量 18）—— 硬件致命错误
        └── 三重故障 → CPU 直接复位
```

**三种中断的关键区别：**

| 维度 | 硬中断 | 软中断 | 异常 |
|------|-------|--------|------|
| **触发源** | CPU 外部（硬件设备） | CPU 执行的 `int` / `syscall` 指令 | CPU 内部（当前指令出错） |
| **与指令流的关系** | **异步**（Asynchronous）——何时发生不可预测 | **同步**——指令执行即触发 | **同步**——当前指令执行时检测到 |
| **可屏蔽性** | 可屏蔽（NMI 除外） | 不可屏蔽（指令本身就是意图） | 不可屏蔽 |
| **处理后的返回位置** | 被打断指令的下一条 | `int` 指令的下一条 | 故障：当前指令（重试）；陷阱：下一条；中止：不返回 |

### 2.3 中断向量表 —— IVT 与 IDT

#### 2.3.1 实模式下的 IVT（Interrupt Vector Table）

在 8086 实模式下，中断向量表（IVT）固定在物理地址 `0x00000` 处，占用 1024 字节（256 个表项，每项 4 字节）。

```
物理地址         内容
0x00000  ┌──────────────┐
         │ 向量 0 的 IP  │  ← 除法错误处理程序入口偏移
0x00002  │ 向量 0 的 CS  │  ← 除法错误处理程序入口段址
0x00004  │ 向量 1 的 IP  │  ← 单步调试
0x00006  │ 向量 1 的 CS  │
0x00008  │ 向量 2 的 IP  │  ← NMI
   ...   │     ...      │
0x003FC  │ 向量 255 的 IP│
0x003FE  │ 向量 255 的 CS│
0x00400  └──────────────┘
```

每项结构（4 字节）：`[IP(2B) | CS(2B)]`。CPU 响应中断时，从 IVT[向量号] 中加载 `CS:IP`，跳转到处理程序。

#### 2.3.2 保护模式 / 长模式下的 IDT（Interrupt Descriptor Table）

现代 CPU（保护模式、IA-32e / long mode）使用**中断描述符表（IDT）**。IDT 的位置不固定——由 CPU 的 `IDTR` 寄存器（IDT Register）指定其**基址和界限**。

```
IDTR 寄存器结构（48 位）：
┌─────────────────────┬──────────────────────┐
│   IDT 基址（32/64 位）│  IDT 界限（16 位）     │
└─────────────────────┴──────────────────────┘

IDT 中的某个表项（门描述符，Gate Descriptor，16 字节）：
┌────────────────────────────────────────────────┐
│  Offset[15:0]          │  Segment Selector     │  ← 目标代码段
├────────────────────────┼───────────────────────┤
│  IST │ 保留 │ Type │ DPL │ P │ Offset[31:16]  │
├────────────────────────┴───────────────────────┤
│  Offset[63:32]                          │ 保留  │
└────────────────────────────────────────────────┘
  关键字段说明：
  · Offset[63:0] — 中断处理程序（ISR）的 64 位入口地址
  · Segment Selector — 目标代码段选择子（通常为内核代码段）
  · Type — 门类型（中断门 / 陷阱门 / 任务门）
  · DPL — 描述符特权级（Descriptor Privilege Level）
  · P — 存在位（Present, 1=有效）
  · IST — 中断栈表索引（Interrupt Stack Table, 0 表示使用普通内核栈）
```

**IDT 中三种门类型：**

| 门类型 | Type 值 | 关键区别 | 用途 |
|--------|---------|---------|------|
| **中断门（Interrupt Gate）** | `0xE` | 进入时**自动关中断**（清除 RFLAGS.IF） | 硬件中断处理——防止同一中断嵌套 |
| **陷阱门（Trap Gate）** | `0xF` | 进入时**保持中断状态**（不清除 IF） | 异常处理和系统调用——允许处理期间响应中断 |
| **任务门（Task Gate）** | `0x5` | 硬件任务切换（通过 TSS） | 几乎废弃（现代 OS 不使用硬件任务切换） |

> **中断门 vs 陷阱门的实际影响**：通过中断门进入 ISR 时，IF 被自动清除，即同一 IRQ 线不会在当前 ISR 执行期间再次触发嵌套（除非 ISR 内部手动 `STI` 开中断）。通过陷阱门进入时 IF 不变，可以响应其他中断。

#### 2.3.3 CPU 如何通过 IDT 定位 ISR

```
CPU 响应中断的硬件查表流程（以向量号 N 为例）：

1. 从 IDTR 获取 IDT 的基址 BASE
2. 计算表项地址: ADDR = BASE + N × 16   （64 位模式下每项 16 字节）
3. 读取门描述符，检查 P 位（不存在则触发 #NP 异常）
4. 检查 DPL（如果由 int 指令触发，需确保 CPL ≤ DPL；硬件中断不受此限制）
5. 从门描述符中提取 Offset（64 位地址）和 Segment Selector
6. 加载 CS 为 Segment Selector（CS.CPL 变为目标代码段的 DPL，通常是 0）
7. 加载 RIP 为 Offset
8. 如果目标代码段是 Ring 0（内核），硬件自动切换到内核栈（通过 TSS）
9. 如果是中断门，自动清除 RFLAGS.IF（关中断）
10. 将旧 SS、RSP、RFLAGS、CS、RIP 压入新栈（构造 iret 返回帧）

现在 CPU 在 ISR 的第一条指令处，运行在 Ring 0。
```

### 2.4 中断处理完整流程（逐步骤）

以下以**时钟中断（Timer Interrupt）** 为例，详细展开每一步。

#### 整体流程图

```
设备发出中断信号
      │
      ▼
中断控制器（PIC/APIC）汇总和优先级判断
      │
      ▼ (INTR 引脚)
CPU 完成当前指令
      │
      ▼
CPU 检查条件 → 如果 IF=0，忽略（可屏蔽中断）→ 如果 IF=1，继续 →
      │
      ▼
CPU 通过中断应答周期获取中断向量号 N
      │
      ▼
保存现场（硬件自动）
      │
      ▼ (查 IDT[N])
跳转到 ISR 入口
      │
      ▼
ISR 第一部分（汇编序曲）：保存寄存器
      │
      ▼
ISR 第二部分（C 语言）：实际处理
      │
      ▼
ISR 第三部分（汇编尾声）：恢复寄存器
      │
      ▼
执行 iretq → 硬件恢复 CS:RIP, RFLAGS, RSP:SS → 回到被打断的程序
```

#### 每一步详解

**Step 1：设备发出中断信号**

系统定时器（如 HPET、APIC Timer 或 PIT）设置的定时到期后，通过电路向中断控制器发送信号。

**Step 2：中断控制器（APIC, Advanced Programmable Interrupt Controller）处理**

现代 x86 系统使用 APIC 体系（Local APIC + I/O APIC）：

```
外设 ──→ I/O APIC ──→ (APIC Bus / System Bus) ──→ Local APIC ──→ CPU Core
                                                      │
                                                      └── 每个 CPU 核都有自己的 Local APIC
```

APIC 的职责：
- 接收来自各外设的中断请求
- 按**优先级**和**目标 CPU** 将中断路由到特定 CPU 核
- 将中断向量号 / IRQ 号传递给 CPU

**Step 3：CPU 完成当前指令并检查中断条件**

```
CPU 在每个指令周期的特定时刻检查：
1. 是否有 NMI 信号？→ 必须响应
2. 是否有 INTR 信号？→ 检查 RFLAGS.IF
   ├── IF=0 → 暂时忽略（pending），等 IF=1 时再响应
   └── IF=1 → 进入中断响应流程
3. 是否有 pending 的异常？→ 必须先处理异常
```

> CPU 在**指令边界（Instruction Boundary）**检查中断，不会在指令执行的中途打断。但对于 REP 前缀的字符串指令（如 `rep movsb`），CPU 可以在迭代之间响应中断。

**Step 4：中断应答（Interrupt Acknowledge）**

```
CPU ──→ INTA 周期 ──→ 中断控制器 ──→ 返回向量号 N
```

CPU 通过中断应答总线周期向 APIC 索取中断向量号。APIC 根据内部优先级仲裁，将最高优先级的中断向量号通过数据总线返回给 CPU。

**Step 5：硬件自动保存现场并切换栈（以长模式为例）**

当目标代码段（从 IDT 门描述符的 Segment Selector）的 CPL < 当前 CPL，即发生特权级切换时（用户态→内核态），CPU 硬件自动执行：

```
硬件自动压栈顺序（从 TSS 加载内核栈后）：
Push SS        ; 用户态 SS
Push RSP       ; 用户态 RSP
Push RFLAGS    ; 用户态 RFLAGS
Push CS        ; 用户态 CS
Push RIP       ; 用户态返回地址（被打断指令的位置）

如果异常有错误码（如 #PF, #GP），额外 Push 错误码
```

此时内核栈布局（进入 ISR 时）：

```
高地址
┌────────────┐
│  旧 SS     │
├────────────┤
│  旧 RSP    │
├────────────┤
│  旧 RFLAGS │  ← 这就是 iretq 返回帧
├────────────┤
│  旧 CS     │
├────────────┤
│  旧 RIP    │
├────────────┤
│  错误码     │  ← 仅部分异常
└────────────┘ ← RSP 指向此处
低地址
```

> **对比**：如果中断发生在内核态（目标 CPL = 当前 CPL = 0），不需要特权级切换，CPU 不切换栈——仅在当前栈上压入 RFLAGS, CS, RIP（和可选的错误码）。

**Step 6：ISR 汇编序曲 —— 保存通用寄存器**

```asm
; Linux 内核 common_interrupt（简化）
common_interrupt:
    ; RFLAGS, CS, RIP 已经被硬件压栈

    pushq  %rdi           ; 保存调用者保存寄存器
    pushq  %rsi
    pushq  %rdx
    pushq  %rcx
    pushq  %rax
    pushq  %r8
    pushq  %r9
    pushq  %r10
    pushq  %r11

    ; 被调用者保存寄存器也需保存（如果 ISR 会用）
    pushq  %rbx
    pushq  %rbp
    pushq  %r12
    pushq  %r13
    pushq  %r14
    pushq  %r15

    ; 至此 pt_regs 构建完成
    movq   %rsp, %rdi      ; rdi = pt_regs 指针
    call   do_IRQ          ; 进入 C 处理层
```

**Step 7：C 语言中断处理核心 —— do_IRQ**

```c
// kernel/irq/handle.c（简化）
void do_IRQ(struct pt_regs *regs, int vector)
{
    // 1. 确认中断（acknowledge）——通知 APIC 中断已被接收
    ack_APIC_irq();

    // 2. 根据向量号找到对应的 irq_desc（中断描述符）
    struct irq_desc *desc = vector_to_irq_desc(vector);

    // 3. 调用该中断线注册的所有 ISR
    //    一条中断线上可以注册多个 handler（共享中断）
    handle_irq(desc, regs);

    // 4. 检查是否需要重新调度（need_resched |= 1）
    //    如果中断处理期间有更高优先级的进程被唤醒
}
```

**Step 8：中断下半部（Bottom Half）**

为避免关闭中断过久，Linux 将中断处理分为两半：

| 阶段 | 说明 | 执行时机 |
|------|------|---------|
| **上半部（Top Half / Hard IRQ）** | 关闭中断执行、只做最紧急的工作（如从设备寄存器读数据、清中断标志） | 中断上下文，立即执行 |
| **下半部（Bottom Half）** | 开中断执行、做非紧急但耗时的工作（如网络数据包处理、写磁盘缓冲区） | 三种机制：softirq / tasklet / workqueue |

```
中断到达
  │
  ▼
上半部（关中断，微秒级）：
  · 读取网卡寄存器，将数据包拷贝到内核缓冲区
  · 清中断标志
  · 调度下半部（softirq 或 tasklet）
  │
  ▼
iretq 返回
  │
  ▼
（在合适的时机，如系统调用返回路径或内核线程上下文）
下半部（开中断，毫秒级）：
  · TCP/IP 协议栈处理数据包
  · 通知等待该 socket 的进程（唤醒）
```

**Step 9：ISR 返回**

```asm
; Linux 内核 ret_from_intr 路径（简化）
ret_from_intr:
    popq    %r15, %r14, %r13, %r12, %rbp, %rbx  ; 恢复通用寄存器
    popq    %r11, %r10, %r9, %r8
    popq    %rax, %rcx, %rdx, %rsi, %rdi         ; 恢复调用者保存寄存器

    ; 检查是否有挂起的信号需要处理
    ; 检查 need_resched 标志，如果需要重新调度则调用 schedule()
    ; （这被称为"中断返回时的调度点"）

    ; 如果有错误码在栈上，需要额外弹出

    iretq       ; ← 硬件恢复 CS:RIP, RFLAGS, SS:RSP

──────────────────────────────────────────

CPU 硬件执行 iretq：
1. 从内核栈 Pop → RIP
2. 从内核栈 Pop → CS
3. 从内核栈 Pop → RFLAGS
4. 如果目标 CS.CPL > 当前 CPL（特权级降低，返回用户态）：
   从内核栈 Pop → RSP
   从内核栈 Pop → SS
5. CPU 加载新的 CS:RIP 和 SS:RSP，继续执行
```

### 2.5 中断优先级

当多个中断同时到达时，CPU 和中断控制器需要决定先处理哪一个。

#### x86 异常/中断的固定优先级

CPU 内部使用固定的向量号优先级（向量号越小不一定优先级越高，x86 有预定义的优先级顺序）：

```
优先级从高到低：
1. 硬件复位（RESET）和机器检查异常（#MC, 向量 18）
2. 任务切换时的陷阱
3. 外部硬件中断——NMI（不可屏蔽中断）
4. 外部硬件中断——INTR（可屏蔽中断）
5. 陷阱标志（TF）单步异常
6. 指令断点（#BP, 向量 3）
7. 无效操作码（#UD, 向量 6）
8. 设备不可用（#NM, 向量 7）
9. 双重故障（#DF, 向量 8）——在前后异常处理期间又发生异常
10. 其他异常 ...
```

> **实际优先级规则**：在同一优先级级别内，较低向量号优先处理。但 NMI 高于 INTR、异常高于 TF 等优先级已经由硬件固定。

#### APIC 中断优先级

APIC 使用**任务优先级寄存器（TPR, Task Priority Register）** 和**处理器优先级寄存器（PPR, Processor Priority Register）**来控制：

```
APIC 接收中断后比较：
  ┌──────────────────────────────────┐
  │ 中断向量号[7:4] （优先级子类）      │
  │        vs                        │
  │ TPR / PPR 中的优先级阈值           │
  └──────────────────────────────────┘

  只有中断向量的优先级 > PPR 中的阈值，中断才会被递送给 CPU
  通过设置 TPR，软件可以屏蔽低于某优先级的所有中断
```

### 2.6 嵌套中断（Nested Interrupts）

#### 概念

**嵌套中断**是指：在某个中断的 ISR 正在执行期间，一个更高优先级的中断到达并被响应，打断了当前 ISR 的执行。

```
正常中断处理：                      嵌套中断处理：
  程序A                               程序A
    │                                   │
    │ 中断X 到达                        │ 中断X（低优先级）
    ▼                                   ▼
  ISR_X                               ISR_X 开始执行
    │                                   │
    │                                   │ 中断Y 到达（高优先级，如 NMI）
    │                                   ▼
    ▼                                 ISR_X 被暂停
  ISR_X 完成                          ISR_Y 开始执行
    │                                   │
    │                                   ▼
    ▼                                 ISR_Y 完成
  程序A 继续                           ISR_X 恢复执行
                                        │
                                        ▼
                                      ISR_X 完成
                                        │
                                        ▼
                                      程序A 继续
```

#### 何时允许嵌套

| CPU 状态 | 是否允许嵌套 |
|----------|-------------|
| 当前 ISR 通过**中断门**进入 → IF 自动清除 | **默认不允许**——新的可屏蔽中断不会打断当前 ISR（但 NMI 仍然可以） |
| 当前 ISR 通过**陷阱门**进入 → IF 保持不变 | 取决于 IF 的原始状态 |
| ISR 内部显式执行了 `STI`（开中断） | **允许**（但新中断进入后会再次关中断，通过中断门的话） |
| NMI 到达 | **总是允许**——NMI 不可屏蔽，会打断任何正在执行的代码（但不能嵌套另一个 NMI，CPU 硬件阻止） |

> **Linux 实践**：为了最小化中断延迟，Linux 的硬件中断上半部（hardirq）默认**关中断**执行——简洁、快速。长耗时操作推迟到**下半部**（开中断执行）。

#### 嵌套中断的风险

| 风险 | 说明 |
|------|------|
| **栈溢出** | 每次嵌套中断都会在内核栈上压入新的 iret 帧。如果深度嵌套无限制，内核栈（通常只有 8KB/16KB）会溢出导致系统崩溃 |
| **死锁** | 如果低优先级的 ISR 持有一把自旋锁，而高优先级的 ISR 也需要这把锁 → 死锁，因为低优先级 ISR 被暂停不可能释放锁 |
| **优先级反转** | 嵌套的 ISR 可能导致关键的低优先级任务永远得不到执行 |
| **调试困难** | 嵌套中断的状态组合复杂，bug 难以复现 |

> **缓解措施**：Linux 内核对嵌套中断有严格的栈深度检查。中断处理函数中尽量避免使用自旋锁跨越可能嵌套的边界。关键数据结构使用 `irqsave` / `irqrestore` 保护。

### 2.7 中断屏蔽机制详解

#### 2.7.1 CPU 级别：CLI / STI

| 指令 | 全称 | 作用 | 使用限制 |
|------|------|------|----------|
| `CLI` | Clear Interrupt Flag | 清除 RFLAGS 中的 IF 位 —— **关中断**。此后 CPU 不再响应可屏蔽中断（INTR） | 用户态执行会触发 #GP（一般保护异常）——必须在内核态使用 |
| `STI` | Set Interrupt Flag | 设置 RFLAGS 中的 IF 位 —— **开中断**。恢复对可屏蔽中断的响应 | 同上，必须在内核态使用 |

> **CLI/STI 的作用范围**：仅对当前 CPU 核生效。在多 CPU 系统中，`CLI` 在 CPU0 上执行不影响 CPU1 响应中断——这是为什么 SMP 下不能仅靠关中断保护临界区。

#### 2.7.2 中断控制器级别：PIC/APIC 屏蔽

除了 CPU 的 IF 标志，中断控制器也可以屏蔽特定 IRQ 线：

```
传统 8259A PIC:
┌────────────────┐
│ IMR (Interrupt │  ← 中断屏蔽寄存器
│ Mask Register) │     每位对应一条 IRQ 线
│  8 位           │     bit=1 → 屏蔽该 IRQ
└────────────────┘

APIC:
┌────────────────┐
│ Local APIC     │
│ LVT (Local     │  ← 本地向量表，可分别屏蔽
│ Vector Table)  │     每个本地中断源（Timer, LINT0, LINT1, Error, PerfMon）
├────────────────┤
│ I/O APIC       │
│ Redirection    │  ← 重定向表，每项有 mask 位控制对应 IRQ
│ Table (RTE)    │     bit=1 → 屏蔽
└────────────────┘
```

#### 2.7.3 不可屏蔽中断（NMI）

**NMI（Non-Maskable Interrupt, 向量 2）** — 无法通过 `CLI` 或 IF 标志屏蔽。触发场景：

| 场景 | 说明 |
|------|------|
| **硬件错误** | 内存 ECC 不可纠正错误、PCI SERR# 信号 |
| **看门狗定时器** | 用于检测系统死锁（NMI watchdog）：如果 CPU 长时间不响应，触发 NMI 来获取内核栈回溯 |
| **电源故障** | 交流电源掉电时，UPS 通过 NMI 通知 OS 尽快保存关键数据 |
| **性能计数器溢出** | Intel PMU（Performance Monitoring Unit）溢出时可选触发 NMI 或普通中断 |
| **调试** | 在系统 hang 住时，通过 NMI 按钮或 IPMI 命令强制内核打印状态 |

> NMI 的另一个特殊之处：CPU 硬件会自动屏蔽**嵌套的 NMI**——在处理一个 NMI 期间，后续 NMI 会被硬件挂起（pending），直到当前 NMI 执行 `iretq` 返回。这防止了 NMI→NMI 的无限嵌套导致栈溢出。

## Part 3：三个概念的关系总结

```
用户态                              |  核心态
                                    |
用户程序执行                          |
    │                               |
    ├── 需要 OS 服务 ──→ syscall ────→ 内核处理 ──→ sysret ──→ 继续执行
    │                               |
    │                               |  任何时刻：
硬件设备事件 ──────→ 中断 ──────────→ ISR 处理 ──→ iret ────→ 继续被打断的程序
    │                               |
    │                               |
程序出错（除零等）──→ 异常 ──────────→ 异常处理 ──→ iret ────→ （重试 / 终止 / 下一条）
```

**三者统一于"CPU 控制权转移"机制**——无论是系统调用、中断还是异常，CPU 最终都是通过 **IDT 查表**（或 `syscall` 的 MSR 跳转）找到内核处理函数，在核心态完成工作后返回。区别仅在于**触发源**（用户请求 / 外部设备 / CPU 自身）和**返回行为**（返回下一条 / 重试当前 / 不返回）。
