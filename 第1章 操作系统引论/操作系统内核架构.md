# 操作系统内核架构

## 1. 内核（Kernel）vs 外壳（Shell / GUI）

```
┌────────────────────────────────────────────┐
│                  用户态（Ring 3）              │
│  ┌──────────────────────────────────────┐  │
│  │          外壳（Shell / GUI）            │  │  ← 用户接口层
│  │  · 命令行解释器（bash, zsh）             │  │     解析用户输入，调用系统调用
│  │  · 图形桌面环境（GNOME, Explorer）       │  │     不能直接操作硬件
│  └──────────────┬───────────────────────┘  │
│                 │ 系统调用（System Call）      │
│  ╔══════════════╧═══════════════════════╗  │
│  ║         核心态（Ring 0 / Kernel Mode）  ║  │
│  ║  ┌─────────────────────────────────┐  ║  │
│  ║  │       内核（Kernel）               │  ║  │  ← 核心模块
│  ║  │  · 进程调度 · 内存管理              │  ║  │     拥有对硬件的完全控制权
│  ║  │  · 文件系统 · 设备驱动 · 网络栈     │  ║  │     执行特权指令
│  ║  └─────────────────────────────────┘  ║  │
│  ╚════════════════════════════════════════╝  │
│              硬件层                          │
└────────────────────────────────────────────┘
```

| 组件 | 运行级别 | 职责 |
|------|---------|------|
| **内核（Kernel）** | 核心态（Ring 0） | 最底层核心，直接控制 CPU/内存/外设，执行特权指令，处理中断和异常 |
| **外壳（Shell / GUI）** | 用户态（Ring 3） | 将用户命令翻译为系统调用序列，将内核返回结果以人类可读形式呈现 |

> Shell 本身只是一个普通用户态程序（如 `/bin/bash`）——它没有任何特殊权限，所有"特权操作"最终都必须通过系统调用委托给内核完成。

## 2. 用户态 vs 核心态

### 2.1 为什么需要两种运行态

如果所有程序都能执行任意指令、访问任意内存地址，那么：
- 一个恶意程序可以读取其他进程的密码
- 一个 buggy 程序可以覆盖内核数据结构，导致系统崩溃
- 任何程序都能直接操作硬盘控制器，绕过文件权限

**解决方案**：CPU 提供硬件级别的保护机制。将 CPU 运行状态分为至少两个级别，只有最高级别才能执行某些"危险的"指令。

### 2.2 x86 架构的四级特权环

```
     ┌─────────────────────────────┐
     │         Ring 3              │  ← 用户态：普通应用程序
     │    （最低特权级）              │     不能执行特权指令
     │   ┌───────────────────┐     │     只能访问自己的地址空间
     │   │     Ring 2         │     │
     │   │  （设备驱动层）     │     │  ← 历史遗留，现代 OS 很少使用
     │   │ ┌───────────────┐ │     │
     │   │ │   Ring 1       │ │     │
     │   │ │ （系统服务层）  │ │     │  ← 同上，现代 OS 通常只使用 Ring 0 和 Ring 3
     │   │ │ ┌───────────┐ │ │     │
     │   │ │ │ Ring 0    │ │ │     │
     │   │ │ │（核心态）  │ │ │     │  ← 内核运行在此
     │   │ │ │ 最高特权  │ │ │     │     可执行所有指令
     │   │ │ │           │ │ │     │     可访问所有内存
     │   │ │ └───────────┘ │ │     │
     │   │ └───────────────┘ │     │
     │   └───────────────────┘     │
     └─────────────────────────────┘
```

> 现代 OS（Linux、Windows、macOS）实际上**只使用 Ring 0 和 Ring 3 两级特权**。Ring 1 和 Ring 2 在现代 x86-64 架构下基本被废弃。这使得虚拟化技术（VMX root/non-root）可以借用这两个级别。

### 2.3 两种态的本质区别

| 维度 | 用户态（User Mode） | 核心态（Kernel Mode） |
|------|--------------------|-----------------------|
| **CPU 特权级** | Ring 3 | Ring 0 |
| **可执行指令** | 仅非特权指令（普通算术、逻辑、控制流指令） | 全部指令，包括特权指令 |
| **可访问内存** | 仅自身进程的虚拟地址空间（用户空间） | 全部物理内存 + 内核地址空间 |
| **可访问外设** | 不可直接访问——必须通过系统调用→内核代为操作 | 可直接通过 I/O 端口或 MMIO 操作任何设备 |
| **中断响应** | 中断发生时自动切换到核心态处理 | 中断处理程序运行在核心态 |
| **代表程序** | 浏览器、文本编辑器、游戏 | 内核代码、设备驱动、中断服务例程（ISR） |
| **崩溃影响** | 仅当前进程崩溃 | 整个系统崩溃（Kernel Panic / BSOD） |

## 3. 特权指令 vs 非特权指令

### 3.1 定义

| 类型 | 定义 | 执行条件 |
|------|------|---------|
| **特权指令（Privileged Instruction）** | 可能危及系统安全或破坏其他进程数据的指令——如果允许任意程序执行，后果不可控 | **只能在核心态（Ring 0）执行**。用户态程序执行时 CPU 会触发**一般保护异常（General Protection Fault, #GP）** |
| **非特权指令（Non-Privileged Instruction）** | 只影响当前进程自身状态的指令，不会破坏系统或其他进程 | 用户态和核心态都可以执行 |

### 3.2 典型特权指令清单

| 类别 | 指令示例（x86） | 说明 | 为什么必须特权化 |
|------|----------------|------|-----------------|
| **I/O 操作** | `IN`, `OUT`, `INS`, `OUTS` | 直接读写 I/O 端口 | 进程 A 可能通过 `OUT` 指令篡改硬盘控制器配置，破坏文件系统 |
| **中断控制** | `CLI`（关中断）, `STI`（开中断） | 启用/禁止 CPU 响应可屏蔽中断 | 进程 A 执行 `CLI` 关中断后陷入死循环——整个系统永远无法恢复，只能重启 |
| **页表操作** | `MOV CR3, reg` | 加载页表基址寄存器 CR3，切换地址空间 | 进程 A 将 CR3 指向自己的页表——绕过了内核的内存保护，可以读写任意物理地址 |
| **MSR 操作** | `WRMSR` | 写入模型特定寄存器（MSR），控制 CPU 全局行为 | 修改 `MSR_LSTAR`（系统调用入口地址）可以劫持所有系统调用 |
| **停机指令** | `HLT` | 使 CPU 停止执行直到下一个中断 | 普通进程不应该能停下整个 CPU |
| **模式切换** | `IRETQ` | 从中断/异常返回时同时恢复 CS（代码段寄存器）和 RFLAGS，可能改变特权级 | 只能由内核在合适时机执行 |
| **缓存控制** | `INVD`, `WBINVD` | 无效化或回写 CPU 缓存 | 影响全系统性能和数据一致性 |

> **关键点**：用户态程序执行特权指令时，CPU 硬件**直接拒绝**，触发 General Protection Fault（#GP，中断向量号 13），控制权**自动**交还给内核。这就是硬件强制的安全边界。

### 3.3 非特权指令举例

普通算术（`ADD`, `SUB`, `MUL`, `DIV`）、逻辑运算（`AND`, `OR`, `XOR`）、数据传输（`MOV`——在自身地址空间内）、控制转移（`JMP`, `CALL`, `RET`, `Jcc`）、栈操作（`PUSH`, `POP`）——这些都可以在用户态直接执行。

## 4. 用户态 → 核心态切换机制

### 4.1 三种切换触发条件

| 触发条件 | 本质 | 说明 | 谁主动 |
|----------|------|------|--------|
| **系统调用（System Call）** | 用户程序的**主动请求** | 应用程序需要 OS 服务（读写文件、创建进程等），通过 `syscall` / `int 0x80` / `svc` 指令**主动陷入**内核 | 用户程序主动 |
| **中断（Interrupt）** | 外部设备的**异步通知** | 时钟芯片、网卡、键盘等硬件通过 INTR/NMI 引脚通知 CPU——CPU 收到后暂停当前指令流，转入中断处理 | 硬件被动触发（对 CPU 而言异步） |
| **异常（Exception）** | CPU 自身的**同步错误** | 当前指令执行出错——除零、缺页、非法指令、断点——CPU 检测到后立即转入异常处理程序 | 当前指令触发（同步） |

### 4.2 切换时 CPU 硬件做了什么

以 x86-64 架构下执行 `syscall` 指令为例，以下是 CPU 硬件**自动完成**的一系列操作（无需任何软件参与）：

```
执行 syscall 指令前                    执行 syscall 指令后
─────────────────────────            ─────────────────────────
RIP = 用户代码下一条指令地址     →     RIP = MSR_LSTAR（内核系统调用入口地址）
CS   = 用户代码段（Ring 3）      →     CS   = 内核代码段（Ring 0）
SS   = 用户栈段（Ring 3）        →     SS   = 内核栈段（Ring 0）
RSP  = 用户栈指针                →     RSP  = 从 TSS 加载的内核栈指针
RFLAGS = 当前标志                →     RFLAGS = RFLAGS & ~MSR_SFMASK（清除中断使能等标志）
RCX  = （被破坏）                →     RCX  = 用户态的 RIP（保存返回地址）
R11  = （被破坏）                →     R11  = 用户态的 RFLAGS（保存标志寄存器）
```

**逐步骤说明：**

| 步骤 | 硬件操作 | 目的 |
|------|---------|------|
| 1 | 将当前 `RFLAGS` 保存到 `R11` | 保护用户态的标志状态（`IF`、`TF`、`IOPL` 等），供返回时恢复 |
| 2 | 将返回地址（`syscall` 下一条指令地址）保存到 `RCX` | 记录"从哪来"，`sysexit` / `iretq` 返回时需要 |
| 3 | 从 `MSR_LSTAR`（Model-Specific Register）加载新的 `RIP` | 跳转到内核预定义的入口函数（Linux 中是 `entry_SYSCALL_64`） |
| 4 | 从 `MSR_STAR` 加载新的 `CS` 和 `SS` 选择子，将特权级设为 Ring 0 | 切换到内核代码段和内核栈段 |
| 5 | 把 `RFLAGS` 与 `MSR_SFMASK` 的**反码**做按位与（即 `RFLAGS = RFLAGS & ~MSR_SFMASK`） | 屏蔽中断使能位（关中断），确保进入内核后不被外部中断打扰（实际 Linux 中设置允许中断） |
| 6 | 从当前任务的任务状态段（TSS, Task State Segment）加载内核栈指针到 `RSP` | 从用户栈切换到内核栈——防止用户栈被内核数据污染，也防止恶意用户栈导致内核崩溃 |

> **核心理解**：`syscall` 指令做了一次原子性的"特权级跃迁"——在一个指令周期内同时完成了**跳转 + 提权 + 换栈 + 关中断**。这保证用户程序不可能"半途插入"内核执行流。

### 4.3 三种架构的陷入指令对比

| 架构 | 指令 | 助记含义 | 说明 |
|------|------|---------|------|
| **x86 (32-bit)** | `int $0x80` | Interrupt Vector 0x80 | 传统方式，通过中断门进入内核。速度较慢（需要查 IDT、权限检查） |
| **x86-64** | `syscall` / `sysenter` | System Call / System Enter | 快速系统调用指令（Fast System Call），直接使用 MSR 跳转，比 `int 0x80` 快 2-3 倍 |
| **ARM (32-bit)** | `svc #0`（曾用名 `swi`） | Supervisor Call / Software Interrupt | 触发软件中断，CPU 切换到 Supervisor 模式 |
| **ARM (64-bit)** | `svc #0` | Supervisor Call | 同上，进入 EL1（Exception Level 1，即内核态） |

## 5. 三种内核架构详解

### 5.1 架构全景对比

```
┌────────────────────────────────────────────────────────────────┐
│                      宏内核（Monolithic Kernel）                   │
│  ┌────────────────────────────────────────────────────────┐    │
│  │            内核空间（Kernel Space / Ring 0）              │    │
│  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌──────┐ │    │
│  │  │ 文件   │ │ 网络   │ │ 进程   │ │ 内存   │ │ 设备  │ │    │
│  │  │ 系统   │ │ 协议栈  │ │ 调度   │ │ 管理   │ │ 驱动  │ │    │
│  │  └────────┘ └────────┘ └────────┘ └────────┘ └──────┘ │    │
│  │     ↑ 彼此通过函数调用（call/jmp）直接通信，无隔离            │    │
│  └────────────────────────────────────────────────────────┘    │
│  特点：全部功能在内核态——代码耦合紧密，但调用速度最快              │
│  代表：Linux, Windows NT, FreeBSD, macOS (XNU)                 │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│                     微内核（Microkernel）                         │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────────┐   │
│  │  文件服务进程    │  │  网络服务进程    │  │  驱动服务进程    │   │
│  │  (Ring 3)      │  │  (Ring 3)      │  │  (Ring 3)      │   │
│  └───────┬────────┘  └───────┬────────┘  └───────┬────────┘   │
│          │         IPC       │          IPC      │            │
│          └───────────────────┼───────────────────┘            │
│                     ╔════════╧════════╗                       │
│                     ║   微内核 (Ring 0) ║                      │
│                     ║ · 中断处理        ║                      │
│                     ║ · IPC 通信机制    ║                      │
│                     ║ · 地址空间管理     ║                      │
│                     ║ · 基本调度        ║                      │
│                     ╚══════════════════╝                       │
│  特点：内核仅保留最少功能——服务进程在用户态，通过 IPC 通信       │
│  代表：MINIX 3, QNX, L4 家族, seL4                          │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│                    混合内核（Hybrid Kernel）                      │
│  ┌────────────────────────────────────────────────────────┐    │
│  │            内核空间（Ring 0）                             │    │
│  │  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐          │    │
│  │  │ 文件   │ │ 网络   │ │ 进程   │ │ 内存   │            │    │
│  │  │ 系统   │ │ 协议栈  │ │ 调度   │ │ 管理   │            │    │
│  │  └────────┘ └────────┘ └────────┘ └────────┘          │    │
│  │     ↑ 核心服务在内核态（类似宏内核）                      │    │
│  └────────────────────────────────────────────────────────┘    │
│                      ↑ IPC（与用户态服务通信）                    │
│  ┌────────────────────────────────────────────────────────┐    │
│  │            用户空间（Ring 3）                             │    │
│  │  ┌────────┐ ┌────────┐ ┌────────┐                      │    │
│  │  │ 窗口   │ │ USB    │ │ 打印   │    ← 部分非性能敏感的    │    │
│  │  │ 系统   │ │ 驱动   │ │ 服务   │      服务放在用户态      │    │
│  │  └────────┘ └────────┘ └────────┘                      │    │
│  └────────────────────────────────────────────────────────┘    │
│  特点：核心服务在内核态（性能），其他服务可运行在用户态（安全）   │
│  代表：Windows NT 架构（部分驱动可运行在用户态）、macOS XNU     │
└────────────────────────────────────────────────────────────────┘
```

### 5.2 三种架构对比表

| 维度 | 宏内核 | 微内核 | 混合内核 |
|------|--------|--------|----------|
| **内核大小** | 大（数万到数百万行） | 极小（数千到数万行） | 中等 |
| **模块间通信** | 函数调用（一两条 CPU 指令） | IPC/消息传递（数百到数千条 CPU 指令） | 核心模块函数调用，非核心模块 IPC |
| **可靠性** | 低——一个驱动崩溃可能拖垮整个系统 | **最高**——服务进程崩溃后重启即可，内核不受影响 | 中等——核心模块崩溃仍会拖垮系统 |
| **性能** | **最高**——直接函数调用，零上下文切换 | 较低——每次服务请求都涉及多次进程间通信和上下文切换 | 较高——核心路径使用函数调用 |
| **可维护性** | 中等——模块间接口不强制隔离 | **极好**——服务隔离，可独立开发/更新 | 中等——核心部分耦合度高 |
| **安全审计** | 困难——代码量巨大 | **容易**——内核极小，可形式化验证（如 seL4） | 困难——核心部分代码量仍大 |
| **典型代表** | Linux, FreeBSD | QNX, seL4, MINIX 3 | Windows NT, macOS XNU |
| **适用场景** | 追求性能：服务器、桌面、移动 | 追求安全：工控、车载、航空、医疗 | 追求平衡：个人桌面、服务器 |

> **Linux 是宏内核吗？** 是的，但 Linux 通过**可加载内核模块（LKM, Loadable Kernel Module）**获得了类似微内核的扩展灵活性——驱动可以编译为 `.ko` 文件，运行时动态加载/卸载，无需重新编译内核。但这不改变其宏内核本质：所有模块运行在同一个地址空间（Ring 0），`rmmod` 卸载掉的只是一个函数指针表。

## 6. 核心态 ↔ 用户态切换的完整过程

### 6.1 用户态 → 核心态（以 `read()` 系统调用为例）

```
用户程序:  fd = open("file.txt", O_RDONLY)
              │
              ▼
glibc 封装: mov $0, %rax        // rax = 0 (read 的系统调用号)
           mov fd, %rdi          // rdi = 文件描述符
           mov buf, %rsi          // rsi = 缓冲区地址
           mov count, %rdx        // rdx = 读取字节数
           syscall                // ← 触发特权级跃迁
              │
    ╔═════════╧═════════════════════════════════╗
    ║  CPU 硬件自动执行：                         ║
    ║  · RCX ← RIP（保存用户态返回地址）           ║
    ║  · R11 ← RFLAGS（保存标志寄存器）           ║
    ║  · RIP ← MSR_LSTAR（跳转到内核入口）        ║
    ║  · CS  ← 内核代码段选择子（Ring 0）         ║
    ║  · RSP ← 从 TSS 加载的内核栈指针            ║
    ╚═══════════════════════════════════════════╝
              │
              ▼ (现在已在核心态，Ring 0)
entry_SYSCALL_64:   // Linux 内核入口
    保存用户态寄存器到内核栈（pt_regs 结构体）
    调用 do_syscall_64(rdi=pt_regs)
              │
              ▼
do_syscall_64:
    根据 rax 查 sys_call_table[rax] → sys_read
    调用 sys_read(fd, buf, count)
              │
              ▼
sys_read:  // 内核中实际完成 read 的代码
    · 从 fd 查文件描述符表 → file 结构体
    · 检查权限（进程是否有读权限）
    · 调用 VFS（虚拟文件系统）层的 vfs_read()
    · VFS 根据文件类型（ext4 / xfs / nfs）分发到具体实现
    · 从磁盘页高速缓存（Page Cache）读取数据，或触发磁盘 I/O
    · 将数据从内核缓冲区 copy_to_user() 到用户空间 buf
              │
              ▼
返回到 entry_SYSCALL_64:
    恢复用户态寄存器（从 pt_regs）
    执行 sysretq / iretq
              │
    ╔═════════╧═════════════════════════════════╗
    ║  CPU 硬件自动执行（sysretq）：               ║
    ║  · RIP ← RCX（恢复到用户态的下一条指令）    ║
    ║  · RFLAGS ← R11（恢复标志寄存器）          ║
    ║  · CS  ← 用户代码段选择子（Ring 3）        ║
    ║  · RSP ← 用户栈指针                        ║
    ╚═══════════════════════════════════════════╝
              │
              ▼
用户程序:  // 继续执行，rax 中存放返回值（读到的字节数）
```

### 6.2 切换时"什么变了"

| 变化项 | 用户态时 | 核心态时 | 说明 |
|--------|---------|---------|------|
| **CS（代码段寄存器）** | 用户代码段选择子（CPL=3） | 内核代码段选择子（CPL=0） | CPL（Current Privilege Level）即当前特权级 |
| **RIP（指令指针）** | 用户程序中的某条指令地址 | 内核入口函数地址 | 即代码执行流从"用户"切换到"内核" |
| **RSP（栈指针）** | 用户栈（在用户地址空间） | 内核栈（在内核地址空间） | 每个进程都有一个独立的内核栈（通常 8KB 或 16KB） |
| **页表（CR3）** | 进程页表（也可能不改，取决于实现） | 进程页表或内核页表 | x86-64 上，内核空间映射在所有进程页表中保持一致（`0xFFFF800000000000` 起），所以通常**不需要**切换 CR3 |
| **RFLAGS.IF（中断使能）** | 通常为 1（开中断） | 取决于 MSR_SFMASK 配置 | Linux 中 `syscall` 后 IF 保持打开，允许中断嵌套 |
| **GS 段基址** | 指向用户态 TLS（线程局部存储） | 指向内核态的 `per-CPU` 数据结构 | `swapgs` 指令用于切换 |

> **关键要点**：`syscall` 不会切换页表（CR3 不变），因为 Linux 使用了**内核空间恒等映射**——每个进程的页表中，用户空间（0x0 ~ 0x00007FFFFFFFFFFF）各自不同，但内核空间（0xFFFF800000000000 ~ 0xFFFFFFFFFFFFFFFF）在所有进程页表中都指向同一块物理内存。这就是为什么内核可以访问任意进程的地址空间。

## 7. 小结

```
用户态程序要执行"特权操作"的唯一路径：

用户态代码
    │
    │ (不能直接 I/O、不能改页表、不能关中断、不能直接访问硬件)
    │
    ▼
执行 syscall / int / svc 指令  ← 唯一合法的"提权"入口
    │
    ▼
CPU 硬件检查 + 切换特权级 + 换栈 + 跳转内核入口
    │
    ▼
内核验证参数 + 执行操作 + 返回结果
    │
    ▼
执行 sysret / iret / eret 指令 ← 退回用户态
    │
    ▼
用户态代码继续执行（结果在 rax 中）
```

**一句话总结**：用户态与核心态的分离是 OS 安全的基石——它是硬件强制的、不可绕过的边界。用户程序必须通过"门"（系统调用指令）才能进入内核，而内核通过特权指令完全控制硬件。这道"门"是单向的——只能通过特定指令进出，无法从用户态"跳"进内核的任意位置。
