## 一、线程 vs 进程

| 维度 | 进程 (Process) | 线程 (Thread) |
|------|---------------|---------------|
| **资源持有** | 资源分配的基本单位 | 不独立拥有资源，共享进程的资源 |
| **调度** | 传统的 CPU 调度单位 | **现代 OS 中 CPU 调度的基本单位** |
| **地址空间** | 独立；每个进程有自己独立的 `mm_struct` 和页表 | **共享**所属进程的地址空间 |
| **通信** | 需要 IPC 机制（管道、共享内存、消息队列等） | 直接通过共享变量（全局变量、堆上的对象）通信 |
| **创建开销** | 大（需分配 PCB、页表、复制/初始化地址空间） | 小（仅分配栈 + 寄存器上下文 + 一个 `task_struct`） |
| **切换开销** | 大（需切换页表 → 写 CR3 → TLB 刷新） | 小（同进程线程切换无需切换页表 → 不写 CR3 → 不刷新 TLB） |
| **独立性** | 健壮性强：一个进程崩溃不影响其他进程 | 脆弱：一个线程的未处理异常会**杀死整个进程** |
| **系统调用开销** | 大（创建进程有较大内核开销） | 小（`clone()` 只需少量标志位指定共享内容） |

> **一句话总结**：线程是进程内部的**执行流 (Execution Flow)**。一个进程可以包含一个或多个线程，这些线程共享进程的地址空间和绝大多数资源，但各自拥有独立的执行上下文（程序计数器、寄存器、栈）。

## 二、线程的共享与独有资源

### 2.1 所有线程共享的资源

| 共享资源 | 说明 |
|----------|------|
| **代码段 (Text)** | 同一进程的所有线程执行同一份代码（虽然各线程 PC 可能指向不同位置） |
| **数据段 (Data)** | 全局变量和静态变量对所有线程可见。多线程同时写同一个全局变量需要同步 |
| **堆 (Heap)** | `malloc`/`new` 分配的内存所有线程都可访问 |
| **打开的文件** | 文件描述符表是进程级的，所有线程共享。线程 A 打开的文件，线程 B 可以用同一个 fd 读写 |
| **信号处理函数** | 信号处理设置是进程级的，所有线程共享同一套 handler 表 |
| **当前工作目录** | 一个线程 `chdir()` 会影响所有线程 |
| **用户 ID / 组 ID** | 权限相关的 uid/gid 是进程级别的 |
| **进程 PID** | 所有线程共享同一个 PID（但在 Linux NPTL 中，每个线程也有自己的 TID） |

### 2.2 每个线程独享的资源

| 独享资源 | 说明 |
|----------|------|
| **线程 ID (TID)** | Linux 中每个线程有独立的线程 ID（由 `gettid()` 获取），`ps -eLf` 可见 |
| **栈 (Stack)** | 每个线程有自己的用户栈（通常 2-8MB，可通过 `pthread_attr_setstacksize` 设置）。栈保存局部变量、函数调用帧和返回地址 |
| **程序计数器 (PC / RIP)** | 每个线程执行到不同的指令位置 |
| **通用寄存器** | 每个线程独立的计算状态 |
| **处理器状态字 (EFLAGS / RFLAGS)** | 条件码、中断使能等 |
| **信号屏蔽字 (Signal Mask)** | 每个线程可以独立设置阻塞哪些信号 |
| **线程局部存储 (TLS — Thread-Local Storage)** | `__thread` 变量 / `thread_local` 变量。每个线程拥有该变量的独立副本 |
| **errno** | C 语言的 `errno` 是线程局部的（在 TLS 中实现），避免一个线程的错误码覆盖另一个线程的错误码 |

> **内核视角**：在 Linux 中，线程和进程本质上都使用 `task_struct`。创建一个线程 = 调用 `clone()` 并设置共享标志位（`CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD` 等），告诉内核哪些资源应该共享而不是复制。

## 三、多线程模型

### 3.1 多对一模型 (Many-to-One)

```
  用户线程  T1   T2   T3   T4   (用户空间)
                \  |  /
                 内核线程 K1        (内核空间)
```

| 项目 | 说明 |
|------|------|
| **原理** | 多个用户线程映射到**一个**内核线程。线程管理（创建、调度、同步）全部在用户空间的线程库中完成，内核完全感知不到多个用户线程的存在 |
| **调度方式** | 用户态线程库自己实现调度器，在用户线程之间切换。内核只看到一个进程/内核线程 |
| **优势** | 线程切换无需陷入内核（模式切换都不需要），切换极快（~几十 ns）。不消耗额外内核资源（一个进程只对应一个 `task_struct`） |
| **致命缺陷** | **一个线程阻塞，全进程阻塞** — 因为内核只看到一个线程，当该线程因为 `read()` 等系统调用阻塞时，内核会阻塞整个进程，所有用户线程都无法执行 |
| **并行局限** | 无法利用多核 — 用户线程库在一个内核线程上运行，一个时刻只有一条指令被执行 |
| **已废弃** | 早期 Java 的"绿色线程"(Green Threads)、GNU Pth 等使用此模型。现代系统基本不使用 |

### 3.2 一对一模型 (One-to-One)

```
  用户线程  T1    T2    T3    T4   (用户空间)
             |     |     |     |
  内核线程  K1    K2    K3    K4   (内核空间)
```

| 项目 | 说明 |
|------|------|
| **原理** | 每一个用户线程对应**一个独立的内核线程**。内核知道所有线程的存在，并由内核调度器直接调度每个线程 |
| **实现在 Linux** | `pthread_create()` → `clone(CLONE_VM | CLONE_FS | CLONE_FILES | ...)` → 创建一个新的 `task_struct`，与父线程共享地址空间等资源 |
| **优势** | (1) **真并发** — 多线程可在多核心上并行执行。(2) 一个线程阻塞不影响其他线程 — 内核只阻塞该线程对应的 `task_struct`。(3) 时钟中断可以精确到线程级别 |
| **劣势** | (1) 创建/销毁需系统调用（`clone`/`exit`），开销较 ULT 大。(2) 多线程切换需要经过内核调度器。(3) 大量线程会消耗内核资源（每个线程有独立的 `task_struct` + 内核栈） |
| **使用** | **Linux (NPTL)** 和 **Windows** 均采用此模型。所有 POSIX 线程 (pthread) 在现代 Linux 上都是一对一 |

### 3.3 多对多模型 (Many-to-Many)

```
  用户线程  T1   T2   T3   T4   T5   T6   (用户空间)
                 \  |  /    \  |  /
                  K1   K2    K3   K4     (内核空间)
```

| 项目 | 说明 |
|------|------|
| **原理** | M 个用户线程映射到 N 个内核线程（M >= N）。线程库可以在无需内核参与的情况下在用户线程间切换；同时可以分配多个内核线程以实现多核并行 |
| **优势** | 兼顾：善用多核（N 个内核线程可并行），同时用户态切换仍较快。N 可以动态调整 |
| **难点** | 实现极其复杂 — 用户级调度器需要和内核调度器协作，何时将用户线程"绑定"到内核线程、何时"解绑"、如何避免优先级反转等问题极难处理 |
| **历史** | Sun Solaris 的早期线程实现（后被一对一取代）。现代 OS 中基本统一采用一对一模型，因为硬件发展使"过多内核线程"不再是瓶颈（每个线程的内核开销在现代内存/CPU 上可以接受） |

> **现代共识**：一对一模型已成为工业界事实标准。多对多模型的实现复杂性 > 其理论收益，在现代多核、大内存的硬件上不再必要。

## 四、线程实现 — ULT vs KLT 深度对比

### 4.1 用户级线程 (ULT — User-Level Thread)

#### 实现原理

```
 ┌────────────────────────────────────────────────────┐
 │                   用户空间                          │
 │  ┌──────┐  ┌──────┐  ┌──────┐                     │
 │  │ TCB1 │  │ TCB2 │  │ TCB3 │   ← 线程控制块      │
 │  └──┬───┘  └──┬───┘  └──┬───┘   在用户空间分配      │
 │     │         │         │                         │
 │  ┌──┴─────────┴─────────┴────┐                     │
 │  │     用户态线程调度器       │  ← 用户态代码实现   │
 │  └────────────┬──────────────┘                     │
 │               │                                   │
 │  ┌────────────┴──────────────────────────┐         │
 │  │          运行时系统 (Run-Time System)  │         │
 │  └────────────┬──────────────────────────┘         │
 └───────────────┼────────────────────────────────────┘
                 │ 系统调用接口
 ┌───────────────┼────────────────────────────────────┐
 │            内核空间                                │
 │         ┌────┴─────┐                               │
 │         │ PCB / K  │   内核只看到一个进程           │
 │         └──────────┘                               │
 └────────────────────────────────────────────────────┘
```

- **线程控制块 (TCB)** 完全在用户空间分配和管理，内核对此一无所知。
- **线程调度器**运行在用户态：通过保存/恢复 TCB 中的 CPU 上下文在用户线程之间切换。切换可以是合作的（线程主动 yield）或抢占的（使用定时信号如 `SIGALRM`）。
- **用户线程库**（如 POSIX `pthread` 的一个 ULT 实现）通过 `setjmp`/`longjmp` 或 `swapcontext` 等机制实现线程切换。

#### 关键技术细节

| 方面 | 实现方法 |
|------|----------|
| **线程创建** | `malloc` 分配 TCB + 栈空间；初始化 TCB（PC 指向线程函数，SP 指向新栈顶），加入线程就绪列表 |
| **线程切换** | `swapcontext(&old_ctx, &new_ctx)` 或 `sigsetjmp`/`siglongjmp`：保存/恢复 PC、SP、通用寄存器（约几十条指令，纯用户空间，~100 ns 量级） |
| **线程调度** | 用户态调度器维护一个就绪 TCB 链表。策略简单（轮转、优先级）。定期用 `SIGALRM` 信号中断当前线程做抢占 |
| **阻塞问题** | **核心难点**：如果线程调用 `read()` 等阻塞系统调用，整个进程被内核阻塞。解决方案：将阻塞式调用包装成非阻塞 + 用户态事件循环，或使用 "jacket" 技术（拦截系统调用，改为非阻塞） |

#### 优缺点汇总

| 优点 | 缺点 |
|------|------|
| 切换极快：纯用户态操作，不陷入内核，~100ns | **一个线程阻塞 → 全进程阻塞**（致命问题） |
| 不依赖内核：任何 OS 上均可实现，可移植性好 | 无法利用多核并行：内核只调度进程，ULT 只能单核执行 |
| 可创建大量线程：不消耗内核 `task_struct` 资源 | 时钟中断是进程级的：无法对不同线程设置不同调度策略 |
| 线程调度策略可由应用定制，不绑定内核调度器 | 缺页异常导致进程阻塞 → 所有线程一起被阻塞 |

### 4.2 内核级线程 (KLT — Kernel-Level Thread)

#### 实现原理

```
 ┌────────────────────────────────────────────────────┐
 │                   用户空间                          │
 │   T1 (栈/寄存器)   T2 (栈/寄存器)   T3 (栈/寄存器)  │
 └───────┬───────────────┬────────────────┬───────────┘
         │               │                │
 ┌───────┼───────────────┼────────────────┼───────────┐
 │   内核空间            │                │            │
 │  ┌────┴─────┐   ┌────┴─────┐   ┌─────┴────┐       │
 │  │task_struct│   │task_struct│   │task_struct│      │
 │  │ (T1)     │   │ (T2)     │   │ (T3)     │       │
 │  │ mm→       │   │ mm→       │   │ mm→      │      │
 │  └────┬──────┘   └────┬──────┘   └────┬──────┘     │
 │       │               │               │             │
 │       └───────────────┼───────────────┘             │
 │                       │  ┌──────────────────┐      │
 │                  mm_struct│  进程地址空间    │      │
 │                       └──────────────────┘         │
 │                                                    │
 │       ┌─────────────────────────────┐              │
 │       │  内核调度器 (CFS / EEVDF)    │              │
 │       │  将所有 task_struct 平等调度 │              │
 │       └─────────────────────────────┘              │
 └────────────────────────────────────────────────────┘
```

- 每个线程在内核中对应一个独立的 `task_struct`。
- 内核调度器按照线程粒度来调度（CFS 将所有 `task_struct` — 不管是进程还是线程都平等地放入红黑树）。
- 所有与线程相关的管理 — 创建、删除、同步、调度 — 均由内核在特权模式下完成。

#### Linux NPTL 中的具体实现细节

| 系统调用 | 说明 |
|----------|------|
| `clone()` | 创建一个新任务（进程/线程）。通过不同的 flags 组合决定资源共享程度。`clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD | ...)` 创建一个线程。`CLONE_VM` = 共享地址空间，`CLONE_THREAD` = 加入同一线程组 |
| `tgid` (Thread Group ID) | 同一线程组的所有线程共享一个 tgid（对用户表现为 PID）。`getpid()` 返回 tgid；`gettid()` 返回该线程自己的 pid (即 `task_struct->pid`) |
| NPTL (Native POSIX Thread Library) | Linux 的线程库。1:1 模型，一个用户线程 = 一个内核 `task_struct`。用 `futex` 系统调用实现用户态的锁操作（无竞争时完全不陷入内核） |

#### 优缺点汇总

| 优点 | 缺点 |
|------|------|
| 真并发：多线程可在多核上同时执行 | 创建/切换需系统调用（模式切换 + 调度开销） |
| 一个线程阻塞不影响其他线程（内核独立调度） | 每个线程消耗内核资源（`task_struct` ~1.7KB + 内核栈 4-16KB） |
| 时钟中断 / 抢占可达线程级别精度 | 大量线程创建（数万个）会使内核调度开销增大 |
| 多处理器亲和性控制可精确到单个线程 | 移植性依赖内核支持 |

## 五、线程上下文切换 vs 进程上下文切换

### 5.1 详细步骤对比

#### 线程上下文切换（同进程内）

```
步骤1: 触发中断/系统调用 → 硬件保存 SS,RSP,RFLAGS,CS,RIP 到内核栈
步骤2: 内核入口代码保存全部通用寄存器到内核栈
步骤3: 保存内核栈指针到 old_task->thread.sp
步骤4: 加载 new_task->thread.sp 到 RSP
步骤5: 恢复 new_task 的通用寄存器
步骤6: ⚠️ 不写 CR3（页表不变！）
步骤7: ⚠️ 不刷新 TLB
步骤8: iretq / sysret 返回用户态，继续执行新线程
```

#### 进程上下文切换

```
步骤1: 触发中断/系统调用 → 硬件保存 SS,RSP,RFLAGS,CS,RIP 到内核栈
步骤2: 内核入口代码保存全部通用寄存器到内核栈
步骤3: 保存内核栈指针到 old_task->thread.sp
步骤4: 保存 FPU/向量寄存器（懒策略：设置 TS 标志而非直接保存）
步骤5: ⚠️ 切换地址空间：写 new_mm->pgd 到 CR3
步骤6: ⚠️ TLB 被刷新（或 PCID 检查失败，间接开销极大）
步骤7: 加载 new_task->thread.sp 到 RSP
步骤8: 恢复 new_task 的通用寄存器
步骤9: 更新当前 CPU 的 current 变量指向新 task_struct
步骤10: iretq / sysret 返回用户态，新进程继续执行
```

### 5.2 核心区别

```
                   线程切换              进程切换
                   ────────              ────────
写 CR3:              ❌ 不需要             ✅ 需要
TLB 刷新:            ❌ 不刷新             ✅ 刷新 (major cost)
FPU 状态:            可能更频繁            频率较低
Cache 局部性:        好 (共享地址空间)      差 (地址空间独立)
内核栈:              需要更换              需要更换
调度队列:            同一棵红黑树          同一棵红黑树
开销 (无缓存效应):    ~0.5 - 1 μs           ~3 - 10 μs
实际完整开销:         ~2 - 5 μs             ~10 - 100+ μs
```

### 5.3 开销量化 (典型 Linux/x86-64)

| 操作 | 线程切换 | 进程切换 | 说明 |
|------|---------|---------|------|
| 保存/恢复通用寄存器 | ~0.1 μs | ~0.1 μs | 相同 |
| 保存/恢复 FPU | ~1-3 μs | ~1-3 μs | 相同（懒策略下经常跳过） |
| 写 CR3 | **无** | ~0.5-1 μs | 进程切换独有 |
| TLB flush + 后续 miss | **无** | **~5-100+ μs** | 上下文切换最大的隐形开销 |
| Cache 冷启动 | **轻微** | **严重** | 线程共享工作集，进程各自独立 |
| **总计** | **~1-5 μs** | **~10-100+ μs** | 实际差异受工作集大小、缓存层数影响 |

> **结论**：线程切换之所以快，不是因为内核代码少，而是因为它避免了地址空间的切换带来的 TLB 失效和缓存冷启动。在数据密集型应用中（工作集很大），进程切换的真实开销可以远远超过 100 μs。

## 六、线程池 (Thread Pool)

### 6.1 设计原理

线程池是一种**预先创建一组工作线程**，等待从任务队列中获取任务执行的并发模式。它避免为每个短小任务频繁创建/销毁线程的开销。

```
         ┌─────────┐
         │  生产者  │  (主线程或其他线程)
         └────┬────┘
              │ submit(task)
              ▼
    ┌─────────────────────┐
    │   任务队列 (Locked)  │  ← 同步队列 (生产者-消费者模式)
    └──┬──┬──┬──┬──┬──┬──┘
       │  │  │  │  │  │
       ▼  ▼  ▼  ▼  ▼  ▼
    ┌──────────────────────────┐
    │     工作线程池            │
    │  T1  T2  T3  T4  T5 ... │  ← 预先创建，循环等待任务
    └──────────────────────────┘
```

### 6.2 核心组件

| 组件 | 说明 |
|------|------|
| **任务队列 (Work Queue)** | 线程安全的有界/无界队列。生产者（提交任务者）将任务入队；消费者（工作线程）从队首取任务。通常使用互斥锁 + 条件变量保护 |
| **工作线程组** | 预先创建的一组线程，每个线程执行循环：从队列取任务 → 执行 → 回到循环。空闲时在条件变量上阻塞等待 |
| **任务 (Task)** | 抽象的可执行对象。C 中用函数指针 + `void*` 参数；C++/Java 中可以是一个 `std::function` 或 `Runnable`/`Callable` 对象 |
| **同步机制** | 条件变量 (`pthread_cond_wait/signal`) 让空闲线程等待；互斥锁 (`pthread_mutex_t`) 保护任务队列的并发访问。注意惊群问题 (thundering herd) — 只唤醒一个线程而非全部 |

### 6.3 线程生命周期管理

```
                ┌──────────────────────────────────────┐
                │          线程池初始化                  │
                │  创建 core_pool_size 个核心线程        │
                └──────────────┬───────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │   线程主循环         │◄─────────────┐
                    │  while (!shutdown)  │              │
                    │    task = dequeue() │              │
                    │    if (task != NULL)│              │
                    │      task->run()    │──────────────┘
                    │    else if (idle>   │ 任务完成，回到循环
                    │      keep_alive)    │
                    │      exit this thread│ ← 多余线程自我销毁
                    └──────────────────────┘
```

| 生命周期阶段 | 说明 |
|-------------|------|
| **初始化** | 创建 `core_pool_size` 个线程（核心线程数），这些线程在池的整个生命周期内保持存活（即使空闲也不销毁） |
| **任务提交** | 应用层调用 `thread_pool_submit(task)`。若队列未满，task 入队，`pthread_cond_signal` 唤醒一个等待线程。若当前所有线程都在忙且线程数 < `max_pool_size`，则临时创建新线程 |
| **空闲线程回收** | 当线程在 `keep_alive_time` 超时时间内未获取到新任务，且当前线程数 > `core_pool_size` 时，该空闲线程自我终止（`exit`），释放资源 |
| **池关闭** | `thread_pool_shutdown()` 设置 `shutdown` 标志，唤醒所有等待中的线程，每个线程看到 `shutdown == true` 后退出主循环并终止。主线程 `pthread_join` 等待所有工作线程结束 |

### 6.4 线程池大小选取的指导原则

| 场景 | 推荐大小 |
|------|----------|
| **CPU 密集型任务** | `N_cpu` (CPU 核心数) 或 `N_cpu + 1`。过多线程导致上下文切换开销超过并行收益 |
| **I/O 密集型任务** | `N_cpu * (1 + W/C)`，其中 W = I/O 等待时间，C = CPU 计算时间。因为 I/O 等待期间线程被阻塞，CPU 空闲下来，需要有其他线程立即补上 |
| **混合型** | 测量 + 调优。实际中通过线程池的监控指标（活跃线程数、队列长度、任务平均等待时间）迭代调整 |

### 6.5 线程池的优点总结

| 优点 | 说明 |
|------|------|
| **降低创建销毁开销** | 避免了每次任务都调用 `pthread_create()`/`pthread_join()` 的系统调用开销和内核资源分配/释放 |
| **控制并发度** | 限制最大线程数，防止系统因线程爆炸而耗尽内存或出现过载崩溃 |
| **响应速度** | 任务到达时线程已就绪，不需等待线程创建，响应延迟稳定 |
| **任务管理** | 集中的任务队列便于监控、限流和故障恢复 |
