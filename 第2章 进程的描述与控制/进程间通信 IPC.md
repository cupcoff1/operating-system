## 一、概述

由于每个进程拥有独立的地址空间，进程 A 无法直接访问进程 B 的内存。**IPC (Inter-Process Communication，进程间通信)** 是操作系统提供的一组机制，使得不同进程之间能够交换数据和同步操作。

### 六种 IPC 方式总览

| 方式 | 通信类型 | 数据量 | 同步性 | 适用场景 | 性能 |
|------|----------|--------|--------|----------|------|
| **管道 (Pipe)** | 字节流 (半双工) | 中等 | 通常阻塞 | 父子进程 / shell 管道 | 中等 |
| **命名管道 (FIFO)** | 字节流 (半双工) | 中等 | 通常阻塞 | 任意有亲缘关系或无亲缘关系的进程 | 中等 |
| **消息队列** | 结构化报文 | 任意 | 阻塞/非阻塞可选 | 需要保留消息边界的多对多通信 | 中等 |
| **共享内存** | 共享内存区域 | 最大 (GB 级) | 无同步 | 大数据量、高频交换 | **最快** |
| **信号 (Signal)** | 异步通知 | 极少 (一个整数) | 异步 | 异常处理、进程控制 | 极低开销 |
| **Socket** | 字节流/数据报 | 任意 | 阻塞/非阻塞可选 | 跨主机或本地精细通信 | 中等 (本地); 高 (网络) |

## 二、管道 (Pipe)

### 2.1 管道的工作方式

管道是 UNIX 中最古老、最经典的 IPC 方式。本质是**内核空间中的一块固定大小的循环缓冲区 (Circular Buffer)**，两个进程分别通过文件描述符对其进行读写。

```
  写进程                        读进程
    │                             │
    │ write(fd[1], buf, n)       │ read(fd[0], buf, n)
    ▼                             ▲
┌───┴─────────────────────────────┴───┐
│          内核空间                    │
│  ┌────────────────────────────────┐ │
│  │   循环缓冲区 (circular buffer) │ │
│  │   默认容量: PIPE_BUF (4096)   │ │
│  │   实际容量: 16页 (64KB Linux) │ │
│  │                                │ │
│  │  ┌──┬──┬──┬──┬──┬──┬──┬──┐    │ │
│  │  │  │  │  │  │  │  │  │  │    │ │
│  │  └──┴──┴──┴──┴──┴──┴──┴──┘    │ │
│  │   ↑head               ↑tail     │ │
│  └────────────────────────────────┘ │
└─────────────────────────────────────┘
```

### 2.2 内核实现机制

**循环缓冲区 (Circular Buffer)** 数据结构：

```c
// 内核中管道的简化表示
struct pipe_buffer {
    struct page *page;         // 指向物理页
    unsigned int offset;       // 数据在页内的偏移
    unsigned int len;          // 有效数据长度
};

struct pipe_inode_info {
    unsigned int head;         // 下一个生产的缓冲区索引
    unsigned int tail;         // 下一个消费的缓冲区索引
    unsigned int max_usage;    // 最大缓冲页数 (默认 16)
    unsigned int ring_size;    // ring 数组大小
    struct pipe_buffer *bufs;  // 环形缓冲区数组 (每个指向一页)
    unsigned int readers;      // 读端打开计数
    unsigned int writers;      // 写端打开计数
};
```

| 实现细节 | 说明 |
|----------|------|
| **数据结构** | 在内核中以 `pipe_inode_info` 表示。数据存储在一个环形数组 (`pipe_buffer` 数组) 中，每个条目指向一个 4KB 物理页 |
| **容量** | Linux 默认 16 页 = 64KB（早期 UNIX 仅 4KB = `PIPE_BUF`）。可通过 `/proc/sys/fs/pipe-max-size` 调整上限 (默认 1MB) |
| **循环写入** | 写入数据填满一页后分配新页到下一个 slot；读进程读完一页释放一页给写端复用 |
| **文件描述符** | `pipe(int fd[2])` 返回两个 fd：`fd[0]` 为读端，`fd[1]` 为写端。管道本身是一个特殊文件系统 `pipefs` 中的 inode |
| **原子写保证** | 当写入长度 <= `PIPE_BUF`(4096 字节)时，写入是原子的——多个进程并发写入 ≤ PIPE_BUF 的数据不会互相交织。超过 PIPE_BUF 的数据无此保证 |

### 2.3 读写的阻塞语义

| 场景 | 读端行为 | 写端行为 |
|------|----------|----------|
| **管道为空** | **阻塞**：`read()` 阻塞直到有数据或写端关闭 (返回 0) | **不阻塞**：可以写入（前提是缓冲区未满） |
| **管道已满** | **不阻塞**：读取数据释放空间 | **阻塞**：`write()` 阻塞直到有读者读取数据腾出空间 |
| **写端已关闭** | `read()` 返回 **0** (EOF) | N/A |
| **读端已关闭** | N/A | 内核向写进程发送 **SIGPIPE** 信号，`write()` 返回 `-EPIPE`。若进程忽略 SIGPIPE 或阻塞了它，写操作失败返回 EPIPE 错误 |

### 2.4 匿名管道 vs 命名管道

| 维度 | 匿名管道 (Anonymous Pipe) | 命名管道 (FIFO / Named Pipe) |
|------|--------------------------|------------------------------|
| **创建方式** | `pipe(fd)` 系统调用 | `mkfifo(pathname, mode)` 或 shell `mkfifo` 命令 |
| **在文件系统中** | 无名字，不出现在文件系统中 | **有文件名**，作为特殊文件类型出现在文件系统目录中 (`ls -l` 显示为 `prw-------` 等) |
| **进程关系** | 仅限**父子进程**或兄弟进程（fork 后通过继承 fd 共享同一管道） | **任意进程**，只要有路径访问权限即可。通过 `open()` 打开 |
| **生命周期** | 随进程终止自动销毁 | 持久化在文件系统中，即使创建者退出仍存在（直到用 `unlink()` 删除） |
| **双向通信** | 半双工 (单向)。需双向则创建两个 pipe | 半双工 (单向)。虽然规范不禁止全双工，但实际实现中建议单向使用 |
| **典型用例** | `ls | grep foo` 的 shell 管道 | 守护进程与服务客户端之间的命名通信通道 |

### 2.5 管道的限制

1. **半双工**：数据只能单向流动。双向通信需要两个管道。
2. **进程亲缘限制**（匿名管道）：只能用于有共同祖先的进程。
3. **字节流，无消息边界**：管道不保留写入的消息边界。如果写入 "hello" 和 "world" 两次，读端可能一次读出 "helloworld"。应用层需自行实现边界分隔（如换行符、长度前缀）。
4. **容量有限**：环形缓冲区满了之后写端阻塞。

## 三、消息队列 (Message Queue)

### 3.1 核心概念

**消息队列** = 内核维护的一个**消息链表**。进程可以向链表中发送带有类型标记的消息，接收者可以按 FIFO 顺序或按指定类型接收。

与管道的关键区别：**消息队列保留消息边界**。

```
  进程A                           进程B
    │                              │
    │ msgsnd(qid, msg, len, flag) │ msgrcv(qid, msg, len, type, flag)
    ▼                              ▲
┌───┴──────────────────────────────┴───┐
│           内核空间                     │
│  ┌─────────────────────────────────┐ │
│  │  消息队列 (kern_ipc_perm + msq) │ │
│  │                                 │ │
│  │  head → ┌────────────┐          │ │
│  │         │ type=1, data│ ──→ ┌────────────┐
│  │         └────────────┘     │ type=1, data│ ──→ ...
│  │                            └────────────┘
│  │         ┌────────────┐     ┌────────────┐
│  │         │ type=5, data│ ──→ │ type=5, data│ ──→ ...
│  │         └────────────┘     └────────────┘
│  └─────────────────────────────────┘ │
└───────────────────────────────────────┘
```

### 3.2 消息边界 (Message Boundaries)

这是消息队列与管道最本质的区别：

| 特性 | 管道 | 消息队列 |
|------|------|----------|
| **数据类型** | 无结构的字节流 | 有结构的消息（每条消息 = 类型标记 + 数据载荷） |
| **消息边界** | 无边界。写入 3 次 10 字节，读端可一次读出 30 字节 | 有边界。发送 3 条消息，需要用 3 次 `msgrcv` 来接收（即使一次读取也能整条接收） |
| **随机访问** | 不支持。只能顺序 FIFO 读写 | 支持按**消息类型**选择性接收 (`msgrcv` 的 `type` 参数) |
| **适用场景** | 流式数据传输（如文本流） | 结构化通信：请求-响应、任务分发（不同 worker 处理不同类型的消息） |

### 3.3 优先级机制

消息队列的独特能力：接收方可以直接按**消息类型 (type)** 筛选，而不是死板地按 FIFO 接收。

```c
// msgrcv 的 type 参数决定接收行为：
// type = 0  → 接收队列中第一条消息 (FIFO)
// type > 0  → 接收该类型的**第一条**消息
// type < 0  → 接收 <= |type| 的**最小类型**的消息（即优先级最低）
```

**优先级用法示例**：

```
消息类型定义:
  type=1 : 最高优先级 (紧急控制消息)
  type=2 : 普通数据
  type=3 : 后台日志

发送: msgsnd(qid, &msg_of_type_1, ...)   // 插入消息
接收: msgrcv(qid, &msg, len, 1, 0)      // 指定只要 type=1 的消息，优先处理
```

> 这意味着不同优先级的消息处理器可以"跳过"低优先级消息，直接处理紧急消息。

### 3.4 System V vs POSIX 消息队列

| 维度 | System V 消息队列 | POSIX 消息队列 (`mq_*`) |
|------|------------------|--------------------------|
| **接口** | `msgget` / `msgsnd` / `msgrcv` / `msgctl` | `mq_open` / `mq_send` / `mq_receive` / `mq_close` / `mq_unlink` |
| **标识符** | 整数 key (`ftok()` 生成) + 消息队列 ID | 名称字符串（如 `"/myqueue"`） |
| **优先级** | 由 `type` 字段实现 | 有独立的 `prio` 参数 (0 到 `MQ_PRIO_MAX`) |
| **通知机制** | `msgrcv` 阻塞等待 | 支持 `mq_notify()` — 异步通知 (信号或线程) 当队列从空变为非空时 |
| **消息长度** | 无固定上限 | `mq_msgsize` (创建时指定，Linux 默认 8192 字节) |
| **队列容量** | 可配置 | `mq_maxmsg` (最多消息数，Linux 默认 10) |
| **可移植性** | 类 UNIX 系统 (Linux/BSD/Solaris) | POSIX 标准，跨平台 |
| **持久性** | 内核持久 (进程退出后仍存在，直到显式删除或重启) | 内核持久，但 `mq_unlink()` 删除 |

> **推荐**：新项目优先使用 POSIX 消息队列（标准化、API 更简洁、有异步通知）。System V 主要用于遗留代码兼容。

### 3.5 阻塞/非阻塞模式

| 调用 | 阻塞模式 (IPC_NOWAIT 未设置) | 非阻塞模式 (IPC_NOWAIT 设置) |
|------|---------------------------|------------------------------|
| `msgsnd` / `mq_send` | 队列满时阻塞等待空间 | 队列满时立即返回 -1，`errno=EAGAIN` |
| `msgrcv` / `mq_receive` | 队列空时阻塞等待消息 | 队列空时立即返回 -1，`errno=ENOMSG` |

## 四、共享内存 (Shared Memory)

### 4.1 为什么共享内存最快

在所有 IPC 方式中，共享内存是**最快的**。原因在于：

| IPC 方式 | 数据传输路径 | 数据拷贝次数 |
|----------|-------------|-------------|
| 管道 / FIFO | 用户空间 → 内核缓冲区 → 用户空间 | **2 次** (用户→内核，内核→用户) |
| 消息队列 | 用户空间 → 内核消息缓冲区 → 用户空间 | **2 次** |
| Socket | 用户空间 → 内核 Socket 缓冲区 → 用户空间 | **2 次** |
| **共享内存** | 用户空间 ↔ 共享物理页 ↔ 用户空间 | **0 次**（映射后直接读写物理页） |

> 共享内存一旦建立映射，进程可以直接通过指针读写共享区域，**完全没有系统调用**，也没有内核缓冲区中转拷贝。对于大数据量高频交换，这是唯一的实用方案。

### 4.2 页表共享机制 — 内核是如何做到让两个进程看到同一块物理内存的

这是共享内存最核心的原理：

```
  进程A 虚拟地址空间              物理内存              进程B 虚拟地址空间

  ┌──────────────┐                                  ┌──────────────┐
  │  0x00000000  │                                  │  0x00000000  │
  │     ...      │                                  │     ...      │
  │  0x7F100000  │──── 页表映射 ──→ ┌──────────┐ ←── 页表映射 ────│  0x7F200000  │
  │ (共享区起始) │                  │ 共享物理  │                 │ (共享区起始) │
  │     ...      │                  │ 页框 #N   │                 │     ...      │
  │  0x7F110000  │──── 页表映射 ──→ ├──────────┤ ←── 页表映射 ────│  0x7F210000  │
  │ (共享区结束) │                  │ 共享物理  │                 │ (共享区结束) │
  │     ...      │                  │ 页框 #N+1 │                 │     ...      │
  └──────────────┘                  └──────────┘                 └──────────────┘

  关键：两个进程的虚拟地址可能不同（0x7F100000 vs 0x7F200000），
  但它们各自的页表项 (PTE) 指向**同一组物理页框**。
```

| 步骤 | 说明 |
|------|------|
| **1. 内核分配物理页** | `shmget()` 调用时，内核分配若干连续或非连续的物理页框，建立一个 `shmid_kernel` 结构体描述这段共享内存，并标记为"共享" (non-swappable 或 special swap handling) |
| **2. 进程附加 (attach)** | `shmat()` 调用时，内核在进程的虚拟地址空间中找一段未使用的区域，创建 VMA (`vm_area_struct`，设置 `VM_SHARED` 标志)，并在此 VMA 中建立页表条目 (PTE)，使其指向共享的物理页框 |
| **3. 设置 PTE 为可写** | 两个进程的 PTE 同时指向同一物理页，且均设置可写权限位 (W=1)。内核通过 `_PAGE_DIRTY` 位跟踪页面是否被修改过，以便在 swap-out 时写回交换区 |
| **4. 直接读写** | 建立映射后，进程通过普通的 load/store 指令即可直接读写共享数据。**整个过程不经过内核，无非拷贝** |
| **5. 解除映射** | `shmdt()` 删除 VMA 中的相关 PTE，但物理页仍然存在。除非所有附加进程都 detach 且调用 `shmctl(IPC_RMID)` 删除共享段 |

### 4.3 System V 共享内存的完整调用流程

```c
// 1. 创建 / 获取共享内存段
int shmid = shmget(key, size, IPC_CREAT | 0666);

// 2. 将共享内存段映射到本进程的地址空间
//    返回指向共享内存起始位置的指针
void *ptr = shmat(shmid, NULL, 0);   // NULL = 让内核选地址

// 3. 直接读写共享内存（最关键——无需系统调用）
strcpy(ptr, "Hello from Process A!");
printf("Data: %s\n", (char*)ptr);

// 4. 解除本进程与共享内存段的映射
shmdt(ptr);

// 5. (可选，通常由最后一个进程执行) 删除共享内存段
shmctl(shmid, IPC_RMID, NULL);
```

### 4.4 POSIX 共享内存 (`shm_open` + `mmap`)

```c
// 更现代、更符合文件系统的使用方式
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, size);              // 设置大小
void *ptr = mmap(NULL, size, PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);
// 之后 ptr 的使用方式和 System V shmat 完全相同
close(fd);                         // mmap 建立后 fd 可关闭，映射保持
// ... 读写 ptr ...
munmap(ptr, size);                 // 解除映射
shm_unlink("/myshm");              // 删除共享内存对象
```

| 对比 | System V | POSIX |
|------|----------|-------|
| 接口风格 | 独立的一套 API (`shmget`/`shmat`/`shmdt`/`shmctl`) | 类文件系统 API (`shm_open`/`mmap`/`munmap`/`shm_unlink`) |
| 标识符 | 整数 key | 字符串名称 (如 `"/my_shm"`) |
| 可见性 | `ipcs / ipcrm` 查看/管理 | `/dev/shm/` 目录中可见文件 |
| 可移植性 | 类 UNIX 特有 | POSIX 标准，更广 |

### 4.5 共享内存的同步问题（为什么需要配合信号量）

> **共享内存自身不提供任何同步机制。**

这意味着：
- 如果进程 A 正在写 `ptr[100]` 的同时进程 B 正在读 `ptr[100]`，会读到不完整的数据（数据竞争）。
- 进程 B 如何知道进程 A 已经写完一批新数据？

**解决方案：共享内存 + 信号量**才是完整的 IPC 方案：
- **POSIX 信号量** (`sem_open` / `sem_wait` / `sem_post`)
- **System V 信号量** (`semget` / `semop`)
- **互斥锁 + 条件变量**（放在共享内存中，但必须标记 `PTHREAD_PROCESS_SHARED`）

## 五、信号 (Signal)

### 5.1 信号是什么

**信号 (Signal)** = 操作系统向进程发送的**异步通知**。它是一种软件中断，告诉进程"某件事发生了"，进程可以选择如何处理：忽略、捕获（调用自定义处理函数）、或默认处理（终止/忽略/停止）。

信号携带的信息极其有限：仅一个信号编号。因此信号适合**通知**和**进程控制**，不适合大数据传递。

### 5.2 内核实现 — 信号如何递送

```
  发送进程                           接收进程
  kill(pid, SIGUSR1)        ┌─────────────────────────┐
       │                    │  task_struct            │
       ▼                    │  ┌───────────────────┐  │
  ┌──────────┐              │  │ pending signal mask│  │ ← 位图：每个bit对应一个信号
  │ 内核检查  │ ──设置bit───→│  │ (sigpending)       │  │   设置SIGUSR1对应的bit=1
  │ 发送权限  │              │  │ 0x...0000000000010 │  │
  └──────────┘              │  └───────────────────┘  │
                            │                         │
                            │  在返回用户态之前检查:   │
                            │  pending & ~blocked != 0? │
                            │      │ yes              │
                            │      ▼                  │
                            │  调用 do_signal()       │
                            │  查找 sighand->action[] │
                            │  → 调用对应的 handler   │
                            └─────────────────────────┘
```

| 步骤 | 说明 |
|------|------|
| **1. 发送** | 进程 A 调用 `kill(pid_B, SIGUSR1)`。内核检查权限（必须同 uid 或特权进程），然后在进程 B 的 `task_struct->pending` 位图中置 SIGUSR1 对应的 bit 为 1 |
| **2. 检查 blocked** | 如果进程 B 当前阻塞了 SIGUSR1 (`task_struct->blocked` 中对应 bit=1)，信号保持 pending，等待进程 B 解除阻塞 (unblock) |
| **3. 递送时机** | 信号**不**会中断正在执行用户态代码的进程。信号的检查和处理发生在**进程从内核态返回用户态**的路径上（在 `do_signal()` / `exit_to_user_mode_loop()` 中）。换言之：每次系统调用返回、中断返回、调度返回后，内核都会检查 `pending & ~blocked` |
| **4. 调用 handler** | 内核在用户栈上伪造一个栈帧：将当前的 PC/SP 等压入用户栈，然后修改 PC 指向信号处理函数地址。处理函数执行完毕后通过 `sigreturn()` 系统调用恢复原始上下文 |

### 5.3 可靠信号 vs 不可靠信号

这是理解 UNIX 信号机制最关键的区分：

| 维度 | 不可靠信号 (Unreliable) | 可靠信号 (Reliable / Real-Time) |
|------|------------------------|--------------------------------|
| **信号编号** | 1-31 (标准信号) | `SIGRTMIN` ~ `SIGRTMAX` (Linux: 34-64) |
| **排队** | **不排队**。同一个信号被发送多次时，接收方可能只收到一次。内核只记录"有/无"而不是"多少次" | **排队**。每个发送实例都被保留，接收方可以收取全部发送次数 |
| **递送顺序** | 不保证顺序（通常是编号从小到大） | 保证**优先顺序**（编号小的优先） |
| **携带数据** | 仅信号编号本身 | 可附带一个额外数据 (`sigqueue` 发送，通过 `siginfo_t` 中的 `si_value` 传递) |
| **丢失风险** | 高：快速发送同一信号时，后来的信号丢失 | 低：有队列保证，直到队列满 (`RLIMIT_SIGPENDING`) |
| **pid 精确度** | 不精确：无法区分是谁发的同一信号 | 接收方通过 `siginfo_t->si_pid` 精确获知发送者的 PID |

**具体例子**：

```c
// 不可靠信号示例
kill(pid, SIGUSR1);   // 发送第1次 SIGUSR1
kill(pid, SIGUSR1);   // 发送第2次 SIGUSR1
kill(pid, SIGUSR1);   // 发送第3次 SIGUSR1
// 接收方可能只收到 1 次或 2 次 SIGUSR1（因为不排队）

// 可靠信号示例
union sigval val = { .sival_int = 42 };
sigqueue(pid, SIGRTMIN+1, val);   // 发送第1次，携带数据 42
sigqueue(pid, SIGRTMIN+1, val);   // 发送第2次，携带数据 42
sigqueue(pid, SIGRTMIN+1, val);   // 发送第3次，携带数据 42
// 接收方保证收到 3 次 SIGRTMIN+1
```

> **为什么有不可靠信号？** 历史原因——最早 UNIX 信号机制使用简单的位图实现（一个 32-bit word 表示信号 pending），没有队列结构。后来 POSIX 扩展了实时信号 (`SIGRTMIN~SIGRTMAX`) 以支持排队和附带数据。

### 5.4 信号处理函数 (Signal Handler)

#### 三种处理方式

| 方式 | 函数 | 说明 |
|------|------|------|
| **默认** | 不设置 | 每个信号有默认行为：终止 (Term)、忽略 (Ign)、停止 (Stop)、继续 (Cont) |
| **忽略** | `signal(SIG, SIG_IGN)` | 该信号被静默丢弃。**SIGKILL 和 SIGSTOP 不可被忽略或捕获** |
| **捕获** | `signal(SIG, handler)` 或 `sigaction(SIG, &act, NULL)` | 当信号到达时，内核调用指定的函数。`sigaction()` 是 **POSIX 推荐的标准方式**（`signal()` 在不同 UNIX 变体中行为不一致） |

#### sigaction — 推荐设置方式

```c
struct sigaction sa;
sa.sa_handler = my_handler;          // 处理函数
// 或 sa.sa_sigaction = my_handler_3param;  // 三参数版本
sigemptyset(&sa.sa_mask);            // 在 handler 执行期间临时阻塞的信号
sa.sa_flags = SA_RESTART             // 被信号中断的系统调用自动重启
            | SA_SIGINFO;           // 使用三参数版本的 handler
sigaction(SIGUSR1, &sa, NULL);
```

#### handler 中可以安全做什么

信号处理函数运行在一个受限的上下文中，**不是所有函数都安全调用**：

- **安全**：读写 `volatile sig_atomic_t` 全局变量、直接系统调用 `write()` 输出日志、设置标志位、`_exit()` 终止
- **不安全**：`printf()` / `malloc()` / `fopen()` ——这些函数通常不是可重入的（reentrant / async-signal-safe）。在 handler 中调用非异步信号安全的函数可能导致死锁或数据损坏
- **最佳实践**：handler 中只设置一个标志位，主循环中检查标志并真正处理

### 5.5 重要信号一览

| 信号 | 编号 | 默认行为 | 说明 | 可捕获/忽略 |
|------|------|----------|------|------------|
| **SIGKILL** | 9 | Term | 强制杀死进程。最"硬"的信号 | **否** |
| **SIGSTOP** | 19 | Stop | 强制暂停进程 | **否** |
| **SIGTERM** | 15 | Term | 礼貌地请求终止（允许清理） | 是 |
| **SIGINT** | 2 | Term | Ctrl+C 触发。相当于"中断当前操作" | 是 |
| **SIGQUIT** | 3 | Core | Ctrl+\ 触发。产生 core dump 用于调试 | 是 |
| **SIGSEGV** | 11 | Core | 段错误：访问无效内存地址 | 是（但 handler 返回后行为未定义） |
| **SIGCHLD** | 17 | Ign | 子进程状态变化（终止、停止、继续） | 是 |
| **SIGPIPE** | 13 | Term | 向已关闭读端的管道写入 | 是 |
| **SIGHUP** | 1 | Term | 控制终端关闭（常用于守护进程重新加载配置） | 是 |
| **SIGUSR1 / SIGUSR2** | 10/12 | Term | 用户自定义用途 | 是 |
| **SIGALRM** | 14 | Term | `alarm()` / `setitimer()` 定时器到期 | 是 |
| **SIGCONT** | 18 | Cont | 继续已停止的进程 | 是 |

### 5.6 信号与线程的交互

| 问题 | 答案 |
|------|------|
| **信号发给哪个线程？** | 如果信号是**进程定向**的（如 `kill` / 终端 Ctrl+C）：内核发送给线程组中任意一个**不阻塞该信号**的线程。如果信号是**线程定向**的（如 `tgkill`）：发送给指定线程 |
| **信号掩码 (blocked mask)** | 每个线程有**独立**的信号阻塞掩码。`pthread_sigmask()` 设置本线程的 blocked 集 |
| **信号处理函数** | 所有线程**共享**同一套 handler 表（`sighand_struct`）。线程 A 设置了 SIGUSR1 的 handler，线程 B 也能使用 |
| **推荐的信号处理模式** | 在一个专门的线程中阻塞所有信号，用 `sigwait()` / `sigwaitinfo()` **同步**等待信号。这避免了异步信号处理的所有可重入性问题 |

## 六、Socket

### 6.1 Unix Domain Socket vs TCP/UDP Socket

| 维度 | Unix Domain Socket | TCP Socket | UDP Socket |
|------|-------------------|------------|------------|
| **通信范围** | 同一台主机 | 跨主机 (网络) | 跨主机 (网络) |
| **寻址** | 文件系统路径 (`/tmp/app.sock`) | IP + 端口 | IP + 端口 |
| **可靠性** | 可靠，有序，无丢失 | 可靠，有序，无丢失 | 不可靠，无序，可能丢失 |
| **数据边界** | 流 (SOCK_STREAM) 或数据报 (SOCK_DGRAM) | 流 (无边界) | 数据报 (有边界) |
| **性能** | 本地最高效的 Socket 通信 (仅次于共享内存和管道) | 中等 (本地仍需走网络栈) | 较快 (无连接，无拥塞控制) |
| **权限控制** | 文件权限 (`chmod` / `chown`) | 网络防火墙 | 网络防火墙 |
| **辅助数据** | 支持传递文件描述符 (`SCM_RIGHTS`) 和进程凭证 (`SCM_CREDENTIALS`) | 不支持 | 不支持 |

> **Unix Domain Socket 的独特能力**：通过 `SCM_RIGHTS` 辅助消息，一个进程可以将其打开的文件描述符**直接传递给另一个进程**。目标进程的 fd 表会增加一个新条目，指向同一个 `file` 结构体。这是 Apache / Nginx 等服务器在 worker 进程间传递连接 fd 的方式。

### 6.2 Socket 通信通用流程

```
  服务端                        客户端
  socket()                      socket()
     ↓                             ↓
  bind()                          │
     ↓                             ↓
  listen()                     connect() ──────────┐
     ↓                             ↓               │ 三次握手
  accept() ←── 三次握手完成 ──→ (连接建立) ←────────┘
     ↓                             ↓
  read()  ←──────── 数据 ────→ write()
     ↓                             ↓
  write() ───────── 数据 ────→  read()
     ↓                             ↓
  close()                        close()
```

## 七、IPC 方式选择决策树

```
需要大数据量高频传输？
  ├── 是 → 共享内存 + 信号量/互斥锁  (最快，但需自行同步)
  └── 否 → 需要消息边界？
            ├── 是 → 消息队列  (结构化通信，支持按类型接收)
            └── 否 → 通信进程有亲缘关系？
                      ├── 是 → 匿名管道 (简单轻量)
                      └── 否 → 需要跨主机？
                                ├── 是 → Socket (TCP/UDP)
                                └── 否 → 命名管道 (FIFO) / Unix Domain Socket
                                          ├── 需传递 fd → Unix Domain Socket
                                          └── 普通本地通信 → FIFO

特殊情况:
  - 仅需异步通知 (不传数据) → 信号
  - 信号需计数/排队 → 可靠信号 (SIGRTMIN+)
```

## 关键概念速查表

| 概念 | 一句话定义 |
|------|-----------|
| **管道** | 内核空间的循环缓冲区，通过 fd 对实现半双工字节流通信 |
| **FIFO** | 有文件系统中名字的管道，支持任意进程间通信 |
| **消息队列** | 内核维护的消息链表，保留消息边界，支持按类型优先级接收 |
| **共享内存** | 两个进程的页表同时映射同一组物理页框，零拷贝读写 |
| **信号** | 异步软件中断，仅传输一个信号编号，用于通知而非数据传递 |
| **可靠信号** | `SIGRTMIN~SIGRTMAX`，支持排队和附带数据，不会丢失 |
| **不可靠信号** | 标准信号 (1-31)，不排队，快速重复发送会丢失 |
| **SCM_RIGHTS** | Unix Domain Socket 的辅助消息机制，可跨进程传递文件描述符 |
| **PIPE_BUF** | 管道原子写保证的阈值 (4096 字节)。写入不超过此值时多进程写不交织 |
