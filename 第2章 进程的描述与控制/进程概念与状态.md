## 一、前驱图

前驱图 (Precedence Graph) 是描述进程（或程序段、语句）之间执行先后顺序的**有向无环图 (DAG — Directed Acyclic Graph)**。

- **节点** = 进程 / 程序段 / 语句
- **有向边** Pi → Pj = Pi 必须在 Pj 开始之前完成（Pi 是 Pj 的前驱，Pj 是 Pi 的后继）
- **无前驱的节点**可以并发执行
- **有环**则存在循环等待，无法执行（逻辑等同于死锁）

> 前驱图是分析并发程序能否正确执行的静态工具。如果图中存在环，说明该组进程不可能被正确调度完成。

## 二、进程定义

### 2.1 进程的定义

**进程 (Process)** = 一个程序的**执行实例**，是操作系统进行**资源分配**和**独立调度**的基本单位。

进程是动态的，有生命周期（创建 → 执行 → 终止）。同一个程序可以被多个进程同时执行（例如：用户打开了三个终端窗口，每个都是同一个 `bash` 程序的不同进程实例）。

### 2.2 进程映像 (Process Image)

进程在内存中的完整布局称为**进程映像**，由五部分组成：

```
┌─────────────────┐ 高地址
│      栈 (Stack)   │  ← 局部变量、函数调用帧、返回地址
│        ↓         │     (自动增长方向：从高地址向低地址)
│        ⋮         │
│        ↑         │
│      堆 (Heap)    │  ← 动态分配内存 (malloc/new)
├─────────────────┤     (从低地址向高地址增长)
│    数据段 (Data)  │  ← 已初始化的全局变量 / 静态变量
├─────────────────┤     (.data section)
│   程序段 (Text)   │  ← 可执行指令（只读）
└─────────────────┘ 低地址    (.text section)

此外，不在进程地址空间中但在内核空间中：
┌─────────────────┐
│      PCB        │  ← 内核维护的数据结构，描述进程的全部元信息
└─────────────────┘
```

| 组成部分 | 存放内容 | 说明 |
|----------|----------|------|
| **程序段 (Text)** | 编译后的机器指令 | 只读、可共享。同一程序的多个进程实例共享同一份物理页 |
| **数据段 (Data)** | 已初始化全局变量、静态变量 (`.data`)，未初始化全局变量 (`.bss`) | 每个进程独立拥有一份 |
| **堆 (Heap)** | 运行时动态分配的内存 (`malloc`/`new`) | 从低地址向高地址增长 |
| **栈 (Stack)** | 局部变量、函数参数、返回地址、保存的寄存器值 | 从高地址向低地址增长；每个函数调用创建一个栈帧 |
| **PCB** | 进程的元数据 | 存储在内核空间中，用户态不可直接访问 |

> **总结公式**：**进程映像 = 程序段 + 数据段 + 堆 + 栈 + PCB**

### 2.3 进程与程序的区别

| 维度 | 程序 (Program) | 进程 (Process) |
|------|---------------|----------------|
| 本质 | 存放在磁盘上的**静态文件** | 内存中的**动态执行实例** |
| 生命周期 | 永久的（除非被删除） | 暂时的（创建到终止） |
| 组成 | 仅指令与数据 | 程序 + 数据 + PCB + 栈 + 堆 |
| 对应关系 | 一个程序可对应多个进程（如多个 `vi` 实例） | 一个进程只能执行一个程序（exec 可替换） |

## 三、PCB 详解

**PCB (Process Control Block，进程控制块)** 是操作系统用来描述和管理进程的**唯一内核数据结构**。操作系统通过 PCB 获知关于某个进程的一切信息。PCB 的大小和具体字段因 OS 而异，但核心信息的分类是统一的。

> 在 Linux 中，PCB 由 `task_struct` 结构体实现（定义在 `<linux/sched.h>`，约 1.7 KB 大小，包含 100+ 个字段）。

### 3.1 进程标识符

| 字段 | 说明 |
|------|------|
| **PID (Process ID)** | 进程的唯一数字标识符。Linux 中 PID 分配采用延迟复用策略，直到环绕后才复用旧 PID |
| **PPID (Parent PID)** | 父进程的 PID。所有用户进程最终可追溯到 `init` (PID=1) |
| **UID / GID** | 进程所属用户的 ID 和组 ID，用于权限检查 |
| **EUID / EGID** | 有效用户/组 ID，决定实际执行权限（setuid 程序的关键） |
| **进程名称 (comm)** | 可执行文件的名称，最多 15 字符，用于 `ps` 等工具显示 |

### 3.2 处理机状态 (Processor State / CPU Context)

当进程被切换出 CPU 时，其 CPU 寄存器现场必须保存在 PCB 中，以便将来恢复执行。

| 字段 | 说明 |
|------|------|
| **通用寄存器 (General-Purpose Registers)** | EAX/RAX, EBX/RBX, ECX/RCX, EDX/RDX, ESI, EDI 等。保存计算的中间结果和临时数据 |
| **程序计数器 (PC / IP / EIP / RIP)** | 指向进程**下一条即将执行的指令**地址。恢复时必须精确还原 |
| **程序状态字 (PSW / EFLAGS / RFLAGS)** | 包含条件码（ZF, CF, OF, SF 等）、中断使能位 (IF)、I/O 特权级 (IOPL) 等状态标志 |
| **栈指针 (SP / ESP / RSP)** | 指向当前进程内核栈或用户栈的栈顶位置 |
| **帧指针 (BP / EBP / RBP)** | 指向当前函数栈帧的基址，用于访问局部变量和参数 |
| **段寄存器 (CS, DS, SS, ES, FS, GS)** | 在分段内存模型中用于定位代码段、数据段、栈段等（x86-64 的长模式下大部分被忽略，FS/GS 仍用于 TLS 等） |

### 3.3 进程调度信息

| 字段 | 说明 |
|------|------|
| **进程状态 (state)** | 当前所处状态：`TASK_RUNNING`(就绪/运行), `TASK_INTERRUPTIBLE`(可中断睡眠), `TASK_UNINTERRUPTIBLE`(不可中断睡眠), `TASK_STOPPED`, `TASK_ZOMBIE` 等 |
| **优先级 (priority / nice)** | 静态优先级 (nice 值: -20 到 +19，越小优先级越高)；动态优先级 (内核根据运行历史自动调整) |
| **调度策略 (policy)** | `SCHED_NORMAL`(CFS), `SCHED_FIFO`(实时先来先服务), `SCHED_RR`(实时轮转), `SCHED_DEADLINE` 等 |
| **时间片 (time_slice)** | 本次分配到的 CPU 时间额度 |
| **实时优先级 (rt_priority)** | 实时进程专用优先级：1-99，数字越大优先级越高 |
| **CPU 亲和性 (cpus_allowed)** | 位掩码，限制进程只能在指定 CPU 核心上运行 |
| **运行队列指针 (run_list)** | 链接到就绪队列的链表节点 |
| **等待队列指针 (wait_queue)** | 阻塞时链接到等待队列的链表节点 |

### 3.4 内存管理信息

| 字段 | 说明 |
|------|------|
| **代码段/数据段基址与限长** | 分段机制的"基址 & 限长"寄存器对，界定进程合法访问的内存范围 |
| **页表指针 (pgd / mm_struct->pgd)** | 指向进程**顶级页表**（x86-64 的 PML4 基地址）的物理地址。当切换进程时，CR3 寄存器加载此值，完成地址空间的切换 |
| **内存描述符 (mm_struct)** | 包含：代码段起止地址、数据段起止地址、堆边界 (brk)、栈边界、mmap 映射列表等完整地址空间布局信息 |
| **VMA 链表 (vm_area_struct)** | 描述进程地址空间中每个虚拟内存区域（代码、数据、堆、栈、mmap 映射区等）的属性与范围 |
| **brk / start_brk** | 堆 (heap) 的起始和当前边界地址 |

### 3.5 I/O 状态信息

| 字段 | 说明 |
|------|------|
| **打开文件表 (files_struct / fdtable)** | 包含指向每个已打开文件的 `file` 结构体指针数组，索引即为文件描述符 (fd)。`fork` 后父子共享同一套 `file` 对象（引用计数 +1） |
| **当前目录 (fs_struct->pwd)** | 进程的当前工作目录 (cwd) 及根目录 (root)，用于解析相对路径 |
| **文件系统掩码 (umask)** | 创建新文件时的默认权限掩码 |
| **已分配设备列表** | 进程当前占用/打开的设备及其状态（如终端、磁盘、网络套接字） |
| **信号信息 (signal_struct / sighand_struct)** | 信号处理函数表 (sigaction 数组)、待处理信号集 (pending signals)、信号阻塞掩码 (blocked) |
| **TTY 指针** | 关联的控制终端设备 |

## 四、进程状态转换

### 4.1 五状态模型

```
    新建 ──────────→ 就绪 ──────────→ 运行 ──────────→ 终止
   (New)   admit    (Ready) dispatch  (Running)  exit   (Terminated)
                       ↑              ↓    ↓
                       │    event_occurs    │ event_wait
                       └──── 阻塞 ←─────────┘
                            (Blocked / Waiting)
```

> 图中箭头上的标注对应下面的转换编号。

#### 各转换的触发条件

| # | 转换 | 触发条件 (Trigger) | 详细说明 |
|---|------|-------------------|----------|
| 1 | **新建 → 就绪** | 操作系统**接纳 (admit)** 进程 | 做完创建进程的全部工作 (分配 PCB、分配资源、初始化)、PCB 插入就绪队列。通常受系统最大进程数的限制。 |
| 2 | **就绪 → 运行** | **调度程序 (dispatcher) 选中**该进程 | CPU 空闲且该进程在就绪队列中优先级最高/被调度算法选中。发生时机：时钟中断导致时间片到期、I/O 完成唤醒、进程主动 yield 等。 |
| 3 | **运行 → 就绪** | **时间片耗尽 (timer interrupt)** 或被**抢占** | 例如 CFS 调度器下，进程的 vruntime 不再是最小值，调度器选了别的进程。 |
| 4 | **运行 → 阻塞** | 进程**主动等待某事件** | 触发系统调用如 `read()` (等待磁盘 I/O)、`wait()` (等待子进程)、`sleep()`、`sem_wait()` (等待信号量)。进程自愿放弃 CPU。 |
| 5 | **阻塞 → 就绪** | **等待的事件发生** | I/O 完成 (DMA 传输完毕，设备发出中断)、子进程结束 (SIGCHLD)、信号量可用、定时器到期等。由内核的**中断处理程序**或**其他进程**（如调用 `signal()` 或 `sem_post()`）来触发。 |
| 6 | **运行 → 终止** | 进程**执行完毕**或被**杀死** | 正常退出 (`return 0` / `exit(0)`)、异常终止 (除零、段错误)、被其他进程杀死 (`kill -9`)。 |

> **核心理解**：进程自己是**不能**从阻塞变回运行的——只能由阻塞变回就绪，再经调度才能运行。同样，进程**不能**自己从就绪变运行——必须由调度程序分派。

### 4.2 七状态模型（含挂起）

当物理内存不足时，操作系统需要将某些进程或部分进程数据**换出 (swap out)** 到外存（磁盘交换区），腾出空间给活跃进程。被换出的进程处于**挂起 (Suspended)** 状态。

```
                             新建 (New)
                                │ admit
                                ↓
  ┌──────── 挂起 ← (suspend) ─── 就绪 (Ready) ───→ 运行 (Running) ───→ 终止 (Terminated)
  │         ↓                     ↑    ↓              ↓   │               (Terminated)
  │   就绪挂起 (Ready/Suspend)     │    │              │   │
  │         │                     │    │ event_wait    │   │
  │         └── (activate) ───→   │    ↓              │   │
  │                               │   阻塞 (Blocked)    │   │
  │                               │    ↓   │           │   │
  │         ┌── (activate) ───→   │    │ suspend        │   │
  │         ↓                     │    ↓   │           │   │
  └── 阻塞挂起 (Blocked/Suspend) ──┘    │   └──→ 阻塞挂起 (Blocked/Suspend)
                ↑ ← (suspend) ──────────┘
                
                当被等待事件发生时 (event occurs):
                阻塞挂起 ──→ 就绪挂起 (事件发生了但仍在挂起状态)
                就绪挂起 ──activate→ 就绪 (被重新调入内存)
```

#### 挂起的两种形式及适用场景

| 挂起类型 | 定义 | 何时使用 |
|----------|------|----------|
| **就绪挂起 (Ready/Suspend)** | 进程已在就绪队列中等待 CPU，但其代码/数据被换出到外存。只要调入内存即可立刻被调度 | 内存紧张时，内核选择**低优先级或等待时间长的就绪进程**换出。这些进程"万事俱备只欠内存"，一旦被激活 (activate) 就可执行。 |
| **阻塞挂起 (Blocked/Suspend)** | 进程在阻塞等待事件的同时，其映像被换出到外存。即便事件发生，也需先激活才能被调度 | 当内存非常紧张且有大量阻塞进程时（例如：多个进程同时等待慢速 I/O），内核会**优先挂起阻塞进程**——因为它们在事件到来之前本来就不可能执行，换出它们释放的内存可以立刻被其他进程使用。 |

> **优先挂起顺序**：阻塞进程 > 低优先级就绪进程 > 高优先级就绪进程。原因是阻塞进程即使在内存在也无法立即运行。
>
> **就绪挂起的激活路径**：就绪挂起 → (activate, 即换入内存) → 就绪 → (dispatch) → 运行
>
> **阻塞挂起的激活路径**：阻塞挂起 → (event occurs) → 就绪挂起 → (activate) → 就绪 → (dispatch) → 运行

#### 挂起的常见原因

| 原因 | 说明 |
|------|------|
| **内存不足** | 系统和用户进程对内存的总需求超过物理内存容量，必须换出 |
| **用户请求** | 用户主动挂起某个进程以便调试或暂停 |
| **父进程控制** | 父进程挂起某个子进程以检查/修改其状态 |
| **定时换出** | 长期不活动的进程 (例如后台批处理) 被定时换出 |

## 五、进程控制原语

**原语 (Primitive)** = 由关中断/开中断（或其他原子性保证机制，如自旋锁）包裹的**不可分割的操作序列**。进程控制原语不允许被中断打断，从而保证 PCB 等关键数据结构的状态一致性。

> 在多核系统中，仅关中断不够——还需配合自旋锁 (spinlock) 来阻止其他核心同时访问同一数据结构。

### 5.1 创建原语 (Create)

**触发**：用户登录、批处理作业提交、用户程序调用 `fork()`、系统服务请求。

**内核步骤**：

1. **分配唯一 PID**：调用 `alloc_pid()` 获取一个新的 PID，在 pid_namespace 和全局 pidmap 中标记已用。
2. **分配 PCB 空间**：从内核内存池 (slab 分配器) 中分配 `task_struct` 内存块。
3. **初始化 PCB 基本信息**：填入 PID、PPID（设为调用者的 PID）、UID/GID（从父进程继承）、进程状态（设为 `TASK_NEW` / "新建"）。
4. **分配必要的资源**：分配内核栈空间、初始化内存描述符 (`mm_struct`)、建立（或复制）页表。
5. **初始化 CPU 上下文**：设置 PC 指向程序入口地址；设置用户栈指针指向分配的栈空间顶部；PSW/EFLAGS 初始化为默认值（用户态、中断使能）。
6. **继承父进程资源**：复制打开的文件描述符表（`files_struct`，引用计数 +1）、复制信号处理设置、复制当前工作目录。
7. **将 PCB 插入就绪队列**：状态改为 `TASK_RUNNING`（就绪态），加入调度器的就绪队列。
8. **返回新进程的 PID 给父进程**（如果是 `fork()`，则内核返回两次：父进程得子 PID，子进程得 0）。

### 5.2 终止原语 (Terminate)

**触发**：进程正常退出 (`exit()`)、异常终止（段错误/除零）、外部杀死 (`kill -9`)。

**内核步骤**：

1. **关中断**（单核）或获取 PCB 自旋锁。
2. **将进程状态改为 "终止"** (`TASK_DEAD` / `EXIT_ZOMBIE`)，从就绪队列/等待队列中移除。
3. **回收进程持有的资源**：
   - 关闭所有打开的文件描述符（引用计数 -1，若为 0 则真正关闭文件）。
   - 释放分配的内存（减少相关内存段的引用计数）。
   - 断开与共享内存段、信号量、消息队列等 IPC 资源的连接。
4. **将 PCB 的退出码 (exit_code) 写入 PCB**，供父进程通过 `wait()` 获取。
5. **处理子进程**：将该进程的所有子进程**过继给 init 进程** (PID=1)。或者如果采用 `prctl(PR_SET_CHILD_SUBREAPER)` 则过继给最近的 subreaper。
6. **向父进程发送 SIGCHLD 信号**，通知父进程子进程已终止。
7. **进程进入僵尸状态 (TASK_ZOMBIE)**：PCB 保留不释放，等待父进程调用 `wait()` 来回收（读取退出码，释放 PCB）。此时进程已不占用内存、CPU 等资源，只有 PCB 仍在内存中。
8. **若父进程不调用 `wait()`** → 进程长期保持僵尸状态，积累过多可能导致 PID 耗尽。

### 5.3 阻塞原语 (Block)

**触发**：进程主动调用阻塞型系统调用，如 `read()`（设备无数据）、`sem_wait()`（信号量计数为 0）、`wait()`（子进程未结束）。

**内核步骤**：

1. **关中断**，获取 PCB 锁。
2. **保存 CPU 寄存器现场到 PCB**（尤其是 PC 指向系统调用返回后的下一条指令，以便恢复后继续执行）。
3. **将进程状态改为阻塞**：根据阻塞原因选择合适的睡眠状态 (`TASK_INTERRUPTIBLE` 或 `TASK_UNINTERRUPTIBLE`)。
4. **将 PCB 从就绪队列移除，插入对应事件的等待队列**（如：磁盘 I/O 等待队列、信号量等待队列）。
5. **调用调度程序 (schedule())**：选择下一个就绪进程获得 CPU。
6. **开中断**（在调度程序切换到新进程时自动完成）。

> **`TASK_INTERRUPTIBLE`** vs **`TASK_UNINTERRUPTIBLE`**：
> - `TASK_INTERRUPTIBLE`：可被信号唤醒（如 `read()` 时按 Ctrl+C），大多数阻塞使用此状态。
> - `TASK_UNINTERRUPTIBLE`：不可被信号打断，只能由等待的事件唤醒。用于短时关键 I/O（如等待磁盘读取完成），避免信号处理导致数据不一致。缺点是进程杀死不掉（D 状态进程）。

### 5.4 唤醒原语 (Wakeup)

**触发**：等待的事件发生。例如：I/O 完成中断到达、信号量释放、子进程终止信号到达。

**内核步骤**：

1. **关中断**，获取等待队列锁。
2. **从等待队列中取出一个（或全部）等待该事件的 PCB**。具体策略取决于事件类型：多数事件唤醒全部等待者然后由它们竞争；某些事件只需唤醒一个。
3. **将 PCB 状态从阻塞改为就绪** (`TASK_RUNNING`)。
4. **将 PCB 插入调度器的就绪队列**。在内核的 CFS 调度器实现中，这意味着将 `task_struct` 加入红黑树 (vruntime 索引)。
5. **如果该进程优先级高于当前运行进程**，设置 `need_resched` 标志，在返回用户态时触发抢占调度。
6. **释放锁，开中断**。

> **注意**：在唤醒原语完成时，被唤醒的进程**并未**立即获得 CPU——它只是回到了就绪队列。能否立即运行取决于调度器的抢占决策。

### 5.5 挂起原语 (Suspend)

**触发**：内存不足时内核主动挂起，或用户请求挂起。

**内核步骤**：

1. **关中断/加锁**。
2. **将进程状态从当前状态改为对应的挂起状态**：就绪 → 就绪挂起；阻塞 → 阻塞挂起。
3. **将进程的代码段、数据段、堆、栈换出到外存的交换区**。记录交换区中的位置。
4. **释放进程占用的物理页面**，更新页表项（标记为 "不在内存" 或 "在交换区"）。
5. **将 PCB 移到挂起队列**。
6. **如果挂起的是当前运行进程**（极少见，但理论上可能），则调用 `schedule()` 切换。
7. **开中断/解锁**。

### 5.6 激活原语 (Activate)

**触发**：内存充足时，或用户请求激活。

**内核步骤**：

1. **关中断/加锁**。
2. **根据 PCB 中记录的交换区位置，将进程映像从外存读回内存**。分配新的物理页框。
3. **重建页表映射**：更新页表项，将虚拟地址映射到新分配的物理页框。
4. **将进程状态从挂起状态改为对应的非挂起状态**：就绪挂起 → 就绪；阻塞挂起 → 阻塞（进程继续等待之前的事件）。
5. **若状态为就绪**，将 PCB 插入就绪队列。
6. **开中断/解锁**。

## 六、上下文切换

### 6.1 定义

**上下文切换 (Context Switch)** = 将 CPU 从一个进程切换到另一个进程执行的过程。它本质上是"保存旧进程的执行现场 + 加载新进程的执行现场"。

### 6.2 具体步骤

#### 阶段一：保存当前进程上下文

1. **触发**：时间片到期（时钟中断）、进程阻塞（自愿放弃 CPU）、抢占（更高优先级进程就绪）。
2. **陷入内核态**（若当前在用户态）：硬件自动将 SS、RSP、RFLAGS、CS、RIP 压入内核栈。
3. **保存通用寄存器**：内核代码将 RAX、RBX、RCX、RDX、RSI、RDI、RBP、R8~R15 等压入内核栈，或直接存入 PCB 的 `thread_struct` 中。
4. **保存浮点/向量寄存器**：懒保存策略——仅当新进程也使用浮点/向量时，才触发 FXSAVE/ XSAVE 指令保存 XMM / YMM / ZMM 寄存器（避免不必要的保存开销）。
5. **保存内核栈指针**：将当前 RSP 值存入 `task_struct->thread.sp`。
6. **将 `thread_struct` 中的 PC (RIP) 设置为 `__switch_to` 函数中的返回地址**，使得当该进程再次被切换回来时，从 switch_to 之后的下一条指令继续执行。

#### 阶段二：切换地址空间（进程级上下文切换专有）

7. **加载新进程的页表**：将新进程的顶级页表物理地址写入 CR3 寄存器。此操作会**清空 TLB (Translation Lookaside Buffer)**，除非使用 PCID (Process-Context Identifier) 标记 TLB 条目（x86 上自 Westmere 架构起支持）。
8. **切换内存描述符**：`current->active_mm` 指向新进程的 `mm_struct`。

#### 阶段三：恢复新进程上下文

9. **恢复内核栈指针**：RSP = `new_task->thread.sp`。
10. **恢复浮点寄存器**：懒恢复策略——等新进程第一次真正使用浮点单元时才触发 XRSTOR 恢复。
11. **恢复通用寄存器**：从内核栈或新进程的 `thread_struct` 中弹出。
12. **返回用户态**：执行 `iretq`（x86-64）或 `sysret` 指令，硬件自动恢复 SS、RSP、RFLAGS、CS、RIP。执行流从新进程上次被中断的位置继续。

### 6.3 开销量化

| 开销来源 | 典型耗时 | 说明 |
|----------|----------|------|
| **保存/恢复寄存器** | ~0.1-0.5 μs | 几十个通用寄存器，操作本身极快 |
| **切换页表 (写 CR3)** | ~0.5-2 μs | 取决于具体架构和 TLB 策略 |
| **TLB 刷新代价** | **间接开销巨大** | TLB 清空后，后续内存访问需要重新走页表（TLB miss × 页表层级数次），这是上下文切换的隐藏大头 |
| **FPU/向量保存恢复** | ~1-5 μs | 仅在使用浮点/SSE/AVX 时触发；AVX-512 上下文可达 2KB+ |
| **Cache 冷启动** | **几十 μs 到 ms** | 新进程的数据不在 L1/L2/L3 缓存中，导致频繁缓存缺失。这是上下文切换最大的隐藏开销 |
| **总计 (无浮点)** | **~1-10 μs** | 典型 Linux 系统上的测量值 |
| **总计 (实际完整开销)** | **~10-100 μs** | 计入缓存冷启动的影响，实际影响可能远超寄存器保存本身 |

> Linux 内核中，`schedule()` → `context_switch()` → `switch_to()` 的汇编实现通常只需几十条指令。但宏观性能影响主要由缓存/TLB 决定，而非指令数量。

### 6.4 上下文切换 vs 模式切换（重要区分）

这两个概念常被混淆，但它们是**完全不同**的操作：

| 维度 | 上下文切换 (Context Switch) | 模式切换 (Mode Switch) |
|------|---------------------------|------------------------|
| **定义** | 将 CPU 从一个进程切换到另一个进程 | 从用户态切换到内核态（或反向），同一个进程内 |
| **触发** | 调度事件：时间片到期、I/O 阻塞、抢占 | 系统调用、中断、异常（缺页、除零） |
| **发生频率** | 通常 100~1000 次/秒 | 可能**10,000~1,000,000 次/秒**（每个系统调用都伴随模式切换） |
| **保存/恢复内容** | 大量：完整寄存器组、页表、内核栈 | 仅特权级相关：RSP 切换、RFLAGS 部分、CS/SS 段选择子 |
| **开销** | 高（μs 级 + 缓存效应） | 极低（~50-200 ns，主要是 MSR 读写和栈切换） |
| **是否切换地址空间** | **是**（进程级切换时必须写 CR3） | **否**（同一个进程，页表不变） |
| **是否一定要过内核** | **是**（调度程序在内核态运行） | 不适用（本身就发生在跨特权级时） |

> **记忆口诀**：一个系统调用 = 一次模式切换 + 零次上下文切换（除非调用导致进程阻塞或时间片到期）。上下文切换通常伴随两次模式切换（切出旧进程时跳到内核，切入新进程时返回用户），再加上一次内核调度逻辑的执行。

### 6.5 线程上下文切换 vs 进程上下文切换

| 步骤 | 进程上下文切换 | 线程上下文切换（同进程内） |
|------|-------------|--------------------------|
| 保存/恢复通用寄存器 | 需要 | 需要 |
| 保存/恢复 FPU/向量寄存器 | 需要 | 需要 |
| 切换内核栈 | 需要（每个进程独立内核栈） | 需要（每个线程独立内核栈） |
| 写 CR3 切换页表 | **需要** | **不需要**（同一进程的线程共享页表，CR3 不变） |
| TLB 刷新 | **通常需要** | **不需要**（TLB 条目仍然有效） |
| Cache 影响 | 较大（进程间不共享数据） | 较小（线程共享地址空间，数据可能已在 Cache 内） |
| **相对开销** | **基准 (100%)** | **约 10-30%** 的进程切换开销 |

> **核心差异**：线程上下文切换省去了写 CR3 和 TLB 刷新的开销，且缓存的局部性更好。这就是为什么线程被称为"轻量级进程"。

## 七、fork() 与 exec()

### 7.1 fork() 的内核流程

`fork()` 创建一个与父进程**几乎完全相同**的子进程。它在内核中的实现极其精巧。

```
用户态调用: pid_t pid = fork();
                          │
                          ▼
         ┌─────────────────────────────────────┐
         │       内核: sys_fork() → do_fork()   │
         │                                     │
         │  1. 分配新 PID (alloc_pid)           │
         │  2. 复制父进程 PCB (copy_process)    │
         │  3. 复制/共享父进程资源              │
         │  4. 将子进程加入调度队列             │
         │  5. 返回 PID (父子不同)              │
         └─────────────────────────────────────┘
```

**copy_process() 详细步骤**：

1. **复制 PCB (task_struct)**：调用 `dup_task_struct()`，分配新的 `task_struct` 和内核栈（`thread_info` 通常在栈底）。将父进程 `task_struct` 的**绝大多数**字段拷贝到子进程。

2. **复制地址空间 (Copy-On-Write)**：
   - 调用 `copy_mm()`，为子进程创建新的 `mm_struct`。
   - **复制页表层级**（不只是拷贝一个 pgd 指针）：为子进程分配新的顶级页表 (PML4/PGD)，然后逐级复制页表条目。
   - **关键 --- COW 标记**：父子进程的**所有**可写页面（数据段、堆、栈）的页表项 (PTE) 在父子两边**同时**被标记为**只读 (Read-Only)**，并设置 COW 标志位。这样当任一进程尝试写入时，触发**缺页异常 (Page Fault)**，内核在异常处理中才真正分配新物理页、复制数据、更新 PTE 为可写。
   - 代码段 (Text) 因为本来就是只读的，保持原样直接共享同一份物理页。
   - 这意味着 `fork()` 之后，父子进程共享同样的物理页面，直到某方写入。

3. **复制文件描述符表**：子进程的 `files_struct` 指向父进程的同一套 `file` 对象，每个 `file` 的引用计数 +1。父子进程共享文件偏移量（这解释了为什么父子进程向同一文件写入时不会互相覆盖，而是追加）。

4. **初始化子进程特定的字段**：
   - 设置 PPID = 父进程 PID。
   - 清空子进程的待处理信号集。
   - 设置 `clear_child_tid` 等。
   - 将状态设为 `TASK_RUNNING` (就绪)。

5. **在内核栈中伪造一个 "返回现场"**：对于子进程，将内核栈上的返回地址设为 `ret_from_fork`，这让子进程在第一次被调度时跳过系统调用的返回路径，直接跳到用户态入口。子进程的 `eax/rax`（fork 的返回值寄存器）被设为 **0**，父进程的则设为子进程的实际 PID。

#### 为什么 fork() 返回两次？

从内核的角度看，`fork()` 返回两次是因为**在 `copy_process` 返回后**，内核队列中现在有**两个** `task_struct`（父和子）。当调度器有机会运行时，它可能选择其中任何一个。父进程从 `sys_fork` 中返回子进程 PID；子进程在内核栈中被设置了返回值寄存器为 0，当它首次被调度运行并从内核态返回用户态时，用户态看到的就是 0。

### 7.2 exec() 的内核流程

`exec()` 系列函数（`execl`, `execv`, `execve` 等）**用新程序替换当前进程的映像**。与 `fork()` 不同，`exec()` **不创建新进程**。

```
用户态调用: execve("/bin/ls", argv, envp);
                          │
                          ▼
         ┌─────────────────────────────────────┐
         │       内核: sys_execve()            │
         │                                     │
         │  1. 打开可执行文件，验证权限         │
         │  2. 解析 ELF 文件头                 │
         │  3. 释放旧的地址空间                │
         │  4. 建立新的地址空间（代码、数据、  │
         │     堆、栈、mmap 区域）              │
         │  5. 设置新 PC 指向 ELF 入口点       │
         │  6. PCB 大部分字段保留（PID、优先级 │
         │     文件描述符、信号等）             │
         │  7. 返回用户态，从新程序入口执行    │
         └─────────────────────────────────────┘
```

**关键点**：
- PCB **不释放** — PID、PPID、文件描述符表（除非 FD 标记了 `O_CLOEXEC`）、信号处理设置、优先级、nice 值等全部保留。
- 地址空间**完全替换** — 旧的 `mm_struct` 被销毁，新的被建立。页表从零开始创建。
- `exec()` **成功后不返回** (调用它的位置不再被执行)，失败才返回 -1。

### 7.3 vfork() vs fork()

`vfork()` 是一个历史优化，用于 `fork()` + `exec()` 这种常见模式。

| 维度 | `fork()` | `vfork()` |
|------|----------|-----------|
| **地址空间处理** | 利用 COW 复制页表，父子共享物理页 | **不复制页表**，子进程直接**借用**父进程的 `mm_struct` |
| **父进程阻塞** | 否，父子可同时运行 | **是**，父进程被阻塞直到子进程调用 `exec()` 或 `exit()` |
| **子进程能否修改数据** | 可以（触发 COW 页面复制） | **严禁**写任何数据，否则污染父进程的栈和数据 |
| **使用场景** | 通用 | `fork()` + 立即 `exec()` 的优化替代 |
| **当前状态** | 现代 OS 的标准实现 | **几乎过时**。在有 COW 的现代内核中，`fork()` 的性能已足够好，且 `vfork()` 的语义危险（子进程错误修改可能引入极其棘手的 bug），已大部分被 `posix_spawn()` 替代 |

> **历史存在的原因**：早期的 `fork()` 会**立即复制全部地址空间**（物理复制每一页），代价巨大。`vfork()` 通过"借用"父进程地址空间来避免复制。现代内核引入 COW 后，`fork()` 的开销大幅降低，`vfork()` 的优势不再明显。

### 7.4 僵尸进程 (Zombie)

**定义**：子进程已终止，但父进程尚未调用 `wait()`/`waitpid()` 来回收其退出状态。此时子进程的 PCB 仍然存在，但所有的内存资源已释放。`ps` 输出显示为 `<defunct>` 或状态标签 `Z`。

**产生过程**：
```
子进程 exit() → 状态变为 TASK_ZOMBIE → 内核向父进程发送 SIGCHLD
→ 父进程忽略或尚未调用 wait() → PCB 滞留在进程表中
```

**危害**：
- 僵尸进程不消耗 CPU、不占用内存（仅一个 PCB）。
- 但 PID 和 PCB 槽位数量有限（PID 在 32 位初始内核中最大 32767，现代内核中通常 4,194,304 = 2^22）。
- 大量僵尸进程会耗尽 PID 空间，导致无法创建新进程。

**清除方式**：
- 父进程调用 `wait()` / `waitpid()` 读取退出码并释放 PCB。
- 父进程显式忽略 SIGCHLD (`signal(SIGCHLD, SIG_IGN)`)，子进程终止时会被自动回收。
- 父进程先于子进程终止 → 子进程变成孤儿进程 → 被 init 收养 → init 定期调用 `wait()` 来回收。

### 7.5 孤儿进程 (Orphan)

**定义**：父进程先于子进程终止，此时子进程变为"孤儿"。内核会将这些孤儿进程**过继给 init 进程** (PID=1)，由 init 承担父职责。

**过继过程**：
1. 父进程退出时，在 `do_exit()` 中调用 `exit_notify()`。
2. `exit_notify()` 遍历该进程的所有子进程，调用 `reparent_leader()`。
3. 将子进程的 PPID 设为 1 (或最近的 subreaper)。
4. `init` 进程的无限循环中包含 `wait()` 调用，会定期回收已终止的孤儿进程。

> **孤儿进程无害**：因为有 init 兜底回收。但僵尸进程有害 —— 父进程健在却不履行 `wait()` 的职责。

## 关键概念速查表

| 概念 | 一句话定义 |
|------|-----------|
| **进程映像** | 程序段 + 数据段 + 堆 + 栈 + PCB |
| **PCB** | 内核中描述进程全部元信息的唯一数据结构（Linux: `task_struct`） |
| **上下文切换** | 保存当前进程寄存器/页表，加载下一进程的完整过程 |
| **模式切换** | 用户态 ↔ 内核态的切换（系统调用/中断触发），不换进程 |
| **COW** | `fork()` 后父子共享同一物理页，页表标记只读，写入时触发复制 |
| **僵尸进程** | 子进程已终止但父进程未 `wait()` 回收，PCB 残留 |
| **孤儿进程** | 父进程先终止，子进程被 init 收养 |
| **挂起 (Suspend)** | 将进程的映像换出到外存交换区，释放占用的物理页 |
| **原语** | 通过关中断/自旋锁保证原子性的不可分割操作序列 |
