# 经典调度算法

> **名词先懂**：**CPU 阵发期（Burst）** = 进程连续使用 CPU 的一段时间，包括 CPU 阵发和 I/O 阵发交替。调度算法关注的是**CPU 阵发期**的分配顺序。

## 知识框架

```
调度算法
├── 非抢占式
│   ├── FCFS（先来先服务）
│   ├── SJF（短作业优先）
│   ├── HRRN（高响应比优先）
│   └── 优先级调度（非抢占变体）
├── 抢占式
│   ├── SRTF（SJF 抢占版）
│   ├── RR（轮转调度）
│   ├── 优先级调度（抢占变体）
│   └── 多级反馈队列
```

## 共用例题

以下例题统一使用 4 个进程：

| 进程 | 到达时间 | CPU 阵发 |
|------|----------|----------|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 2 | 9 |
| P4 | 3 | 5 |

---

## 1. FCFS — 先来先服务

**名词**：按进程**到达就绪队列**的先后顺序分配 CPU，非抢占。

**特点**：最简单的调度算法，用 FIFO 队列实现。一个进程获得 CPU 后一直运行到完成或阻塞。

### 例题演算（到达顺序 P1→P2→P3→P4）

```
时间 0: P1 到达，CPU 空闲 → P1 运行
时间 1: P2 到达，排队
时间 2: P3 到达，排队
时间 3: P4 到达，排队
时间 8: P1 完成 → P2 运行
时间 12: P2 完成 → P3 运行
时间 21: P3 完成 → P4 运行
时间 26: P4 完成
```

```
甘特图：| P1 (0-8) | P2 (8-12) | P3 (12-21) | P4 (21-26) |
```

| 进程 | 到达 | 执行 | 完成 | 周转时间 | 带权周转 |
|------|------|------|------|----------|----------|
| P1 | 0 | 8 | 8 | 8 | 1.0 |
| P2 | 1 | 4 | 12 | 11 | 2.75 |
| P3 | 2 | 9 | 21 | 19 | 2.11 |
| P4 | 3 | 5 | 26 | 23 | 4.6 |

**平均周转**：(8+11+19+23)/4 = **15.25**
**平均等待**：(0+7+10+18)/4 = **8.75**

> **护航效应（Convoy Effect）**：一个长进程挡在前面，后面所有短进程排队等候，就像护卫舰跟在航母后面。严重拉高平均等待时间。

---

## 2. SJF — 短作业优先

**名词**：每次从就绪队列中选出**估计 CPU 阵发最短**的进程，分配 CPU。非抢占版本也叫 SPF（最短进程优先）。

**特点**：可证明 SJF 在**所有进程同时就绪**时平均等待时间最小（理论最优）。

### 例题演算

```
时间 0: 只有 P1（8），P1 运行
时间 1: P2(4) 到达，但 P1 还在运行，非抢占不切换
时间 2: P3(9) 到达
时间 3: P4(5) 到达
时间 8: P1 完成 → 就绪队列: P2(4), P3(9), P4(5) → 选最短的 P2
时间 12: P2 完成 → 就绪队列: P3(9), P4(5) → 选 P4
时间 17: P4 完成 → 选 P3
时间 26: P3 完成
```

```
甘特图：| P1 (0-8) | P2 (8-12) | P4 (12-17) | P3 (17-26) |
```

| 进程 | 到达 | 执行 | 完成 | 周转 | 带权周转 |
|------|------|------|------|------|----------|
| P1 | 0 | 8 | 8 | 8 | 1.0 |
| P2 | 1 | 4 | 12 | 11 | 2.75 |
| P3 | 2 | 9 | 26 | 24 | 2.67 |
| P4 | 3 | 5 | 17 | 14 | 2.8 |

**平均周转**：14.25（优于 FCFS 的 15.25）
**平均等待**：(0+7+15+9)/4 = **7.75**

> **饥饿（Starvation）**：长进程永远排不过短进程，可能一直得不到 CPU。SJF 需预估 CPU 阵发长度（通常用指数平均法估算）。

---

## 3. SRTF — 最短剩余时间优先

**名词**：SJF 的**抢占版**。每当新进程到达或当前进程完成时，比较所有就绪进程的**剩余 CPU 阵发**，选最短的执行。

**抢占条件**：新到达进程的 CPU 阵发 < 当前进程的剩余时间，则抢占。

### 例题演算

```
时间 0: P1 运行（剩余 8）
时间 1: P2(4) 到达 → 4 < P1剩余8 → 抢占！P1 剩余 7，P2 运行
时间 2: P3(9) 到达 → P2 剩余 3 < 9，不抢占
时间 3: P4(5) 到达 → P2 剩余 2 < 5，不抢占
时间 5: P2 完成（运行了 1~5 = 4）→ 就绪: P1(7), P3(9), P4(5) → 选 P4
时间 10: P4 完成 → 选 P1
时间 17: P1 完成 → 选 P3
时间 26: P3 完成
```

```
甘特图：| P1(0-1) | P2(1-5) | P4(5-10) | P1(10-17) | P3(17-26) |
```

| 进程 | 到达 | 执行 | 完成 | 周转 |
|------|------|------|------|------|
| P1 | 0 | 8 | 17 | 17 |
| P2 | 1 | 4 | 5 | 4 |
| P3 | 2 | 9 | 26 | 24 |
| P4 | 3 | 5 | 10 | 7 |

**平均周转**：(17+4+24+7)/4 = **13.0**（三种中最优）
**平均等待**：((7+3)+0+15+2)/4 = **6.75**

> SRTF 平均等待最小，但上下文切换次数多，且需要知道剩余时间的精确估计。

---

## 4. HRRN — 高响应比优先

**名词**：**响应比 R** = (等待时间 + 要求服务时间) / 要求服务时间 = 1 + 等待时间/服务时间。

- R = 1：刚到达，只服务时间
- R > 1：等得越久 R 越大，优先级越高
- 非抢占：每次 CPU 空闲时，计算所有就绪进程的 R，选最大者

**设计思想**：FCFS 偏向长进程，SJF 偏向短进程。HRRN 折中——短作业天然 R 大（分母小），长作业等久了 R 也会变大获得补偿。

### 例题演算

```
时间 0: P1 运行（只有 P1）

时间 8: P1 完成，就绪: P2, P3, P4
  响应比计算（等待了多久 / 需要多久）：
  P2: 1 + (8-1)/4 = 1 + 7/4 = 2.75
  P3: 1 + (8-2)/9 = 1 + 6/9 = 1.67
  P4: 1 + (8-3)/5 = 1 + 5/5 = 2.0
  → R(P2)=2.75 最大 → P2 运行

时间 12: P2 完成，就绪: P3, P4
  P3: 1 + (12-2)/9 = 1 + 10/9 = 2.11
  P4: 1 + (12-3)/5 = 1 + 9/5 = 2.8
  → P4 运行

时间 17: P4 完成 → P3 运行
时间 26: P3 完成
```

| 进程 | 到达 | 完成 | 周转 |
|------|------|------|------|
| P1 | 0 | 8 | 8 |
| P2 | 1 | 12 | 11 |
| P3 | 2 | 26 | 24 |
| P4 | 3 | 17 | 14 |

**平均周转**：14.25。结果接近 SJF 但避免了长作业饥饿。

## 5. 优先级调度

**名词**：每个进程分配一个**优先级**（整数，通常越小越高），CPU 分配给最高优先级的就绪进程。

- **静态优先级**：创建时确定，不变
- **动态优先级**：运行中可调整

### 抢占 vs 非抢占

| 类型 | 说明 |
|------|------|
| 非抢占 | 高优先级到达时，等当前进程自愿放弃 CPU |
| 抢占 | 高优先级到达立即抢占当前进程 |

> **Aging（老化）**：随时间推移逐渐提高等待中进程的优先级，防止低优先级进程**饥饿**。是静态优先级到动态优先级的升级。

### 例题（非抢占，优先级 P3>P4>P1>P2，数字越小越高）

假设优先级：P3=1, P4=2, P1=3, P2=4

```
时间 0: P1 运行（只有 P1）
时间 8: P1 完成 → 就绪 P2(pri4), P3(pri1), P4(pri2) → P3 最高优先
时间 17: P3 完成 → P4 运行
时间 22: P4 完成 → P2 运行
时间 26: P2 完成
```

> 优先级反转：高优先级进程等低优先级进程释放资源，中等优先级进程反而先执行。解法：**优先级继承**。

## 6. RR — 轮转调度

**名词**：每个进程分配固定**时间片 q**（也叫时间量子），运行 q 后被抢占，排到队尾。

- q 太小 → 上下文切换过多，CPU 有效利用率下降
- q 太大 → 退化为 FCFS
- 经验值：10~100ms，至少大于上下文切换时间（~10μs）

### 例题演算（q=4）

```
就绪队列初始: P1
时间 0-4: P1 执行 4（剩余 4）
  P2(1), P3(2), P4(3) 陆续到达
时间 4: P1 时间片到 → 排到队尾
  队列: P2(4), P3(9), P4(5), P1(4)
时间 4-8: P2 执行 4（刚好完成）
时间 8-12: P3 执行 4（剩余 5）
  队列: P4(5), P1(4), P3(5)
时间 12-16: P4 执行 4（剩余 1）
  队列: P1(4), P3(5), P4(1)
时间 16-20: P1 执行 4（刚好完成）
时间 20-24: P3 执行 4（剩余 1）
时间 24-25: P4 执行 1（完成）
时间 25-26: P3 执行 1（完成）
```

| 进程 | 完成 | 周转 |
|------|------|------|
| P1 | 20 | 20 |
| P2 | 8 | 7 |
| P3 | 26 | 24 |
| P4 | 25 | 22 |

**平均周转**：(20+7+24+22)/4 = **18.25**。周转时间比之前都大，但**响应时间**最小——每个进程都能快速获得首次 CPU。

> **RR 的核心价值**：追求**响应时间**最短，适合**分时系统**。不适合需要长连续 CPU 的计算密集型任务。

---

## 7. 多级队列（Multilevel Queue）

**名词**：将就绪队列按进程类型**永久性**地分为多个独立队列，每个队列有自己的调度算法，队列之间有严格的**优先级**。

### 与多级反馈队列的关键区别

| | 多级队列 | 多级反馈队列 |
|------|----------|-------------|
| 进程移动 | **固定**——进程一旦分配进某个队列就不再迁移 | **动态**——进程可跨队列升降级 |
| 适应性 | 无，需要人工分类 | 有，自动根据行为调整 |

### 队列划分

```
系统进程 ──→ Q0（最高优先级，RR）
交互进程 ──→ Q1（RR）
批处理 ──→ Q2（FCFS，最低优先级）
```

### 队列间调度

- **固定优先级抢占**：高优先级队列非空时绝不调度低优先级。交互进程可能饿死批处理。
- **时间片分配**：给每个队列分配 CPU 时间比例（如 Q0 80%、Q1 15%、Q2 5%），防止低优先级完全饿死。

> 多级队列是**静态**分类策略，依赖预先知道进程类型。现代通用 OS 更倾向用多级反馈队列自适应。

---

## 8. 多级反馈队列

**名词**：将就绪队列分为多个优先级的子队列，新进程进入最高优先级队列；用光时间片未完成则降级。综合 FCFS、RR、优先级调度的优点，是**现代通用 OS 的标准调度器**。

### 框架结构

```
          ┌── 新进程 ──┐
          │            │
          ↓            │ CPU 用完 → 降级
  ┌─────────────┐      │
  │ Q0: RR q=8  │ ─────┘
  │ 最高优先级   │
  └─────────────┘
          │ CPU 用完 → 降级
          ↓
  ┌─────────────┐
  │ Q1: RR q=16 │
  │ 中等优先级   │
  └─────────────┘
          │ CPU 用完 → 降级
          ↓
  ┌─────────────┐
  │ Q2: FCFS    │
  │ 最低优先级   │
  └─────────────┘
```

### 四条核心规则

1. 优先级 Q0 > Q1 > Q2，高优先级队列非空时不调度低优先级
2. Q0 用 RR（q=8），短作业快速完成
3. 用光时间片未完成 → 降级到下一队列
4. 最低层用 FCFS，保证长作业最终完成

### 为什么好？

- **短作业**在 Q0 就完成了，周转时间接近 SJF
- **I/O 密集型**经常用不完时间片就被阻塞，停留在高优先级 → 快速响应
- **CPU 密集型**逐步降级，但最终在底层获得 CPU → 不饥饿
- 无需预估 CPU 阵发长度

## 算法对比

| 算法 | 抢占 | 平均周转 | 平均等待 | 响应时间 | 饥饿 | 需预知 CPU |
|------|------|----------|----------|----------|------|------------|
| FCFS | 否 | 差 | 差 | 差 | 无 | 否 |
| SJF | 否 | 优 | 优 | 差 | 长作业 | **是** |
| SRTF | 是 | 最优 | 最优 | 差 | 长作业 | **是** |
| HRRN | 否 | 良 | 良 | 差 | 无 | 是 |
| 优先级 | 均可 | 中 | 中 | 中 | 低优先级 | 否 |
| RR | 是 | 偏大 | 偏大 | **优** | 无 | 否 |
| 多级队列 | 均可 | 中 | 中 | 中 | 低优先级 | 否 |
| 多级反馈 | 是 | 良 | 良 | 优 | 无 | **否** |

> **考试核心**：多级反馈队列是唯一同时做到"周转好、响应快、不饥饿、不需预估"的算法，也是实际 OS 的首选。多级队列是多级反馈的简化静态版——进程不可跨队列迁移。
