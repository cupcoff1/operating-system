## 磁盘物理结构

### 基本术语定义

- **磁盘（Disk）**：由一组盘片叠加而成的存储设备，内部包含多个盘面。
- **盘面（Platter）**：单个磁性圆盘，数据记录在盘面表面的磁性介质上。一个盘片有两个盘面（上、下）。
- **磁道（Track）**：盘面上以主轴为圆心的同心圆环，数据按磁道组织。外圈磁道比内圈磁道长，现代磁盘利用此特性在外圈存储更多扇区（ZBR, Zone Bit Recording，区域位密度记录）。
- **扇区（Sector）**：磁道上的一段弧，是磁盘读写的最小可寻址单元，传统大小为 **512 字节**，现代磁盘多采用 **4KB**（Advanced Format, 高级格式化）。
- **磁头（Head）**：每个盘面配有一个磁头，负责该盘面的读写，磁头在盘面上方以气垫方式浮动（约几个纳米间距），不与盘面接触。
- **柱面（Cylinder）**：所有盘片上同一半径位置的磁道集合。柱面编号相同意味着可同时通过切换磁头读写而无需移动磁臂，故柱面是许多磁盘调度算法的组织单位。

### 存储容量

$$
\text{存储容量} = \text{柱面数} \times \text{磁头数} \times \text{每道扇区数} \times \text{每扇区字节数}
$$

### 磁盘访问时间模型

一次磁盘 I/O 的总延迟由三部分组成：

$$
T_{\text{access}} = T_{\text{seek}} + T_{\text{rotational}} + T_{\text{transfer}}
$$

| 组成部分 | 符号 | 典型值 | 含义 |
|----------|------|--------|------|
| **寻道时间** | $T_{\text{seek}}$ | 2~20ms（HDD） | 磁臂从当前磁道移动到目标磁道所需的时间。由加速—匀速—减速三段组成，跨柱面越多时间越长。 |
| **旋转延迟** | $T_{\text{rotational}}$ | 半圈平均（7200RPM 约 4.2ms） | 磁盘旋转使目标扇区到达磁头下方的时间。平均为半圈时间。 |
| **传输时间** | $T_{\text{transfer}}$ | 极短（微秒级） | 实际从扇区读取或写入数据的时间，由旋转速度和数据量决定。 |

> **核心结论**：寻道时间是机械运动，在三个因子中占绝对主导（数个毫秒 vs 微秒级）。因此磁盘调度算法的核心目标是**最小化磁头移动距离**以降低寻道时间。

## 磁盘格式化

磁盘从出厂到能存储文件需经历三级处理：

### 第一阶段：低级格式化（Low-Level Formatting）

工厂或专业工具完成。在空白的盘面上划分物理扇区边界：

- 在每个扇区写入**扇区头**（含扇区编号、柱面和磁头信息）与**扇区尾**（校验码/ECC）
- 填充**前导码**、**同步字节**和**间隔**（Gap）用于控制器识别和对齐
- 为每个扇区预留 **ECC（Error Correction Code，纠错码）** 字段，用于检测和纠正介质的位错误
- 工厂还会标记并映射**坏块**（Bad Block），在固件层将坏块重映射到备用扇区

> 现代 HDD 中低级格式化不可由终端用户执行，坏块管理由磁盘固件内部完成（P-list 和 G-list）。

### 第二阶段：分区（Partitioning）

将整块磁盘划分为一个或多个独立的逻辑区域（分区，Partition）：

- 分区表记录每个分区的**起始扇区**、**大小**和**类型**
- 传统 **MBR（Master Boot Record，主引导记录）**：位于 0 扇区，支持最多 4 个主分区，最大 2TB
- 现代 **GPT（GUID Partition Table）**：UEFI 标准，支持最多 128 分区，最大 9.4ZB，含 CRC 校验和备份表
- 分区可以挂载为独立的文件系统（Linux 下 /boot、/home、/ 分离分区）

### 第三阶段：逻辑格式化（Logical Formatting）

在分区上建立文件系统元数据结构：

- 创建**根目录**（Root Directory）
- 初始化**空闲空间管理结构**（空闲块位图 / 空闲链表 / 成组链接，见第 8 章）
- 写入**超级块**（Superblock）：文件系统类型、块大小、inode 总数、空闲块数
- 建立 **inode 表**（Unix/Linux）或 **FAT 表**（Windows）

## 外存组织方式

文件的物理存储组织方式，详见 [[文件结构]]。此处仅列出三种基本方式：

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **连续组织（Contiguous）** | 文件占连续的物理块 | 顺序读写快，寻道少 | 外部碎片严重，扩展困难 |
| **链接组织（FAT, Linked）** | 每个块含有下一块的指针 | 无外部碎片，易于扩展 | 随机访问慢（需遍历链），FAT 大文件占用内存 |
| **索引组织（inode, Indexed）** | 单独的索引块存储文件所有块指针 | 支持大文件（直接/一级/二级间接块），随机访问 O(1) | 间接块增加一次 I/O；小文件浪费索引块空间 |

## 提高磁盘 I/O 速度

| 技术 | 原理 | 关键细节 |
|------|------|----------|
| **磁盘高速缓存（Disk Cache）** | 在内存中开辟缓冲区，缓存近期访问的磁盘块 | 利用局部性原理；需要替换算法（LRU、Clock）；OS 和磁盘控制器两级缓存。写回 vs 写透策略影响一致性与性能。 |
| **提前读（Read-Ahead）** | 检测到顺序访问模式时，预先将后续块读入缓存 | 空间局部性利用；典型预取窗口为 16~64KB。访问模式检测失败（随机访问）则预取无效。 |
| **延迟写（Delayed Write）** | 写操作暂存缓存中，积攒一定量后批量写回磁盘 | 减少磁盘 I/O 次数；风险是**掉电丢失**——脏块未写回的数据永久丢失。配合 UPS 或文件系统日志（journaling）缓解。 |
| **虚拟盘（RAM Disk）** | 将一部分内存模拟为块设备 | 极速（内存速度），掉电全失；适合编译缓存、临时文件。Linux 为 ramfs/tmpfs。 |
| **RAID（Redundant Array of Independent Disks，独立磁盘冗余阵列）** | 多块磁盘协作提供并行性和冗余 | 见下节详述。 |

## RAID 级别详解

**RAID（Redundant Array of Independent Disks，独立磁盘冗余阵列）** 的核心思想：将多块低成本磁盘组合成一个逻辑盘，通过**条带化**（Striping）并行读写提升性能，通过**奇偶校验**（Parity）或**镜像**（Mirroring）提供冗余容错。

### 基础概念定义

- **条带化（Striping）**：将连续数据按固定大小（条带块/Chunk Size，常见 64KB~256KB）交替分布到多块磁盘。例如数据块 0 在磁盘 0，块 1 在磁盘 1，块 2 在磁盘 2，块 3 在磁盘 3。
- **奇偶校验（Parity）**：对数据块的各位进行异或（XOR）运算，得到一个校验值。若某块磁盘丢失，可以利用剩余数据块与校验值通过 XOR 重建丢失的数据。$P = D_0 \oplus D_1 \oplus D_2 \oplus \dots$，反过来 $D_k = P \oplus \sum_{i \neq k} D_i$。
- **镜像（Mirroring）**：将相同数据写入两块磁盘，提供 1:1 冗余。
- **n-1 容错**：RAID 组可以容忍 n 块磁盘同时故障而**不丢失数据**。

---

### RAID 0：条带化（Striping）

**数据布局**：数据以条带块为单位交替分布于 N 块磁盘，**无任何冗余数据**。

```
磁盘 0: 块 0 | 块 4 | 块 8  | ...
磁盘 1: 块 1 | 块 5 | 块 9  | ...
磁盘 2: 块 2 | 块 6 | 块 10 | ...
磁盘 3: 块 3 | 块 7 | 块 11 | ...
```

| 维度 | 评估 |
|------|------|
| 容错能力 | **0 块磁盘** — 任何一块磁盘故障，所有数据不可恢复 |
| 读性能 | **优秀** — 全部磁盘可并行读取，聚合带宽接近 N × 单盘带宽 |
| 写性能 | **优秀** — 无校验计算，全部磁盘并行写入，聚合带宽 N × 单盘带宽 |
| 有效容量 | N × 单盘容量（100% 可用） |
| 最小磁盘数 | 2 |

> 适用场景：视频渲染、科学计算的临时/中间数据，对速度要求高但对可靠性不敏感。

---

### RAID 1：镜像（Mirroring）

**数据布局**：每块数据同时写入两块磁盘（主盘和镜像盘），两块盘内容完全一致。

```
磁盘 0: 块 0 | 块 1 | 块 2 | 块 3 | ...  (主盘)
磁盘 1: 块 0 | 块 1 | 块 2 | 块 3 | ...  (镜像)
```

| 维度 | 评估 |
|------|------|
| 容错能力 | **至少 1 块（即一块镜像盘）** — 实际上最多可容忍 N-1 块（只要每组镜像至少存活一块） |
| 读性能 | **好** — 可向两块盘并发分发不同读请求，读带宽约 2×；某些实现仅从主盘读 |
| 写性能 | **与单盘持平（甚至略差）** — 同一数据需写两次，写带宽 = 单盘带宽 |
| 有效容量 | 总容量的 50% |
| 最小磁盘数 | 2 |

> 写性能的关键瓶颈：两个副本都需写完成才算成功（写穿透），不能异步，否则不一致时没法恢复。

---

### RAID 5：分布式奇偶校验（Distributed Parity）

**数据布局**：数据和奇偶校验块轮流分布在所有磁盘上（**奇偶校验不固定在某一块盘**，避免校验盘成为写热点）。每个条带包含 N-1 个数据块 + 1 个校验块。校验值 $P = D_0 \oplus D_1 \oplus \dots \oplus D_{N-2}$。

```
      条带 0        条带 1        条带 2
磁盘0:  D0            D3            P2
磁盘1:  D1            P1            D4
磁盘2:  P0            D2            D5
```

| 维度 | 评估 |
|------|------|
| 容错能力 | **1 块磁盘** — 任意一块故障后，可用剩余数据 + 校验通过 XOR 在线重建 |
| 读性能 | **好** — 所有磁盘可参与并行读（除校验盘外 N-1 块并行），聚合读带宽 ≈ (N-1)×单盘 |
| 写性能 | **有代价（Read-Modify-Write）** — 写惩罚：每次小写需要 2 读 + 2 写 + XOR 计算。流程详见下文 |
| 有效容量 | (N-1)/N × 总容量 — 每 N 块盘有 1 块的容量用于校验 |
| 最小磁盘数 | 3 |

#### RAID 5 写惩罚（Write Penalty）详解

当要更新单个数据块 $D_0'$ 时，不能只写新数据和校验块（因为旧校验 $P$ 包含所有数据块的 XOR，不知道旧 $D_0$ 的值则无法算出新 $P'$）。两种策略：

**方法一：Read-Modify-Write（小写优化）**
1. 读出旧数据块 $D_0$ 和旧校验块 $P$
2. 计算新校验：$P' = P \oplus D_0 \oplus D_0'$（仅 XOR 差异）
3. 写入新数据 $D_0'$ 和新校验 $P'$
4. 总共：**2 次读 + 2 次写**

**方法二：Reconstruct-Write（满条带写）**
1. 若整个条带数据都需更新，读出其余 N-2 个旧数据块
2. 计算新校验 $P' = D_0' \oplus D_1' \oplus \dots \oplus D_{N-2}'$
3. 写入所有数据块和新校验块
4. 总共：**(N-2) 次读 + (N-1) 次写**

> 因此 RAID 5 适合**读多写少**的场景，随机小写密集的工作负载（如数据库 OLTP）会受到写惩罚的严重影响。

#### RAID 5 降级模式（Degraded Mode）

一块盘故障后，RAID 组进入**降级模式**：
- 读操作：若目标块在故障盘上，需读取剩余 N-1 块盘的同条带数据并 XOR 计算重建，**读惩罚**。
- 写操作：对故障盘对应的块直接忽略（无需写入），仅写其余盘 + 更新校验。
- 用户感知：性能下降，但仍可使用。

---

### RAID 6：双重奇偶校验（Dual Parity）

**数据布局**：每个条带含 N-2 个数据块 + 2 个校验块（通常记为 P 和 Q）。P 沿用 XOR，Q 采用 Reed-Solomon 编码（或 RAID-6 专用编码如 EVENODD、RDP 等）提供第二个线性无关的校验。两个校验块同样**分布式**排列。

| 维度 | 评估 |
|------|------|
| 容错能力 | **2 块磁盘** — 任意两块同时故障，数据可恢复 |
| 读性能 | **好**（与 RAID 5 类似） |
| 写性能 | **比 RAID 5 更差的写惩罚** — 每次小写需计算 P 和 Q 两个校验，RMW 为 3 读 + 3 写 |
| 有效容量 | (N-2)/N × 总容量 |
| 最小磁盘数 | 4 |

> **为什么要 RAID 6？** 当磁盘容量增大时，RAID 5 重建时间变长（数小时甚至数天），在此期间若发生第二块盘故障（或重建过程中遇到 UER —— 见下文可靠性一节），数据将永久丢失。RAID 6 用额外一个校验块的代价换取了"重建期间也能再坏一块"的安全边界。

---

### RAID 10（RAID 1+0）：镜像 + 条带化

**数据布局**：先将每两块磁盘组成 RAID 1 镜像对，再将多个镜像对组成 RAID 0 条带。

```
      RAID 0 (条带化)
      ┌───────────┴───────────┐
   RAID 1 (镜像)          RAID 1 (镜像)
   ┌────┴────┐           ┌────┴────┐
 磁盘0    磁盘1        磁盘2    磁盘3
(块0,2,4) (块0,2,4)   (块1,3,5) (块1,3,5)
```

| 维度 | 评估 |
|------|------|
| 容错能力 | **至少 1 块** — 若每个镜像对至少存活一块，则可容忍多达 N/2 块故障；但若同一镜像对两块同时坏则数据丢失 |
| 读性能 | **优秀** — 条带化 + 镜像读分发 |
| 写性能 | **好** — 条带化并行，镜像需写两份（但可并行写镜像对中的两块盘） |
| 有效容量 | 50% |
| 最小磁盘数 | 4 |
| 重建速度 | **比 RAID 5/6 快得多** — 重建时只需从镜像对的好盘整盘拷贝到新盘，无需读取全阵列做 XOR 计算 |

#### RAID 10 vs RAID 0+1 对比

这一点非常重要且常被混淆：

| 项目 | RAID 10（先镜像再条带） | RAID 0+1（先条带再镜像） |
|------|---------------------------|----------------------------|
| 构造顺序 | N/2 个镜像对 → 条带化 | 2 个条带组（各 N/2 块）→ 镜像 |
| 容错特性 | 每个镜像对独立，**允许不同镜像对各坏一块** | 任一条带组中坏一块，该条带组整体失效 |
| 坏一块盘影响 | 仅需重建该镜像对的一块盘 | 整个条带组（N/2 块盘）进入降级 |
| **结论** | **RAID 10 显著优于 RAID 0+1** | |

**具体例子（4 块盘）**：
- RAID 10：磁盘 0 坏 → 磁盘 1 正常，系统正常；磁盘 2 再坏 → 磁盘 3 正常，系统仍正常（三块盘正常 → 可容忍）。
- RAID 0+1：磁盘 0 坏 → 条带组 A 故障（降级，还可靠另一条带组 B 的镜像）；磁盘 1（另一条带组）再坏 → 条带组 B 也故障 → **全阵列崩溃**。

### RAID 级别速查表

| 级别 | 最小盘数 | 容错（容忍坏盘数） | 读性能 | 写性能 | 有效容量 | 典型场景 |
|------|----------|---------------------|--------|--------|----------|----------|
| RAID 0 | 2 | 0 | N× | N× | 100% | 临时数据、渲染 |
| RAID 1 | 2 | 1 (最多 N-1) | 2×(并发) | 1× | 50% | 操作系统盘、小型数据库 |
| RAID 5 | 3 | 1 | (N-1)× | RMW 惩罚 | (N-1)/N | 文件服务器、读密集型数据库 |
| RAID 6 | 4 | 2 | (N-2)× | RMW 惩罚更重 | (N-2)/N | 大容量归档、冷存储 |
| RAID 10 | 4 | 1~N/2 | N× | N/2× | 50% | OLTP 数据库、虚拟化存储 |

## 磁盘可靠性

磁盘可靠性的量化对 RAID 级别的选择和系统设计至关重要。

### 核心指标定义

- **MTBF（Mean Time Between Failures，平均故障间隔时间）**：可修复设备两次故障间的平均运行时间。以小时计，企业级 HDD 典型值 120 万~250 万小时（≈137~285年）。**注意**：MTBF 是统计学时间而非单块盘实际寿命 —— 100 万小时 MTBF 表示若 1000 块同类盘同时运行，平均每 1000 小时（约 42 天）发生一次故障。
- **AFR（Annualized Failure Rate，年化故障率）**：一块硬盘在一年内发生故障的概率。$AFR = 1 - e^{-8760/MTBF} \approx \frac{8760}{MTBF}$。

- **BER（Bit Error Rate，位错误率）** 或 **UER（Unrecoverable Error Rate，不可恢复错误率）**：每读取多少位数据会发生一次不可恢复的读取错误。典型消费级 HDD UER 为 $10^{-14}$（每读 10¹⁴ 位约出错一次，约 12.5TB），企业级 HDD UER 为 $10^{-15}$ 或更好。
  - **关键影响**：若一块 4TB 的盘故障，RAID 5 重建需读出剩余所有盘的**全量数据**（例如 3 块 4TB 盘 = 12TB）。如果 UER = $10^{-14}$ ≈ 12.5TB，则重建过程中遇到不可恢复读错误的概率**接近 100%**，导致重建失败和数据丢失。这正是 **RAID 5 不适合大容量磁盘**的根本原因，也是 RAID 6 存在的核心理由。

- **MTTDL（Mean Time To Data Loss，平均数据丢失时间）**：在特定 RAID 配置下，从系统启动到**首次发生无法恢复的数据丢失**的平均时间。MTTDL 综合考虑磁盘故障率、重建时间和 RAID 级别。例如：
  - RAID 5（N 块盘）：$MTTDL \approx \frac{MTBF^2}{N \times (N-1) \times MTTR}$
  - 其中 MTTR（Mean Time To Repair，平均修复时间）包括故障检测时间 + 备件准备时间 + 重建时间

- **MTTR（Mean Time To Repair，平均修复时间）**：从磁盘故障到完全恢复冗余所需的时间。对 HDD，重建时间通常是瓶颈 —— 一块 10TB 盘的重建可能需要 10~24 小时。

### UER / BER 的实际意义

```
场景：4x4TB 磁盘组 RAID 5，UER = 10⁻¹⁴

重建需要读取的位数 = 3 × 4TB × 8 bits/byte = 9.6 × 10¹³ bits
不遇到不可恢复错误的概率 ≈ exp(-读取位数 / UER倒数)
                          ≈ exp(-9.6 × 10¹³ / 10¹⁴)
                          ≈ exp(-0.96) ≈ 38%

重建成功概率仅 38%，失败概率 62%。
→ 这就是 RAID 5 为什么在大容量场景被 RAID 6 和 RAID 10 取代。
```

## 磁盘容错

### S.M.A.R.T.（自监控、分析与报告技术）

**S.M.A.R.T.（Self-Monitoring, Analysis, and Reporting Technology）** 是嵌入在磁盘固件中的健康监控系统。

**监控的参数（属性）类型：**

| 类别 | 属性示例 | 含义 |
|------|----------|------|
| 机械指标 | 启停次数（Start/Stop Count）、通电时间（Power-On Hours） | 反映磁盘使用历史 |
| 盘面质量 | 重分配扇区计数（Reallocated Sector Count）、待处理扇区计数（Current Pending Sector Count） | 表面介质损坏或退化 |
| 机械健康 | 寻道错误率（Seek Error Rate）、旋转重试次数（Spin Retry Count） | 磁臂和主轴健康状态 |
| 环境 | 温度（Temperature）、G-Sense 错误率（冲击检测） | 环境风险 |

**工作流程：**
1. 磁盘固件持续在后台采集各种参数（通电小时数、坏扇区数、ECC 纠错次数等）。
2. 每个属性有**当前值**、**最差值**和**阈值（Threshold）**。当前值低于阈值时触发告警。
3. OS 或磁盘管理软件（如 smartd）定期轮询 S.M.A.R.T. 数据。
4. 当某个属性值接近或低于阈值时，提前通知管理员**在磁盘彻底故障前进行替换**。

> S.M.A.R.T. 不是万能的：有相当比例的磁盘故障（约 30-50%）在故障前**没有任何 S.M.A.R.T. 预警**。它只能作为补充手段，不能替代备份策略。

### 热备份盘（Hot Spare）工作流程

**热备份盘（Hot Spare）** 是 RAID 阵列中安装但不参与正常数据存储的空闲磁盘，专门用于自动替换故障磁盘。

**完整工作流程：**

1. **正常运行阶段**：热备份盘处于空闲（Spun Down 或 Spun Up idle）状态，不属于任何 RAID 卷。
2. **故障检测**：RAID 控制器通过以下机制检测到活动磁盘故障：
   - 磁盘响应超时、I/O 错误、S.M.A.R.T. 告警
   - 心跳检查（控制器定期向各磁盘发送健康查询）
3. **自动接管**：控制器立即将热备份盘插入故障盘在 RAID 组中的位置。
4. **重建（Rebuild）**：控制器根据 RAID 校验或镜像数据在新盘上重建故障盘的数据：
   - RAID 1/RAID 10：从镜像盘整盘复制
   - RAID 5/RAID 6：读取全条带数据，通过 XOR（RAID 5）或 Reed-Solomon（RAID 6）计算重建
5. **重建完成**：阵列恢复完全冗余状态。管理员替换物理损坏的盘作为新的热备份盘。
6. **回拷（Copyback）**（可选）：部分实现支持在新盘替换后，将数据从热备份盘拷回原槽位以保持物理槽位一致性。

**热备份策略：**

| 策略 | 说明 |
|------|------|
| **全局热备份（Global Hot Spare）** | 一块备份盘可服务于多个 RAID 组，哪个组先故障就给哪个 |
| **专用热备份（Dedicated Hot Spare）** | 一块备份盘仅为某一个 RAID 组预留 |

### 热插拔（Hot Swap / Hot Plug）

支持在系统不断电的情况下，安全移除或插入磁盘。要求：
- 磁盘接口支持（SAS、SATA AHCI）
- 背板/控制器支持电源管理和信号中断
- OS 需支持设备热移除（通知上层卸载文件系统/停止 I/O）

## SSD 固态硬盘

**SSD（Solid State Drive，固态硬盘）** 使用 NAND 闪存（NAND Flash）作为存储介质，无任何机械部件。

### SSD vs HDD 对比

| 维度 | HDD（机械硬盘） | SSD（固态硬盘） |
|------|------------------|------------------|
| 介质 | 磁性盘片 + 磁头 + 主轴电机 | NAND 闪存单元（Floating Gate 晶体管） |
| 寻道时间 | 2~20ms（机械运动） | ~0.1ms（纯电子，无机械延迟） |
| 顺序读写 | 150~250 MB/s | 500~7000 MB/s（SATA 到 NVMe） |
| 随机 4K 读取 | ~100 IOPS | 10K~1000K IOPS |
| 随机 4K 写入 | ~100 IOPS | 10K~1000K IOPS |
| 寿命 | 极长（无擦写限制，但机械部件会磨损） | 有限（每个单元可编程/擦除 P/E Cycle 次数，SLC ~100K、MLC ~10K、TLC ~3K、QLC ~1K） |
| 功耗 | 5~10W | 2~5W |
| 噪声 | 存在（盘片旋转、磁臂移动） | 无（完全静音） |
| 抗震 | 脆弱（磁头撞击盘面风险） | 优秀（无活动部件） |

### SSD 核心机制

- **写放大（Write Amplification, WA）**：SSD 不能原地覆盖写入（需先擦除整个块再编程），因此 OS 写入 1 个扇区的数据，SSD 内部可能需要搬运整个块（擦除 → 复制有效页 → 写入新数据 → 擦除旧块），实际写入的物理 NAND 数据量 > 主机写入量。**写放大因子 WA = 实际写入 NAND 量 / 主机写入量**，优质控制器可控制在 1.2~2.0。

- **磨损均衡（Wear Leveling）**：SSD 控制器将写入均匀分布到所有 NAND 块以避免部分块提前"磨损失效"。分为：
  - **动态磨损均衡**：仅在"空闲块池"中轮换，不主动搬移静态数据
  - **静态磨损均衡**：定期将"长期不动"的冷数据搬移到高擦写次数的块，腾出低擦写次数的块
  - 控制器维护每个块的**擦写次数（Erase Count）**，写入时优先选择擦写次数最少的块

- **垃圾回收（Garbage Collection, GC）**：失效页（已被主机标记删除或覆盖）占用的物理块在擦除前不能重新使用。GC 后台回收含大量失效页的块：搬移有效页 → 擦除整块 → 加入空闲块池。

- **Over-provisioning（预留空间）**：SSD 出厂容量小于实际 NAND 总容量（例如标称 512GB 实际 512GiB 或更多），差额用于：GC 的工作缓冲区、坏块替换、写放大缓冲、控制器性能优化。

- **TRIM / UNMAP 命令**：OS 通知 SSD "某逻辑块已被文件系统释放"。SSD 收到后可将对应的物理页标记为无效（Stale），在 GC 时无需搬移这些页，显著降低写放大。无 TRIM 时 SSD 以为所有逻辑块都在使用，GC 时浪费地搬移实际上已释放的数据。

### SSD 与磁盘调度

由于 SSD 没有机械磁臂和旋转延迟，传统的寻道优化调度算法（SSTF、SCAN、C-SCAN）在 SSD 上**毫无意义，只会浪费 CPU**。Linux 下 SSD 通常使用：

- **NOOP 调度器**（或不使用 CFQ/BFQ）：FIFO 请求，或做最轻量的合并
- **MQ-Deadline / Kyber**：NVMe 多队列下保证延迟的轻量级调度器

> 但随机 vs 顺序写入在 NAND 层面仍有差异（顺序写有利于减少 GC 和写放大），不过这属于 SSD 控制器的 FTL（Flash Translation Layer，闪存翻译层）管理范围，操作系统通常不直接感知。
