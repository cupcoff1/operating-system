## 虚拟化理论基础

### 什么是虚拟化

**虚拟化（Virtualization）** 是利用软件（Hypervisor）将一台物理计算机的硬件资源（CPU、内存、I/O 设备）抽象为多台逻辑上独立的**虚拟机（Virtual Machine, VM）**。每个 VM 运行自己的操作系统（Guest OS，客户操作系统），Guest OS 以为自己在独占硬件，实际上由 Hypervisor 在底层进行调度和资源分派。

### Hypervisor 分类

| 类型 | 名称 | 架构 | 特点 | 代表产品 |
|------|------|------|------|----------|
| **Type 1（裸机型 / 原生型）** | Bare-Metal Hypervisor | Hypervisor 直接运行在物理硬件之上，VM 运行在 Hypervisor 之上，没有 Host OS | 性能高、开销低，生产环境通用 | VMware ESXi、Microsoft Hyper-V、Xen、**KVM**（将 Linux 变成 Type 1） |
| **Type 2（宿主型 / 托管型）** | Hosted Hypervisor | Hypervisor 作为普通程序运行在宿主操作系统（Host OS）之上，VM 在 Hypervisor 之上 | 安装方便、便于调试，但有 Host OS 一层额外开销 | VMware Workstation / Fusion、Oracle VirtualBox、QEMU（TCG 模式） |

> 注意：KVM 常引发争论。KVM 本身是一个 Linux 内核模块，利用 Linux 内核作为 Type 1 Hypervisor 的基础设施（调度 VM 即调度 vCPU 线程，使用内核内存管理子系统等）。它被归类为 Type 1，因为它直接利用硬件虚拟化扩展且 VM 直接运行在根模式之下。

## Popek & Goldberg 虚拟化定理

1974 年，Gerald J. Popek 和 Robert P. Goldberg 在论文《Formal Requirements for Virtualizable Third Generation Architectures》中给出了一个形式化的判断标准，定义了什么样的计算机体系结构可以被**高效地**虚拟化。

### 两类关键指令

**敏感指令（Sensitive Instruction）**：
- **控制敏感（Control-Sensitive）**：试图修改系统状态或资源配置的指令（如修改特权级别、操纵页表基址寄存器 CR3、操纵中断标志）
- **行为敏感（Behavior-Sensitive）**：其行为或结果依赖于系统当前特权级别或资源配置的指令（如读取当前特权级的指令，其返回值随运行在 Guest 还是 Host 而变化）

**特权指令（Privileged Instruction）**：
在非最高特权级（用户态）执行时会**发生自陷（Trap）** 进入最高特权级（内核态）的指令。即 CPU 硬件本身保证"用户态执行此指令 = 自动陷入内核"。

### 定理陈述

> **一个计算机体系结构可以被高效地虚拟化，当且仅当该体系结构的所有敏感指令同时都是特权指令。**

**等价论述**：如果存在某条指令是"敏感的"但不是"特权的"（即用户态也能执行且不自动陷入，但执行结果却依赖于特权级），则这个架构**不能**构建出高效的"陷阱并模拟（Trap-and-Emulate）" Hypervisor。

### 陷阱并模拟（Trap-and-Emulate）

经典高效虚拟化策略：
1. Guest OS 运行在比 Hypervisor 更低特权级的 CPU 模式下（"去特权化"，De-privileging）
2. Guest OS 执行敏感指令 → 因是特权指令，自动陷入（Trap）Hypervisor
3. Hypervisor 检查指令、模拟该指令的效果（如修改虚拟的页表而非真实页表）
4. Hypervisor 返回 Guest，Guest 以为指令直接执行成功

这是经典虚拟化的**"陷阱并模拟"**循环，要求硬件完全配合。

### x86 的缺陷：17 条非特权敏感指令

不幸的是，早期的 x86 架构（在 Intel VT-x 和 AMD-V 出现之前）**不满足 Popek & Goldberg 定理**，因为存在大约 **17 条敏感但非特权的指令**。

典型例子：
- **SGDT / SLDT / SIDT**（存储全局/局部/中断描述符表寄存器）：在用户态可以执行（不 Trap），但它们返回的值依赖于当前特权级或系统配置（行为敏感），Guest OS 读到错误的值会误判系统状态。
- **POPF**（弹出标志寄存器）：在用户态执行时，修改 IF（中断标志）位会被**静默忽略**（不报错、不 Trap），但 Guest OS 希望它能开关中断。
- **PUSHFD**（压入标志寄存器）：在用户态返回的标志值与在内核态返回的不同（IF/IOPL 等位被屏蔽），造成 Guest 读到假的 CPU 状态。

**后果**：在 VT-x/AMD-V 出现之前，x86 无法实现真正的"陷阱并模拟"虚拟化，必须采用更复杂的方案。

### 三种绕过方案

| 方案 | 原理 | 是否需要修改 Guest OS？ | 性能 | 代表 |
|------|------|------------------------|------|------|
| **二进制翻译（Binary Translation）** | Hypervisor 扫描 Guest OS 的指令流，运行时动态地将"有问题的指令"替换为安全的指令序列或 Hypercall | 否（Guest 无感知） | 中等（翻译有开销，但配合缓存和自适应优化可做得较好） | VMware ESXi（早期）、QEMU TCG |
| **半虚拟化（Paravirtualization）** | 修改 Guest OS 内核代码，将对敏感指令的依赖改为主动调用 Hypervisor 提供的接口（Hypercall） | **是**（必须修改 Guest 内核） | 好（避免了大量 Trap，直接用 Hypercall 批量操作） | Xen（PV 模式） |
| **硬件辅助虚拟化（Hardware-Assisted）** | CPU 硬件新增"根模式（Root Mode）"和"非根模式（Non-Root Mode）"，Guest 在非根模式下运行，硬件自动截获敏感指令并交给 Hypervisor | 否 | 最好（现代方案） | Intel VT-x、AMD-V |

## Intel VT-x 硬件辅助虚拟化详解

Intel VT-x（Virtual Technology for x86）于 2005 年引入，通过新增 CPU 模式和控制结构，使 x86 满足 Popek & Goldberg 的条件。

### VMX 操作模式

Intel VT-x 定义了两种新的 CPU 操作模式，统称为 **VMX 操作（VMX Operation）**：

| 模式 | 说明 | 谁在运行 |
|------|------|----------|
| **VMX 根模式（VMX Root Mode）** | 完全控制硬件的最高特权模式，与传统的 CPU 最高特权级（Ring 0）基本等同，但具有启动/管理 VM 的能力 | **Hypervisor（VMM）** |
| **VMX 非根模式（VMX Non-Root Mode）** | 受限的操作模式，Guest OS 在此模式下运行。当 Guest 执行某些敏感指令或遇到事件时，硬件自动从非根模式退出到根模式 | **Guest OS 及其应用程序** |

> VMX Root 和 Non-Root 各自都有 Ring 0 ~ Ring 3，使得 Guest OS 自身可以在"真正的 Ring 0"下运行，而其"自己是最高特权"的错觉得以保持。硬件在非根模式中拦截"不应暴露给 Guest"的操作。

### VMCS（Virtual Machine Control Structure，虚拟机控制结构）

**VMCS** 是每个虚拟 CPU（vCPU）对应的一个内存数据结构，充当 CPU 在根模式和非根模式之间切换时的"状态保存/恢复区"。Hypervisor 通过 VMCS 配置 VM 的行为。

**VMCS 包含六个逻辑区域：**

| 区域 | 内容 |
|------|------|
| **Guest-State Area（客户状态区）** | 进入 VM（VM-Entry）时，CPU 从此区域加载 Guest 的寄存器状态（通用寄存器、CR0/CR3/CR4、IDTR/GDTR、RIP、RSP 等） |
| **Host-State Area（宿主状态区）** | 退出 VM（VM-Exit）时，CPU 从此区域加载 Hypervisor 的寄存器状态 |
| **VM-Execution Control Fields（VM 执行控制域）** | 配置哪些 Guest 操作会触发 VM-Exit（是否拦截 CPUID、IO 指令、MSR 访问、异常等） |
| **VM-Exit Control Fields（VM 退出控制域）** | 配置 VM-Exit 时的行为细节 |
| **VM-Entry Control Fields（VM 进入控制域）** | 配置 VM-Entry 时的行为细节（是否注入中断/异常、MSR 位图加载） |
| **VM-Exit Information Fields（VM 退出信息域）** | VM-Exit 时硬件自动填入原因和详情（退出原因码、指令长度、错误码等），只读 |

每个 vCPU 有一个独立的 VMCS。CPU 通过 **VMPTRLD** 指令指定当前与哪个 VMCS 关联（"绑定"到逻辑 CPU）。

### VM-Entry 与 VM-Exit

```
┌──────────────────────────────────┐
│      VMX Root（Hypervisor）       │
│   ┌──────────────────────────┐   │
│   │    VM-Exit 处理程序       │   │
│   │ (解释退出原因，模拟操作)   │   │
│   └──────────────────────────┘   │
│         ↑ VM-Exit     ↓ VM-Entry │
│   ┌──────────────────────────┐   │
│   │   VMX Non-Root (Guest)   │   │
│   │   Guest OS + Applications│   │
│   └──────────────────────────┘   │
└──────────────────────────────────┘
```

**VM-Entry**：Hypervisor 执行 **VMLAUNCH**（首次启动 vCPU）或 **VMRESUME**（恢复之前中止的 vCPU）指令，CPU 从 VMCS 的 Guest-State 区加载 Guest 的状态，将 CPU 从根模式切换到非根模式，Guest 开始执行。Hypervisor 也可以选择向 Guest **注入（Inject）** 事件（如中断、异常）。

**VM-Exit**：Guest 运行时，若发生 VMCS 中配置的拦截条件（执行了敏感指令、收到外部中断、访问特定 MSR/IO 端口、发生异常、执行了 HLT/INVD 等），硬件自动：
1. 将当前 Guest CPU 状态保存到 VMCS 的 Guest-State 区
2. 从 VMCS 的 Host-State 区加载 Hypervisor 状态
3. 将退出原因（Exit Reason）和详情填入 VMCS 的 Exit-Information 区
4. 将 CPU 从非根模式切换到根模式
5. RIP 跳转到 Hypervisor 指定的 VM-Exit 处理入口

> VM-Exit 是虚拟化中**最大的性能开销**（每次退出数百到数千个 CPU 周期）。因此虚拟化优化的核心是**尽量减少 VM-Exit**：如大页映射减少 EPT 缺页、virtio 减少 I/O 退出次数、中断直通（Posted Interrupt）减少外部中断退出等。

### VM-Exit 常见原因

| 退出原因（Exit Reason） | 场景 |
|------------------------|------|
| 执行 CPUID 指令 | Guest 查询 CPU 能力（Hypervisor 应返回虚拟化的 CPU 特性而非物理 CPU 特性） |
| MSR 读写（RDMSR / WRMSR） | 访问模型特定寄存器（如 EFER, STAR） |
| 执行 HLT 指令 | Guest 让 vCPU 空闲（Hypervisor 应调度其他 vCPU 使用物理 CPU） |
| I/O 指令（IN / OUT / INS / OUTS） | Guest 访问 I/O 端口（Hypervisor 模拟设备行为） |
| EPT Violation（EPT 违例） | Guest 访问未建立映射的 GPA → HPA 关系（Hypervisor 建立映射） |
| 外部中断（External Interrupt） | 物理中断到达（Hypervisor 决定是否注入 Guest） |
| 异常 | Guest 发生缺页异常、除零等 |
| CR 访问（MOV to CR） | Guest 修改 CR0 / CR3 / CR4（Hypervisor 需虚拟化页表和 CPU 模式） |

## 内存虚拟化

**二层地址映射**是内存虚拟化的核心问题：Guest OS 管理自己的页表，只认识 GVA（Guest Virtual Address）到 GPA（Guest Physical Address）的映射，而真正的物理地址是 HPA（Host Physical Address）。

```
GVA（Guest 虚拟地址）──→ GPA（Guest 物理地址）──→ HPA（宿主机物理地址）
    Guest 页表 负责               Hypervisor 负责
```

### 影子页表（Shadow Page Table, SPT）

VT-x/AMD-V 出现前的主流方案：Hypervisor 维护一份"影子页表"，直接将 **GVA → HPA** 的一条龙映射。Hypervisor 截获 Guest 对页表的修改，将 GVA→GPA→HPA 链展开为直接映射写入影子页表。缺点：维护复杂度高，频繁截获开销大。

### EPT / NPT：硬件加速的二层页表翻译

- **Intel EPT（Extended Page Table，扩展页表）**：CPU 在非根模式下自动完成两级页表遍历（GVA → GPA 经由 Guest 页表，GPA → HPA 经由 EPT 页表）。Hypervisor 配置 EPT，Guest 页表由 Guest 自行管理不拦截。EPT 缺失时触发 **EPT Violation** VM-Exit 而非缺页异常（#PF），Hypervisor 处理 GPA→HPA 映射，然后重启 Guest。
- **AMD NPT（Nested Page Table，嵌套页表）**：与 EPT 功能等价，不同厂商。

> EPT/NPT 使**缺页异常在 Guest 内部直接处理**而不触发 VM-Exit（#PF 只在 Guest 的页表管理中处理）。仅当 GPA→HPA 映射缺失时才发生 VM-Exit。这大幅减少了内存虚拟化的 VM-Exit 频率。

## I/O 虚拟化

| 方式 | 原理 | 性能 | 兼容性 |
|------|------|------|--------|
| **设备模拟（Device Emulation）** | 软件模拟一个已知硬件设备的完整接口（如 Intel E1000 网卡、IDE 磁盘），Guest 用标准驱动即可。每次 I/O 指令（in/out、MMIO）→ VM-Exit → Hypervisor 软件模拟该操作。 | 最慢 | 最高（无需 Guest 特殊驱动） |
| **virtio（半虚拟化 I/O）** | Guest 加载专用的 virtio 驱动（virtio-net, virtio-blk 等），通过共享环形缓冲区（Vring）与 Host 后端（vhost-net, vhost-scsi 等）批量交换 I/O 请求，减少 VM-Exit 次数。 | 接近原生 | 需要 Guest 安装 virtio 驱动 |
| **PCI 直通（PCI Passthrough）** | 将一块物理 PCI/PCIe 设备独占分配给特定 VM，利用 IOMMU（VT-d）实现 DMA 地址翻译和安全隔离，Guest 直接读写设备无 Hypervisor 介入。 | 原生性能 | 仅服务于一个 VM（设备独占） |
| **SR-IOV（Single Root I/O Virtualization）** | 单个物理 PCIe 设备（如网卡）暴露一个 **PF（Physical Function，物理功能）** 和多个 **VF（Virtual Function，虚拟功能）**。每个 VF 可直接分配给不同 VM，Hypervisor 仅管理 PF。 | 原生性能 | 需要 PF 驱动 + VF 驱动（不同的 Guest 驱动），硬件需支持 SR-IOV |

## KVM 深度解析

**KVM（Kernel-based Virtual Machine，基于内核的虚拟机）** 是 Linux 内核的虚拟化子系统，2007 年并入主线。它将 Linux 内核转变为 Type 1 Hypervisor。

### 内核模块架构

```
┌──────────────────────────────────┐
│   用户空间（QEMU / firecracker）    │
│   I/O 模拟、设备实现、配置管理       │
├──── ioctl(/dev/kvm) ─────────────┤ ← KVM API
│   内核空间                        │
│   ┌────────────┐ ┌─────────────┐ │
│   │  kvm.ko    │ │ kvm-intel.ko│ │ ← 通用框架 + 平台代码
│   │  (通用模块) │ │  (or AMD)   │ │
│   └────────────┘ └─────────────┘ │
│   利用 Linux 核心内核机制           │
│   (调度器、内存管理、RCU 等)       │
└──────────────────────────────────┘
```

- **`kvm.ko`**：架构无关的通用虚拟化框架，提供 `/dev/kvm` 设备接口、VM 生命周期管理、vCPU 抽象、内存插槽（KVM Memory Slot）机制、IRQ 路由等。
- **`kvm-intel.ko` / `kvm-amd.ko`**：与底层硬件虚拟化扩展（VT-x/AMD-V）对接，实现 VMCS 的配置、VM-Entry/VM-Exit 处理、EPT/NPT 管理等。

### /dev/kvm 设备

用户态程序通过打开 `/dev/kvm` 获得一个文件描述符，然后通过 **ioctl** 执行所有虚拟化操作：

| ioctl 命令 | 功能 |
|------------|------|
| `KVM_CREATE_VM` | 创建一个新的虚拟机（返回 VM fd） |
| `KVM_CREATE_VCPU` | 在指定 VM 中创建一个虚拟 CPU（返回 vCPU fd） |
| `KVM_SET_USER_MEMORY_REGION` | 为 VM 注册一块内存区域（HVA）作为其物理内存（GPA） |
| `KVM_RUN` | 启动 vCPU 执行（进入 Guest 模式），触发 VM-Entry |
| `KVM_SET_REGS` / `KVM_GET_REGS` | 设置/读取 vCPU 的寄存器状态 |
| `KVM_SET_SREGS` / `KVM_GET_SREGS` | 设置/读取 vCPU 的特殊寄存器（段寄存器、MSR 等） |

### KVM + QEMU 协作模型

KVM 负责 CPU 和内存的虚拟化（性能关键路径），QEMU 负责 I/O 设备的模拟和用户态配置：

```
工作流程（以一次完整的 vCPU 运行为例）：
1. QEMU 打开 /dev/kvm，创建 VM 和 vCPU
2. QEMU 为 VM 分配并注册内存区域（通过 ioctl(KVM_SET_USER_MEMORY_REGION)）
3. QEMU 配置 vCPU 初始寄存器（包括初始化页表、实模式/保护模式设置）
4. QEMU 调用 ioctl(KVM_RUN) → 系统调用 → kvm.ko → kvm-intel.ko → VMLAUNCH / VMRESUME
5. 硬件执行 Guest 代码（在 VMX Non-Root 模式）
6. 发生 VM-Exit → 硬件跳转到 KVM 的 VM-Exit 处理代码
   - 若退出原因是 KVM 内部可处理的（如 EPT violation）→ KVM 直接处理，然后 VMRESUME
   - 若退出原因是用户态需要介入的（如 I/O 指令）→ KVM 将 ioctl(KVM_RUN) 返回给 QEMU
7. QEMU 检查 VM-Exit 原因，模拟 I/O 操作，然后再次调用 ioctl(KVM_RUN) 继续执行
```

### vCPU 调度

KVM 将每个 vCPU 暴露为一个 Linux 线程（进程上下文）。vCPU 调度的本质就是 Linux 调度器调度普通线程，支持：
- SMP Guest（多个 vCPU 线程可并发在不同物理核上运行）
- vCPU 亲和性/绑定
- CFS 调度的公平性与优先级
- 实时 vCPU 线程支持（`SCHED_FIFO` / `SCHED_RR`）

## 容器技术

### 容器 vs 虚拟机

| 维度 | 虚拟机 | 容器（Linux Container） |
|------|--------|------------------------|
| 隔离单元 | 完整 OS（独立内核） | 进程组（共享宿主机内核） |
| 隔离实现 | Hypervisor + 硬件辅助 | Linux Namespace + Cgroup + Capabilities + Seccomp |
| 引导时间 | 秒 ~ 分钟（须启动内核和 init） | 毫秒 ~ 秒（只创建进程隔离环境） |
| 内存开销 | 每个 VM 都有独立内核镜像（数百 MB） | 每个容器仅应用 + 依赖（MB 级） |
| CPU 开销 | 虚拟化开销（VM-Exit 等） | 几乎无额外开销（直接系统调用） |
| 密度（单机可运行数量） | 低（数十台） | 高（数千台） |
| 灵活性 | 支持不同内核 / 不同 OS | 必须与宿主机同内核 |
| 安全性 | 强隔离（独立内核） | 弱隔离（共享内核，逃逸风险更高） |
| 代表技术 | KVM, VMware ESXi, Hyper-V | Docker, containerd, Podman, LXC/LXD |

### Linux Namespace 类型

**Namespace（命名空间）** 是 Linux 内核提供的一种资源隔离机制，使不同进程组看到完全独立的系统资源视图。

| Namespace | 内核标志 | 隔离资源 | 作用 |
|-----------|----------|----------|------|
| **PID** | `CLONE_NEWPID` | 进程 ID 编号空间 | 容器内进程有自有的 PID 编号（如 init 为 PID 1），看不到宿主机进程，容器间 PID 不可见 |
| **Network** | `CLONE_NEWNET` | 网络栈（网卡、IP、路由表、防火墙规则、端口号空间） | 每个容器拥有独立的网络接口、IP 地址、iptables 规则、监听端口；可通过 veth pair / bridge 互联 |
| **Mount** | `CLONE_NEWNS` | 文件系统挂载点视图 | 容器有自己的 `/proc`、`/sys`、根文件系统，对挂载点的变更不影响宿主机或其他容器 |
| **IPC** | `CLONE_NEWIPC` | System V IPC 对象（信号量、消息队列、共享内存）、POSIX 消息队列 | 防止容器间通过 IPC 对象意外通信或互扰 |
| **UTS** | `CLONE_NEWUTS` | 主机名和 NIS 域名 | 每个容器可以有独立的 hostname（`utsname` 结构中的 nodename 和 domainname） |
| **User** | `CLONE_NEWUSER` | 用户 ID / 组 ID 空间 | 容器内的 root（UID 0）可映射为容器外的普通用户，极大提升安全性（"非特权容器"） |
| **Cgroup** | `CLONE_NEWCGROUP` | cgroup 文件系统视图 | 容器内的进程看到的是相对于自身 cgroup 层级的 cgroup 文件系统，而非宿主机完整层级 |

### Cgroup（Control Group，控制组）子系统

**Cgroup** 是 Linux 内核提供的一种资源限制与统计机制，对一组进程的资源使用进行限额、优先、监控。

| Cgroup 子系统 | 控制对象 | 典型用途 |
|--------------|----------|----------|
| **cpu** | CPU 使用时间配额 | 限制一个容器最多使用 0.5 核 CPU (`--cpus=0.5`) 或额定的 CPU 份额 (`--cpu-shares`)，支持 CFS 带宽控制 |
| **cpuset** | CPU 核心与 NUMA 节点绑定 | 将容器绑定到特定 CPU 核和 NUMA 节点，用于实时系统、CPU 隔离 |
| **memory** | 内存使用上限（含 RSS + Page Cache） | 限制容器最大内存（超过则 OOM Kill），统计内存使用，支持 swap 限制 |
| **blkio / io** | 块设备 I/O 速率限制（读/写 BPS 或 IOPS） | 限制容器的磁盘 I/O 带宽（`--device-read-bps`, `--device-write-iops`）防止一个容器把磁盘打满 |
| **freezer** | 挂起/恢复进程组 | 暂停/恢复整个容器的所有进程（Cgroup Freezer），用于 CRIU 检查点/迁移 |
| **pids** | 进程数量上限 | 限制一个 cgroup 内（即容器内）可创建的 PID 总数，防止 fork bomb |
| **devices** | 设备文件访问控制（mknod / read / write） | 控制容器可以访问哪些设备（如 `/dev/sda`），使用白名单/黑名单机制 |
| **net_prio** | 网络流量优先级 | 配置不同 cgroup 产生的网络数据包的优先级值 |
| **net_cls** | 网络数据包类别标记 | 为 cgroup 的数据包标记一个 classid，配合 tc（Traffic Control）做 Qos 流控 |
| **hugetlb** | 大页（HugePage）内存限额 | 限制 cgroup 可使用的 HugeTLB 大页数量 |
| **rdma** | RDMA / InfiniBand 资源限额 | 限制 RDMA 设备的使用资源 |
| **perf_event** | 性能计数器监控 | 对 cgroup 内所有进程做性能计数 |

### UnionFS / OverlayFS（分层联合文件系统）

Docker 等容器引擎利用 **UnionFS（联合文件系统）** 实现轻量的分层镜像构建：

```
容器文件系统视图：
  ┌───────────────────────┐
  │ Container Layer (R/W) │  ← 容器创建时的可写层（仅保存增量修改）
  ├───────────────────────┤
  │ Image Layer 3 (R/O)   │  ← 应用层 (apt install nginx)
  ├───────────────────────┤
  │ Image Layer 2 (R/O)   │  ← 运行时层 (apt install python)
  ├───────────────────────┤
  │ Image Layer 1 (R/O)   │  ← 基础层 (Ubuntu 22.04)
  └───────────────────────┘
```

- 所有只读层（Image Layers）分层堆叠，联合为一个统一视图
- 容器创建时最上面挂一层可读写层（Copy-on-Write, COW）
- **修改文件**：从下层的只读层拷贝到可写层，然后在可写层修改（写时复制）
- **删除文件**：在可写层创建"白描文件"（Whiteout File）覆盖下层
- **相同文件**：上层的文件覆盖下层的同名文件

Linux 上的具体实现：**OverlayFS**（已进内核主线），Docker 默认 `overlay2` 存储驱动。

### 服务模型

## 无服务器计算（Serverless / FaaS）

### 概念

**无服务器计算（Serverless Computing）** 并非真的没有服务器，而是指开发者不再关心服务器的存在（不需管理 OS、补丁、系统配置、扩缩容）。**FaaS（Function as a Service，函数即服务）** 是 Serverless 的实现形式：开发者编写和上传**函数代码**，云平台在事件触发时自动执行函数，按实际调用次数和执行时间计费（从 0 开始，不用就不收钱）。

### 工作流程

```
事件源（HTTP 请求、消息队列、定时器、对象存储事件）
    │
    ▼
FaaS 平台 自动：
  1. 分配一个运行时容器（或复用已有热实例）
  2. 注入事件数据
  3. 执行函数代码
  4. 返回结果
  5. 若一段时间无调用则关闭实例（节省成本）
```

### 关键特性

| 特性 | 说明 |
|------|------|
| **事件驱动（Event-Driven）** | 函数由事件（HTTP 请求、队列消息、文件上传事件等）触发执行，没有事件则不执行 |
| **自动扩缩** | 从零实例到成千上万实例无需人工干预，自动应对流量变化 |
| **按用量计费** | 仅对 CPU 时间 × 内存分配量计费，不调用 = 不收费 |
| **无状态** | 函数实例无状态，状态需存储在外部服务（数据库、对象存储）。同一函数的多次调用、不同实例均没有共享内存 |
| **运行时限制** | 有最大执行时长（如 15 分钟）、内存上限（如 10GB）和临时空间限制 |

### 冷启动问题（Cold Start）

当一个函数长时间未被调用时，平台会回收其运行实例（释放资源）。下次有请求到来时需要**新的容器启动 + 初始化运行时 + 加载函数代码**，这会导致额外的延迟（通常在 100ms ~ 数秒）。这是 Serverless 的主要技术挑战之一。缓解方案包括预留并发、定时预热、镜像缓存、轻量化运行环境（如 Firecracker microVM）。

### 代表平台

- **AWS Lambda**：最成熟的 FaaS 方案，支持 Node.js, Python, Go, Java 等多种运行时及自定义 runtime
- **Azure Functions**：与 Azure 生态深度集成
- **Google Cloud Functions** / **Cloud Run**：与 Google 的 BigQuery, Pub/Sub, Cloud Storage 紧密配合
- **Knative**：Kubernetes 上的开源 Serverless 框架

## 边缘计算与雾计算

### 边缘计算（Edge Computing）

将计算、存储、网络功能从集中式的大型数据中心**下移到数据产生源附近的网络边缘**（基站、交换机、网关、IoT 设备旁），减少数据传输延迟和回传带宽压力。

**核心动机**：
- 降低**延迟**：自动驾驶汽车需要毫秒级障碍物识别，不能等数据走到云中心再返回
- 降低回传带宽：工业 IoT 每天产生 TB 级传感器数据，全部回传云端不现实
- 数据隐私：医院影像数据不必离开局域网即可完成初步处理
- 离线能力：偏远场景的网络连接不稳定，边缘需有本地自主决策能力

### 雾计算（Fog Computing）

**雾计算** 是 Cisco 2014 年提出的术语，在边缘设备和云数据中心之间增加**中间层**计算节点（路由器、交换机、网关、小型服务器），形成一个"雾层"（Fog Layer）。

| 维度 | 边缘计算 | 雾计算 |
|------|----------|--------|
| 计算位置 | 物联网设备旁（最接近数据源） | 边缘与云之间（局域网路由器/网关） |
| 延迟 | 最低（毫秒级） | 低（数毫秒 - 数十毫秒） |
| 计算能力 | 受限（小型嵌入式处理器） | 中等（小型服务器 / 边缘节点） |
| 典型节点 | 传感器、智能摄像头、Raspberry Pi、手机 | 接入点/网关、交换机、uCPE 设备、小规模边缘机架 |
| 关系 | 最靠近物 | 连接边缘到云端，分层协调 |

**三层协作模型**：
```
IoT 设备层 → 边缘层（Edge） → 雾层（Fog） → 云数据中心
  传感器      实时处理        聚合/过滤      大数据分析/AI 训练
```

> 实际应用中常将"边缘"和"雾"混用（OpenFog Consortium 已与 IEEE 的雾计算标准合并），均强调计算下沉、降低延迟。课程和教科书常简化为"边缘计算"总称。
