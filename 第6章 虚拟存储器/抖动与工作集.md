# 抖动与工作集

## 知识框架树

```
虚拟存储器 ── 性能问题
├── 抖动 (Thrashing)
│   ├── 定义 — 换页时间 >> 执行时间，系统瘫痪
│   ├── 原因 — 分配的页框数 < 工作集大小
│   ├── 恶性循环 — 更多进程 → CPU 利用率下降 → OS 再加新进程 → 更严重抖动
│   └── 检测 — 缺页率飙升 + CPU 利用率骤降
│
├── 工作集模型 (Working Set Model)
│   ├── 定义 — WS(t, Δ) = { 最近 Δ 次访存引用的页面集合 }
│   ├── 性质 — 局部性越好，工作集越稳定
│   ├── 用途 — 检测抖动风险：Σ 所有进程 WS > 可用帧数 → 风险
│   └── 缺点 — 每页需维护时间戳，开销较大
│
├── 缺页率控制 (Page Fault Frequency, PFF)
│   ├── 上界 → 缺页率太高→分配更多帧
│   ├── 下界 → 缺页率太低→回收多余帧
│   └── 机制 — 每个进程独立监控，动态调整帧分配
│
└── 内存映射文件 (mmap)
    ├── 原理 — 文件映射到虚拟地址空间，按需换页
    ├── 优势 — 零拷贝 / 共享内存 / 延迟加载
    └── 实现 — 内核 page cache + 缺页中断驱动
```

---

## 1. 抖动（Thrashing）

### 1.1 定义

**抖动**：进程在**换页上的时间多于实际执行的时间**，CPU 忙于处理缺页中断而几乎不执行有效指令，系统吞吐率急剧下降，近乎瘫痪。

直观感受：一个进程刚把某页换入，立刻又要把它换出去换入另一页——来回"倒腾"，CPU 一直在等待磁盘 I/O。

### 1.2 抖动的成因与恶性循环

抖动的核心机制是一个**正反馈环路**——系统越忙，OS 越容易做出恶化局面的决策：

```
                        ┌──────────────────┐
                        │  CPU 利用率下降    │ ← 所有进程都在等磁盘 I/O
                        └───────┬──────────┘
                                │
              OS 调度器误判：     │
        "CPU 空闲，说明进程不够多"  │
                                │
                        ┌───────▼──────────┐
                        │  OS 引入更多进程   │ ← 换入新进程到内存
                        └───────┬──────────┘
                                │
                        每个进程分到的页框更少
                                │
                        ┌───────▼──────────┐
                        │  缺页率急剧上升    │ ← 每个进程的工作集装不下
                        └───────┬──────────┘
                                │
                        更多磁盘 I/O，进程阻塞
                                │
                        ┌───────▼──────────┐
                        │  CPU 利用率进一步降 │ ← 循环加剧（回到起点）
                        └──────────────────┘
```

**关键误判**：长程调度器（Long-term Scheduler / 中级调度）看到 CPU 利用率降低，误以为系统负载不足，于是引入新的进程。但实际 CPU 空闲是因为所有进程都在等待换页 I/O。新进程的加入进一步瓜分本就紧张的物理页框，恶化缺页情形。

### 1.3 判断是否发生抖动

| 指标 | 正常系统 | 抖动状态 |
|:---:|:------:|:------:|
| CPU 利用率 | 高（> 70%） | 极低（< 10%） |
| 磁盘 I/O | 适量 | 满负荷（持续换页） |
| 缺页率 | 低 | 极高 |
| 进程状态 | 多数 Running | 多数 Blocked (I/O wait) |

## 2. 工作集模型（Working Set Model）

### 2.1 定义

**工作集**（Working Set）是 Denning 于 1968 年提出的概念，用于量化和预测进程对物理内存的需求。

> **WS(t, Δ) = { 进程在时间区间 [t − Δ + 1, t] 内访问过的页面集合 }**

参数含义：
- **t**：当前时刻（或当前访问的序号）。
- **Δ**（Delta）：**工作集窗口大小**——往回看多少次访问。通常取一个合理值（如 1000 次访问）。
- **|WS|**：工作集大小（Working Set Size, WSS）——窗口内不同页面的个数。

### 2.2 工作集的直观含义

- Δ 太小 → 工作集不能完整反映进程的局部性。
- Δ 太大 → 工作集会包含一些已经退出局部性的页面，高估内存需求。
- Δ 适中 → 工作集能捕捉进程当前的**活跃页面集合**，准确反映其内存需求。

**关键关系**：

> 进程的物理页框数 >= 其工作集大小 → 进程在长期运行中将保持低缺页率。
> 进程的物理页框数 < 其工作集大小 → 频繁缺页，可能引发抖动。

### 2.3 具体示例（Δ = 5）

### 2.4 观察与分析

从上面的追踪中可以观察到三个关键现象：

**（1）强局部性阶段 — 工作集收缩**

t=6 到 t=10，进程密集访问页面 7 和 5。工作集从 5 收缩到 2~3。此时如果给该进程分配 3 个以上页框，缺页极少。

**（2）局部性切换阶段 — 工作集急剧变化**

t=11 到 t=14，进程从 {5,7} 的局部性切换到 {1,2,3,4} 的局部性。工作集从 2 急剧增大到 5。如果有足够的空闲页框可以平稳过渡；如果页框不足，此时会大量缺页。

**（3）抖动风险判断**

假设系统有 8 个空闲页框，同时运行 2 个进程，每个进程的当前工作集大小为 5。则：

> **总需求 = 5 + 5 = 10 > 8（可用帧）→ 抖动风险**

解决方法：挂起一个进程（换出到磁盘），将其页框分配给另一个，等局势缓解后再唤醒。

### 2.5 工作集策略的优缺点

| 优点 | 缺点 |
|:----|:----|
| 数学上严格定义，可量化 | 每页需记录"最近访问时间"，硬件开销大 |
| 能检测抖动并预防 | Δ 的值难以选择——太小不准确，太大高估需求 |
| 适合分时系统 / 批处理系统 | 工作集过渡期（如进程切换局部性）需要额外页框支持 |

## 3. 缺页率控制（Page Fault Frequency, PFF）

### 3.1 核心思想

PFF 方法不再直接计算工作集大小，而是**监控每个进程的缺页频率**，通过反馈调节来动态分配页框。

> 定义一个**缺页率** = 最近一段时间内的缺页次数 / 访问次数。

### 3.2 上下界机制

系统为每个进程设定两个阈值：

```
  缺页率
    ↑
    │         ┌──────────┐
    │         │  过多缺页  │ ← 缺页率 > 上界：分配更多帧
上界 ─────────┤  (需加帧)   │
    │         │  正常区间  │ ← 缺页率在上下界之间：维持不变
下界 ─────────┤  (不需调整)  │
    │         │  帧有冗余  │ ← 缺页率 < 下界：回收多余帧
    │         └──────────┘
    │
    └──────────────────────→ 时间
```

- **缺页率 > 上界**：说明进程分配的页框太少，频繁缺页。如果系统有空闲帧，则分配给该进程；如果没有，则可能需要挂起另一个进程来释放帧。
- **缺页率 < 下界**：说明给该进程的页框过多，有冗余。回收一些帧，让它们服务于更需要内存的进程。
- **缺页率在上下界之间**：理想状态，维持当前的帧分配不变。

### 3.3 PFF 的优势

1. **不需要预先计算工作集**。系统通过观察实时缺页率间接推断工作集是否得到了满足。
2. **自适应性强**。进程在不同执行阶段（如进入新的局部性）缺页率会自然波动，PFF 会自动调整。
3. **实现相对简单**。只需记录每个进程的缺页中断发生时间，计算最近一段时间内的频率。

### 3.4 PFF 与工作集策略的对比

| 维度 | 工作集策略 | PFF 控制 |
|:---:|:--------:|:------:|
| **监控对象** | 工作集大小 \|WS\| | 缺页率（缺页数/时间） |
| **决策依据** | \|WS\| vs 可用帧数 | 缺页率 vs 上下界阈值 |
| **调整方式** | 确保分配帧 >= \|WS\| | 缺页率高→加帧，低→减帧 |
| **优点** | 理论上精确 | 实现简单，自适应 |
| **缺点** | 需要维护时间戳，开销大 | 阈值选择需要经验 |

## 4. 内存映射文件（Memory-Mapped Files）

### 4.1 什么是 mmap

**内存映射文件**（Memory-Mapped File）是一种将磁盘文件映射到进程虚拟地址空间的机制。通过 `mmap()` 系统调用，进程可以将文件视为一块内存，直接通过指针或数组索引来读写，而不需要先 `read()` 到用户缓冲区再处理。

```
                传统 read() 方式                           mmap() 方式
              ┌──────────────┐                     ┌──────────────┐
              │   进程 (用户态)  │                     │   进程 (用户态)  │
              │  buffer[]  ←──│── 拷贝到用户空间      │  ptr ─────────│── 直接访问
              └──────┬───────┘                     └──────┬───────┘
                     │                                    │
              ┌──────▼───────┐                     ┌──────▼───────┐
              │  页缓存 (内核)  │                     │  页缓存 (内核)  │
              │  page cache   │                     │  page cache   │
              └──────┬───────┘                     └──────┬───────┘
                     │                                    │
              ┌──────▼───────┐                     ┌──────▼───────┘
              │   磁盘文件     │                     │   磁盘文件     │
              └──────────────┘                     └──────────────┘

           两次拷贝（磁盘→页缓存→用户buffer）        零拷贝（磁盘↔页缓存，用户直接访问页缓存）
```

### 4.2 mmap 的工作原理

1. **调用 `mmap()`**：进程请求将文件的某个区域映射到其虚拟地址空间的某个区域。内核创建对应的 VMA（Virtual Memory Area，虚拟内存区域）结构，记录映射关系（文件、偏移、长度、访问权限），但此时**并不实际读取文件内容**。

2. **初次访问触发缺页中断**：当进程首次读写映射区域内某个地址时，MMU 发现该虚拟页没有对应的物理页框，触发缺页中断。

3. **缺页中断处理**：
   - 内核识别出这是一个文件映射页（而不是匿名页或 swap 页）。
   - 内核从磁盘读取对应的文件块到 page cache（页缓存）。
   - 将进程的对应页表项指向该 page cache 中的物理页框。
   - 设置页表项的访问权限（可读/可写，取决于 `mmap` 的 `prot` 参数）。

4. **后续访问**：直接命中 page cache，不产生磁盘 I/O。

5. **写回磁盘**：
   - 如果映射为 `MAP_SHARED`，对映射区域的修改最终会写回磁盘文件。内核标记被修改的页为"脏"，由后台 `pdflush`（或较新内核的 writeback 线程）定期刷回磁盘，也可通过 `msync()` 显式同步。
   - 如果映射为 `MAP_PRIVATE`，修改采用**写时复制**（Copy-on-Write, COW）机制：修改的页被复制一份私有副本，原始文件不受影响。

6. **调用 `munmap()`**：解除映射。内核释放相关的 VMA，清理页表项，脏页写回磁盘（SHARED 映射）。

### 4.3 mmap 的优势

| 优势 | 说明 |
|:----|------|
| **减少拷贝次数** | 数据直接在 page cache 中被用户进程访问，免去了 `read()`/`write()` 中"内核空间→用户空间"的一次拷贝 |
| **延迟加载** | 文件内容按需加载（demand paging），只有真正访问的页才从磁盘读入，适合处理大文件 |
| **共享内存** | 多个进程映射同一文件（`MAP_SHARED`）时，共享同一份 page cache，内存中只有一份数据副本，任一进程的修改对其他进程立即可见 |
| **简化编程模型** | 像操作数组一样读写文件，`ptr[offset] = value` 等价于写文件的某个位置，无需管理 buffer、lseek |
| **零拷贝 I/O** | 配合 `sendfile()` 等系统调用可实现网络发送零拷贝 |

### 4.4 mmap 的适用场景

- **读取大文件**：只需映射需要的部分，内核按需加载页，内存占用可控。
- **数据库系统**：将数据文件映射到内存，直接通过指针读写记录，效率极高（如 LMDB、SQLite 可选 mmap 模式）。
- **动态链接库加载**：`ld.so` 使用 mmap 将 `.so` 文件的代码段和数据段映射到进程地址空间。
- **进程间通信（IPC）**：通过映射同一个文件实现高效共享内存。

### 4.5 mmap 与虚拟内存的统一

mmap 完美体现了**虚拟存储器的统一抽象**：文件和内存被统一对待——文件映射到虚拟地址空间后，就像普通的匿名内存页一样按需换入换出。内核的 page cache 充当了文件与内存之间的桥梁。这种设计使得操作系统可以用同一套缺页中断机制同时处理：
- 匿名页（进程堆/栈 → swap 分区）
- 文件映射页（文件内容 → 磁盘文件本身）
- 代码页（可执行文件 → 按需加载）

## 5. 总结：抖动预防的完整策略链

```
1. 分配策略
   为每个进程分配 ≥ 其工作集大小的物理页框。
   → 使用工作集模型计算 WS(t, Δ)

2. 全局监控
   确保 ∑ 所有活跃进程的 WS ≤ 系统总可用帧数。
   → 如果超出，挂起某些进程（中级调度换出），释放帧给活跃进程。

3. 动态调节 (PFF)
   为每个进程监控缺页率。
   → 缺页率 > 上界 → 分配更多帧（或挂起其他进程）。
   → 缺页率 < 下界 → 回收多余帧。

4. 兜底机制
   如果系统仍然陷入抖动，换出（swap out）低优先级进程，减少并发度，
   让剩余的进程有足够的物理内存平稳运行。
```
