# 分段与段页式

## 一、知识框架树

```
存储器管理
├── 连续分配
│   ├── 单一连续分配
│   ├── 固定分区分配
│   └── 动态分区分配（首次适应 / 最佳适应 / 最差适应）
├── 离散分配
│   ├── 分页存储管理 ← 基础分页
│   ├── 分段存储管理 ← 本章核心（语义划分）
│   └── 段页式存储管理 ← 本章核心（混合方案）
└── 虚拟存储器
    ├── 请求分页
    ├── 请求分段
    └── 请求段页式
```

本章是离散分配管理的下半部分。分页解决"如何高效利用内存"的问题，分段解决"如何按程序逻辑组织内存"的问题，段页式是二者的融合——取各自所长。

## 二、分段存储管理

### 2.1 为什么需要分段 —— 程序员的视角

分页对程序员是**不可见**的：程序员看到的是一个连续的、一维的地址空间，操作系统和硬件在背后把地址空间切成固定大小的页，映射到物理帧。这种划分完全由硬件决定，与程序的逻辑结构无关。

但现实中的程序不是一维数组。一个进程的地址空间天然由几个**语义上独立的区域**组成：

```
高地址 ┌─────────────┐
       │    栈 (Stack)  │  ← 局部变量、函数调用，向低地址增长
       ├─────────────┤
       │      ↓       │
       │     空闲      │
       │      ↑       │
       ├─────────────┤
       │   堆 (Heap)   │  ← 动态分配（malloc），向高地址增长
       ├─────────────┤
       │   数据段 (Data)│  ← 全局变量、静态变量
       ├─────────────┤
       │   代码段 (Code)│  ← 可执行指令（通常只读）
低地址 └─────────────┘
```

每个段的性质完全不同：

- **代码段**：只读、可执行、固定大小、多进程可共享（同一个程序的不同进程可以共享代码段）。
- **数据段**：可读写、不可执行、固定大小（全局变量数量编译时确定）。
- **堆**：可读写、动态增长/收缩（brk/sbrk）、不可执行。
- **栈**：可读写、动态增长/收缩（函数调用深度决定）、不可执行。

分页无法表达这些语义。所有的页都是"大小相同的内存块"，操作系统不知道哪一页是代码、哪一页是数据，也就无法做细粒度的保护（比如：代码页不可写、数据页不可执行）。

**分段的核心动机**：让操作系统的内存管理方案匹配程序员的心理模型（代码、数据、堆、栈），从而自然支持：
- **保护**：在段表条目中设置"读/写/执行"权限位。
- **共享**：两个进程可以共享同一个代码段——因为段是对程序员有意义的数据单元。
- **动态增长**：栈和堆可以在运行时伸缩，只需要修改段表中的 limit 字段。

### 2.2 段表结构

每个进程拥有一张**段表**，存储在主存中，由**段表基址寄存器**指向。每个段表条目结构如下：

```
段表条目（Segment Table Entry）:
┌──────────┬──────────┬──────────────────┐
│  base   │  limit  │  protection bits │
│ (基址)  │ (段长)  │  (保护位)        │
│  32/64位 │  32位   │    3-4位         │
└──────────┬──────────┬──────────────────┘
```

| 字段 | 含义 |
|------|------|
| **base** | 该段在主存中的起始物理地址。段内偏移 d 加上 base 即得物理地址。|
| **limit** | 段的最大合法长度（字节数）。d ≥ limit 时，硬件产生"越界中断"（Segment Fault — 正是我们熟悉的 Segfault 的硬件根源）。|
| **protection bits** | 读(R) / 写(W) / 执行(X) 权限。越权访问同样触发保护中断。|

保护位的典型设置：

| 段 | R | W | X |
|----|---|---|---|
| 代码段 | 1 | 0 | 1 |
| 数据段 | 1 | 1 | 0 |
| 栈 | 1 | 1 | 0 |
| 堆 | 1 | 1 | 0 |

### 2.3 地址变换 —— 数值演算

**逻辑地址格式**：

```
逻辑地址（二维）：[ 段号 s ] [ 段内偏移 d ]
```

段号是"第几个段"——程序员/编译器在生成指令时知道数据属于哪个段（比如：`MOV AX, [DS:100]` 中的 `DS` 就是数据段的选择子）。

**变换公式**：

```
if (s >= 段表长度)       → 越界中断
if (d >= 段表[s].limit)  → 段内越界中断（Segmentation Fault）
if (访问类型非保护位允许)  → 保护中断
物理地址 = 段表[s].base + d
```

**数值示例**：

假设某进程段表如下：

| 段号 s | base | limit | R | W | X |
|--------|------|-------|---|---|---|
| 0 | 0x0000_4000 | 2048 (0x800) | 1 | 0 | 1 |  ← 代码段
| 1 | 0x0000_6000 | 1024 (0x400) | 1 | 1 | 0 |  ← 数据段
| 2 | 0x0000_A000 | 4096 (0x1000)| 1 | 1 | 0 |  ← 堆段
| 3 | 0x0001_0000 | 2048 (0x800) | 1 | 1 | 0 |  ← 栈段

**题目**：将逻辑地址 `(s=2, d=150)` 转换为物理地址。

**解题步骤**：
1. 段号 s = 2，段表大小 = 4，s < 4 ✓，通过。
2. 查段表条目 [2]：base = 0x0000_A000，limit = 4096。
3. 检查：d = 150 < 4096 ✓，通过（若 d = 5000 则越界 → Segfault）。
4. 检查保护位：假设该访问为写操作，段 2 的 W = 1 ✓，通过。
5. 物理地址 = 0x0000_A000 + 150 = 0x0000_A096。

**地址变换硬件流程**：

```
逻辑地址 (s, d)
      │
      ▼
┌─────────────┐    段表基址寄存器
│ s ≥ 段表长度? │── 是 → 越界中断
└──────┬──────┘
       │否
       ▼
┌─────────────────┐
│ 读段表[s]：      │
│ base, limit, RWE│
└──────┬──────────┘
       │
       ▼
┌─────────────┐
│ d ≥ limit ?  │── 是 → Segfault
└──────┬──────┘
       │否
       ▼
┌─────────────────┐
│ 保护位检查通过?   │── 否 → 保护中断
└──────┬──────────┘
       │是
       ▼
  base + d ──→ 物理地址
```

### 2.4 段的共享

分段使得共享非常自然。假设两个进程 P1 和 P2 都运行同一个文本编辑器程序：

```
P1 段表:                          P2 段表:
┌───┬─────────────┬───────┐      ┌───┬─────────────┬───────┐
│ 0 │ → 物理地址 A │ 代码段│      │ 0 │ → 物理地址 A │ 代码段│  ← 同一物理地址！
│ 1 │ → 物理地址 B │ 数据段│      │ 1 │ → 物理地址 C │ 数据段│  ← 各自独立
│ 2 │ → 物理地址 D │ 栈   │      │ 2 │ → 物理地址 E │ 栈   │  ← 各自独立
└───┴─────────────┴───────┘      └───┴─────────────┴───────┘
```

两个进程的段 0 的 base 指向**同一个物理地址 A**，共享一份代码。各自的数据段和栈指向不同的物理地址，保持隔离。这就是现代操作系统共享库（.so / .dll）的底层原理。

### 2.5 分段的缺点

1. **外部碎片**：段的大小可变，分配和释放后留下零散的空洞。
2. **段必须连续存放**：若一个段很大，可能需要通过紧凑操作（移动所有段，类似磁盘碎片整理）来腾出足够大的连续空间。
3. **内存利用率不如分页**：分页只有内部碎片（最后一页浪费最多半个页框），分段的外部碎片可能浪费大量空间。

## 三、分段 vs 分页 —— 深入对比

### 3.1 表面对比

| 维度 | 分页 | 分段 |
|------|------|------|
| **划分依据** | 固定大小（硬件决定，如 4KB） | 可变大小（语义决定，一个代码段的 natural size） |
| **划分者** | 硬件 / OS | 程序员 / 编译器 |
| **对用户透明** | 是，用户看到一维地址空间 | 否，用户/编译器使用二维地址（段号 + 偏移） |
| **大小单位** | 页 (page) → 页框 (frame)，等长 | 段 (segment)，任意长度 |
| **碎片类型** | 内部碎片（最后一页未用满） | 外部碎片（段间空隙） |
| **地址空间维度** | **一维**（地址从 0 到 max） | **二维**（需指定段号和段内偏移） |
| **共享** | 困难（一页可能包含代码和数据）| 自然（以段为单位共享） |
| **保护** | 粗粒度（页级权限） | 细粒度（段级，可按语义 R/W/X） |
| **动态增长** | 不支持（页固定大小） | 支持（修改 limit 即可） |
| **地址变换速度** | 快（页表 + TLB） | 较慢（段表 + 加法，段表条目少但检查多） |

### 3.2 哲学差异 —— 核心理解

这是最重要的一点，不只是一张对比表，而是两种截然不同的设计哲学：

**分页 = 硬件视角（Machine-Oriented）**

分页的设计哲学是：<u>内存是由许多等大小槽位组成的物理资源</u>。操作系统的工作是把进程需要的内存需求"打散"填入这些等大小槽位中，这样不管程序怎么分配，总能把物理内存填满，不会出现"有 10MB 空闲但不能用"的情况（= 消除外部碎片）。

分页把每个程序都视为一个**无结构的线性字节序列**。硬件不关心哪部分是代码、哪部分是数据——对所有页一视同仁。这就像把所有东西打成一样大小的箱子，搬运时只管箱子，不问内容。

**分段 = 程序员视角（Programmer-Oriented）**

分段的设计哲学是：<u>程序天然由不同性质的模块组成</u>，每个模块有自己独立的生存周期、保护需求、增长模式。操作系统应该辨认并尊重这种结构，而不是强行把程序压平成无结构的字节流。

分段让程序员可以"告诉"OS：这块是代码，只读可执行；那块是数据，读写但不可执行。这种语义信息可以被 OS 用来做智能决策（比如：代码段在内存紧张时可以丢弃——因为可以从磁盘重新加载，数据段则需要换出到交换分区）。

**一句话总结**：分页问的是"有没有物理槽位放得下这一页"，分段问的是"这段数据应该被如何对待"。

## 四、段页式存储管理

### 4.1 动机：取二者之长

分段语义好但内存利用率差（外部碎片）。分页内存利用率好但没有语义。段页式的思路是**先分段，再分页**——外部碎片问题在段内就被分页解决了，而保护/共享在段层面处理。

**组合策略**：
- 段层面：按语义划分地址空间（代码段、数据段、栈段），做保护和共享决策。
- 页层面：把每个段切成等大小的页，页可以装在任意空闲页框中，消除外部碎片。

### 4.2 地址结构

段页式的逻辑地址由三部分组成：

```
    段号 s        │      段内页号 p      │   页内偏移 d
└── 选段 ──┘     └───── 段内分页 ─────┘    └── 页内定位 ┘
```

举例（32 位地址，实际分割可配置）：

```
│31      24│23            12│11         0│
│  段号 s  │   页号 p       │ 页内偏移 d  │
│  8 bits  │   12 bits      │  12 bits    │
```

段号 8 位 → 最多 256 个段，页号 12 位 → 每段最多 4096 页，页内偏移 12 位 → 每页 4KB（与纯分页一致）。

### 4.3 地址变换路径 —— 三次访存

段页式需要**三次内存访问**才能取到数据：

```
逻辑地址: [s=2 | p=3 | d=150]

第一次访存：读段表
  段表基址寄存器 + s×段表条目大小 → 找到段表[2]
  段表[2] 内容：{ 页表地址 PTR2, 段长 limit2 }
  检查：p 对应的地址是否在本段内（即 p×页面大小 + d < limit2）

第二次访存：读页表
  PTR2 + p×页表条目大小 → 找到页表[3]
  页表[3] 内容：{ 页框号 f=7 }
  注意：此时页号是"段内页号"，每个段有自己的页表！

第三次访存：读数据
  页框号 f=7 × 4KB + d=150 → 物理地址 = 7×4096 + 150 = 28822

最终：物理地址 28822 处读到所需数据
```

**流程图**：

```
                   逻辑地址
          ┌─────────┼──────────┐
          │         │          │
          ▼         ▼          ▼
        段号 s   段内页号 p   页内偏移 d
          │
          ▼
    ┌───────────┐
    │  段表[?]   │ ← 段表基址寄存器 + s × 条目大小
    │  找到段 s   │   (第 1 次访存)
    │  的页表地址 │
    └─────┬─────┘
          │
          ▼
    ┌───────────┐
    │  页表[?]   │ ← 段 s 的页表基址 + p × 条目大小
    │  找到页 p   │   (第 2 次访存)
    │  的页框号 f │
    └─────┬─────┘
          │
          ▼
    f × 页大小 + d  ───→ 物理地址  (第 3 次访存)
          │
          ▼
    ┌───────────┐
    │ 数据在此！ │
    └───────────┘
```

**三次访存的开销非常大**。在实际实现中，必须依赖 TLB（快表）来减少访存次数——绝大多数地址变换命中 TLB 后只需一次访存。

### 4.4 段页式的优缺点

| 优点 | 缺点 |
|------|------|
| 兼具分段的语义保护/共享能力 | 三次访存，开销大 |
| 兼具分页的无外部碎片特性 | 实现复杂，硬件支持要求高 |
| 段可以动态伸缩（增加/减少页表条目） | 每个段一张页表，内存开销大 |
| 权限检查自然（段层 + 页层双重检查） | TLB 命中率可能低于纯分页 |

## 五、真实案例：Intel x86 的内存模型

Intel x86（32 位保护模式）是分段 + 分页组合的现实教材。整个机制分两阶段：

### 5.1 逻辑地址 → 线性地址（分段阶段）

```
逻辑地址: [ 段选择子 (16 bits) : 段内偏移 (32 bits) ]

段选择子:
┌─────────────┬──┬────┐
│ 索引 (13位)  │TI │RPL │
│ 在 GDT/LDT   │ │  │ 请求特权级│
│ 中的条目编号 │表 │  │  (2位)   │
└─────────────┴──┴────┘
          TI = 0 → GDT（全局描述符表）
          TI = 1 → LDT（局部描述符表）
```

**段描述符**（GDT/LDT 条目，即 x86 的"段表条目"）：

```
段描述符（8 字节）:
┌────────────────────────────────────────────────────┐
│ Base 31..24 │G│D│L│ Limit 19..16 │...│ Base 23..0 │...
│ Base 31..0 → 段基址（32 位，4GB 地址空间）        │
│ Limit 19..0 → 段限长（20 位，G=0 时单位 1B，G=1 时单位 4KB）│
│ DPL → 描述符特权级（保护）                         │
│ Type → 代码/数据/系统段 + R/W/E 权限              │
└────────────────────────────────────────────────────┘
```

分段变换：`线性地址 = GDTR/LDTR 指向的表[选择子.索引].base + 段内偏移`

### 5.2 线性地址 → 物理地址（分页阶段）

x86 分页典型配置（启用 PAE 等扩展后有所不同）：

```
线性地址（32 位）:
│31     22│21      12│11      0│
│ 页目录索引 │ 页表索引  │ 页内偏移 │
│ (10 bits)  │ (10 bits) │ (12 bits)│
```

两级页表：CR3 → 页目录 → 页表 → 物理页框。

### 5.3 完整双阶段转换

```
逻辑地址 (selector:offset)
   │                      ← 编程可见
   ▼
[ 分段单元 ]
   │ 查 GDT/LDT，段基址 + 偏移
   ▼
线性地址 (32-bit flat)
   │                      ← 对大多数现代 OS 是"透明"的
   ▼                      ← （Linux/Windows 将 base 设为 0，limit 设为 4GB）
[ 分页单元 ]
   │ 查页目录 → 页表
   ▼
物理地址
```

### 5.4 Linux 的做法

Linux 内部把 x86 的段基址全部设为 0、段限长全部设为 4GB（"平坦模式"），使得逻辑地址 = 线性地址——本质上绕过了分段，只依赖分页。但对 VDSO、TLS（线程局部存储）等少数场景，仍会使用段寄存器（如 `fs`、`gs`）指向特定数据结构。

## 六、三者关系总结

```
                ┌─────────────────────────┐
                │      分页 (Paging)      │
                │  消除外部碎片            │
                │  硬件驱动，对用户透明    │
                │  一维地址                │
                └────────────┬────────────┘
                             │ 融合
                ┌────────────▼────────────┐
                │   段页式 (Seg+Page)     │
                │ 分段语义 + 分页效率     │
                │ 三次访存，需 TLB        │
                │ x86 真实实现            │
                └────────────┬────────────┘
                             │ 继承语义
                ┌────────────▼────────────┐
                │     分段 (Segmentation) │
                │  按语义组织内存          │
                │  程序员可见              │
                │  二维地址，可变大小      │
                │  自然支持共享/保护       │
                └─────────────────────────┘
```

- **纯分段**：最贴合程序员心智模型，但存在外部碎片，管理复杂。
- **纯分页**：最高效的内存利用，但丢失了程序的逻辑结构信息。
- **段页式**：实际工程中最常见的折中，x86 架构是典型实例。现代操作系统在软件层面采平坦模式（弱化分段），但硬件一直保留着两层转换的能力。
