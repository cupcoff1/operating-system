# 连续内存分配

## 知识框架

```
连续内存分配
├── 1. 程序装入
│   ├── 绝对装入：编译时绑定物理地址
│   ├── 可重定位装入：装入时完成地址映射
│   └── 动态运行时装入：运行时 MMU 映射（现代 OS）
├── 2. 程序链接
│   ├── 静态链接：装入前，每个程序完整拷贝库
│   ├── 装入时动态链接：装入时链接共享库
│   └── 运行时动态链接：执行时 dlopen（插件机制）
├── 3. 交换技术（Swapping）
│   ├── 中级调度：挂起/激活进程
│   ├── 对换区：磁盘上连续空间
│   └── 决策：换出阻塞进程、换入就绪进程
├── 4. 动态分区分配算法
│   ├── 首次适应（First Fit）：从头找第一个够大的
│   ├── 最佳适应（Best Fit）：找大小最接近的最小分区
│   ├── 最差适应（Worst Fit）：找最大空闲区
│   └── 邻近适应（Next Fit）：从上次分配位置继续找
└── 5. 碎片问题
    ├── 外部碎片：总空闲够但无不连续大块
    ├── 内部碎片：分配块大于实际需要
    └── 紧凑（Compaction）：移动分区合并空闲空间
```

---

## 一、程序装入：地址如何从磁盘到内存

源程序要执行，必须装入内存，而装入过程的核心问题是**地址绑定** —— 何时将逻辑地址转换为物理地址。

| 装入方式 | 绑定时机 | 原理 | 特点 |
|----------|----------|------|------|
| **绝对装入** | 编译时 | 编译器直接生成物理地址 | 程序只能固定在指定地址运行，灵活性最差；早期单道系统使用 |
| **可重定位装入（静态重定位）** | 装入时 | 加载器根据当前空闲空间，将逻辑地址统一加上偏移量 | 装入后地址固定，运行时不可移动 |
| **动态运行时装入（动态重定位）** | 执行时 | CPU 每次访存时，MMU 将逻辑地址 + 重定位寄存器值 → 物理地址 | 支持运行时移动、交换、紧凑；现代 OS 标准 |

> **关键认知**：动态重定位使得进程在内存中的位置可以在运行时改变，这是交换和紧凑技术的硬件基础。

---

## 二、程序链接：代码和库如何组装

| 链接方式 | 时机 | 原理 | 特点 |
|----------|------|------|------|
| **静态链接** | 编译后、装入前 | 将用到的库函数目标模块直接拷贝到可执行文件中 | 每个进程都有一份库的副本，n 个进程 = n 份拷贝，磁盘和内存双重浪费 |
| **装入时动态链接** | 装入时 | 装入时发现未解析的符号，查找并链接对应共享库 | 多个进程可共享同一份库的物理内存，但修改库需要重新装入所有依赖进程 |
| **运行时动态链接** | 执行中（`dlopen` / `LoadLibrary`） | 程序运行到特定代码时才加载目标库 | 最灵活：插件机制、按需加载、热更新；但增加运行时开销和复杂度 |

> **示例**：Windows 的 `.dll` 和 Linux 的 `.so` 均为共享库，支持装入时或运行时动态链接。浏览器加载 Flash 插件即为运行时动态链接的典型场景。

## 三、交换技术（Swapping）

### 3.1 为什么需要交换

内存容量有限，当并发进程数增加导致内存不足时，不能简单地拒绝新进程（会降低吞吐量）。**交换（Swapping）** 将暂时无法运行的进程从内存换出到磁盘，腾出空间给当前需要运行的进程。

### 3.2 交换的完整流程

```
    内存                         磁盘对换区（Swap Space）
    ┌─────────────┐             ┌─────────────────────┐
    │  进程 A      │── 换出 ──→ │  进程 A (完整镜像)   │
    │  (被阻塞)    │             │                     │
    ├─────────────┤             ├─────────────────────┤
    │  进程 B      │←── 换入 ── │  进程 C (之前被换出) │
    │  (就绪运行)  │             │                     │
    └─────────────┘             └─────────────────────┘
```

**步骤拆解**：
1. **触发条件**：内存空闲空间不足，或有更高优先级进程需要进入内存
2. **选择换出进程**：优先选择阻塞态（等待 I/O）或低优先级进程；选择原则类似于进程调度，但交换开销极大（毫秒级 vs 纳秒级），所以选择必须谨慎
3. **保存进程映像**：将进程的整个地址空间（代码 + 数据 + 栈 + 堆）写入磁盘的**对换区（Swap Space）**
4. **更新 PCB**：标记进程为"挂起"状态，记录其在磁盘上的位置
5. **回收内存**：释放该进程占用的所有物理内存
6. **加载新进程**：从对换区或可执行文件中将另一个进程读入内存
7. **运行**：CPU 调度该进程执行

### 3.3 交换的决策因素

| 因素 | 说明 |
|------|------|
| **对换区位置** | 通常为磁盘的独立分区或专用文件（Linux 的 swap partition / swap file），直接连续读写，无文件系统元数据开销，速度比普通文件 I/O 快 |
| **交换粒度** | 连续分配下必须整进程交换；分页系统中可以按页交换（更精细、开销更分散） |
| **交换时机** | 只在内存压力确实严重时触发。轻微不足时优先通过紧凑或回收缓存缓解 |
| **中级调度（Medium-term Scheduler）** | 负责换入/换出决策，也称为**挂起调度**；不同于长期调度（作业调度）和短期调度（CPU 调度） |

### 3.4 交换的性能代价

```
内存访问：        ~100 ns
磁盘 I/O（HDD）：  ~10 ms  = 10,000,000 ns
磁盘 I/O（SSD）：  ~0.1 ms =    100,000 ns
```

一次交换涉及两次磁盘 I/O（换出 + 换入），即使 SSD 也要付出约 20 万倍于内存访问的代价。因此交换是**最后的保底手段**，系统会优先尝试其他回收策略。

## 四、动态分区分配算法：完整推演

**初始内存布局**：假设系统有 5 个空闲块，按地址从低到高排列：

```
[100KB] [200KB] [150KB] [80KB] [300KB]
  ↑       ↑       ↑      ↑       ↑
 块1     块2     块3    块4     块5     总空闲: 830KB
```

**进程请求序列**（按到达顺序）：

| 进程 | 所需大小 | 到达顺序 |
|------|----------|----------|
| P1 | 90KB | ① |
| P2 | 180KB | ② |
| P3 | 60KB | ③ |
| P4 | 100KB | ④ |

### 4.1 首次适应（First Fit）

> **策略**：从低地址开始扫描，选择第一个大小满足要求的空闲块。

```
初始: [100KB] [200KB] [150KB] [80KB] [300KB]

P1=90KB:  从块1开始扫描 → 块1(100KB) >= 90KB ✓
           分配块1, 剩余 10KB
           结果: [10KB] [200KB] [150KB] [80KB] [300KB]

P2=180KB: 从块1(10KB)开始扫描 → 10KB ✗ → 块2(200KB) >= 180KB ✓
           分配块2, 剩余 20KB
           结果: [10KB] [20KB] [150KB] [80KB] [300KB]

P3=60KB:  从块1(10KB)开始扫描 → 10KB ✗ → 20KB ✗ → 块3(150KB) >= 60KB ✓
           分配块3, 剩余 90KB
           结果: [10KB] [20KB] [90KB] [80KB] [300KB]

P4=100KB: 从块1(10KB)开始扫描 → 10KB ✗ → 20KB ✗ → 90KB ✗ → 80KB ✗ → 块5(300KB) >= 100KB ✓
           分配块5, 剩余 200KB
           结果: [10KB] [20KB] [90KB] [80KB] [200KB]
```

**特点**：
- 低地址区域堆积大量无法利用的微小碎片（10KB、20KB），高地址保留大块
- 查找速度平均较快（前半段通常就能找到合适的）
- 碎片集中在低地址，但总量不大

### 4.2 最佳适应（Best Fit）

> **策略**：扫描所有空闲块，选择**容量满足要求且大小最小**的那个（即大小与请求最接近的）。

```
初始: [100KB] [200KB] [150KB] [80KB] [300KB]

P1=90KB:  遍历全部, 所有 >=90KB 的: 100KB, 200KB, 150KB, 300KB
           最小的是 100KB → 分配, 剩余 10KB
           结果: [10KB] [200KB] [150KB] [80KB] [300KB]

P2=180KB: 遍历全部, 所有 >=180KB 的: 200KB, 150KB✗, 300KB
                                      (150KB 不够)
           最小的是 200KB → 分配, 剩余 20KB
           结果: [10KB] [20KB] [150KB] [80KB] [300KB]

P3=60KB:  遍历全部, 所有 >=60KB 的: 10KB✗, 20KB✗, 150KB, 80KB, 300KB
           最小的是 80KB → 分配, 剩余 20KB
           结果: [10KB] [20KB] [150KB] [20KB] [300KB]

P4=100KB: 遍历全部, 所有 >=100KB 的: 10KB✗, 20KB✗, 150KB, 20KB✗, 300KB
           最小的是 150KB → 分配, 剩余 50KB
           结果: [10KB] [20KB] [50KB] [20KB] [300KB]
```

**特点**：
- 每次选择最"紧身"的块，产生大量极小碎片（三个 < 50KB 的小块）
- 需要对空闲区按大小排序或每次全扫描，效率较低
- 大块（300KB）始终得以保留，但低地址全是碎片

> **要害问题**：最佳适应虽然名字好听，但长期运行后会积累大量无法分配任何进程的微小碎片 —— 每次裁掉一小块布，满桌子都是碎布头，大块布料被"蚕食"殆尽。

### 4.3 最差适应（Worst Fit）

> **策略**：扫描所有空闲块，选择**容量最大**的那个。

```
初始: [100KB] [200KB] [150KB] [80KB] [300KB]

P1=90KB:  遍历全部, 最大的是 300KB → 分配, 剩余 210KB
           结果: [100KB] [200KB] [150KB] [80KB] [210KB]

P2=180KB: 遍历全部, 最大的是 210KB → 分配, 剩余 30KB
           结果: [100KB] [200KB] [150KB] [80KB] [30KB]

P3=60KB:  遍历全部, 最大的是 200KB → 分配, 剩余 140KB
           结果: [100KB] [140KB] [150KB] [80KB] [30KB]

P4=100KB: 遍历全部, 最大的是 150KB → 分配, 剩余 50KB
           结果: [100KB] [140KB] [50KB] [80KB] [30KB]
```

**特点**：
- 每次分割最大块，剩余部分仍可能足够大
- 理论意图：减少碎片 —— 但本例中最终产生了 30KB、50KB 等碎片
- **真实问题**：大块是稀缺资源（未来可能有 P5=250KB 的大进程），最差适应会过早耗尽大块
- 同样需要每次遍历全部空闲区，效率不高

### 4.4 邻近适应（Next Fit / 循环首次适应）

> **策略**：与首次适应类似，但**不从开头扫描**，而是从上次分配的位置继续向后找。维护一个**查询指针**，指向上次扫描结束的地方。

```
初始: [100KB] [200KB] [150KB] [80KB] [300KB]
      ↑指针

P1=90KB:  从指针(块1)开始 → 100KB >= 90KB ✓ → 分配, 剩余10KB
           指针移至此处, 结果: [10KB] [200KB] [150KB] [80KB] [300KB]
                                ↑指针

P2=180KB: 从指针(块1剩余)开始: 10KB ✗ → 块2 200KB >= 180KB ✓ → 分配, 剩余20KB
           指针移至此处, 结果: [10KB] [20KB] [150KB] [80KB] [300KB]
                                       ↑指针

P3=60KB:  从指针(块2剩余)开始: 20KB ✗ → 块3 150KB >= 60KB ✓ → 分配, 剩余90KB
           指针移至此处, 结果: [10KB] [20KB] [90KB] [80KB] [300KB]
                                               ↑指针

P4=100KB: 从指针(块3剩余)开始: 90KB ✗ → 块4 80KB ✗ → 块5 300KB >= 100KB ✓
           分配, 剩余200KB, 指针移至此处
           结果: [10KB] [20KB] [90KB] [80KB] [200KB]
                                                  ↑指针
```

**特点**：
- 分配均匀分布在内存各个区域，避免低地址过度碎片化
- 不需要回绕扫描到开头（除非整个环都没有合适块），减少了重复扫描
- 缺点是可用空间分布更零散，大块的保留不如首次适应好

### 4.5 四种算法对比总结

| 维度 | 首次适应 | 最佳适应 | 最差适应 | 邻近适应 |
|------|----------|----------|----------|----------|
| **搜索方式** | 从开头顺序找 | 全扫描取最小 | 全扫描取最大 | 从上次位置找 |
| **碎片倾向** | 低地址多小碎片 | 极多微小碎片 | 大块耗尽早 | 碎片分布均匀 |
| **大块保留** | 高地址保留大块 | 大块保留最好 | 大块保留最差 | 一般 |
| **查找速度** | 较快 | 慢（每次全扫描） | 慢（每次全扫描） | 较快 |
| **适用场景** | 通用（实际使用最多） | 请求大小相近 | 请求大小差异极大 | 需要均衡分布 |

## 五、碎片：内存的"锯齿问题"

### 5.1 外部碎片（External Fragmentation）

```
总空闲 = 50KB                     但最大连续块只有 20KB
┌──┬─────┬──┬─────┬──┬─────┐      进程请求 30KB → 分配失败
│已│ 20KB │已│ 15KB │已│ 15KB │     （虽然总空闲 50KB > 30KB）
└──┴─────┴──┴─────┴──┴─────┘
```

> **定义**：空闲空间的总量足够满足请求，但由于空闲块不连续，没有单独一个块足够大。**是连续分配的固有缺陷。**

**产生原因**：不断分配和释放不同大小的分区，空闲区被已分配区隔开，像"奶酪孔洞"一样分散在内存各处。

**衡量指标**：50% 规则（经验公式）—— 即使有 n 个已分配块，平均而言外部碎片占用的空间约为总空闲的一半。换言之，连续分配下约 **1/3 的内存可能因外部碎片而浪费**。

### 5.2 内部碎片（Internal Fragmentation）

```
进程请求 95KB, 分配 100KB 块
┌────────────┬───┐
│   进程     │ 5KB│  ← 这 5KB 分配了但进程用不上 → 内部碎片
└────────────┴───┘
```

> **定义**：分配给进程的内存块**大于**进程实际需要，多余的部分无法被该进程使用，也无法分配给其他进程。

**产生原因**：分配粒度问题 —— 分配时按固定块大小（或找"刚好够"的块本身就不可能），多余部分只能浪费。

**与外部碎片的关键区别**：
- 外部碎片：在已分配块**之间**，空闲但不可用（因为不连续）
- 内部碎片：在已分配块**内部**，已分配但不用

### 5.3 紧凑（Compaction）

> **定义**：通过移动内存中已分配的进程，将所有空闲碎片拼接成一个连续的大块。相当于"内存整理 / 磁盘碎片整理"。

```
紧凑前:  [ OS | A | 10K | B | 20K | C | 15K ]
                       ↑外碎    ↑外碎    ↑外碎
                      
紧凑后:  [ OS | A | B | C |        45K         ]
                             ↑ 一块连续空闲空间
```

**代价与前提**：
1. **必须支持动态重定位** —— 移动进程后，需要通过重定位寄存器修正其地址映射；静态绑定的程序无法移动
2. **时间开销大** —— 复制整个进程的内存内容，I/O 密集型进程尤其痛苦（可能有数 GB）
3. **时机选择** —— 通常在系统相对空闲时触发，或在分配请求失败后作为最后手段

> 现代 OS 中，分页机制从根源上避免了外部碎片，因此紧凑主要存在于**嵌入式实时系统**（无 MMU）中。

### 5.4 动态重定位

> **硬件支撑**：CPU 中有一对寄存器 —— **重定位寄存器（Base Register）** 和 **界限寄存器（Limit Register）**。

```
逻辑地址 0x1000          物理地址 0x8000 + 0x1000 = 0x9000
      ↓                                          ↓
    ┌───┐       MMU        ┌──────┐
    │CPU│ ─── 0x1000 ───→ │ +    │ ─── 0x9000 ──→ 内存
    └───┘                  │ Base=│                  │
                           │0x8000│             进程 B (被移动后)
                           └──────┘
```

每次 CPU 发出逻辑地址时，MMU 自动加上重定位寄存器的值得到物理地址。当进程被移动（交换或紧凑）后，只需更新重定位寄存器的值 —— 进程内部的逻辑地址完全不需要改变。

## 六、小结：连续分配的本质困局

| 问题 | 根因 | 解决思路 |
|------|------|----------|
| 外部碎片 | 各进程占用不连续分区 | 紧凑（治标）、分页（治本） |
| 内部碎片 | 分配粒度与需求不匹配 | 分页（固定大小页，内部碎片可控） |
| 交换代价高 | 必须整进程搬移 | 分页（按页交换，粒度细） |
| 分配算法各有缺陷 | 难以预知未来请求 | 首次适应在多数场景下实践最优 |

> **核心结论**：连续内存分配虽简单直观，但碎片问题和交换效率问题使其无法支撑现代多道程序系统。第 5 章的分页技术正是为了解决这些根本性问题而提出的。
