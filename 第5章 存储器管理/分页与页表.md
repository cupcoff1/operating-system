# 分页与页表

## 知识框架

```
分页与页表
├── 1. 基础概念
│   ├── 帧（Frame）：物理内存的固定大小块
│   ├── 页（Page）：逻辑地址空间的固定大小块，与帧等大
│   ├── 页表（Page Table）：每个页→帧的映射关系
│   └── 核心优势：消除外部碎片，按需换入换出
├── 2. 地址翻译（Address Translation）
│   ├── 逻辑地址 → [页号 p | 偏移 d]
│   ├── 查页表：页号 p → 帧号 f
│   └── 物理地址 = f × 页大小 + d
├── 3. 页表项（PTE）
│   ├── 帧号：映射到的物理帧
│   ├── 有效位（Present）：该页是否在内存中
│   ├── 保护位（R/W/X）：访问控制
│   ├── 脏位（Dirty）：被修改过，换出需写回磁盘
│   └── 引用位（Accessed）：用于页面置换算法
├── 4. TLB（Translation Lookaside Buffer）
│   ├── 硬件缓存：<页号, 帧号> 的快速映射
│   ├── 局部性原理：99%+ 访存命中 TLB
│   └── EAT 公式：h·(T_TLB+T_mem) + (1-h)·(T_TLB+2·T_mem)
├── 5. 多级页表
│   ├── 为什么：单级页表太大（32-bit, 4KB页 → 4MB/进程）
│   ├── 两级拆分：[外页 p1 | 内页 p2 | 偏移 d]
│   └── 节省原理：未用页表部分不分配物理帧
├── 6. 倒置页表（Inverted Page Table）
│   ├── 全系统一张表：帧号 → <pid, 页号>
│   ├── 查找：哈希加速
│   └── 优势：省内存；代价：查找慢、难以共享
├── 7. 页的共享
│   └── 纯代码（可重入码）→ 不同进程映射到同一帧
└── 8. 内存保护
    ├── 越界检查：页号 < 页表长度
    └── 保护位：每页 R/W/X 控制
```

## 一、核心概念定义

### 1.1 帧（Frame） vs 页（Page）

| 概念 | 所在空间 | 含义 |
|------|----------|------|
| **页（Page）** | 逻辑地址空间 | 进程视角的固定大小块，是逻辑地址的划分单位 |
| **帧（Frame）** | 物理地址空间 | 物理内存的固定大小块，是物理内存的划分单位 |
| **页大小 = 帧大小** | — | 大小必须相等，这样一页恰好装入一帧，无浪费 |

```
逻辑地址空间（进程视角）          物理内存
┌──────┐                        ┌──────┐
│ Page0 │ ──────────────→       │Frame7 │  (任意帧，由页表决定)
├──────┤                        ├──────┤
│ Page1 │ ──────────────→       │Frame2 │
├──────┤                        ├──────┤
│ Page2 │ ──────────────→       │Frame0 │
├──────┤                        ├──────┤
│ Page3 │ ───────→ 磁盘（缺页）  │ ...   │
└──────┘                        └──────┘

关键：页表中的映射关系使得连续的逻辑页可以映射到不连续的物理帧
```

> **关键认知**：分页的核心洞察是 —— 既然进程不需要连续的物理内存，外部碎片就从根本上被消除了。每个页都是固定大小的，任何空闲帧都能装下任何页。

### 1.2 页表项（PTE — Page Table Entry）

页表中的每一行称为一个**页表项**，记录一页的映射和元信息。

| 位域 | 说明 | 用途 |
|------|------|------|
| **帧号（Frame Number）** | 该页映射到哪个物理帧 | 地址翻译的核心字段 |
| **有效位（Present / Valid）** | 1 = 该页在物理内存中；0 = 缺页（在磁盘） | 触发缺页中断 |
| **保护位（Protection）** | 按位控制：Read / Write / Execute | 防止越权访问（如写只读代码页导致 segfault） |
| **脏位（Dirty）** | 1 = 该页被写入了新数据 | 换出时：脏页必须写回磁盘，干净页可直接丢弃 |
| **引用位（Accessed）** | 1 = 该页近期被访问过 | 页面置换算法（如 Clock 算法）的决策依据 |
| **禁止缓存位（Cache Disable）** | 1 = 不放入 CPU 缓存 | 内存映射 I/O 时使用 |

```c
// PTE 的逻辑结构（简化表示）
struct pte {
    uint32_t frame_number : 20;  // 物理帧号（高位）
    uint32_t present      : 1;   // 有效位
    uint32_t writable     : 1;   // 可写
    uint32_t user         : 1;   // 用户态可访问
    uint32_t accessed     : 1;   // 引用位
    uint32_t dirty        : 1;   // 脏位
    uint32_t reserved     : 7;   // 保留/平台相关
};
```

## 二、地址翻译：从逻辑地址到物理地址的完整过程

### 2.1 地址格式拆分

**前提**：32 位逻辑地址空间，页大小 4KB = 4096 B = 2^12。

```
┌─────────────────────────────┬──────────────────────────────┐
│      页号 p (20 bits)       │      页内偏移 d (12 bits)     │
│      bit 31 ~ bit 12        │      bit 11 ~ bit 0           │
└─────────────────────────────┴──────────────────────────────┘

为什么是 20 + 12？
  - 页大小 = 4KB = 2^12  → 偏移需要 12 位来区分页内的每个字节
  - 总 32 位 - 12 位 = 20 位用于页号
  - 最大页数 = 2^20 = 1,048,576 页（约 100 万页）
```

物理地址同样拆分：
```
┌─────────────────────────────┬──────────────────────────────┐
│      帧号 f (可变 bits)      │      帧内偏移 d (12 bits)     │
└─────────────────────────────┴──────────────────────────────┘
帧内偏移 = 页内偏移（因帧大小 = 页大小）
```

### 2.2 具体数值推演

**已知条件**：
- 逻辑地址：`20515`（十进制）
- 页大小：4KB = 4096
- 页表（部分条目）：

| 页号 | 帧号 | 有效位 |
|------|------|--------|
| 0 | 7 | 1 |
| 1 | 3 | 1 |
| 2 | 0 | 0（缺页） |
| 3 | 9 | 1 |
| 4 | 1 | 1 |
| **5** | **12** | **1** |
| ... | ... | ... |

**Step 1 — 提取页号和偏移**：

```
页号 p = 逻辑地址 / 页大小 = 20515 / 4096 = 5      （整数除法，截断）
偏移 d = 逻辑地址 % 页大小 = 20515 % 4096 = 35     （模运算）

验证：5 × 4096 + 35 = 20480 + 35 = 20515 ✓
```

用位运算视角（与硬件实现一致）：
```
20515 的二进制 = 0101 0000 0010 0011  (14 bits，高位补零至 32 bits)

32 bits: 0000 0000 0000 0000 0101   0000 0010 0011
         └── 高 20 bits = 页号 ──┘  └─ 低 12 bits = 偏移 ─┘
                 = 5                         = 35
```

**Step 2 — 查页表获取帧号**：

```
页号 = 5 → 查页表第 5 项 → 帧号 = 12, 有效位 = 1
（若有效位 = 0，则触发缺页中断，OS 从磁盘加载该页到空闲帧）
```

**Step 3 — 合成物理地址**：

```
物理地址 = 帧号 × 帧大小 + 偏移
        = 12 × 4096 + 35
        = 49152 + 35
        = 49187

用位运算：帧号 12 = 0b1100 高位拼接偏移 35 = 0b001000100011
          → [00000000000000001100] [000000100011] = 49187
```

**翻译流程总结**：

```
逻辑地址 20515
    │
    ├──→ 页号 p = 20515 >> 12     = 5      (右移12位取高20位)
    ├──→ 偏移 d = 20515 & 0xFFF  = 35     (掩码取低12位)
    │
    ├──→ 查进程页表, 页表基址寄存器(CR3) + p × PTE大小
    │    → 读取PTE: 有效位=1, 帧号=12
    │
    └──→ 物理地址 = (12 << 12) | 35 = 49187

硬件实际完成：两次乘法(除法) → 改为：位移 + 掩码 + 拼接
开销为 O(1)，每次访存都需要经过此翻译
```

## 三、TLB（快表）与有效访问时间

### 3.1 为什么需要 TLB

页表存储在内存中。没有 TLB 时，**每一次** CPU 访存都需要**两次**内存访问：
1. 第一次访存：读页表，获取帧号
2. 第二次访存：读/写实际数据

每次数据访问都翻倍的访存次数是无法接受的。**TLB（Translation Lookaside Buffer）** 是 CPU 芯片内部的高速硬件缓存，专门存储最近使用的 `<页号, 帧号>` 映射。

```
                ┌─── 命中(hit) ──→ 直接使用 TLB 中的帧号 ──→ 访问物理内存
                │                              (只需1次访存)
CPU 发出逻辑地址 ──┤
                │
                └─── 未中(miss) ──→ 查内存中的页表(1次访存)
                                    → 得到帧号
                                    → 访问实际数据(再1次访存)
                                    → 更新 TLB
                                    (共2次访存)
```

### 3.2 有效访问时间（EAT）公式与数值计算

**变量定义**：
- `h` = TLB 命中率（Hit Ratio）
- `T_tlb` = TLB 查询时间
- `T_mem` = 一次内存访问时间

```
EAT = h × (T_tlb + T_mem) + (1-h) × (T_tlb + 2 × T_mem)
```

**逐项解释**：
- 命中路径：查询 TLB（`T_tlb`）→ 获得帧号 → 一次访存取数据（`T_mem`）。总时间 `T_tlb + T_mem`
- 未中路径：查询 TLB（`T_tlb`）→ 未命中 → 访存读页表（`T_mem`）→ 获得帧号 → 访存取数据（`T_mem`）。总时间 `T_tlb + 2 × T_mem`

**数值实例**：

```
假设: h=95%, T_tlb=20ns, T_mem=100ns

EAT = 0.95 × (20 + 100) + 0.05 × (20 + 2×100)
    = 0.95 × 120 + 0.05 × 220
    = 114 + 11
    = 125 ns

对比无TLB: 每次访问 = 2 × 100ns = 200ns
加速比: 200 / 125 = 1.6x

若命中率提高到 99%:
EAT = 0.99 × 120 + 0.01 × 220 = 118.8 + 2.2 = 121 ns
加速比: 200 / 121 ≈ 1.65x

若命中率 99.9%:
EAT = 0.999 × 120 + 0.001 × 220 = 119.88 + 0.22 ≈ 120.1 ns
加速比: 200 / 120.1 ≈ 1.67x（接近于无开销的 1 次访存）
```

> **为什么 TLB 能达到 99%+ 的命中率？** 局部性原理（Locality of Reference）：程序倾向于反复访问同一页（时间局部性）和相邻页（空间局部性）。一个 128 条的 TLB 基本上就能覆盖典型程序的活跃页集合。

### 3.3 TLB 的组成

| 字段 | 说明 |
|------|------|
| **页号（Tag）** | 逻辑页号，用于匹配 |
| **帧号（Data）** | 对应的物理帧号 |
| **保护位** | 与 PTE 一致，用于权限检查 |
| **有效位** | 该 TLB 条目是否有效 |
| **ASID（Address Space ID）** | 区分不同进程的 TLB 条目（避免进程切换时全刷新） |

TLB 是**全相联**或**组相联**的硬件查找表，单次查找在 1 个 CPU 周期内完成（约 0.2ns~1ns）。

## 四、多级页表：解决单级页表过大的问题

### 4.1 单级页表的空间开销

```
32-bit 地址空间, 4KB 页
  → 页数 = 2^32 / 2^12 = 2^20 = 1,048,576 页
  → 每个 PTE 4 字节(假设), 页表总大小 = 2^20 × 4B = 4MB

每个进程 4MB 的页表！若 100 个进程 → 400MB 仅用于页表
而且必须连续存放（单级页表要求），这在物理内存中几乎不可能
```

### 4.2 两级页表的结构

**核心思路**：对页表本身再进行分页。页表被拆分成多个 4KB 的"页表页"，只将当前需要的页表页保留在内存中。

**地址拆分**（32-bit, 4KB 页, 两级页表）：

```
┌───────────────┬───────────────┬──────────────────────────────┐
│  外页号 p1    │  内页号 p2    │       页内偏移 d              │
│  (10 bits)    │  (10 bits)    │       (12 bits)               │
│  bit 31~22    │  bit 21~12    │      bit 11~0                 │
└───────────────┴───────────────┴──────────────────────────────┘

为什么是 10 + 10 + 12？
  - 每级页表恰好占一个 4KB 页,
    每级 = 4KB / 4B(每个PTE) = 1024 个条目 = 2^10
  - 所以每级需要 10 bits 来索引 2^10 个条目
  - 2^10(外) × 2^10(内) = 2^20 页 ← 覆盖全部逻辑地址空间
  - 偏移 12 bits ← 页大小 4KB
```

### 4.3 完整翻译过程推演

**假设逻辑地址**：`0x0040A0C8`

```
0x0040A0C8 的二进制:
  0000 0000 01  00 0000 1010  0000 1100 1000
  └── p1 ───┘  └──── p2 ────┘ └── d ──────┘
  = 1          = 10            = 200
```

**Step 1 — 查外页表（一级页表）**：

```
CR3 寄存器(一级页表基址)
  │
  ├── p1 = 1 → 偏移 = 1 × 4B = 4B
  │    读取一级页表第1项 PTE: {有效位=1, 帧号=0x0010, ...}
  │    含义: 二级页表存放在物理帧 0x0010 中
```

**Step 2 — 查内页表（二级页表）**：

```
物理帧 0x0010 (二级页表存放处)
  │
  ├── p2 = 10 → 偏移 = 10 × 4B = 28B
  │    读取二级页表第10项 PTE: {有效位=1, 帧号=0x00AB, ...}
  │    含义: 目标数据存放在物理帧 0x00AB 中
```

**Step 3 — 合成物理地址**：

```
物理地址 = 帧号 × 帧大小 + 偏移
        = 0x00AB << 12 | d
        = 0x00AB_0000 + 0xC8
        = 0x00AB_00C8
```

**完整路径**：

```
CR3 → [一级页表] → 帧 0x0010 → [二级页表] → 帧 0x00AB → [数据页 + offset 0xC8]
       ↑ 1次访存              ↑ 1次访存              ↑ 1次访存

总共 3 次内存访问才能读取实际数据（无 TLB 情况下）
```

### 4.4 为什么多级页表节省内存

**关键机制**：只有被进程实际使用的地址范围才分配页表页。

```
场景: 进程只用 0~16MB 和 3GB~3GB+16MB 两个区段
逻辑地址空间(4GB):
┌──────────────────────┐ 0x00000000
│   代码/数据 (16MB)    │   → 一级页表项 p1=0~3 有对应的二级页表(4页=16KB)
├──────────────────────┤ 0x01000000
│   未使用 (大量空洞)    │   → 一级页表中对应位置有效位=0, 不分配二级页表
│                      │     (省掉 1020 个二级页表, 每个4KB, 共省 4080KB)
├──────────────────────┤ 0xC0000000
│   栈/内核映射 (16MB)  │   → 一级页表项 p1=768~771 有对应的二级页表
└──────────────────────┘ 0xFFFFFFFF

单级页表: 必须分配完整 4MB (1024 个 PTE 全部连续)
两级页表: 一级页表 4KB + 仅8个二级页表(32KB) = 36KB → 节省 99%+
```

### 4.5 进一步扩展到多级

| 地址位宽 | 页大小 | 典型级数 | 备注 |
|----------|--------|----------|------|
| 32-bit | 4KB | 1~2 级 | 两级已足够 |
| 64-bit | 4KB | 4~5 级 | Linux(x86-64): 4 级; 最新: 5 级(支持 57-bit 虚拟地址) |
| 64-bit | 2MB | 3~4 级 | Huge Page 减少页表深度 |

## 五、倒置页表（Inverted Page Table）

### 5.1 传统页表的空间问题

```
传统(正向)页表: 每个进程各有自己的页表
索引方式: 页号 → 帧号

100 个进程, 每个使用 4GB 虚拟地址空间(用得不多也想定址):
  → 100 × 4MB = 400MB 仅用于页表
```

### 5.2 倒置页表的结构

> **定义**：全系统只有**一张**页表，按物理帧号索引，每个表项记录哪个进程的哪个页映射到了这个帧。

```
正向页表（每个进程一份）:
  进程A 页表: [页0→帧3] [页1→帧7] [页2→帧1] ...
  进程B 页表: [页0→帧2] [页1→帧5] ...
  → 100个进程 = 100张表

倒置页表（全系统一份）:
  系统倒置页表 (按帧号索引):
  ┌────────┬────────┬──────┐
  │ 帧号   │ PID    │ 页号 │
  ├────────┼────────┼──────┤
  │ 0      │ A      │ 5    │  ← 帧0 被进程A的页5占用
  │ 1      │ A      │ 0    │  ← 帧1 被进程A的页0占用
  │ 2      │ B      │ 3    │  ← 帧2 被进程B的页3占用
  │ 3      │ C      │ 7    │
  │ ...    │ ...    │ ...  │
  └────────┴────────┴──────┘
  
  大小: 帧总数 × PTE 大小 = (物理内存大小 / 4KB) × PTE 大小
  例如: 256MB 物理内存 → 256MB/4KB = 65536 帧 × 4B = 256KB
  100 个进程也只要 256KB，不会增长！
```

### 5.3 地址翻译过程（哈希加速）

倒置页表无法像正向页表那样直接按页号索引，必须**搜索**匹配 `<PID, 页号>` 的表项。

```
翻译: 进程A, 逻辑地址(页号=3, 偏移=0x200)
  │
  ├── 正向页表: 直接读页表[3] → 立即得到帧号 (O(1))
  │
  └── 倒置页表: 搜索整个表, 查找 (PID=A, 页号=3) 在哪一帧
       │
       ├── 遍历所有 65536 个条目 → O(n) → 不可接受
       │
       └── 实际做法: 哈希表加速
           hash(PID, 页号) → 哈希桶 → 链表遍历查匹配
           平均 O(1) ~ O(链长)
```

```
倒置页表哈希查找过程:
  ┌─────────────────────────────────────────────┐
  │ 输入: PID=A, 页号=3                         │
  │ hash(A,3) → 哈希值 = 42                     │
  │                                             │
  │ 哈希桶 42:                                  │
  │  ├→ [帧号=17, PID=A, 页号=3] ← 命中！     │
  │  └→ [帧号=89, PID=D, 页号=12]              │
  │                                             │
  │ 返回帧号 17                                 │
  │ 物理地址 = (17 << 12) | 0x200              │
  └─────────────────────────────────────────────┘
```

### 5.4 优缺点对比

| 维度 | 正向页表 | 倒置页表 |
|------|----------|----------|
| **表大小** | 与虚拟地址空间成正比（32-bit: 4MB/进程） | 与物理内存大小成正比（固定） |
| **多进程开销** | 每个进程一份，线性增长 | 全系统一份，不随进程数增长 |
| **查找速度** | O(1) 直接索引 | 需要哈希查找，最坏 O(n) |
| **共享页** | 简单：多个进程 PTE 指向同一帧 | 难：一个帧只能有一个表项（一个帧只被一个进程的某一页占用） |
| **换页支持** | 简单：缺页 = 有效位为 0 | 需额外数据结构跟踪空闲帧和磁盘页 |
| **实际采用** | x86, ARM, RISC-V（主流） | PowerPC, IA-64 (Itanium), 一些嵌入式 |

> **要害**：倒置页表以查找速度换取空间效率。在物理内存远小于虚拟地址空间的场景下（如 64 位系统 + 小物理内存），节省效果显著；但 64 位系统通过多级页表也能有效压缩，因此主流架构（x86、ARM）未采用倒置页表。

## 六、页的共享与保护

### 6.1 页的共享

多个进程运行同一程序（如多个 bash 实例、共享库 libc.so），它们的**代码页**可以映射到相同的物理帧，避免重复占用内存。

```
进程A 页表: [页0→帧5] [页1→帧8] [页2→帧3] ...
进程B 页表: [页0→帧9] [页1→帧8] [页2→帧3] ...
                          ↑         ↑
                    共享代码页   共享库页
```

**前提条件**：
- 共享代码必须是**纯代码（Reentrant Code / Pure Code）** —— 代码自身不可被修改（无自修改指令）
- 共享只对只读段（`.text`）有效；数据段（`.data`, `.bss`）各进程必须有独立副本
- 不同进程中，共享页的**页号可以不同**，但帧号必须相同

### 6.2 内存保护

| 保护机制 | 原理 | 触发条件 |
|----------|------|----------|
| **越界检查** | 页号 ≥ 页表长度 → 非法地址 | 访问超出进程地址空间 |
| **保护位（R/W/X）** | PTE 中记录每页的访问权限 | 写只读页 → segfault；执行数据页 → NX bit 拦截 |
| **用户/内核隔离** | PTE 中 U/S 位标记 | 用户态访问内核页 → 权限异常 |

现代 CPU 同时支持 **NX（No-eXecute）** 位：标记某页不可执行，防御缓冲区溢出攻击（在数据页上注入代码也无法执行）。

## 七、关键术语速查

| 术语 | 英文 | 一句话定义 |
|------|------|------------|
| **页（Page）** | Page | 逻辑地址空间划分的固定大小块，进程视角的存储单元 |
| **帧（Frame）** | Frame | 物理内存划分的固定大小块，硬件视角的存储单元；页与帧大小相同 |
| **页表** | Page Table | 每个进程的数据结构，记录页号→帧号的映射关系 |
| **页表项（PTE）** | Page Table Entry | 页表中的一行，含帧号、有效位、保护位、脏位、引用位 |
| **TLB / 快表** | Translation Lookaside Buffer | CPU 片上硬件缓存，存储最近使用的页→帧映射，命中时避免查页表 |
| **命中率（h）** | Hit Rate | TLB 命中的比例；典型值 99%~99.9%，由程序局部性保证 |
| **有效位** | Present / Valid Bit | PTE 中的标志位，0 = 该页不在内存（在磁盘），访问时触发缺页中断 |
| **脏位** | Dirty Bit | PTE 中的标志位，1 = 该帧被修改过，换出时必须写回磁盘 |
| **引用位** | Accessed Bit | PTE 中的标志位，页面置换算法的参考依据 |
| **缺页中断** | Page Fault | 访问有效位为 0 的页时触发，OS 从磁盘加载页到空闲帧 |
| **外围碎片** | — | 分页系统**无外部碎片**（每页等大，任意帧可装入任意页），仅存在极小的内部碎片（最后一页未用完） |

## 八、总结：分页解决了什么

| 连续分配的问题 | 分页的解决方案 |
|----------------|----------------|
| 外部碎片 | 逻辑连续→物理可分散，任意空闲帧均可分配 → 不存在外部碎片 |
| 交换必须整进程搬移 | 按页粒度换入换出，只交换需要的页 |
| 进程必须全部在内存 | 仅活跃页在内存（按需调页 / Demand Paging） |
| 内存利用率低 | 虚拟内存让进程以为自己拥有全部空间，实际只分配正在使用的部分 |

> 页号 → 帧号的翻译是每次 CPU 访存的必经之路，因此 TLB 的加速至关重要。两级/多级页表解决了页表本身的内存占用问题，而倒置页表则是另一种空间换时间的折中方案。
